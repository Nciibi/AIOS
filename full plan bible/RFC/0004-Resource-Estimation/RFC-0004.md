## Problem Statement

Resource allocation (tokens, compute, storage) is currently performed at worker creation time through ROS, based on static template defaults. Sou has no mechanism to estimate resource requirements before committing to a mission plan, and no feedback loop to adjust allocations based on actual consumption.

This leads to two failure modes:

1. **Over-allocation**: Tokens and compute are reserved but unused, wasting capacity that other missions could use.
2. **Under-allocation**: Workers run out of tokens mid-mission, causing stalls, failures, or emergency re-allocation calls.

Without resource estimation, Sou plans missions blind — unable to answer "Can I afford this mission?" or "Which configuration minimizes cost?"

**Affected**: Sou (mission planning), ROS (allocation), Workers (execution), Token accounting.

**Evidence**: `Research/000-Phase-2-Orchestration.md` (Area 4: Token & Compute Budget Estimation). `Bible/02-Core/ROS/005-Budget.md` defines budget but not estimation. Missions fail with `ROS_003` (budget exceeded) approximately 40% of the time in early trials.

## Proposed Solution

Introduce the **Resource Estimation Protocol** — a pipeline that estimates resource requirements before allocation, monitors actual consumption during execution, and feeds consumption data back into future estimates.

### Protocol Components

```
┌─────────────────────────────────────────────────────┐
│                  Sou (Planner)                       │
│  Defines mission, tasks, worker types                │
└──────────────┬──────────────────────────────────────┘
               │ EstimationRequest
               ▼
┌─────────────────────────────────────────────────────┐
│              Resource Estimator                      │
│  - Historical data analysis                          │
│  - Template-based estimation                         │
│  - Task complexity scoring                           │
│  - Confidence calculation                            │
└──────────────┬──────────────────────────────────────┘
               │ BudgetRecommendation
               ▼
┌─────────────────────────────────────────────────────┐
│                    ROS                               │
│  - Allocates budget per recommendation               │
│  - Enforces caps per worker/org/mission              │
│  - Tracks consumption                                │
└──────────────┬──────────────────────────────────────┘
               │ ConsumptionData
               ▼
┌─────────────────────────────────────────────────────┐
│            Consumption Analyzer                      │
│  - Records actual vs. estimated                      │
│  - Updates historical baseline                       │
│  - Detects anomalies                                 │
│  - Generates estimation adjustments                  │
└─────────────────────────────────────────────────────┘
```

### Estimation Request Schema

```yaml
resource_estimation_request:
  mission_type: string
  tasks:
    - task_id: string
      task_type: code_generation | code_review | analysis | planning | documentation
      complexity: low | medium | high | critical
      estimated_lines_of_code: integer          # For code tasks
      estimated_files_affected: integer          # For code tasks
      context_size: small | medium | large       # Approximate context window needed
      num_rounds_estimated: integer              # Expected worker-model interaction rounds
      preferred_runtime: string                  # Optional, for runtime-specific estimates
  workers:
    - template_id: UUID
      count: integer
      runtime: string                            # Preferred runtime
  resource_profile:
    quality: draft | standard | high             # Lower quality = cheaper estimate
    max_budget_placeholder: float                # Optional cap on proposed budget
```

### Estimation Response Schema

```yaml
resource_estimation_response:
  mission_estimate:
    tokens_total:
      estimated: integer
      min: integer                               # Lower bound (5th percentile)
      max: integer                               # Upper bound (95th percentile)
      confidence: float                          # 0.0–1.0 based on historical accuracy
    compute_units:
      estimated: integer
      min: integer
      max: integer
      confidence: float
    storage_mb:
      estimated: integer
      min: integer
      max: integer
      confidence: float
    estimated_duration_seconds: integer
    estimated_cost: float                        # In resource units

  per_worker:
    - worker_index: integer
      template_id: UUID
      tokens_estimated: integer
      tokens_min: integer
      tokens_max: integer
      cost_estimated: float

  per_task:
    - task_id: string
      tokens_estimated: integer
      tokens_min: integer
      tokens_max: integer
      duration_estimated_seconds: integer

  metadata:
    estimation_method: historical | template | heuristic | hybrid
    historical_basis_points: integer              # Number of historical data points used
    similar_missions_found: integer               # Number of similar past missions
    estimator_latency_ms: integer
    warning: string                               # Optional warning
```

### Estimation Methods

| Method | Data Source | Confidence | Use Case |
|--------|-------------|------------|----------|
| **Historical** | Past missions with same template_id and task_type | High (≥0.85) | Mature templates with execution history |
| **Template-based** | Template defaults × complexity multiplier | Medium (0.6–0.8) | New templates with no execution history |
| **Heuristic** | Task type × complexity × context_size × rule-of-thumb constants | Low (0.3–0.6) | Novel task types with no history |
| **Hybrid** | Weighted combination of above | Varies | Default — uses all available data |

```yaml
consumption_record:

    started_at: timestamp

      description: string

```

### Budget Adjustment Protocol

When actual consumption deviates from estimates, the Protocol supports three adjustment modes:

| Mode | Behavior | Trigger |
|------|----------|---------|
| **Soft Cap** | Warning at 80%, alert at 95%, auto-increase on approval | Default |
| **Hard Cap** | Worker stops at budget limit, requires Sou intervention | Critical missions |
| **Elastic** | Budget grows within bounds, capped at max_estimate | Low-priority missions |

### Feedback Loop

Periodically (or on mission completion), the Analyzer generates:

```yaml
estimation_adjustment:
  template_id: UUID
  task_type: string
  runtime: string
  adjustment_factor: float                    # 1.0 = accurate, 1.2 = 20% low, 0.8 = 20% high
  confidence_adjustment: float                # Updated confidence for future estimates
  historical_data_points: integer
  trend: stable | increasing | decreasing
  recommended_baseline_update:
    tokens: integer
    compute_units: integer
```

### Alternatives Considered

1. **Static allocation (current approach)**: Rejected — no estimation, no feedback. Allocates blindly from template defaults.

2. **Worker self-reporting**: Rejected — violates Law 1 (Non-Interference). Workers should not be responsible for resource accounting.

3. **Post-hoc analysis only**: Rejected — estimation before allocation is the primary value. Post-hoc analysis alone prevents over-allocation but not under-allocation.

4. **Machine learning based estimation**: Deferred — feasible for v2. Heuristic/historical methods sufficient for v1.

### Affected Services

- **New service**: Resource Estimator (can be a module within ROS)
- **ROS**: Estimation endpoints, consumption tracking, budget adjustment
- **LMS**: Consumption records linked to lifecycle events
- **AGS**: Historical consumption indexed by template

- ROS allocation API: `allocate_budget` gains an optional `estimation_id` parameter. Backward compatible.
- No existing API removed.

## Evidence

1. **`Research/000-Phase-2-Orchestration.md` (Area 4)**: Identifies token/compute budget estimation as critical gap. Notes that current allocation is reactive (responds to OOM/budget exceeded) rather than proactive.

2. **`Bible/02-Core/ROS/005-Budget.md`**: Defines budget structure and allocation but has no estimation, no historical feedback, and no consumption tracking beyond raw totals.

3. **Mission failure analysis**: Missions aborting with `ROS_003` (budget exceeded) ~40% of the time suggests systemic under-estimation. Template defaults are likely too low.

4. **`Bible/02-Core/Brain/Sou/003-Missions.md`**: Mission phases (Intake → Intention → Intuition → Incubation → Insight → Implementation) have no resource estimation step between Intention and Implementation.

## Constitutional Review

### Law Compliance

- Does this RFC violate any Law? **No** — Resource Estimator is an analytical service. It informs Sou's decisions (Law 2) but does not execute. It tracks resource consumption (Law 6) without interfering with worker operation (Law 1).
- Does this RFC require a Law amendment? **No**
- Does this RFC require a Physics change? **No**

### Design DNA Compliance

| Rule | Assessment |
|------|------------|
| R1 — Modulsingularity | Resource Estimator has one responsibility: estimate resources and track consumption. Does not allocate or enforce — that remains ROS. |
| R2 — Dependency Order | Depends on ROS (allocation data), AGS (templates), LMS (lifecycle events) — all more primitive. |
| R3 — DRY | Estimation logic currently scattered (planning tools, manual calculations, PostHog queries). Centralized in one service. |
| R7 — Tests Exist | Estimation accuracy tests, historical data regression tests, consumption tracking tests, anomaly detection tests. |
| R8 — Fast Tests | Estimation is a calculation — <50ms per request. |
| R10 — Simpler Over Complex | v1 uses historical averages + heuristic multipliers. No ML. No distributed state. No real-time streaming (batch consumption updates). |
| R12 — Embrace Errors | Estimator errors: EST_001 (no historical data), EST_002 (insufficient context for estimate), EST_003 (confidence below threshold). These are informational, not fatal — caller decides whether to proceed. |
| R13 — Design for Failure | If Estimator is unavailable, ROS falls back to template defaults (current behavior). Graceful degradation. |
| R14 — Paved Path | Estimation becomes the paved path for all allocations. Direct allocation without estimation is the power-user path. |
| R15 — Open/Closed | New estimation methods can be added by implementing the `EstimationMethod` interface. No Estimator core changes needed. |

## Migration Plan

1. **Phase 1** (collection): Enable consumption tracking on all existing missions. Build historical baseline.
2. **Phase 2** (estimation): Deploy Estimator alongside existing allocation. New missions can use estimation; existing missions continue with static allocation.
3. **Phase 3** (enforcement): Soft/hard cap modes enabled. All new allocations require estimation.
