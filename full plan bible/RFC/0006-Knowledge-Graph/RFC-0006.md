## Problem Statement

The system currently has no structured knowledge representation. Information exists in:

- **Logs** (PostHog): Event streams without semantic relationships.
- **Documents** (Bible, Research, Case Law): Human-readable text without machine-traversable links.
- **Worker outputs**: Ephemeral artifacts destroyed when workers terminate.
- **Sou's internal state**: In-memory context not persisted as shareable knowledge.

Without a knowledge graph, there is no way to:
- Traverse relationships between missions, workers, artifacts, and outcomes.
- Answer "What did worker X produce that relates to component Y?"
- Discover implicit connections (e.g., "All workers that use template Z produce high-quality tests").
- Preserve institutional knowledge beyond individual session context.

**Affected**: Sou, Learning Engine (RFC-0005), Mission Planning, All Workers, Search/Discovery.

**Evidence**: `Research/001-Phase-3-Intelligence.md` (Area 2: Knowledge Graph Nodes & Edges). The system generates thousands of artifacts with no semantic indexing.

## Proposed Solution

Introduce the **Knowledge Graph** — a semantic graph database that indexes all system entities (missions, workers, artifacts, templates, organizations, decisions) and their relationships, making knowledge machine-traversable.

### Graph Schema

```
Nodes (Entities)
  ├── Mission
  │   ├── Properties: id, type, status, created_at, completed_at
  │   └── Relationships: HAS_TASK, PRODUCES_ARTIFACT, USED_TEMPLATE
  │
  ├── Task
  │   ├── Properties: id, type, complexity, status
  │   └── Relationships: PART_OF_MISSION, ASSIGNED_TO_WORKER, PRODUCES_ARTIFACT
  │
  ├── Worker
  │   ├── Properties: id, template, runtime, model, lifecycle_state
  │   └── Relationships: BELONGS_TO_ORG, EXECUTED_TASK, USED_TEMPLATE
  │
  ├── Artifact
  │   ├── Properties: id, type, size, hash, content_summary
  │   └── Relationships: PRODUCED_BY, REFERENCES_ARTIFACT, RELATED_TO_COMPONENT
  │
  ├── Template
  │   ├── Properties: id, name, version, capability_profile
  │   └── Relationships: USED_BY_WORKER, EFFECTIVE_FOR_TASK, SUPERSEDES_TEMPLATE
  │
  ├── Organization
  │   ├── Properties: id, name, type, autonomy_level
  │   └── Relationships: CONTAINS_DEPARTMENT, EXECUTED_MISSION, OWNS_RESOURCES
  │
  ├── LearningRecord (from RFC-0005)
  │   ├── Properties: id, observation, pattern_type, confidence
  │   └── Relationships: DERIVED_FROM_MISSION, APPLIES_TO_TEMPLATE, SUPERSEDES_RECORD
  │
  ├── Decision
  │   ├── Properties: id, type, rationale, outcome
  │   └── Relationships: MADE_BY, BASED_ON_EVIDENCE, RESULTED_IN
  │
  └── ResourcePool
      ├── Properties: id, type, total, used, available
      └── Relationships: ALLOCATED_TO_ORG, CONSUMED_BY_MISSION, OWNED_BY
```

### Edge Types

| Edge Type | Source → Target | Semantics |
|-----------|----------------|-----------|
| `HAS_TASK` | Mission → Task | Mission decomposition |
| `PART_OF_MISSION` | Task → Mission | Task membership (inverse) |
| `ASSIGNED_TO_WORKER` | Task → Worker | Task execution assignment |
| `EXECUTED_TASK` | Worker → Task | Worker executed this task |
| `PRODUCES_ARTIFACT` | Entity → Artifact | Entity produced this artifact |
| `PRODUCED_BY` | Artifact → Entity | Artifact origin |
| `USED_TEMPLATE` | Entity → Template | Template used |
| `BELONGS_TO_ORG` | Entity → Organization | Organizational membership |
| `CONTAINS_DEPARTMENT` | Organization → Organization | Hierarchy |
| `REFERENCES_ARTIFACT` | Artifact → Artifact | Cross-artifact reference |
| `RELATED_TO_COMPONENT` | Entity → Component | Domain-specific relationship |
| `DERIVED_FROM_MISSION` | LearningRecord → Mission | Pattern source |
| `APPLIES_TO_TEMPLATE` | LearningRecord → Template | Pattern applicability |
| `SUPERSEDES_TEMPLATE` | Template → Template | Template version chain |
| `SUPERSEDES_RECORD` | LearningRecord → LearningRecord | Knowledge evolution |
| `MADE_BY` | Decision → Entity | Decision maker |
| `BASED_ON_EVIDENCE` | Decision → LearningRecord | Evidence basis |
| `RESULTED_IN` | Decision → Entity | Decision outcome |
| `ALLOCATED_TO_ORG` | ResourcePool → Organization | Resource ownership |
| `CONSUMED_BY_MISSION` | ResourcePool → Mission | Resource consumption |
| `SIMILAR_TO` | Entity → Entity | Semantic similarity (ML-derived) |

### Query Examples

```yaml
# Find all artifacts from Rust migration missions
query:
  match:
    - node: Mission
      where: { type: "software_migration", target_language: "Rust" }
    - edge: PRODUCES_ARTIFACT
    - node: Artifact
  return: Artifact.id, Artifact.content_summary

# Find which templates produce the highest quality test artifacts
query:
  match:
    - node: Template
      where: { capability_profile: { contains: "testing" } }
    - edge: USED_TEMPLATE
    - node: Worker
    - edge: EXECUTED_TASK
    - node: Task
      where: { type: "testing" }
    - edge: PART_OF_MISSION
    - node: Mission
    - edge: PRODUCES_ARTIFACT
    - node: Artifact
      where: { type: "test_report", quality_score: { gt: 0.9 } }
  return: Template.id, count(Artifact.id)
  group_by: Template.id
  order_by: desc(count)

# Trace decision lineage for a failed mission
query:
  match:
    - node: Mission
      where: { id: "mission-123", status: "failed" }
    - edge: HAS_TASK
    - node: Task
    - edge: ASSIGNED_TO_WORKER
    - node: Worker
    - edge: EXECUTED_TASK
    - node: Task
    - edge: { type: null, direction: any }
    - node: Decision
  return: path
```

### Ingestion

```yaml
knowledge_graph_ingestion:
  source_type: artifact | event | decision | learning_record
  source_id: UUID

  nodes:
    - type: string
      id: UUID
      properties: object
      ttl_seconds: integer              # Optional time-to-live (ephemeral nodes)

  edges:
    - source_id: UUID
      target_id: UUID
      type: string
      properties: object
      weight: float                     # Optional edge weight (0.0–1.0)
```

### Graph Maintenance

```yaml
graph_maintenance_policy:
  indexing:
    full_rebuild_interval_days: 30      # Periodic full re-index
    incremental_interval_seconds: 60    # Near-real-time incremental updates

  pruning:
    ephemeral_node_ttl: 86400           # 24 hours for temporary nodes
    unlinked_node_retention: 2592000    # 30 days before removing orphan nodes
    edge_decay_enabled: true            # Older edges get lower weight

  access_control:
    read_roles: [sou, worker, security_council]
    write_roles: [ingestion_pipeline, security_council]
    admin_roles: [security_council]

  backup:
    full_backup_daily: true
    point_in_time_recovery_days: 7
```

### Alternatives Considered

1. **No structured knowledge (current approach)**: Rejected — information is trapped in logs and documents. Not machine-traversable.

2. **Relational database only**: Rejected — relational models struggle with variable-depth relationships (path queries) and evolving schema. Graph DB is purpose-built for this.

3. **Document store (MongoDB) with manual references**: Rejected — loses graph traversal capability. Every relationship query requires application-level JOIN logic.

4. **Full RDF/OWL ontology**: Overengineered for v1. Simple property graph with typed nodes and edges is sufficient.

## Constitutional Review

### Law Compliance

- Does this RFC violate any Law? **No** — Knowledge Graph is a passive index. It stores and retrieves; it does not execute, decide, or interfere.
- Does this RFC require a Law amendment? **No**
- Does this RFC require a Physics change? **No**

### Design DNA Compliance

| Rule | Assessment |
|------|------------|
| R1 — Modulsingularity | Knowledge Graph stores and serves relationships. Query execution is the only active behavior. |
| R2 — Dependency Order | Depends on all services that produce entities (AGS, LMS, Sou, Workers). Not depended upon by critical path. |
| R3 — DRY | Relationship traversal logic currently duplicated across manual analysis tools. Centralized in graph. |
| R7 — Tests Exist | Schema validation tests, query execution tests, ingestion pipeline tests, pruning correctness tests. |
| R8 — Fast Tests | Graph queries <100ms for typical traversals. |
| R10 — Simpler Over Complex | Property graph model (nodes + edges + properties) — not RDF, not hypergraph, not temporal graph (v1). |
| R11 — Refactor over Rewrite | Existing artifact storage preserved. Graph indexes alongside, not replacing. |
| R12 — Embrace Errors | Graph errors: GRAPH_001 (node not found), GRAPH_002 (query timeout), GRAPH_003 (schema violation). |
| R13 — Design for Failure | Graph reads are cacheable. If graph is down, services operate without knowledge traversal. |
| R15 — Open/Closed | New node types and edge types defined by configuration, not code. |

## Migration Plan

1. **Phase 1** (schema): Deploy graph database with initial schema. Index existing entities (missions, workers, templates).
2. **Phase 2** (ingestion): Connect ingestion pipeline to all producing services.
3. **Phase 3** (queries): Enable graph queries from Sou and Learning Engine.
