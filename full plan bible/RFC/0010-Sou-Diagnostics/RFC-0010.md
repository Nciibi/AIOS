## Problem Statement

Sou's internal state — goals, decisions, reasoning, plans, memory, and strategic intent — is opaque to external observation. When Sou makes a suboptimal decision or enters an unexpected state, there is no mechanism to:

- Inspect Sou's current goals and priorities.
- Trace decision lineage: "Why did Sou choose this mission structure?"
- Diagnose planning failures: "Why is Sou stuck in the Intention phase?"
- Audit Sou's compliance with constitutional constraints.
- Monitor Sou's cognitive load and resource consumption.

**Affected**: Security Council, System Agent (RFC-0009), Human operators, Constitutional compliance auditing.

**Evidence**: `Research/002-Phase-4-Integration.md` (Area 2: Sou Diagnostics). PostHog captures Sou's external actions but not internal state. Debugging Sou requires restart with debug flags — no runtime introspection.

## Proposed Solution

Introduce the **Sou Diagnostics Protocol** — a structured introspection interface into Sou's internal state, providing read-only access to goals, decisions, memory, planning state, and constitutional reasoning.

### Diagnostic Endpoints

```
Sou Diagnostics API
  │
  ├── GET /sou/state
  │   └── Current operational state, phase, sub-phase
  │
  ├── GET /sou/goals
  │   └── Active goals, priorities, progress, blocked goals
  │
  ├── GET /sou/decisions
  │   └── Recent decisions, rationale, alternatives considered
  │
  ├── GET /sou/plans
  │   └── Current plan, pending tasks, completed tasks, stuck tasks
  │
  ├── GET /sou/memory
  │   └── Active memory slots, context window utilization
  │
  ├── GET /sou/constitution
  │   └── Law compliance status, active governance constraints
  │
  ├── GET /sou/threads
  │   └── Active reasoning threads, depth, status
  │
  ├── GET /sou/resources
  │   └── Cognitive resource utilization, token consumption
  │
  ├── GET /sou/log
  │   └── Diagnostic event stream (filterable)
  │
  └── GET /sou/trace?decision_id=<uuid>
      └── Full decision lineage with supporting evidence
```

### State Schema

```yaml
sou_state_snapshot:
  # Identity
  sou_id: UUID
  version: string
  uptime_seconds: integer
  session_id: UUID

  # Operational phase
  phase: idle | planning | executing | reviewing | sleeping
  sub_phase: string                          # Phase-specific
  phase_started_at: timestamp
  phase_goal: string                         # What Sou is trying to achieve

  # Mission context
  active_mission_id: UUID
  active_mission_type: string
  active_mission_status: string
  mission_progress_pct: float

  # Goals
  goals:
    - goal_id: UUID
      description: string
      priority: critical | high | medium | low
      status: active | completed | blocked | abandoned
      progress_pct: float
      blocked_by: string                     # If blocked, what's the blocker
      created_at: timestamp
      target_completion: timestamp

  # Cognitive metrics
  cognitive:
    context_window_used_pct: float
    tokens_consumed_session: integer
    tokens_consumed_total: integer
    reasoning_depth: integer
    active_threads: integer
    decision_queue_depth: integer

  # Recent decisions
  recent_decisions:
    - decision_id: UUID
      type: mission_structure | worker_selection | resource_allocation | strategy
      description: string
      rationale: string
      alternatives_considered: integer
      selected_option: string
      status: pending | executing | completed | failed
      timestamp: timestamp
```

### Decision Trace Schema

```yaml
decision_trace:
  decision_id: UUID
  decision_type: string
  timestamp: timestamp

  # Context at decision time
  context:
    phase: string
    active_mission: UUID
    active_goals: [UUID]
    memory_slots_active: integer
    external_input: string                   # What triggered this decision

  # Reasoning
  reasoning_chain:
    - step_number: integer
      deliberation: string                   # Sou's internal reasoning text
      evidence_considered: [string]
      laws_applied: [string]
      alternatives_generated: [string]
      timestamp: timestamp

  # Outcome
  selected_option: string
  rejected_options:
    - option: string
      rejection_reason: string
  expected_outcome: string
  actual_outcome: string                     # Populated post-execution
  outcome_success: boolean | null            # null until outcome known

  # Compliance
  constitutional_review:
    laws_checked: [string]
    compliance: pass | warning | fail
    warnings: [string]
    override_applied: boolean
```

  short_term:
    - slot_id: string

      priority: float                        # 0.0–1.0 eviction priority
      created_at: timestamp
      last_accessed: timestamp

    used_slots: integer

      recall_count: integer

### Diagnostics Access Control

```yaml
diagnostics_access:
  read_roles:
    security_council:
      access_level: full                     # All endpoints, all data
      audit_logged: true
    system_agent:
      access_level: operational              # State, goals, resources only
      audit_logged: true
    human_operator:
      access_level: full                     # With Security Council approval
      audit_logged: true
    worker:
      access_level: none                     # Workers cannot inspect Sou
  rate_limits:
    requests_per_minute: 60
    burst_limit: 10
  audit:
    all_access_logged: true
    retention_days: 90
```

### Diagnostic Event Stream

Sou emits structured events for real-time diagnostics:

```yaml
sou_diagnostic_event:
  event_type: state_change | decision_made | goal_updated | phase_transition | error | warning
  timestamp: timestamp
  sou_id: UUID
  data: object                              # Event-specific payload
  sequence_number: integer                  # Monotonically increasing
  cause_event_id: UUID                      # Chain of causality
```

### Alternatives Considered

1. **No introspection (current approach)**: Rejected — Sou is a black box. Debugging requires restart with debug flags.

2. **Full state persistence (every thought logged)**: Rejected — too expensive, too noisy. Diagnostic protocol provides structured snapshots without logging every token.

3. **Screen recording / REPL-style observation**: Rejected — violates Law 1 (Non-Interference). Observation should not alter Sou's execution.

## Constitutional Review

### Law Compliance

- Does this RFC violate any Law? **No** — Diagnostics is read-only observation. It does not interfere with Sou's execution (Law 1) or make decisions for Sou (Law 2).
- Does this RFC require a Law amendment? **No**
- Does this RFC require a Physics change? **Yes** — Add monitoring & introspection standards to `Physics/010-Monitoring.md`.

### Design DNA Compliance

| Rule | Assessment |
|------|------------|
| R1 — Modulsingularity | Diagnostics is a read-only view into Sou. It does not modify, execute, or decide. |
| R2 — Dependency Order | Diagnostics depends on Sou's internal state. Nothing depends on diagnostics. |
| R3 — DRY | State introspection logic centralized rather than scattered across debug scripts. |
| R7 — Tests Exist | Diagnostic endpoint response validation, access control tests, event emission tests. |
| R8 — Fast Tests | Snapshot endpoints <100ms. |
| R10 — Simpler Over Complex | REST API reading in-memory state. No complex query engine, no state reconstruction. |
| R11 — Refactor over Rewrite | Existing debug mode preserved. Diagnostics provide structured alternative. |
| R12 — Embrace Errors | Diagnostics errors: DIAG_001 (state not available), DIAG_002 (access denied). Informational, not critical. |
| R13 — Design for Failure | If diagnostics endpoint fails, Sou continues unaffected. Diagnostics is non-critical path. |
