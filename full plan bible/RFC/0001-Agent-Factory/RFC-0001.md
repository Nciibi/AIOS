## Problem Statement

Currently, every Worker must be individually created through OSYS with explicit identity assignment (IDS), genome selection (AGS), resource allocation (ROS), and lifecycle registration (LMS). This manual process does not scale when Sou needs to deploy 10, 50, or 200 workers for a complex mission. There is no mechanism for Sou to request "spawn N workers of type X" and have the system handle provisioning, identity, resource allocation, and lifecycle automatically.

Without an Agent Factory, Sou becomes the bottleneck — every worker requires Sou's direct involvement in creation. Law 2 (Non-Execution) prohibits Sou from executing, but the current worker creation process forces Sou to micromanage provisioning.

**Affected**: Sou, OSYS, IDS, AGS, ROS, LMS, all mission types that require multiple workers.

**Evidence**: Phase 2 research in `Research/000-Phase-2-Orchestration.md` identifies the Agent Factory as the critical missing mechanism for scaling orchestration. The `Bible/03-Institutions/Workers/000-Overview.md` defines the 10-state worker lifecycle but assumes workers are created one at a time.

## Proposed Solution

Introduce the **Agent Factory** — a new service that accepts bulk worker creation requests from Sou and handles provisioning, identity, resource allocation, template instantiation, and lifecycle registration atomically.

### Architecture

```
Sou
  │
  ├── Request: "Spawn 7 workers: 3 Rust, 2 UI, 2 QA"
  │
  ▼
Agent Factory
  ├── Receives: template_id, count, org_id, mission_id, capability_overrides
  ├── For each worker:
  │   ├── 1. AGS: Validate and load template genome
  │   ├── 2. IDS: Generate unique identity (batch UUID allocation)
  │   ├── 3. ROS: Allocate resource budget (per-worker or pool)
  │   ├── 4. LMS: Register lifecycle (batch creation → Planned → Assigned)
  │   ├── 5. ATS: Generate authentication/authorization tokens
  │   └── 6. Runtime binding: Select and bind runtime
  ├── Returns: [worker_id, worker_id, ...] (all in Assigned state)
  └── Events emitted per worker for audit trail
```

### Factory Request Schema

```yaml
agent_factory_request:
  mission_id: UUID                    # Parent mission
  org_id: UUID                        # Owning organization
  template_id: UUID                   # AGS genome template
  count: integer                      # Number of workers to create (1–100)
  capability_overrides:               # Optional overrides to genome defaults
    - capability: string
      bounds: object
  resource_allocation:                # Per-worker or pool allocation
    mode: per_worker | pool
    budget:
      tokens_max: integer
      compute_units: integer
      storage_mb: integer
  runtime_constraints:                # Optional runtime selection criteria
    preferred_runtime: string         # e.g., "claude-code", "codex"
    fallback_runtimes: [string]
    model_criteria:                   # If runtime not specified
      min_capability: string
      max_cost_per_token: float
  naming_pattern: string              # e.g., "backend-{n}" → "backend-3"
  parallelism: sequential | parallel  # Create workers sequentially or in parallel batches
  autoscale: boolean                  # If true, factory monitors and adjusts count
```

### Factory Lifecycle

```
Idle → RequestReceived → Provisioning → Complete → Idle
                            │
                            ▼
                         PartialFailure (some workers created, some failed)
                            │
                            ▼
                         Failed (all workers failed)
```

### Batch Identity Allocation

Rather than calling IDS N times sequentially, the factory requests a block of UUIDs:

```yaml
factory_batch_request:
  count: 7
  purpose: "worker-creation"
  owner_org: UUID
  owner_mission: UUID

factory_batch_response:
  identity_block: [UUID, UUID, ...]  # Pre-allocated, contiguous block
  allocated_at: timestamp
  expires_at: timestamp               # Unused identities released if factory fails
```

### Ticket-Based Resource Allocation

Instead of ROS allocating per-worker (N calls), the factory requests a pool:

```yaml
factory_resource_ticket:
  type: pool
  total_tokens: 100000
  pool_mode: flexible                # Workers draw from pool as needed
  max_per_worker: 20000
  org_owner: UUID
  mission_owner: UUID
```

### Alternatives Considered

1. **Sequential worker creation (current approach)**: Rejected — Sou must make N separate API calls. Does not scale to 50+ workers. Violates R10 (Simpler Over Complex).

2. **Single monolithic worker with internal parallelism**: Rejected — breaks the constitutional invariant that a Worker is a single Session. A single worker cannot be in multiple states simultaneously.

3. **Worker pools managed by OSYS**: Rejected — OSYS manages Organizations, not worker creation. Adding factory logic to OSYS violates R1 (Modulsingularity).

## Evidence

The Agent Factory concept is derived from:

1. **`ChatGPT-souuSouu Agent System Design.md`**: The original design conversation describes "Agent Factory creates unlimited sessions" with the Docker-container metaphor — workers created on demand, configured, launched, and destroyed when finished.

2. **`Research/000-Phase-2-Orchestration.md`**: Section "Area 1: Dynamic Agent Factory" defines the research baseline with key questions about optimal worker count, template languages, heterogeneous runtime handling, and session lifecycle.

3. **Worker creation flow in `Bible/03-Institutions/Workers/000-Overview.md`**: The existing 9-step creation flow requires 6 separate system calls. A factory reduces this to 1 call.

## Constitutional Review

### Law Compliance

- Does this RFC violate any Law? **No** — the Agent Factory executes operations; it does not make strategic decisions. Sou still decides how many workers and of what type.
- Does this RFC require a Law amendment? **No**
- Does this RFC require a Physics change? **No**

### Design DNA Compliance

| Rule | Assessment |
|------|------------|
| R1 — Modulsingularity | Agent Factory has a single responsibility: batch worker creation. Separate from worker lifecycle management (OSYS). |
| R2 — Dependency Order | Factory depends on IDS, AGS, ROS, LMS, ATS — all more primitive services. No circular dependencies. |
| R3 — DRY | Worker creation logic currently duplicated across callers. Factory centralizes it. |
| R4 — Builder Pattern | Factory is a Builder for Workers. Construction (batch provisioning) is separate from use (worker execution). |
| R5 — Liskov Substitution | Factory works with any runtime that implements the Worker interface. Runtime-neutral. |
| R6 — Dependency Injection | Factory receives service dependencies (IDS, AGS, ROS) via injection. |
| R7 — Tests Exist | Batch creation tests, partial failure tests, identity leak tests, pool allocation tests. |
| R8 — Fast Tests | Batch creation tests should complete in <100ms per batch of 100. |
| R9 — Deterministic | Factory must handle partial failures deterministically — same inputs produce same result or same error. |
| R10 — Simpler Over Complex | Factory is the simplest mechanism for scaling worker creation. Avoids distributed coordination complexity. |
| R11 — Refactor over Rewrite | Existing single-worker creation is preserved. Factory delegates to existing services. |
| R12 — Embrace Errors | Error codes: FACTORY_001 (identity batch exhausted), FACTORY_002 (resource pool insufficient), FACTORY_003 (template invalid), FACTORY_004 (partial failure with N successes). |
| R13 — Design for Failure | Partial failure is first-class: factory reports which workers succeeded and which failed. Failed workers are cleaned up (identities released, resources returned). |
| R14 — Paved Path | Factory becomes the paved path for worker creation. Single-worker creation becomes the edge case. |
| R15 — Open/Closed | Factory is extensible: new worker types require only new AGS templates, not factory changes. |
