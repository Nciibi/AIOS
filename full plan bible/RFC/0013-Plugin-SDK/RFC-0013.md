## Problem Statement

The system's capabilities are currently defined by built-in templates and runtimes. There is no mechanism for third-party developers to extend the system with custom plugins — new worker templates, capability modules, runtime adapters, or integration connectors.

Without a Plugin SDK:
- Every new capability must be built into the core system.
- External developers cannot contribute extensions.
- The system cannot tap into ecosystem innovation.
- Custom integrations require modifying core code.

**Affected**: External developers, System extensibility, Capability catalog, AGS templates.

**Evidence**: `Research/003-Phase-5-Ecosystem.md` (Area 1: Plugin SDK for Developers). Multiple requests for custom templates and integrations that cannot be fulfilled without core changes.

## Proposed Solution

Introduce the **Plugin SDK** — a development kit and runtime for building, packaging, and distributing plugins that extend system capabilities.

### Plugin Architecture

```
Plugin
  ├── Manifest (plugin.yaml)
  │   ├── name, version, author
  │   ├── type: worker_template | capability_module | runtime_adapter | connector
  │   ├── dependencies: [other plugins]
  │   ├── permissions: [scopes, capabilities]
  │   └── lifecycle: install, activate, deactivate, uninstall
  │
  ├── Runtime module
  │   ├── Plugin API implementation
  │   ├── Sandboxed execution
  │   └── Resource limits
  │
  ├── Capability declarations
  │   ├── What the plugin can do
  │   ├── Input/output schemas
  │   └── Quality guarantees
  │
  └── Assets
      ├── Templates (for worker_template type)
      ├── Prompts (for LLM-based capabilities)
      └── Configuration defaults
```

### Plugin Manifest Schema

```yaml
plugin_manifest:
  # Identity
  id: "com.example.my-plugin"             # Reverse-domain uniqueness
  name: "My Plugin"
  version: "1.0.0"
  author:
    name: "Example Corp"
    email: "plugins@example.com"
    website: "https://example.com/plugins"

  # Type
  type: worker_template | capability_module | runtime_adapter | connector

  # Description
  description: string
  documentation_url: string
  repository_url: string
  license: "MIT"

  # Dependencies
  dependencies:
    - plugin_id: string
      min_version: string
      max_version: string
    system_api_version: "1.0"

  # Permissions
  permissions:
    scopes:                               # API scopes the plugin needs
      - missions:read
      - workers:read
      - artifacts:read
    capabilities:                         # Capability declarations
      - capability_id: string
        constraints: object
    resource_limits:
      max_tokens_per_minute: 10000
      max_storage_mb: 100

  # Lifecycle
  lifecycle:
    install:
      pre_install: string                 # Script or hook
      post_install: string
    activate:
      on_activate: string
    deactivate:
      on_deactivate: string
    uninstall:
      pre_uninstall: string
      post_uninstall: string

  # Capabilities (for worker_template type)
  worker_template:
    default_runtime: string
    supported_runtimes: [string]
    capability_profile:
      - capability: string
        level: integer
        bounds: object
    input_schema: object                  # JSON Schema
    output_schema: object                 # JSON Schema
    prompts:
      system_prompt: string
      user_prompt_template: string

  # Capabilities (for capability_module type)
  capability_module:
    functions:
      - name: string
        description: string
        input_schema: object
        output_schema: object
        execution_mode: sync | async | streaming
        timeout_seconds: integer

  # Capabilities (for runtime_adapter type)
  runtime_adapter:
    supported_models: [string]
    capabilities: [string]
    health_check: string                  # Endpoint or command
    max_concurrent_sessions: integer

  # Capabilities (for connector type)
  connector:
    protocol: http | grpc | websocket | file
    endpoints:
      - name: string
        url: string
        method: GET | POST | PUT | DELETE
        authentication: none | api_key | oauth | custom
    authentication_config:
      type: api_key | oauth2 | basic
      credentials_required: [string]
```

### Plugin Runtime

```yaml
plugin_runtime:
  execution:
    sandbox: container | wasm | vm
    resource_limits:
      cpu_shares: integer
      memory_mb: integer
      network_access: isolated | restricted | full
      filesystem_access: read_only | restricted | full
    lifecycle_hooks:
      - hook: on_install | on_activate | on_deactivate | on_uninstall
        timeout_seconds: 30
        failure_action: abort | warn | ignore

  api_access:
    allowed_apis:
      - api: missions
        methods: [list, get]
      - api: workers
        methods: [list, get]
      - api: artifacts
        methods: [list, get, create]
    denied_apis:
      - api: system
        methods: [all]
    rate_limits:
      requests_per_minute: 60

  isolation:
    process: container | subprocess | thread
    namespace: pid, network, mount
    readonly_root: true
    temp_directory: true
    network_rules:
      egress_allowed: boolean
      egress_hosts: [string]
      ingress_allowed: false
```

### Plugin Lifecycle

```
Registered → Installed → Activated → Running → Deactivated → Uninstalled
                  │           │                      │
                  ▼           ▼                      ▼
              InstallFailed  ActivationFailed    Deactivated (error)
```

| State | Description |
|-------|-------------|
| Registered | Plugin uploaded, metadata validated |
| Installed | Assets deployed, dependencies resolved |
| Activated | Plugin loaded and ready for use |
| Running | Plugin actively serving requests |
| Deactivated | Plugin stopped, resources released |
| Uninstalled | Plugin removed from system |
| InstallFailed | Installation error |
| ActivationFailed | Activation error |
| Deactivated (error) | Plugin crashed or exceeded limits |

### Alternatives Considered

1. **No plugin system (current approach)**: Rejected — all extensions require core changes. Limits ecosystem growth.

2. **Shared library / DLL-style plugins**: Rejected — no isolation. Plugin crash takes down core service.

3. **Sidecar process plugins**: Accepted for high-performance plugins. Sandboxed execution is default for safety.

4. **WebAssembly (Wasm) plugins**: Accepted as a sandbox technology option alongside containers.

## Constitutional Review

### Law Compliance

- Does this RFC violate any Law? **No** — Plugins operate within capability bounds (Law 4). They do not interfere with core execution (Law 1). Plugin isolation prevents violations.
- Does this RFC require a Law amendment? **No**
- Does this RFC require a Physics change? **No**

### Design DNA Compliance

| Rule | Assessment |
|------|------------|
| R1 — Modulsingularity | Plugin Runtime manages plugin lifecycle; Plugin Registry manages discovery. Separate concerns. |
| R2 — Dependency Order | Plugin Runtime depends on system APIs. System does not depend on any specific plugin. |
| R3 — DRY | Plugin framework centralizes extension logic rather than each extension modifying core. |
| R5 — Liskov Substitution | Plugin workers are substitutable for built-in workers if they implement the same capability profile. |
| R6 — Dependency Injection | Plugins receive API access and capabilities via injection. |
| R7 — Tests Exist | Sandbox isolation tests, plugin lifecycle tests, API access control tests, resource limit enforcement tests. |
| R10 — Simpler Over Complex | Plugin API is a standard REST/gRPC interface. No custom protocol, no complex IPC. |
| R12 — Embrace Errors | Plugin errors: PLUGIN_001 (permission denied), PLUGIN_002 (resource exceeded), PLUGIN_003 (dependency missing), PLUGIN_004 (crashed). Errors isolate to plugin, not core. |
| R13 — Design for Failure | Plugin crash does not affect core. Plugin runtime restarts crashed plugins with exponential backoff. |
| R15 — Open/Closed | Plugin SDK is the open-for-extension mechanism. Core system is closed for modification. |

## Migration Plan

1. **Phase 1** (runtime): Deploy Plugin Runtime with sandboxed execution.
2. **Phase 2** (SDK): Publish Plugin SDK documentation and tooling.
3. **Phase 3** (registry): Deploy Plugin Registry for discovery and installation.
4. **Phase 4** (marketplace): Connect to Marketplace (RFC-0014) for distributed plugin distribution.
