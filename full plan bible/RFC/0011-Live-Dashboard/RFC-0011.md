## Problem Statement

System operators and the Security Council have no unified real-time view of system state. Operational awareness requires checking multiple sources:

- PostHog dashboards (historical metrics, not real-time).
- Log files (text-only, no visualization).
- Manual service probing (ad-hoc, no aggregation).
- Sou diagnostics (RFC-0010) output (API responses, not rendered).

Without a live dashboard:
- Incident detection lags by minutes (PostHog ingestion delay).
- Correlation between events (e.g., ROS budget spike + LMS state errors) requires manual cross-referencing.
- No visual representation of system topology, mission progress, or resource utilization.
- Security Council has no operational picture for governance decisions.

**Affected**: Security Council, System Agent (RFC-0009), Human operators, Sou oversight.

**Evidence**: `Research/002-Phase-4-Integration.md` (Area 3: Live Dashboard & CLI). Operators currently maintain personal Grafana instances with inconsistent configurations.

## Proposed Solution

Introduce a **Live Dashboard** — a web-based real-time visualization of system state, health, missions, and resources, consuming data from all services and the System Agent.

### Dashboard Architecture

```
System Services
  ├── Sou (RFC-0010 diagnostics)
  ├── System Agent (RFC-0009 health)
  ├── ROS (resource metrics)
  ├── LMS (lifecycle events)
  ├── IDS (identity metrics)
  ├── ACF (communication metrics)
  ├── AGS (template metrics)
  ├── PostHog (analytics data)
  └── Worker runtimes (worker metrics)
       │
       ▼
Dashboard Backend
  ├── Event ingestion (WebSocket + REST)
  ├── State aggregation
  ├── Alert processing
  ├── Data cache
  └── GraphQL API
       │
       ▼
Dashboard Frontend
  ├── System Overview
  ├── Mission View
  ├── Health View
  ├── Resource View
  ├── Worker View
  ├── Compliance View
  └── Alert Center
```

### Data Sources

| View | Data Sources | Update Frequency |
|------|-------------|-----------------|
| System Overview | System Agent (RFC-0009), Sou (RFC-0010) | 5s polling |
| Mission View | Sou diagnostics, LMS, ROS | 5s polling |
| Health View | System Agent, all services | 10s polling |
| Resource View | ROS, System Agent | 5s polling |
| Worker View | LMS, Sou, System Agent | 10s polling |
| Compliance View | Sou constitution check, ATS audit trail | 30s polling |
| Alert Center | System Agent | Real-time (WebSocket) |

```yaml
  event_type: state_change | metric_update | alert | lifecycle_event | mission_progress

  source: string

```

### Access Control

```yaml
dashboard_access:
  roles:
    security_council:
      views: [all]
      actions: [acknowledge_alert, view_sou_state, view_worker_logs]
      real_time: true
    human_operator:
      views: [system_overview, health, resources, alerts]
      actions: [acknowledge_alert]
      real_time: true
    system_agent:
      views: [all]                           # Programmatic access
      actions: [all]
      real_time: true
```

### Alternatives Considered

1. **PostHog dashboards only**: Rejected — PostHog has 30–60s ingestion delay, no real-time mission view, no alert center, no system topology visualization.

2. **CLI-only dashboard**: Rejected — text-only cannot convey topology, heat maps, or real-time status effectively. CLI as a supplement (see below).

3. **Grafana + PostHog**: Accepted as a supplement for historical analytics. Live Dashboard provides operational real-time view that Grafana cannot.

## Constitutional Review

### Law Compliance

- Does this RFC violate any Law? **No** — Dashboard is a passive visualization. It observes but does not execute, interfere, or decide.
- Does this RFC require a Law amendment? **No**
- Does this RFC require a Physics change? **Yes** — Add real-time visualization and data presentation standards to `Physics/010-Monitoring.md`.

### Design DNA Compliance

| Rule | Assessment |
|------|------------|
| R1 — Modulsingularity | Dashboard visualizes. It does not execute, monitor, or decide. System Agent (RFC-0009) monitors; Dashboard displays. |
| R7 — Tests Exist | Rendering tests, data source integration tests, real-time update tests, access control tests. |
| R10 — Simpler Over Complex | REST + WebSocket backend. React/Svelte frontend. No complex streaming infrastructure (Kafka not required). |
| R12 — Embrace Errors | Dashboard gracefully handles missing data sources — stale data indicators, not crashes. |
| R13 — Design for Failure | If a data source is down, dashboard shows "stale" indicator for that widget. Other widgets continue updating. |

## Migration Plan

1. **Phase 1** (backend): Deploy dashboard backend with System Agent (RFC-0009) and Sou diagnostics (RFC-0010) integration.
2. **Phase 2** (frontend): Deploy web dashboard with System Overview, Health, and Alert views.
3. **Phase 3** (mission view): Integrate mission tracking from Sou diagnostics.
4. **Phase 4** (resource view): Integrate ROS metrics.
5. **Phase 5** (CLI): Deploy CLI dashboard for terminal access.
