## Problem Statement

Organizations are currently created through OSYS with manual specification of structure, departments, roles, and resource bounds. Sou cannot dynamically form Organizations in response to mission complexity — it must either use a pre-existing Organization or request manual creation. This limits Sou's ability to match organizational structure to mission requirements.

For complex missions requiring multiple departments (backend, frontend, QA, security, documentation), Sou should be able to describe the organizational structure and have it created automatically. Without this, Sou is limited to flat worker pools or must reuse suboptimal existing Organizations.

**Affected**: Sou, OSYS, Organizations, Missions.

**Evidence**: `Research/000-Phase-2-Orchestration.md` (Area 2: Hierarchical Organization Formation) identifies the gap. Examples in `Examples/000-Example-Orgs.md` show the desired pattern but no API exists to realize it.

## Proposed Solution

Introduce a **Dynamic Organization Formation Protocol** that allows Sou to request Organization creation by describing the structure, departments, capability requirements, and resource allocation in a single request. OSYS handles provisioning, verification, and activation.

### Formation Request

```yaml
org_formation_request:
  name: string
  type: ORG | OHS | OPE                  # Organization type per Bible/03-Institutions
  parent_org_id: UUID                     # Parent Organization (root if null)
  mission_id: UUID                        # Mission this Org serves
  sou_goals:                              # Strategic goals from Sou
    - goal_id: string
      description: string
      priority: critical | high | medium | low

  departments:
    - name: string
      type: ODS                            # Department type
      purpose: string
      workers:
        template_id: UUID                  # AGS template
        count: integer                     # Number of workers
        runtime: string                    # Preferred runtime
        model: string                      # Preferred model
        autonomy: L0 | L1 | L2
        capability_bounds: [string]

  resource_budget:
    mode: per_org | per_dept              # Allocation mode
    tokens_total: integer
    compute_units: integer
    storage_mb: integer
    per_worker_max: integer                # Max draw per worker

  governance:
    supervisor: sou | org_admin | security_council
    autonomy_level: L0 | L1 | L2
    review_frequency: continuous | milestone | completion
    escalation_path:
      - trigger: string                    # e.g., "budget_exceeded"
        escalate_to: string                # e.g., "security_council"

  lifecycle:
    lifespan: mission_bound | timed | indefinite
    auto_dissolve: boolean                 # Auto-dissolve on mission completion
    retention_days: integer                # Archive retention after dissolve
```

### Formation Flow

```
Sou describes Org structure
  │
  ▼
OSYS Formation Engine
  ├── 1. Validate structure against constitutional bounds
  │   ├── Depth ≤ 7 levels
  │   ├── No cycle in hierarchy
  │   └── All types valid per Org taxonomy
  │
  ├── 2. Agent Factory creates workers per department
  │   ├── Batch identity allocation
  │   ├── Resource pool allocation
  │   └── Template instantiation
  │
  ├── 3. LMS registers Org lifecycle
  │   ├── Created → Verified → Active
  │   └── Departmental lifecycle linked to parent Org
  │
  ├── 4. ACF configures communication channels
  │   ├── Departmental topics
  │   ├── Cross-department routing
  │   └── Supervisor subscription
  │
  ├── 5. Sou receives Org identity + worker roster
  │
  └── Returns: org_id, department_ids, worker_ids[], status
```

### Formation Lifecycle

```
Draft → Validating → Provisioning → Active → Running → Dissolving → Archived
                          │
                          ▼
                      ValidationFailed
                          │
                          ▼
                      Rejected
```

| State | Description |
|-------|-------------|
| Draft | Structure defined by Sou, not yet submitted |
| Validating | OSYS checks constitutional compliance, hierarchy rules, resource availability |
| Provisioning | Agent Factory creates workers, ROS allocates budget, IDS assigns identities |
| Active | Org is operational, workers are in Running state, Mission progresses |
| Running | Org executing mission tasks |
| Dissolving | Mission complete, workers terminated, resources returned |
| Archived | Record preserved, all identities retired |

### Alternatives Considered

1. **Manual Org creation (current approach)**: Rejected — requires human-level specification of every department and role. Does not scale to dynamic mission requirements. Violates R14 (Paved Path).

2. **Flat worker pool without Org structure**: Rejected — loses departmental isolation, governance boundaries, and capability scoping. Violates constitutional separation of powers.

3. **Pre-created Org templates with parameterization**: Accepted as a supplement (see below). Templates handle common patterns; dynamic formation handles novel structures.

### Org Templates

For common patterns, pre-approved Org templates provide a faster path:

```yaml
org_template:
  id: "software-dev-org"
  name: "Software Development Organization"
  type: ORG
  departments:
    - name: "Backend"
      type: ODS
      worker_template: "rust-backend-dev"
      default_count: 3
    - name: "Frontend"
      type: ODS
      worker_template: "frontend-dev"
      default_count: 2
    - name: "QA"
      type: ODS
      worker_template: "qa-engineer"
      default_count: 1
```

Sou can use a template directly or extend it with additional departments.

## Evidence

1. **`Research/000-Phase-2-Orchestration.md` (Area 2)**: Defines the hierarchical formation concept with multi-division examples, commander selection, resource cascading, and teardown protocols.

2. **`Examples/000-Example-Orgs.md` (Example 1)**: The messaging app example shows the exact pattern — Sou creates a 3-department, 7-worker Organization. This RFC provides the API to realize that example.

3. **`Bible/03-Institutions/Organizations/000-Overview.md`**: Current Org taxonomy (ORG, ODS, OHS, OOM, OPE, EEE, OIS, DOM) provides the type system. Formation Engine maps requested structure to these types.

4. **`Bible/02-Core/OSYS/002-Org-Lifecycle.md`**: Existing 7-state lifecycle (Created → Verified → Active → Suspended → Restored → Dissolved → Archived) needs Draft, Validating, and Provisioning states for dynamic formation.

### Law Compliance

### Design DNA Compliance

| Rule | Assessment |
|------|------------|
| R1 — Modulsingularity | Formation Engine has one responsibility: translating Sou's structural intent into a verified, provisioned Organization. |
| R2 — Dependency Order | Formation Engine depends on OSYS, Agent Factory, AGS, IDS, ROS, LMS — all simpler services. |
| R3 — DRY | Formation centralizes Org creation logic currently scattered across manual processes. |
| R4 — Builder Pattern | Formation Engine is a Builder for Organizations. Construction is separate from operation. |
| R5 — Liskov Substitution | Any Organization type (ORG, OHS, OPE) can be formed through the same interface. |
| R10 — Simpler Over Complex | Templates handle 80% of cases. Dynamic formation handles the remaining 20%. No distributed orchestration required. |
| R11 — Refactor over Rewrite | Existing Org creation is preserved. Formation Engine delegates to existing OSYS APIs. |
| R13 — Design for Failure | Partial provisioning rolls back — if 2 of 3 departments form successfully, the failed department doesn't orphan the Org. |
| R14 — Paved Path | Org templates are the paved path. Dynamic formation without a template is the power-user path. |
| R15 — Open/Closed | New Org types and department types can be defined without modifying the Formation Engine. |
