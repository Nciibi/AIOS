## Problem Statement

Workers currently declare a single runtime and model at creation time through AGS template selection. There is no mechanism for routing a worker to a different model based on task requirements, availability, cost, or capability constraints. This creates a rigid binding: each worker type is coupled to a specific model/runtime combination.

When the optimal model for a task is unavailable (rate-limited, degraded, or at capacity), the worker has no fallback. When a cheaper model would suffice, the worker overpays. When a more capable model is needed, the worker is underpowered.

**Affected**: All Workers, Sou (mission planning), Runtime selection, Cost optimization.

**Evidence**: `Research/000-Phase-2-Orchestration.md` (Area 3: Model Router & Runtime Load Balancer) identifies runtime coupling as the key limitation. Current AGS templates hardcode runtime (`AGS/templates/rust-dev-v1.json` → `"runtime": "claude-code"`).

## Proposed Solution

Introduce the **Model Router** — a routing layer between Workers and Runtimes that decouples worker capability requirements from runtime/model selection. The Router evaluates task requirements, runtime availability, cost constraints, and capability bounds to select the optimal runtime for each task or session.

### Architecture

```
Worker (wants to run a task)
  │
  ├── Sends: CapabilityRequest to Model Router
  │   ├── task_type: string
  │   ├── required_capabilities: [string]
  │   ├── priority: critical | normal | low
  │   ├── max_cost_per_token: float
  │   ├── preferred_model: string (optional)
  │   └── context_window_required: integer
  │
  ▼
Model Router
  ├── 1. Evaluate available runtimes
  │   ├── claude-code (available, cost: medium, capabilities: [code, reasoning])
  │   ├── codex (available, cost: low, capabilities: [code])
  │   └── gpt-4 (rate-limited, cost: high, capabilities: [code, reasoning, planning])
  │
  ├── 2. Score and rank
  │   ├── claude-code: 92/100 (optimal match)
  │   ├── codex: 65/100 (cheaper but less capable)
  │   └── gpt-4: 80/100 (best but rate-limited)
  │
  ├── 3. Select runtime
  │   └── claude-code (selected)
  │
  └── Returns: RuntimeAssignment
      ├── runtime_id: "claude-code-session-441"
      ├── model: "claude-4-opus"
      ├── endpoint: URL
      ├── tokens_allocated: 64000
      └── fallback_chain: [codex, gpt-4]
```

### Router Request Schema

```yaml
model_router_request:
  worker_id: UUID
  task_type: code_generation | code_review | analysis | planning | documentation | qa
  required_capabilities:
    - capability: string              # From Capabilities taxonomy
      min_level: integer
  priority: critical | normal | low
  constraints:
    max_cost_per_token: float
    max_total_cost: float
    min_context_window: integer
    preferred_runtime: string          # Optional
    preferred_model: string            # Optional
    excluded_runtimes: [string]        # Optional
    latency_tolerance: low | medium | high
  session_context:
    estimated_tokens: integer
    estimated_rounds: integer
    shared_session_allowed: boolean    # Can share session with other workers?
```

### Router Response Schema

```yaml
model_router_response:
  assignment:
    runtime_id: string
    runtime_type: string
    model: string
    endpoint: string
    session_token: string
    tokens_allocated: integer
    cost_budget: float
    fallback_chain:
      - runtime_id: string
        model: string
        trigger: rate_limited | cost_exceeded | capability_required
  metadata:
    score: float                       # Match score 0–100
    alternatives_considered: integer
    selection_reason: string
    router_latency_ms: integer
```

### Routing Strategies

| Strategy | Behavior | Use Case |
|----------|----------|----------|
| `best_match` | Highest capability-to-cost ratio | Default |
| `cheapest_viable` | Cheapest runtime meeting minimum requirements | Budget missions |
| `most_capable` | Best runtime regardless of cost | Critical missions |
| `preferred` | Use specified runtime if available, else fail | Strict requirements |
| `fallback_chain` | Try primary, fall through chain on failure | Reliability-sensitive |

### Runtime Registry

The Router maintains a registry of available runtimes:

```yaml
runtime_registry_entry:
  runtime_id: string
  runtime_type: string                  # e.g., "claude-code", "codex", "custom"
  status: available | degraded | offline | rate_limited
  models:
    - model_id: string
      capabilities: [string]
      cost_per_token: float
      context_window: integer
      status: healthy | degraded | overloaded
  max_concurrent_sessions: integer
  current_sessions: integer
  health_endpoint: string
  last_checked: timestamp
```

### Runtime Load Balancing

The Router tracks runtime load and routes accordingly:

```yaml
runtime_health:
  runtime_id: string
  status: available | degraded | offline
  current_load_pct: float               # 0–100
  avg_latency_ms: integer
  error_rate_pct: float
  tokens_per_second: integer
  estimated_queue_time_ms: integer
```

### Alternatives Considered

1. **Static runtime binding (current approach)**: Rejected — no flexibility. Each worker is permanently married to one runtime/model.

2. **Worker-initiated runtime switching**: Rejected — breaks Law 1 (Non-Interference). Workers should not manage infrastructure decisions.

3. **Sou as router**: Rejected — violates Law 2 (Non-Execution). Sou is strategic, not operational.

### Affected Services

- **New service**: Model Router microservice
- **AGS**: Capability profiles replace hardcoded runtime references
- **Runtime adapters**: Implement health check and capabilities reporting
- **ROS**: Cost tracking per routing decision

### Breaking Changes

- AGS templates: `runtime` field becomes advisory (preferred_runtime). Primary routing is through capability profiles.
- Migration path: Existing templates without capability profiles fall back to hardcoded runtime field.

## Evidence

1. **`Research/000-Phase-2-Orchestration.md` (Area 3)**: Defines the runtime load balancing concept with heterogeneous runtime pooling, health monitoring, cost optimization, and runtime-agnostic template design.

2. **`Bible/02-Core/AGS/001-Templates.md`**: Current templates hardcode `"runtime": "claude-code"`. This RFC replaces that with capability-based routing.

3. **Worker creation examples**: The 10-state lifecycle assumes runtime is set at creation. Router decouples these concerns.

## Constitutional Review

### Law Compliance

- Does this RFC violate any Law? **No** — Router is an infrastructure service making operational routing decisions. Strategic model selection (which models to support) remains with Sou. Law 1 (Non-Interference) preserved.
- Does this RFC require a Law amendment? **No**
- Does this RFC require a Physics change? **No**

### Design DNA Compliance

| Rule | Assessment |
|------|------------|
| R1 — Modulsingularity | Router has one job: match task requirements to runtime capabilities. Not a general orchestrator. |
| R2 — Dependency Order | Router depends on Runtime Registry, Capabilities taxonomy, AGS — all simpler or equal complexity. |
| R3 — DRY | Runtime selection logic currently duplicated across worker creation, mission planning, and manual overrides. Router centralizes it. |
| R4 — Builder Pattern | Router builds the Worker-Runtime binding. The binding is used by the worker, not the Router. |
| R5 — Liskov Substitution | All runtimes implement the same health/capability reporting interface. Any runtime can substitute for any other with matching capability profile. |
| R6 — Dependency Injection | Runtime registry entries are injected from runtime adapters. Router doesn't hardcode runtime knowledge. |
| R7 — Tests Exist | Routing strategy tests, fallback chain tests, load-based routing tests, cost optimization tests. |
| R10 — Simpler Over Complex | Routing is a scoring function: capabilities × cost × availability. No distributed consensus, no machine learning (initially). |
| R12 — Embrace Errors | Router errors: ROUTER_001 (no eligible runtime), ROUTER_002 (all runtimes degraded), ROUTER_003 (cost exceeds budget). |
| R13 — Design for Failure | Fallback chains handle runtime failure mid-session. Router can hot-reassign if runtime degrades. |
| R15 — Open/Closed | New runtimes register through the runtime registry interface. No Router code changes needed. |

## Migration Plan

1. **Phase 1** (parallel run): Router introduced alongside existing static binding. New workers can use Router; existing workers retain static binding.
2. **Phase 2** (capability profiles): AGS templates add `capability_profile` field. Router becomes primary routing mechanism.
3. **Phase 3** (deprecation): Static runtime field deprecated. All routing through Router.
