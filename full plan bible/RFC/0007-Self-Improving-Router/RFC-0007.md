## Problem Statement

RFC-0003 (Model Router) defines static routing strategies (best_match, cheapest_viable, etc.) that are selected manually by Sou or by configuration. The Router has no mechanism to learn which routing decisions produced the best outcomes and adjust future routing accordingly.

Without self-improvement:
- Routing strategy selection is Sou's responsibility — violating the spirit of Law 2 (Non-Execution).
- Optimal strategies for specific task/runtime combinations are discovered manually, never systemically.
- The Router cannot adapt to changing runtime conditions (new models, price changes, capability shifts).

**Affected**: Model Router (RFC-0003), Sou, Mission Planning, Resource Optimization.

**Evidence**: `Research/001-Phase-3-Intelligence.md` (Area 3: Self-Improving Model & Agent Router). The Router's static strategies are already suboptimal — early data shows `best_match` underperforms `cheapest_viable` for documentation tasks but overperforms for code generation.

## Proposed Solution

Augment the Model Router (RFC-0003) with a **Self-Improving Feedback Loop** that learns from routing outcomes and automatically adjusts routing strategy weights, fallback chain selection, and runtime preference scoring.

### Architecture

```
Model Router (RFC-0003)
  │
  ├── Receives: CapabilityRequest
  ├── Executes: RoutingStrategy → RuntimeAssignment
  └── Emits: RoutingOutcome (post-execution)
       │
       ▼
Self-Improving Layer
  ├── Outcome Collector
  │   ├── Routing result (selected runtime vs. alternatives)
  │   ├── Execution quality (task success, output quality, duration)
  │   ├── Resource efficiency (actual vs. estimated cost)
  │   └── User satisfaction (implicit: rework needed? explicit: Sou rating)
  │
  ├── Strategy Evaluator
  │   ├── For each strategy: win/loss record per task/runtime combination
  │   ├── Statistical significance testing
  │   └── Confidence scoring
  │
  ├── Strategy Adjuster
  │   ├── Update strategy weights
  │   ├── Tweak scoring parameters
  │   └── Generate new strategy candidates
  │
  └── Router Config Store
      ├── strategy_weights: { best_match: 0.8, cheapest_viable: 0.2 }
      ├── scoring_parameters: { capability_weight: 0.4, cost_weight: 0.3, ... }
      └── experiment_config: { exploration_rate: 0.1 }
```

### Outcome Feedback Schema

```yaml
routing_outcome:
  routing_request_id: UUID
  worker_id: UUID
  mission_id: UUID
  task_id: UUID

  # What was selected
  selected_runtime: string
  selected_model: string
  routing_strategy: string

  # What was available (for counterfactual analysis)
  alternatives:
    - runtime: string
      model: string
      score: float
      was_selected: boolean

  # How it went
  execution:
    success: boolean
    quality_score: float                # 0.0–1.0 (from artifact review)
    duration_seconds: integer
    tokens_consumed: integer
    cost: float
    estimated_cost: float
    rework_required: boolean
    user_rating: float                  # If Sou provided explicit feedback

  # Task context
  task_type: string
  task_complexity: low | medium | high
  required_capabilities: [string]
```

### Learning Mechanism

**Multi-Armed Bandit** approach:

```
For each (task_type, capability_profile) pair:
  - Maintain Q-values for each strategy: Q(strategy)
  - After each routing outcome, update Q:
    Q_new = Q_old + α × (reward - Q_old)
    where reward = f(quality_score, -cost, -duration, -rework)
  - Exploration: ε-greedy selection (ε = 0.1 exploration rate)
  - Exploitation: softmax selection over Q-values
```

### Feedback Loop Constraints

```yaml
feedback_constraints:
  min_trials_per_configuration: 20       # Statistical significance floor
  exploration_rate: 0.10                 # ε in ε-greedy
  exploration_rate_min: 0.02             # Never fully stop exploring
  learning_rate: 0.15                    # α in Q-learning
  reward_discount: 0.95                  # γ — prefer near-term outcomes
  window_size: 1000                      # Rolling window for metrics
  confidence_threshold: 0.95             # p-value for strategy superiority
  cooldown_period_hours: 24              # Min time between parameter changes
```

### Alternatives Considered

1. **Static routing (RFC-0003 baseline)**: Rejected — no adaptation. Router cannot improve without manual intervention.

2. **Manual strategy tuning by Human/Security Council**: Rejected — defeats the purpose of autonomous routing. Creates human bottleneck.

3. **Full reinforcement learning (deep RL)**: Overengineered for v1. Multi-armed bandit provides sufficient improvement with explainability. Deep RL deferred to v2.

4. **A/B testing framework with manual analysis**: Rejected — analysis is automated by the Evaluator. No manual analysis step.

## Evidence

1. **`Research/001-Phase-3-Intelligence.md` (Area 3)**: Defines self-improving routing with multi-armed bandit, win/loss tracking, and automatic strategy refinement.

2. **Early routing data**: Internal analysis of 500+ routing decisions shows `cheapest_viable` outperforms `best_match` for documentation (82% vs 74% success) but underperforms for code generation (71% vs 88%). A static strategy misses these differences.

3. **`Bible/01-Constitution/000-Law-of-Governance.md` (Law 7)**: Iteration requires the system to learn. Static routing violates this law by never adapting.

- Does this RFC require a Law amendment? **No**

### Design DNA Compliance

| Rule | Assessment |
|------|------------|
| R1 — Modulsingularity | Self-Improving Layer has one job: learn from routing outcomes and tune parameters. Routing execution remains in Model Router. |
| R2 — Dependency Order | Depends on Model Router (RFC-0003), Learning Engine (RFC-0005). Not depended upon by critical path. |
| R3 — DRY | Outcome analysis logic centralized rather than duplicated across mission analysis, cost analysis, and manual reviews. |
| R7 — Tests Exist | Bandit update correctness tests, exploration/exploitation balance tests, parameter convergence tests, strategy superiority detection tests. |
| R8 — Fast Tests | Mathematical updates — <10ms per outcome. |
| R10 — Simpler Over Complex | Multi-armed bandit with ε-greedy. No neural networks, no deep RL, no distributed training. |
| R12 — Embrace Errors | Self-improving errors: SIROUTER_001 (insufficient data for significance), SIROUTER_002 (diverging parameters). Non-fatal — Router continues with current config. |
| R13 — Design for Failure | If Self-Improving Layer fails, Router falls back to last known good configuration. Graceful degradation. |
| R14 — Paved Path | Self-improving routing becomes default. Explicit strategy selection becomes advanced usage. |

## Migration Plan

1. **Phase 1** (collect): Enable outcome feedback collection. Router logs outcomes without changing routing behavior.
2. **Phase 2** (analyze): Deploy outcome analysis. Generate strategy performance reports.
3. **Phase 3** (adjust): Enable automatic strategy weight adjustment with manual override.
4. **Phase 4** (autonomous): Full self-improving routing with exploration. Sou override available.
