## Problem Statement

The system currently has no mechanism for monitoring, maintaining, or self-healing its own infrastructure. Services (IDS, ROS, LMS, AGS, ATS, ACF, Sou) run independently without a coordinating agent that watches for failures, resource exhaustion, or configuration drift.

When a service degrades or fails:
- No automated detection or alerting.
- No recovery procedure — human must intervene.
- No root cause analysis linking failure to system changes.
- No preventive maintenance (e.g., log rotation, cache warming, connection pool refresh).

**Affected**: All system services, System reliability, Operational continuity.

**Evidence**: `Research/002-Phase-4-Integration.md` (Area 1: System Agent for Internal Architecture). The system has experienced 12 unplanned outages in the past 90 days, all requiring human intervention.

## Proposed Solution

Introduce the **System Agent Framework** — a self-monitoring, self-healing layer that watches system services, detects anomalies, executes recovery procedures, and reports system health.

### Architecture

```
System Agent
  │
  ├── Health Monitors (per service)
  │   ├── IDS Monitor: identity allocation rate, collision rate, latency
  │   ├── ROS Monitor: resource utilization, budget exhaustion rate
  │   ├── LMS Monitor: lifecycle transition latency, state consistency
  │   ├── AGS Monitor: template load time, validation accuracy
  │   ├── ATS Monitor: token issuance rate, revocation rate
  │   ├── ACF Monitor: message throughput, delivery latency
  │   ├── Sou Monitor: mission completion rate, planning latency
  │   └── PostHog Monitor: event ingestion rate, query latency
  │
  ├── Health Evaluator
  │   ├── Aggregate health scores per service and overall
  │   ├── Anomaly detection (threshold-based and deviation-based)
  │   ├── Trend analysis (degrading? improving? stable?)
  │   └── Impact assessment (which missions/workers affected)
  │
  ├── Recovery Engine
  │   ├── Predefined recovery procedures
  │   ├── Escalation path (auto → Security Council → Human)
  │   └── Rollback capability for configuration changes
  │
  ├── Maintenance Scheduler
  │   ├── Log rotation, cache warming, connection refresh
  │   ├── Backup verification, integrity checks
  │   └── Certificate renewal, credential rotation
  │
  └── Reporting Interface
      ├── Health dashboard data
      ├── Incident reports
      ├── Post-mortem generation
      └── SLA compliance tracking
```

### Health Check Schema

```yaml
health_check:
  service_id: string
  service_type: core | auxiliary | external
  check_type: liveness | readiness | deep

  # Liveness: Is the service running?
  liveness:
    status: healthy | degraded | down
    last_heartbeat: timestamp
    uptime_seconds: integer
    pid: integer

  # Readiness: Can the service serve requests?
  readiness:
    status: healthy | degraded | down
    current_load_pct: float
    avg_latency_ms: integer
    p99_latency_ms: integer
    error_rate_pct: float
    queue_depth: integer
    connections_active: integer
    connections_idle: integer

  # Deep: Internal consistency checks
  deep:
    status: healthy | degraded | critical
    consistency_checks:
      - check_name: string
        status: pass | fail | warning
        details: string
    resource_usage:
      cpu_pct: float
      memory_mb: integer
      disk_usage_pct: float
      network_io_bytes_per_sec: integer
```

### Alert Schema

```yaml
system_alert:
  alert_id: UUID
  severity: critical | warning | info
  timestamp: timestamp
  service: string
  check_type: liveness | readiness | deep
  metric: string                          # e.g., "error_rate_pct"
  observed_value: float
  threshold_value: float
  duration_seconds: integer               # How long threshold exceeded

  status: firing | acknowledged | resolved | escalated
  acknowledged_by: string
  resolved_at: timestamp

  impact:
    affected_missions: [UUID]
    affected_workers: [UUID]
    affected_orgs: [UUID]
    severity_assessment: string

  related_alerts: [UUID]                  # Alert correlation
  related_changes: [UUID]                 # Configuration changes before alert
```

### Recovery Procedure Schema

```yaml
recovery_procedure:
  procedure_id: UUID
  trigger: alert_type | schedule | manual
  service: string
  automation_level: auto | semi_auto | manual

  steps:
    - step_number: integer
      action: restart_service | clear_cache | rotate_logs | scale_up | rollback_config | run_script
      parameters: object
      timeout_seconds: integer
      expected_result: string
      fallback: string                    # What to do if step fails

  prerequisites:
    - condition: string                   # e.g., "service is down"
      verify_before_execution: boolean

  rollback:
    procedure_id: UUID                    # Rollback procedure
    auto_rollback_on_failure: boolean

  post_recovery:
    verify_healthy: boolean
    notify: [string]                      # Who to notify
    generate_report: boolean
```

### Autonomous Recovery Levels

| Level | Name | Behavior |
|-------|------|----------|
| L0 | Manual | Alert only. Human must act. |
| L1 | Semi-autonomous | System recommends action. Human approves. |
| L2 | Autonomous with notification | System acts, then notifies. Human can override. |
| L3 | Fully autonomous | System acts. Notification only on failure. |

### Alternatives Considered

1. **Manual monitoring (current approach)**: Rejected — human must detect, diagnose, and recover every incident. Does not scale to 24/7 operation.

2. **External monitoring service (e.g., Datadog, Prometheus)**: Accepted as complementary. System Agent provides domain-specific recovery procedures that generic monitoring cannot.

3. **AI-driven self-healing (ML models for anomaly detection)**: Deferred to v2. v1 uses threshold-based and deviation-based detection. Simpler and more predictable.

## Evidence

1. **`Research/002-Phase-4-Integration.md` (Area 1)**: Defines System Agent with health checks, maintenance scripts, recovery procedures, and the internal architecture monitoring.

2. **12 unplanned outages in 90 days**: PostHog incident log shows all required human intervention. Average recovery time: 47 minutes. Auto-recovery target: <2 minutes.

3. **No health check endpoints**: Current services have no standardized health reporting. Each incident diagnosis starts with manual service probing.

## Constitutional Review

### Law Compliance

- Does this RFC violate any Law? **No** — System Agent monitors and recovers. It does not execute missions (Law 2), interfere with workers (Law 1), or set strategy. Recovery is infrastructure maintenance, not strategic decision-making.
- Does this RFC require a Law amendment? **No**
- Does this RFC require a Physics change? **Yes** — Add `Physics/010-Monitoring.md` for health check standards, alert severity definitions, and recovery procedures.

### Design DNA Compliance

| Rule | Assessment |
|------|------------|
| R1 — Modulsingularity | System Agent monitors and recovers. It does not execute missions, route tasks, or manage workers. |
| R2 — Dependency Order | System Agent depends on all services for health data. Services do not depend on System Agent for their operation. |
| R3 — DRY | Recovery procedures centralized rather than duplicated across service-specific scripts. |
| R7 — Tests Exist | Health check endpoint tests, recovery procedure execution tests, escalation path tests. |
| R10 — Simpler Over Complex | v1 uses threshold-based alerts and scripted recovery procedures. No ML, no predictive analytics. |
| R12 — Embrace Errors | Errors from health checks are data, not failures. System Agent errors: AGENT_001 (service unreachable), AGENT_002 (recovery timeout), AGENT_003 (rollback required). |
| R13 — Design for Failure | System Agent itself is monitored by a watchdog. If agent fails, services continue operating without monitoring. Graceful degradation. |
| R15 — Open/Closed | New services register health check endpoints via configuration. New recovery procedures added without agent core changes. |

## Migration Plan

1. **Phase 1** (endpoints): All core services implement health check endpoints (liveness + readiness).
2. **Phase 2** (agent): Deploy System Agent with health monitoring. Begin collecting baseline.
3. **Phase 3** (alerts): Configure alert thresholds. Enable alerting.
4. **Phase 4** (recovery): Deploy auto-recovery procedures at L1 (semi-autonomous).
5. **Phase 5** (autonomous): Gradual progression to L2/L3 for non-critical services.
