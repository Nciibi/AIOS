## Problem Statement

The system is currently limited to its own resources — built-in runtimes, templates, and compute capacity. There is no mechanism to connect to external providers of compute, models, storage, or specialized capabilities.

Without a provider network:
- All compute must be provisioned internally.
- Model access limited to pre-configured runtimes.
- No access to specialized external services (APIs, data sources, computation).
- Resource scaling is bounded by local infrastructure.

**Affected**: Resource capacity, Model diversity, Compute scalability, Cost optimization.

**Evidence**: `Research/003-Phase-5-Ecosystem.md` (Area 3: Provider Network for Models & Runtimes). System currently runs on fixed local capacity with 3 pre-configured models.

## Proposed Solution

Introduce the **Provider Network** — a federated network of external resource providers offering models, compute, storage, and specialized capabilities, with standardized contracts, billing, and quality of service.

```
AIOS Instance
  │
  ├── Provider Network Client
  │   ├── Provider discovery
  │   ├── Contract negotiation
  │   ├── Resource acquisition
  │   └── Payment/billing

  ├── Compute Providers

  │   ├── Edge compute providers
  │   └── Community compute
  │
  ├── Storage Providers
  │   ├── Object storage
  │   ├── Database providers
  │   └── CDN providers
  │
  └── Capability Providers
      ├── Specialized APIs
      ├── Data sources
      └── Computation services
```

### Resource Acquisition Flow

```
Sou (needs compute for a mission)
  │
  ├── 1. Query: "Need 500K tokens, claude-4-opus, <50ms latency"
  │
  ▼
Model Router (RFC-0003, extended)
  ├── 2. Check internal runtimes: insufficient capacity
  ├── 3. Query Provider Network: available providers?
  │
  ▼
Provider Network Client
  ├── 4. Discover providers matching requirements
  │   ├── Provider A: claude-4-opus, $15/M tokens, 45ms, available
  │   ├── Provider B: claude-4-opus, $12/M tokens, 55ms, available
  │   └── Provider C: claude-4-haiku, $3/M tokens, 30ms, available
  ├── 5. Score and select (cost × latency × reliability)
  ├── 6. Acquire resource: contract or pay-per-use
  ├── 7. Provision session
  │
  ▼
Worker runs on external provider
  ├── Worker is runtime-agnostic (RFC-0003)
  ├── Provider handles model execution
  ├── Usage metered and billed
  └── Session returned to pool on completion
```

### Alternatives Considered

1. **Local-only resources (current approach)**: Rejected — limited to local capacity. Cannot scale or access diverse models.

2. **Direct API integration for each provider**: Rejected — N providers × M integrations. Provider Network standardizes the interface.

3. **Centralized broker model**: Rejected — creates single point of control. Federated model allows direct provider relationships.

## Constitutional Review

### Law Compliance

- Does this RFC violate any Law? **No** — External resources are accessed through standard interfaces. Law 4 (Resource Bounds) is respected via budgets and rate limits. Law 1 (Non-Interference) preserved — workers use external providers same as internal.
- Does this RFC require a Law amendment? **No**
- Does this RFC require a Physics change? **Yes** — Add external resource provider standards to `Physics/008-Resources.md`.

### Design DNA Compliance

| Rule | Assessment |
|------|------------|
| R1 — Modulsingularity | Provider Network handles external resource contracts. Routing (RFC-0003) handles selection. Budget (ROS) handles cost. |
| R2 — Dependency Order | Provider Network depends on external APIs. Internal services depend on Provider Network abstraction, not external APIs directly. |
| R3 — DRY | External provider integration logic centralized rather than duplicated per service. |
| R5 — Liskov Substitution | External providers are substitutable for internal providers through the same interface. Worker does not know where model runs. |
| R7 — Tests Exist | Provider discovery tests, contract negotiation tests, metering accuracy tests, failover tests. |
| R10 — Simpler Over Complex | REST API for provider communication. Standard OAuth2 for auth. No blockchain, no decentralized consensus. |
| R12 — Embrace Errors | Provider errors: PROV_001 (provider unreachable), PROV_002 (rate limited), PROV_003 (budget exceeded), PROV_004 (contract expired). |
| R13 — Design for Failure | Automatic provider failover. Circuit breaker for degraded providers. Fallback to internal resources. |
| R15 — Open/Closed | New provider types implement ProviderAdapter interface. No core changes needed. |

## Migration Plan

1. **Phase 1** (discovery): Enable provider discovery. Catalog available external providers.
2. **Phase 2** (contracting): Enable contract negotiation with selected providers.
3. **Phase 3** (routing): Model Router (RFC-0003) extended with external provider routing.
4. **Phase 4** (metering): Usage metering and billing integration.
