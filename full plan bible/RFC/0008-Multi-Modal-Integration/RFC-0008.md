## Problem Statement

Workers currently operate exclusively in text-based modalities — code generation, analysis, documentation, and planning. The system has no mechanism for workers to process or produce non-text content: images, diagrams, audio, structured data, or other binary artifacts.

This limits the system's ability to:
- Analyze UI screenshots or architecture diagrams.
- Generate visual artifacts (charts, diagrams, wireframes).
- Process audio inputs or outputs.
- Handle binary data formats (PDF, CSV, images).
- Produce multi-modal reports combining text, images, and data.

**Affected**: All Workers, Mission Planning, Artifact types, Runtime selection.

**Evidence**: `Research/001-Phase-3-Intelligence.md` (Area 4: Multi-Modal Agent Integration). Current workers fail when missions involve image analysis or diagram generation. Runtime capabilities for vision/audio are present in models but not exposed through the system.

## Proposed Solution

Introduce **Multi-Modal Integration** — a capability framework that extends the system to support non-text modalities as first-class artifacts, with modality-aware routing, processing, and storage.

### Modality Taxonomy

```yaml
modalities:
  text:
    types: [code, markdown, plaintext, json, yaml, toml, csv, xml]
    mime_types: [text/*]
    default_runtime: any
  image:
    types: [png, jpeg, svg, webp, gif, bmp]
    mime_types: [image/*]
    capabilities_required: [vision, image_analysis, image_generation]
    default_runtime: claude-code (vision-capable)
  diagram:
    types: [mermaid, plantuml, drawio, excalidraw]
    mime_types: [text/vnd.mermaid, application/vnd.plantuml]
    capabilities_required: [diagram_generation, diagram_analysis]
    default_runtime: claude-code (diagram-capable)
  structured_data:
    types: [parquet, avro, arrow, protobuf]
    mime_types: [application/octet-stream]
    capabilities_required: [data_analysis]
    default_runtime: custom (data-analysis)
  audio:
    types: [wav, mp3, ogg, flac]
    mime_types: [audio/*]
    capabilities_required: [audio_transcription, audio_analysis]
    default_runtime: custom (audio-capable)
  document:
    types: [pdf, docx, xlsx, pptx]
    mime_types: [application/pdf, application/vnd.*]
    capabilities_required: [document_analysis, document_generation]
    default_runtime: claude-code (document-capable)
  archive:
    types: [zip, tar, gz, 7z]
    mime_types: [application/zip, application/gzip]
    capabilities_required: [archive_extraction, archive_creation]
    default_runtime: system-tool
```

### Modality-Aware Artifact Schema

```yaml
artifact:
  id: UUID
  type: artifact
  modality: text | image | diagram | structured_data | audio | document | archive
  mime_type: string
  format: string                          # Specific format within modality

  # Content
  content:
    inline: string                        # For text-based modalities
    reference: string                     # URI/URL for binary content
    size_bytes: integer
    hash: string                          # SHA-256 for integrity
    preview_available: boolean            # Thumbnail/summary for non-text

  # Modality-specific metadata
  modality_metadata:
    text: { encoding, language, line_count }
    image: { width, height, color_depth, dpi }
    diagram: { diagram_type, element_count }
    structured_data: { schema, row_count, column_count, compression }
    audio: { duration_seconds, sample_rate, channels }
    document: { page_count, has_images, has_tables }
    archive: { entry_count, compression_ratio, entry_names[] }

  # Processing
  capabilities_required: [string]         # What capabilities needed to process
  processed_by: UUID                      # Worker that produced/transformed this
  transforms:                             # Processing history
    - from_artifact_id: UUID
      transform_type: string              # e.g., "image_to_text", "diagram_to_code"
      timestamp: timestamp

  # Relationships
  parent_artifact_id: UUID                # For derived artifacts
  child_artifact_ids: [UUID]              # For decomposed artifacts
  mission_id: UUID
  task_id: UUID
```

### Modality-Aware Model Router

Extends RFC-0003 Model Router with modality requirements:

```yaml
model_router_request:
  # ... existing fields from RFC-0003 ...
  modality:                              # NEW
    input_modalities: [text, image]      # What modalities worker will receive
    output_modalities: [text, diagram]   # What modalities worker will produce
    required_mime_types: [image/png, text/markdown]
    max_file_size_bytes: 10485760         # 10MB
```

Router returns runtimes that support the required modality combinations.

### Modality Transformation Pipeline

```yaml
modality_transform:
  transform_id: UUID
  source_modality: string
  target_modality: string
  transform_type: string                  # Specific transformation

  known_transforms:
    image_to_text:
      description: "OCR or image description to text"
      required_capabilities: [vision, ocr]
      quality: medium                     # Lossy
    diagram_to_code:
      description: "Architecture diagram to infrastructure code"
      required_capabilities: [diagram_analysis, code_generation]
      quality: medium                     # Interpretive
    code_to_diagram:
      description: "Code structure to Mermaid diagram"
      required_capabilities: [code_analysis, diagram_generation]
      quality: high                       # Deterministic
    pdf_to_text:
      description: "PDF content extraction"
      required_capabilities: [document_analysis]
      quality: variable                   # Depends on PDF structure
    audio_to_text:
      description: "Speech to text transcription"
      required_capabilities: [audio_transcription]
      quality: high                       # Depends on audio quality
    structured_data_to_chart:
      description: "Data to visualization"
      required_capabilities: [data_analysis, diagram_generation]
      quality: variable                   # Depends on visualization choices
```

```yaml
worker_modality_config:
  worker_id: UUID
  supported_input_modalities: [string]
  supported_output_modalities: [string]
  modality_capabilities:
    - modality: string

```

### Storage Strategy

| Modality | Storage | Access Pattern |
|----------|---------|----------------|
| Text | Inline in graph + object store | Queryable, searchable |
| Image | Object store + thumbnail | Reference in graph |
| Diagram | Source text + rendered image | Both stored |
| Structured data | Object store + schema | Schema in graph |
| Audio | Object store | Reference in graph |
| Document | Object store + text extract | Extract in graph |
| Archive | Object store + manifest | Manifest in graph |

### Alternatives Considered

1. **Text-only system (current approach)**: Rejected — runtime models increasingly support vision, audio, and document analysis. System must expose these capabilities.

2. **Binary blob storage without modality awareness**: Rejected — loses semantic understanding of content type, processing requirements, and transformation possibilities. Cannot route modality-specific tasks.

3. **Universal format conversion (everything to text)**: Rejected — lossy for images, diagrams, and structured data. Destroys modality-specific information.

### Affected Services

- **Artifact Store**: New binary object store for non-text artifacts
- **Model Router** (RFC-0003): Modality-aware runtime selection
- **Worker provisioning**: Modality capability declaration
- **Knowledge Graph** (RFC-0006): Artifact nodes include modality properties
- **Transformation Pipeline**: New modality transformation service

### Breaking Changes

- Artifact schema changes: `content` field may be a reference rather than inline text.
- Backward compatible: Existing text artifacts remain inline.

## Evidence

1. **`Research/001-Phase-3-Intelligence.md` (Area 4)**: Defines multi-modal integration with image analysis, diagram generation, PDF/CSV processing, and multi-modal agent orchestration.

2. **Model capability growth**: Claude, GPT-4, Gemini all support vision. Some support audio. The system's text-only constraint is an artificial limitation.

3. **Mission requests**: Multiple mission requests involve UI screenshot analysis, architecture diagram generation, and PDF report processing — currently impossible.

## Constitutional Review

### Law Compliance

- Does this RFC violate any Law? **No** — Modality is a capability extension. It does not change how workers execute (Law 1), how Sou decides (Law 2), or how identity is assigned (Law 5).
- Does this RFC require a Law amendment? **No**
- Does this RFC require a Physics change? **No**

### Design DNA Compliance

| Rule | Assessment |
|------|------------|
| R1 — Modulsingularity | Multi-modal support is a capability cross-cutting concern, not a service. Modality-aware storage (Artifact Store), routing (Model Router), and provisioning (AGS) each handle their aspect. |
| R2 — Dependency Order | Modality taxonomy is a data schema — depends on nothing. Modality-aware services depend on the taxonomy. |
| R3 — DRY | Modality definitions centralized in taxonomy rather than duplicated across services. |
| R5 — Liskov Substitution | Modality-aware workers are substitutable for text-only workers in text-only tasks. Backward compatibility. |
| R7 — Tests Exist | Modality routing tests, artifact storage/retrieval tests, transformation pipeline tests. |
| R10 — Simpler Over Complex | Modality taxonomy is a configuration file, not a complex inference engine. |
| R15 — Open/Closed | New modalities registered by adding to taxonomy — no code changes required. |

## Migration Plan

1. **Phase 1** (taxonomy): Define modality taxonomy. Update Artifact schema to support binary references.
2. **Phase 2** (storage): Deploy object store for non-text artifacts. Backfill existing text artifacts as "just text" modality.
3. **Phase 3** (routing): Update Model Router (RFC-0003) to consider modality constraints.
4. **Phase 4** (workers): Workers declare modality capabilities. Modality-aware tasks dispatched.
