## Problem Statement

The system currently has no mechanism for learning from past missions, worker outcomes, or operational patterns. Each mission starts from scratch — Sou has no institutional memory, and workers cannot benefit from the successes and failures of prior missions.

Without learning:
- Successful strategies are repeated manually, never codified.
- Failed approaches are repeated unknowingly.
- Worker configurations that produce superior results are not identified or promoted.
- Pattern detection (anomalies, bottlenecks, optimization opportunities) requires human analysis.

The companion **Knowledge Graph** (RFC-0006) provides the semantic storage for learned patterns; without it, learned knowledge has no structured persistence.

**Affected**: Sou, All Workers, Mission Planning, Resource Optimization, Quality Assurance.

**Evidence**: `Research/001-Phase-3-Intelligence.md` (Area 1: Outcome Learning Engine). The system has accumulated 200+ missions with no systematic learning pipeline. PostHog analytics exist but are not integrated into operational feedback.

## Proposed Solution

Introduce the **Learning Engine** — a service that ingests mission artifacts, worker outputs, and operational metrics to generate patterns, recommendations, and institutional knowledge.

### Architecture

```
Mission Complete
  │
  ├── Artifacts: worker outputs, code, decisions, logs
  ├── Metrics: tokens used, duration, quality scores
  └── Outcomes: success/failure, quality rating, rework needed
  │
  ▼
Learning Engine
  ├── Ingestion Pipeline
  │   ├── Worker output collector
  │   ├── Mission outcome collector
  │   ├── Resource consumption collector
  │   └── Interaction log collector
  │
  ├── Analysis Pipeline
  │   ├── Pattern detection
  │   ├── Success factor analysis
  │   ├── Failure root cause analysis
  │   ├── Template optimization suggestions
  │   └── Anomaly detection
  │
  ├── Knowledge Store (backed by Knowledge Graph — RFC-0006)
  │   ├── Learned patterns
  │   ├── Best practices (extracted)
  │   ├── Anti-patterns (extracted)
  │   ├── Worker effectiveness profiles
  │   └── Template effectiveness scores
  │
  └── Query Interface
      ├── Sou queries: "What patterns work for Rust migrations?"
      └── System queries: "Suggest template improvements"
```

### Learning Record Schema

```yaml
learning_record:
  record_id: UUID
  source_type: mission | worker | org | system
  source_id: UUID

  # What was learned
  observation: string                    # Natural language description
  pattern_type: success_factor | failure_cause | optimization | anomaly | best_practice
  confidence: float                      # 0.0–1.0
  evidence_count: integer                # Number of supporting observations

  # Context
  mission_type: string
  worker_templates: [UUID]
  runtimes: [string]
  models: [string]

  # Impact
  impact_metric: string                  # e.g., "mission_success_rate"
  impact_delta: float                    # e.g., +0.15 (15% improvement)
  impact_significance: float             # Statistical significance p-value

  # Application
  applicability:
    - condition: string
      recommendation: string
  automation_possible: boolean           # Can system auto-apply this?
  auto_applied: boolean

  # Metadata
  created_at: timestamp
  expires_at: timestamp                  # Learning decays if not reinforced
  reinforcement_count: integer
  last_reinforced: timestamp
```

### Analysis Pipeline Stages

```
Ingested Data
  │
  ▼
Stage 1: Classification
  ├── Categorize by mission type, worker type, runtime
  └── Tag with metadata
  │
  ▼
Stage 2: Pattern Extraction
  ├── Frequent subsequence mining (action patterns)
  ├── Correlation analysis (what co-occurs with success)
  ├── Deviation detection (what differs in failures)
  └── Temporal pattern detection (time-series)
  │
  ▼
Stage 3: Hypothesis Generation
  ├── "Rust workers with claude-code succeed 30% more often than with codex"
  ├── "Missions with 3+ QA workers have 50% fewer bugs"
  └── "Token budgets exceeding 100K have diminishing returns"
  │
  ▼
Stage 4: Validation
  ├── Statistical significance test
  ├── Cross-validation across mission types
  └── Confidence scoring
  │
  ▼
Stage 5: Knowledge Store Update
  ├── Insert/update learning record
  └── Adjust confidence of related records
```

### Pattern Types

| Pattern Type | Description | Example | Auto-Applicable? |
|-------------|-------------|---------|-----------------|
| `success_factor` | Conditions correlating with success | "3+ review rounds → 90% fewer defects" | Advisory |
| `failure_cause` | Conditions correlating with failure | "Missing capability bound → hallucination" | Pre-check |
| `optimization` | Configuration improvements | "claude-4-haiku sufficient for doc tasks" | Auto-config |
| `anomaly` | Unusual patterns requiring attention | "Token consumption spike in testing worker" | Alert |
| `best_practice` | Extracted workflow patterns | "Always run lint before review" | Template update |
| `anti_pattern` | Patterns to avoid | "Parallel code gen without conflict detection" | Pre-check |

### Query Interface

Sou queries the Learning Engine:

```yaml
learning_query:
  intent: planning | configuration | troubleshooting | improvement
  mission_type: string
  worker_templates: [UUID]
  constraints:
    min_confidence: float                # Default 0.7
    max_age_days: integer                # Default 90
    limit: integer                       # Default 10

learning_response:
  patterns:
    - pattern_id: UUID
      observation: string
      pattern_type: string
      confidence: float
      evidence_count: integer
      recommendation: string
      applicability_score: float         # How relevant to this query
  meta_learning:
    data_quality_score: float            # How good is the underlying data
    coverage_gaps: [string]              # Areas with insufficient data
```

### Lifecycle of a Learning Record

```
Discovered → Validated → Published → Applied → Reinforced → Archived
                  │                                      │
                  ▼                                      ▼
              Rejected (insufficient evidence)      Decayed (not reinforced)
```

| State | Description |
|-------|-------------|
| Discovered | Pattern detected but not validated |
| Validated | Statistical significance confirmed |
| Published | Available for queries |
| Applied | Used in at least one real decision |
| Reinforced | Confirmed by additional evidence |
| Archived | Superseded or decayed |
| Rejected | Failed validation |
| Decayed | Confidence fell below threshold |

### Alternatives Considered

1. **No learning (current approach)**: Rejected — 200+ missions with no accumulation. Each mission is independent. Violates Law 7 (Iteration).

2. **Log-based manual analysis (PostHog only)**: Rejected — PostHog provides raw data but no automated pattern extraction, no closed-loop feedback into operations.

3. **Worker-level learning only**: Rejected — workers have ephemeral lifecycles. Learning must persist beyond individual workers.

4. **ML model-based learning pipeline**: Deferred to v2. v1 uses statistical analysis (correlation, frequency, deviation) — simpler, explainable, auditable.

## Constitutional Review

### Law Compliance

- Does this RFC violate any Law? **No** — Learning Engine observes and recommends. It does not execute (Law 2), interfere (Law 1), or make strategic decisions.
- Does this RFC require a Law amendment? **No**
- Does this RFC require a Physics change? **Yes** — Create `Physics/013-Learning.md` to define learning feedback loops, knowledge retention, data decay policies, and pattern lifecycle standards.

### Design DNA Compliance

| Rule | Assessment |
|------|------------|
| R1 — Modulsingularity | Learning Engine analyzes and recommends. It does not enforce or execute. |
| R2 — Dependency Order | Depends on PostHog, LMS, ROS for input data. Not depended upon by critical path services. |
| R3 — DRY | Pattern extraction is centralized rather than duplicated across mission planning, template management, and QA. |
| R7 — Tests Exist | Pattern extraction accuracy tests, confidence calibration tests, decay simulation tests, query relevance tests. |
| R8 — Fast Tests | Analysis is offline/async. Query path <200ms. |
| R10 — Simpler Over Complex | v1 uses correlation and frequency analysis — no neural networks, no deep learning, no NLP models. |
| R12 — Embrace Errors | Learning errors are non-fatal: LEARN_001 (insufficient data), LEARN_002 (pattern below confidence threshold), LEARN_003 (query too broad). |
| R13 — Design for Failure | If Learning Engine is down, missions proceed without pattern lookup. Graceful degradation. |
| R15 — Open/Closed | New analysis methods (e.g., ML models in v2) implement the `AnalysisMethod` interface. |

## Migration Plan

1. **Phase 1** (data collection): Enable learning record ingestion from all missions. Establish baseline.
2. **Phase 2** (analysis): Deploy analysis pipeline. Generate initial pattern set from historical data.
3. **Phase 3** (integration): Sou integrates Learning Engine queries into mission planning.
