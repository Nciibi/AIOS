## Purpose

CSP is the central cryptographic service for AIOS. All cryptographic operations route through CSP. No entity performs cryptography outside CSP. CSP provides hashing, encryption, signing, key generation, and random number generation as atomic, auditable operations. CSP enforces algorithm policy, manages algorithm deprecation, and integrates with HSM for sensitive operations.

## Cryptographic Operations

### Core Operations

| Operation | Description | Parameters |
|-----------|-------------|------------|
| `hash(data, algorithm)` | Produce cryptographic hash | data: bytes, algorithm: Algorithm |
| `encrypt(data, key_id, algorithm)` | Symmetric or asymmetric encryption | data: bytes, key_id: KeyID, algorithm: Algorithm |
| `decrypt(ciphertext, key_id)` | Decrypt ciphertext | ciphertext: bytes, key_id: KeyID |
| `sign(data, key_id)` | Produce digital signature | data: bytes, key_id: KeyID |
| `verify(data, signature, public_key)` | Verify digital signature | data: bytes, signature: bytes, public_key: PublicKey |
| `generateKey(algorithm, purpose)` | Generate cryptographic key | algorithm: Algorithm, purpose: KeyPurpose |
| `generateRandom(count)` | Generate cryptographically secure random bytes | count: uint |

### Supported Algorithms

| Category | Algorithms | Status |
|----------|------------|--------|
| Hashing | SHA-256, SHA-384, SHA-512, BLAKE2b, SHA-3 | Active |
| Symmetric | AES-256-GCM, ChaCha20-Poly1305 | Active |
| Asymmetric | RSA-4096 OAEP | Legacy |
| Signing | ECDSA P-256, ECDSA P-384, Ed25519, RSA-4096 | Active |
| Key Exchange | X25519, ECDH P-256, ECDH P-384 | Active |

### Algorithm Selection Policy

CSP selects the strongest available algorithm the entity's capability allows. Algorithm selection is deterministic for identical inputs. Deprecated algorithms (MD5, SHA-1) are blocked at the CSP boundary and produce no result.

### CSP Clustering

CSP operates in active-passive configuration with HSM backing. The active instance handles all cryptographic operations. The passive instance synchronises state from the active instance and HSM. Failover is automatic within 30 seconds.

## Cross-Cutting Concerns

### Security
CSP is the sole cryptographic gateway. All entities call CSP; no entity performs cryptography inline. CSP enforces algorithm policy centrally. Compromise of CSP is a system-level emergency triggering automatic key rotation and HSM lockdown.

### Evidence
Every cryptographic operation produces a CSP.Event logged to the Event Store. Operation parameters (algorithm, key_id, entity_id) are recorded. Plaintext data is never logged.

### Lifecycle
CSP has two lifecycles: software lifecycle (version upgrades through ACF rolling update) and algorithm lifecycle (introduced → active → deprecated → removed). Algorithm lifecycle is governed by the Security Council.

### Capability Bounds
Entities may only use algorithms their capability profile permits. CSP enforces this at the operation level. An entity cannot request SHA-512 if its capability only allows SHA-256.

### Communication
All CSP calls use ACF request/response with mTLS. CSP does not expose HTTP or gRPC endpoints directly.

### Design DNA Compliance

| Rule | Compliance |
|------|------------|
| R1 — Modulsingularity | CSP does exactly one thing: cryptography. Sub-engines (hash, encrypt, sign, key) are internal. |
| R2 — Dependency Order | CSP depends on HSM, KMS, and ACF. It is depended on by all services. |
| R3 — DRY | Algorithm selection policy exists in one place: the Algorithm Registry. |
| R4 — Builder Pattern | Key objects are built by the Key Engine; callers receive validated key references. |
| R5 — LSP | All algorithm implementations satisfy the same Operation interface. |
| R6 — DI over Singletons | CSP dependencies (HSM adapter, KMS client) are injected. |
| R7 — Tests Exist | Every operation has unit tests. Every algorithm has integration tests against HSM. |
| R8 — Fast Tests | Software operations complete in <10ms. HSM operations complete in <100ms. |
| R9 — Deterministic | Hashing, encryption (given same IV), signing are deterministic. |
| R10 — Simpler Over Complex | CSP provides 6 core operations. No branching, no plugins, no extensibility. |
| R11 — Refactor over Rewrite | Algorithm additions extend the registry; no CSP core modification. |
| R12 — Embrace Errors | Every failure mode has a unique code (CSP_ERR_001-050). |
| R13 — Design for Failure | HSM unavailable → CSP degrades to software-only for non-sensitive ops. |
| R14 — Paved Path | The paved path is: entity → ACF → CSP → HSM/KMS. No other path exists. |
| R15 — Open/Closed | New algorithms implement Algorithm interface without modifying CSP core. |
