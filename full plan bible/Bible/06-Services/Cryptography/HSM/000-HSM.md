## Purpose

This document specifies Hardware Security Module integration for AIOS cryptography. HSM provides hardware-backed security for all sensitive cryptographic operations. Keys generated in HSM never leave HSM in plaintext. HSM is the root of trust for all cryptographic operations.

## HSM Types

| Type | Examples | Use Case |
|------|----------|----------|
| Cloud HSM | AWS CloudHSM, Azure Dedicated HSM | Production (cloud deployment) |
| Hardware HSM | Thales Luna 7, YubiHSM 2 | Production (on-premises) |
| Software HSM | SoftHSM2 | Development and testing only |

## HSM Operations

| Operation | Location | Description |
|-----------|----------|-------------|
| Key Generation | Inside HSM | Generate asymmetric key pairs, symmetric keys. Private key never leaves HSM. |
| Signing | Inside HSM | Private key signing operations execute inside HSM boundary. |
| Decryption | Inside HSM | Private key decryption operations execute inside HSM boundary. |
| Random Generation | HSM RNG | Hardware random number generator for cryptographic randomness. |
| Key Wrapping | Inside HSM | Export a key encrypted under a wrapping key (key is encrypted at export). |
| Key Deletion | Inside HSM | Destroy key material inside HSM. Cryptographic delete. |

### Sensitive vs. Non-Sensitive Operations

| Operation | Sensitive | Location |
|-----------|-----------|----------|
| CA key generation | Yes | HSM only |
| CA key signing | Yes | HSM only |
| Code signing key gen | Yes | HSM only |
| TLS key generation | No | Software (keys stored encrypted) |
| TLS key signing | Conditional | HSM for high-security, software for standard |
| Random generation | Yes | HSM hardware RNG for crypto, OS entropy for non-crypto |
| Data encryption (DEK) | No | Software |
| Hash computation | No | Software |

## HSM Clustering

HSM operates in active-active configuration for key operations and active-passive for HSM-backed key storage.

| Configuration | Purpose | Failover |
|---------------|---------|----------|
| Active-Active | Signing, decryption, random generation | Instant (load-balanced) |
| Active-Passive | Key storage, key generation | 30 seconds |

## HSM Health Monitoring

| Health Check | Frequency | Failure Action |
|-------------|-----------|---------------|
| Connectivity | Every 5 seconds | Log, retry, failover to passive |
| Operation latency | Every operation | Alert if >100ms average |
| Key availability | Every 60 seconds | Alert if keys missing |
| RNG entropy | Every 30 seconds | Alert if entropy low |
| Certificate expiry | Every 5 minutes | Alert at 30 days before expiry |

## Cross-Cutting Concerns

### Security
HSM keys never leave HSM in plaintext. HSM firmware is verified at boot. HSM audit logs are exported to the Event Store. Physical HSM tamper detection triggers automatic key zeroization.

### Evidence
All HSM operations produce events logged by CSP. HSM audit logs are exported and stored in the Event Store for compliance.

### Lifecycle
HSM firmware updates follow the standard RFC process. Key lifecycle is managed by KMS; HSM enforces key material lifecycle at the hardware level.

### Capability Bounds
Entities requiring HSM-backed keys must have HSM capability in their profile. Development instances may use SoftHSM as a substitute.

### Design DNA Compliance
| Rule | Compliance |
|------|------------|
| R1 | HSM does exactly one thing: hardware cryptographic operations. |
| R10 | CSP routes sensitive/non-sensitive deterministically. Simple dispatch. |
| R13 | HSM failure → CSP degrades to software for non-sensitive ops only. |
| R14 | Paved path: sensitive ops → HSM. No software fallback for sensitive. |
