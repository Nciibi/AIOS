## Purpose

SMS provides secure storage for static secrets (API keys, passwords, database credentials, configuration secrets, service account tokens). SMS is distinct from SSM (Session & Secret Manager, 04-Execution/Security/SSM/): SMS stores persistent secrets with rotation policies, while SSM manages ephemeral session secrets with TTL measured in minutes.

## Secret Types

| Type | Description | Example | Rotation Interval |
|------|-------------|---------|-------------------|
| Database Credential | Username/password for internal databases | `db.aios_events.rw` | 90 days |
| API Token | External service access tokens | `aws_access_key_id` | 180 days |
| TLS Private Key | Private key material (encrypted) | TLS server private key | Certificate renewal |
| Configuration Secret | Sensitive configuration values | `jwt_signing_secret` | 90 days |
| Service Account | Internal service credentials | `acf_service_token` | 30 days |

## Secret Lifecycle

```
Created → Active → Rotating → Retired → Destroyed
                ↑         │
                └─────────┘
              (re-rotate)
```

| State | Description | Allowed Operations |
|-------|-------------|-------------------|
| Created | Secret stored, not yet usable | Get metadata, rotate |
| Active | Secret is usable | Get secret, get metadata, rotate |
| Rotating | Old value being replaced | Get old value, complete rotation |
| Retired | Secret no longer usable | Get metadata, destroy |
| Destroyed | Secret material deleted | Get metadata (without value) |

## SMS Operations

| Operation | Description | Parameters |
|-----------|-------------|------------|
| createSecret | Store a new secret | name: string, value: bytes, type: SecretType, rotation_policy: RotationPolicy |
| getSecret | Retrieve a secret value | secret_id: SecretID, context: AccessContext |
| rotateSecret | Trigger secret rotation | secret_id: SecretID |
| listSecrets | List secrets matching filter | filter: SecretFilter |
| destroySecret | Permanently delete a secret | secret_id: SecretID |

### Rotation Process

```
1. Security Council or rotation timer triggers SMS.rotateSecret(secret_id)
2. SMS generates new secret value via CSP.generateRandom() or external API
3. SMS stores new version (N+1) as Active, marks old version (N) as Rotating
4. SMS notifies consuming services of rotation via ACF event
5. Consuming services switch to new value (grace period: 5 minutes)
6. After grace period, SMS marks old version as Retired
7. After 24 hours in Retired, SMS marks old version as Destroyed
```

## Access Control

| Aspect | Specification |
|--------|--------------|
| Granting Authority | Security Council |
| Scope | Per service, per secret type |
| Auditing | Every access logged (who, when, which secret, context) |
| Default | Deny all (explicit grant required) |
| Emergency Access | Security Council override, logged with reason |

## Encryption

Secrets are encrypted at rest using CSP envelope encryption:
- Secret value encrypted with AES-256-GCM under a DEK
- DEK wrapped under KEK managed by KMS
- KEK stored in HSM (never software-accessible)
- In-transit: ACF secure channels (mTLS + CSP encryption)

## Secret Access Patterns

### Standard Access

```
1. Service calls SMS.getSecret(secret_id, context)
2. SMS validates: service identity, capability, context, authorization
3. SMS retrieves encrypted secret from store
4. SMS calls CSP.decrypt(encrypted_secret, context) 
5. CSP returns plaintext secret
6. SMS logs access event (without secret value)
7. SMS returns plaintext to service (ACF encrypted channel)
```

### Emergency Access

```
1. Security Council member authorizes emergency access
2. SMS.getSecret() called with emergency authorization token
3. SMS bypasses normal authorization, logs emergency flag
4. Secret returned with automatic rotation trigger
5. Security Council notified of emergency access within 5 minutes
```

## Rotation Policy Configuration

| Field | Type | Description | Default |
|-------|------|-------------|---------|
| interval_days | uint | Days between automatic rotation | 90 |
| grace_period_minutes | uint | Minutes both old and new values active | 5 |
| retain_versions | uint | Number of old versions to retain | 2 |
| auto_rotate | boolean | Automatic rotation enabled | true |
| notify_services | boolean | Notify consuming services on rotation | true |

## Cross-Cutting Concerns

### Security
Secrets are encrypted at rest and in transit. Access is explicitly granted by Security Council. Rotation is enforced by policy. Emergency access is audited. Secret values are never logged.

### Evidence
Every secret operation (create, get, rotate, destroy) produces an SMS event. Access is audited with full context (who, when, which secret, reason).

### Lifecycle
Secret lifecycle follows Law 6. Rotation is mandatory for all secret types. Retired secrets are destroyed after a retention period. Secrets cannot be recovered after destruction.

### Capability Bounds
Services can only access secret types their capability profile permits. A database service may access database credentials but not API tokens.

### Design DNA Compliance
| Rule | Compliance |
|------|------------|
| R1 | SMS stores static secrets. SSM stores ephemeral session secrets. Clear separation. |
| R3 | Secret type definitions in one place. Rotation policy in one place. |
| R9 | Secret access is deterministic: same identity + same secret_id = same result. |
| R10 | Simple lifecycle: 5 states, deterministic transitions. |
| R12 | Every error has a unique code (SMS_ERR_001-010). |
| R13 | Secret unavailable → service fails closed (no fallback to insecure mode). |
| R14 | Paved path: create → active → rotate → retire → destroy. |
