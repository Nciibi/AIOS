## Purpose

KMS manages the full lifecycle of all cryptographic keys in AIOS. KMS governs key generation, storage, rotation, deletion, and auditing. Keys are the foundational security primitive; KMS ensures their integrity, availability, and proper lifecycle management.

## Key Types

| Type | Abbreviation | Description | Storage |
|------|-------------|-------------|---------|
| Key Encryption Key | KEK | Encrypts other keys (DEKs) | HSM only, never software-accessible |
| Data Encryption Key | DEK | Encrypts data | Encrypted under KEK in CSP |
| Signing Key | SIG | Digital signature private key | HSM (high-security) or encrypted store |
| Authentication Key | AUTH | Entity authentication (mTLS, token signing) | Encrypted store |
| Transport Key | TRANS | Key exchange between instances | Encrypted store, short-lived |

## Key Lifecycle

```
Generated → Enabled → Disabled → Scheduled_for_Deletion → Deleted
                ↑          │
                └──────────┘
               (Re-enabled)
```

| State | Description | Allowed Operations |
|-------|-------------|-------------------|
| Generated | Key created, not yet usable | Get metadata, schedule deletion |
| Enabled | Key is active and usable | Encrypt, decrypt, sign, verify |
| Disabled | Key exists but cannot be used | Get metadata, enable, schedule deletion |
| Scheduled_for_Deletion | Key will be deleted after waiting period | Cancel deletion, get metadata |
| Deleted | Key material destroyed | None |

## Key Operations

| Operation | Description | Parameters |
|-----------|-------------|------------|
| createKey | Create a new key | type: KeyType, algorithm: Algorithm, purpose: KeyPurpose |
| enableKey | Enable a disabled key | key_id: KeyID |
| disableKey | Disable an enabled key | key_id: KeyID |
| scheduleKeyDeletion | Schedule key for deletion | key_id: KeyID, days: uint (default 7) |
| cancelKeyDeletion | Cancel pending deletion | key_id: KeyID |
| rotateKey | Rotate key (create new version) | key_id: KeyID |
| getKeyMetadata | Get key attributes | key_id: KeyID |
| listKeys | List keys matching filter | filter: KeyFilter |

## Key Rotation

### Automatic Rotation Schedule

| Key Type | Rotation Interval | Trigger |
|----------|-------------------|---------|
| KEK (Root) | 1 year | Timer |
| KEK (Regional) | 6 months | Timer |
| DEK | Per encryption operation | Auto (new DEK each encrypt) |
| Signing Key | 1 year | Timer |
| Authentication Key | 90 days | Timer |
| Transport Key | 24 hours | Timer |
| Emergency Rotation | Immediate | Security Council order |

### Rotation Process

```
1. KMS.createKey() generates new key version (N+1)
2. KMS.rotateKey() promotes N+1 to primary
3. KMS triggers reEncrypt for all data under old key version
4. Old key version (N) is disabled after all data re-encrypted
5. Old key version is scheduled for deletion (7-day waiting period)
6. After waiting period, old key version is deleted
```

## Key Storage Hierarchy

```
┌──────────────────────────────────────────────┐
│                  HSM                           │
│  ┌──────────────────────────────────────────┐ │
│  │  Root KEK (never leaves HSM)              │ │
│  │  └── Regional KEK 1 (wrapped by Root)    │ │
│  │  └── Regional KEK 2 (wrapped by Root)    │ │
│  └──────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
         │ (KEK wrapping/unwrapping)
         ▼
┌──────────────────────────────────────────────┐
│                  CSP / KMS                     │
│  ┌──────────────────────────────────────────┐ │
│  │  DEK v1 (encrypted under KEK)             │ │
│  │  DEK v2 (encrypted under KEK)             │ │
│  │  Signing Keys (encrypted under KEK)       │ │
│  └──────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
```

## Key Deletion Rules

- KEK deletion: requires Security Council approval (unanimous). 30-day waiting period.
- DEK deletion: requires entity owner approval. 7-day waiting period.
- Signing key deletion: requires Security Council approval. 14-day waiting period.
- Emergency deletion: immediate, Security Council order required. Evidence logged.
- Deleted keys: metadata retained for 7 years (key_id, algorithm, creation, deletion dates). Key material destroyed.

## Cross-Cutting Concerns

### Security
KEKs never leave HSM. DEKs exist in memory only during encryption operations. Key deletion is cryptographic (key material destroyed). Key rotation renders previous key versions unusable for new operations.

### Evidence
Every key lifecycle transition (create, enable, disable, rotate, delete) produces a KMS event. Key access is audited per operation.

### Lifecycle
Key lifecycle is governed by Law 6. Each transition is authorized, recorded, and verifiable. Keys cannot skip lifecycle states.

### Capability Bounds
Entities can only create key types their capability profile permits. A Worker entity cannot create a KEK.

### Design DNA Compliance
| Rule | Compliance |
|------|------------|
| R1 | KMS manages keys only. CSP uses keys. |
| R6 | KMS dependencies injected (HSM adapter, audit logger). |
| R10 | Simple lifecycle: 5 states, deterministic transitions. |
| R13 | KMS unavailable → existing keys cached in CSP for read-only operations. |
| R14 | Paved path: create → enable → use → rotate → disable → delete. |
