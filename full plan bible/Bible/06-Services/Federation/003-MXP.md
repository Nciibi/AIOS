## Purpose

MXP enables Missions to span multiple AIOS instances. A Mission initiated on one instance may delegate sub-Missions to other instances. Results are synchronised across instances through MXP.

## Protocol Flow

```
Lead Instance                     Partner Instance
   │                                   │
   │──── proposeCollaboration ────────►│
   │◄── accept(terms) ─────────────────│
   │──── assignSubMission(spec) ──────►│
   │◄── acknowledge ───────────────────│
   │                                   │ (partner executes sub-Mission)
   │◄── syncProgress(status) ──────────│
   │──── syncAcknowledged ────────────►│
   │                                   │
   │◄── complete(result) ──────────────│
   │──── completeAcknowledged ────────►│
```

## Operations

| Operation | Description | Parameters |
|-----------|-------------|------------|
| proposeMission | Propose collaborative Mission | mission_spec: MissionSpec, partners: InstanceID[] |
| acceptMission | Accept collaboration proposal | proposal_id: ProposalID, terms: CollaborationTerms |
| assignSubMission | Delegate sub-Mission to partner | partner_id: InstanceID, sub_mission_spec: SubMissionSpec |
| syncProgress | Report sub-Mission progress | sub_mission_id: SubMissionID, status: MissionStatus, checkpoint: Checkpoint |
| completeMission | Report sub-Mission completion | sub_mission_id: SubMissionID, result: bytes |
| abortMission | Abort collaborative Mission | mission_id: MissionID, reason: string |

## Mission Collaboration Terms

| Field | Description | Default |
|-------|-------------|---------|
| delegation_depth | Max sub-Mission nesting | 2 |
| max_partners | Max participating instances | 5 |
| sync_interval | Progress sync frequency | 60 seconds |
| timeout | Max sub-Mission wall-clock time | 24 hours |
| abort_policy | What happens on partner abort | cleanup |
| result_sharing | Which results are shared | all |

## Sub-Mission Lifecycle

```
Assigned → Acknowledged → Running → Sync → Complete → Verified
                              │                        │
                              └── Aborted ──────────────┘
```

| State | Description | Owner |
|-------|-------------|-------|
| Assigned | Lead assigns sub-Mission to partner | Lead |
| Acknowledged | Partner accepts sub-Mission | Partner |
| Running | Partner executes sub-Mission | Partner |
| Sync | Partner reports progress to lead | Both |
| Complete | Partner submits final result | Partner |
| Verified | Lead verifies result | Lead |
| Aborted | Either side terminates sub-Mission | Both |

## MXP Error Codes

| Code | Condition | Recovery |
|------|-----------|----------|
| MXP_001 | Partner instance at max mission capacity | Retry later or choose different partner |
| MXP_002 | Sub-Mission exceeds partner capability | Decompose mission into smaller scopes |
| MXP_003 | Progress sync timeout | Increase sync_interval or check connectivity |
| MXP_004 | Sub-Mission execution timeout | Increase timeout or optimize execution |
| MXP_005 | Result verification failure | Partner re-executes or lead reassigns |
| MXP_006 | Abort not acknowledged | Force abort after escalation timeout |

## MXP Events

| Event Type | Produced When | Fields |
|-----------|--------------|--------|
| MXP.MissionProposed | Collaboration proposal sent | proposal_id, mission_spec, partners |
| MXP.MissionAccepted | Proposal accepted | proposal_id, terms |
| MXP.SubMissionAssigned | Sub-Mission delegated | sub_mission_id, partner_id, spec |
| MXP.ProgressSynced | Progress checkpoint reported | sub_mission_id, status, checkpoint_data |
| MXP.MissionCompleted | Sub-Mission completed | sub_mission_id, result_hash |
| MXP.MissionAborted | Mission or sub-Mission aborted | mission_id, reason, aborting_instance |

## Cross-Cutting Concerns

### Security
Sub-Mission specifications are signed by the lead instance. Partner instance verifies signature before execution. Result verification ensures partner completed assigned work correctly.

### Evidence
All sub-Mission state transitions produce events in both instances. Progress checkpoints provide evidence of ongoing work.

### Lifecycle
Sub-Missions follow the Mission lifecycle. Aborted sub-Missions trigger cleanup processes in both instances.

### Capability Bounds
Partner instances may only accept sub-Missions within their capability profile. Lead may not delegate operations the partner cannot perform.

### Design DNA Compliance
| Rule | Compliance |
|------|------------|
| R1 | MXP handles cross-instance Mission collaboration only. |
| R10 | Linear flow: propose → accept → assign → execute → sync → complete. |
| R12 | Every error has a unique code (MXP_001-006). |
| R13 | Partner abort → lead reassigns or completes remaining work locally. |
| R14 | Paved path: propose → accept → assign → sync → complete → verify. |
