## Purpose

The API Specifications document defines the canonical framework for all Application Programming Interfaces within AIOS. It establishes the contract for how internal systems, external services, SDK consumers, and domain Workers communicate programmatically. Every API â€” whether ACF topic-based, RPC-style, RESTful, or streaming â€” must conform to the specifications defined herein.

This document does not specify individual APIs (those are defined in their respective component documents). Instead, it defines the *framework* â€” the rules, patterns, schemas, versioning, authentication, error handling, and documentation standards that every AIOS API must follow.

## API Categories

AIOS defines four API categories, each with distinct characteristics:

| Category | Transport | Use Case | Examples |
|----------|-----------|----------|----------|
| **ACF Topics** | ACF (pub/sub) | Event-driven communication, streaming, broadcast | Academy knowledge events, ROS budget updates |
| **RPC Services** | ACF (request/reply) | Synchronous operations, queries, commands | DTS confidence scoring, IDS identity lookup |
| **REST Gateways** | HTTPS (external) | External integrations, user-facing services | Chat API, Admin API, Webhook receivers |
| **Streaming** | ACF (streams) | High-throughput, ordered, partitioned data | Market data feeds, system metrics, audit logs |

## API Schema Standards

Every AIOS API payload must conform to these schema standards:

### Envelope

All API messages use a canonical envelope:

```
{
  "api_version": "1.0",
  "message_id": "uuid-v7",
  "correlation_id": "uuid-v7",
  "timestamp": "ISO8601",
  "source_entity_id": "string",
  "target_entity_id": "string",
  "auth_token": "string (JWT)",
  "payload": { ... }
}
```

| Field | Description | Required | Validation |
|-------|-------------|----------|------------|
| `api_version` | Semantic version of the API | Yes | Must match major version of contract |
| `message_id` | Globally unique message identifier | Yes | UUIDv7, generated by sender |
| `correlation_id` | Links related messages in a flow | Yes | Same as initiating message ID |
| `timestamp` | When the message was created | Yes | ISO 8601 UTC, microsecond precision |
| `source_entity_id` | Identity of the sending entity | Yes | Valid IDS identity |
| `target_entity_id` | Identity of the target entity | Conditional | Required for addressed messages |
| `auth_token` | Authentication token from ATS | Yes | JWT with entity claims |
| `payload` | The message body | Yes | Schema-validated per API contract |

### Error Response Schema

Every API must return errors in a standardized format:

```
{
  "error": {
    "code": "API_XXX",
    "message": "Human-readable description",
    "details": { ... },
    "correlation_id": "uuid-v7"
  }
}
```

| Error Code Range | Category | Example |
|-----------------|----------|---------|
| API-001 â€“ API-099 | General | API-001 (InvalidEnvelope), API-002 (SchemaValidationFailed) |
| API-100 â€“ API-199 | Authentication | API-101 (InvalidToken), API-102 (TokenExpired) |
| API-200 â€“ API-299 | Authorization | API-201 (UnauthorizedEntity), API-202 (InsufficientPermissions) |
| API-300 â€“ API-399 | Rate Limiting | API-301 (RateLimitExceeded), API-302 (BudgetExhausted) |
| API-400 â€“ API-499 | Resource Not Found | API-401 (EntityNotFound), API-402 (SchemaNotFound) |
| API-500 â€“ API-599 | Business Logic | API-501 (InvalidState), API-502 (Conflict) |
| API-900 â€“ API-999 | Internal | API-901 (InternalError), API-902 (DependencyFailure) |

## API Versioning

All AIOS APIs follow semantic versioning per Foundations/009-Versioning.md:

| Component | Scheme | Example | Change Triggers |
|-----------|--------|---------|-----------------|
| Major | Integer | v1, v2 | Breaking schema changes, removed fields |
| Minor | Integer | v1.1, v1.2 | Added optional fields, new endpoints |
| Patch | Integer | v1.0.1, v1.0.2 | Bug fixes, documentation updates |

API versions are specified in two places:
1. The `api_version` field in every message envelope
2. The API contract document published in the Knowledge Registry

Backward-incompatible changes require an RFC and a migration period with both old and new versions supported simultaneously.

## API Contract Requirements

Every API must be documented with a contract that includes:

| Requirement | Description | Validation |
|-------------|-------------|------------|
| Purpose | One-paragraph description of what the API does | Review |
| Category | ACF Topic, RPC, REST, or Streaming | Defined at design |
| Endpoint | Topic name, RPC method, or URL path | Registry |
| Schema | Request and response payload schemas (JSON Schema) | Automated schema validation |
| Authentication | Required token type and claims | Security review |
| Authorization | Required permissions for each operation | Security review |
| Rate Limits | Maximum calls per time window, per entity | ROS budget config |
| Error Codes | All possible error codes for this API | Documentation |
| Latency SLO | P50 and P99 latency targets | Monitoring |
| Examples | At least one request/response example | Review |
| Version History | Changelog of API versions | Registry |

## API Lifecycle

Every API follows a defined lifecycle from design through retirement:

| Phase | Description | Gate |
|-------|-------------|------|
| **Designed** | API contract drafted, schema defined | Architecture review |
| **Contract Drafted** | Full specification including examples, errors, SLOs | Team review |
| **Reviewed** | Contract reviewed by API governance board | Approval |
| **Published** | Contract registered in Knowledge Registry | Automated validation |
| **Active** | API available for consumption | Monitoring |
| **Deprecated** | API marked for retirement; consumers notified | Migration period |
| **Retired** | API removed from service | Final archive |

An API spends a minimum of 6 months in Deprecated state before retirement. Exceptions require Security Council approval.

## API Rate Limiting

Rate limits are enforced per entity, per API:

| Limit Type | Scope | Default | Enforced By |
|------------|-------|---------|-------------|
| Requests per second | Per entity, per API | 100 | ACF Gateway / API Gateway |
| Requests per minute | Per entity, per API | 5,000 | ROS budget |
| Concurrent requests | Per entity | 10 | ACF Gateway |
| Daily quota | Per entity, per API | Varies by budget | ROS budget |
| Burst allowance | Per entity | 2x RPS for 5 seconds | ACF Gateway |

Rate limit violations produce an `API.RateLimitExceeded` Event and return error code API-301.

## API Governance

API governance is enforced through the ADG (Architectural Decision Gateway):

| Principle | Enforcement | Escalation |
|-----------|-------------|------------|
| No shadow APIs | All APIs must be registered in the Knowledge Registry | Security Council |
| No breaking changes without RFC | Backward-incompatible changes require RFC approval | ADG |
| Schema-first design | All API contracts must include JSON Schema | Architecture review |
| Authentication required | No unauthenticated APIs (except public health endpoints) | Security Council |
| Rate limits mandatory | Every API must define rate limits | ROS |

## Cross-Cutting Concerns

### Security

Every API request is authenticated and authorized. ACF topic messages use ACF's built-in authentication. REST gateways require JWT tokens issued by ATS. Streaming APIs authenticate at connection establishment. API keys for external services are managed by KMS and scoped to specific APIs and entities. (Physics/008-Security.md)

### Evidence

Every API request and response produces an Event. The complete request lifecycle â€” received, authenticated, authorized, processed, responded â€” is recorded in the Event Store. Schema validation failures are logged as Events for audit. Rate limit violations are recorded. (PHI-008)

### Lifecycle

APIs follow a lifecycle: Designed â†’ Contract Drafted â†’ Reviewed â†’ Published â†’ Active â†’ Deprecated â†’ Retired. API versions follow the versioning policy (Foundations/009-Versioning.md). Deprecated APIs have a minimum support window before retirement. (Physics/006-Lifecycles.md)

### Capability Bounds

APIs expose only the capabilities authorized for the requesting entity. No API may allow an entity to exceed its capability bounds. APIs enforce rate limits defined by ROS budget allocations. Streaming API throughput is bounded by partition count and consumer group limits. (Physics/007-Capabilities.md)

### Communication

All internal APIs communicate through ACF (Law 3 â€” Communication). External REST APIs go through API gateways that enforce authentication and route to internal ACF services. Streaming APIs use ACF stream partitions for ordered, durable delivery. (Law 3 â€” Communication)

### Design DNA

| Rule | Compliance |
|------|-----------|
| R1 (Modulsingularity) | Each API contract covers exactly one capability â€” no monolithic APIs |
| R3 (DRY) | Schema components are defined once and referenced across contracts |
| R5 (Liskov) | All API implementations of the same contract are interchangeable |
| R7 (Testability) | Every API contract has an automated conformance test suite |
| R9 (Deterministic) | Same request to same API produces identical response (idempotent where specified) |
| R10 (Simpler Over Complex) | APIs expose coarse-grained operations â€” no chatty interfaces |
| R13 (Design for Failure) | All APIs return structured errors â€” no silent failures |
| R14 (Paved Path) | Single paved path: define contract â†’ validate schema â†’ implement â†’ test â†’ publish |
