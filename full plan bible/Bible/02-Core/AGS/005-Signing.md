## Purpose

Genome Signing provides cryptographic integrity for every Genome in AIOS. Every Genome, once composed and validated, is cryptographically signed by AGS. The Security Council verifies this signature before instantiating any Session. Signing ensures that a Genome has not been tampered with between creation and instantiation.

## Signing Process

Every Genome is signed at creation time using the following process:

```
Genome JSON    Salt (per-genome)    AGS Private Key
      │               │                    │
      ▼               ▼                    │
┌──────────────────────────────────────┐   │
│  hash = SHA256(genome_json + salt)   │   │
└──────────────────┬───────────────────┘   │
                   │                       │
                   ▼                       ▼
┌──────────────────────────────────────────────┐
│  signature = ECDSA(AGS_private_key, hash)    │
└──────────────────────┬───────────────────────┘
                       │
                       ▼
        Signed Genome { genome_id, json, salt, signature }
```

## Signature Components

| Component | Description | Format |
|-----------|-------------|--------|
| genome_json | Canonical JSON representation of the Genome | UTF-8 JSON |
| salt | Per-genome random nonce (64 bytes) | Hex-encoded |
| hash | SHA-256 hash of (genome_json + salt) | 64 hex characters |
| signature | ECDSA signature (P-256 curve) over the hash | DER-encoded |
| public_key_id | Identifier of the AGS public key used to sign | UUID |
| signed_at | Timestamp of signing operation | ISO 8601 |

## Signing Key Management

AGS signing keys are managed by the Hardware Security Module (HSM) system:

| Key | Type | Managed By | Rotation |
|-----|------|------------|----------|
| AGS Private Key | ECDSA P-256 | HSM (06-Services/Cryptography/HSM.md) | Every 90 days or on compromise |
| AGS Public Key | ECDSA P-256 | Published to IDS registry | Updated on rotation |

The private key NEVER leaves the HSM. All signing operations are performed inside the HSM boundary.

## Signature Verification (Security Council)

The Security Council verifies the Genome signature before instantiating a Session:

```
Signed Genome
    │
    ▼
1. Load AGS Public Key (from IDS or local cache)
2. Verify signature = ECDSA_verify(public_key, hash, signature)
3. Verify hash = SHA256(genome_json + salt) matches stored hash
4. Verify public_key_id matches current AGS key (check for compromise)
5. Verify signed_at is within acceptable time window
    │
    ▼
VerificationResult { valid: bool, reason: string }
```

## Signing Operations

### signGenome(genome_id)

```
Input:  genome_id (must be in Validated state)
Process:
  1. Load Genome JSON from Registry
  2. Generate per-genome salt (via HSM RNG)
  3. Compute hash = SHA256(genome_json + salt)
  4. Submit hash to HSM for ECDSA signing
  5. Store signature, salt, and metadata with Genome record
Output: SignedGenome { genome_id, signature, salt, public_key_id }
Event: AGS.GenomeSigned
```

### verifySignature(genome_id, signature)

```
Input:  genome_id, signature
Process:
  1. Load Genome JSON and salt from Registry
  2. Compute hash = SHA256(genome_json + salt)
  3. Load AGS Public Key by public_key_id
  4. Run ECDSA_verify(public_key, hash, signature)
Output: VerificationResult { valid, reason }
Event: (called externally — Security Council produces Event)
```

### rotateSigningKey(old_key_id, new_key_id)

```
Input:  old_key_id, new_key_id
Process:
  1. Verify new key is generated by HSM
  2. Add new key to active key set
  3. Set old key to "retired" status
  4. Re-sign all active Genomes with new key (if policy requires)
Output: KeyRotationResult { old_key_id, new_key_id, rotated_at }
Event: AGS.SigningKeyRotated
```

## Salt Generation

The salt is generated per-genome at signing time:

| Property | Value |
|----------|-------|
| Algorithm | HSM random number generator (FIPS 140-2 compliant) |
| Length | 64 bytes |
| Uniqueness | No two Genomes may have the same salt |
| Storage | Stored alongside Genome in Registry (not in the signature) |
| Verification | Salt is loaded from Registry during verification |

The salt ensures that two identical Genomes produce different signatures. This prevents signature replay attacks and provides non-repudiation.

## Signature Schema

The signed Genome record contains:

```json
{
  "genome_id": "wkr-001-v2",
  "genome": { ... },
  "signing_metadata": {
    "salt": "a1b2c3d4...64bytes...",
    "hash": "e5f6a7b8...64hex...",
    "signature": "3045022100...DER...",
    "public_key_id": "ags-key-2026-q2",
    "algorithm": "ECDSA-P256",
    "signed_at": "2026-07-09T12:00:00Z",
    "signed_by": "AGS-HSM-01"
  }
}
```

## Key Rotation Process

AGS signing keys are rotated every 90 days or immediately on compromise:

```
1. Security Council notifies AGS of key rotation (scheduled or compromise)
2. HSM generates new ECDSA P-256 key pair
3. New public key registered with IDS
4. Old public key set to "retired" status (not used for new signatures)
5. If compromise: All Genomes MUST be re-signed with new key
6. If scheduled rotation: New Genomes use new key, old Genomes retain old key
7. Old public key retained for verification of existing signatures
8. Verification uses the public_key_id stored in the Genome's signature metadata
```

## Signature Verification for Session Instantiation

The flow from signed Genome to Session instantiation:

```
1. AGS composes, validates, signs Genome
2. Session creation request arrives at Security Council
3. Security Council loads SignedGenome
4. Security Council calls verifySignature(genome_id, signature)
5. If verification passes → Security Council authorizes Session creation
6. If verification fails → Session creation is denied, Security Event produced
```

## Signing Events

| Event Type | Produced When | Fields |
|-----------|--------------|--------|
| `AGS.GenomeSigned` | A Genome is signed | genome_id, signature_hash, public_key_id |
| `AGS.SigningKeyRotated` | The signing key is rotated | old_key_id, new_key_id, rotated_at |
| `AGS.SignatureVerificationPassed` | Signature verification succeeds | genome_id, verifier_id |
| `AGS.SignatureVerificationFailed` | Signature verification fails | genome_id, verifier_id, reason |
| `AGS.KeyCompromiseReported` | A signing key is reported compromised | key_id, report_source |

## Error Codes (R12)

| Code | Description |
|------|-------------|
| AGS_SIG_001 | HSM unavailable for signing operation |
| AGS_SIG_002 | Genome not in Validated state — cannot sign |
| AGS_SIG_003 | Signature verification failed — tampered Genome |
| AGS_SIG_004 | Unknown public_key_id — key not found |
| AGS_SIG_005 | Signing key expired or compromised |
| AGS_SIG_006 | Salt generation failed — HSM RNG error |

## Cross-Cutting Concerns

### Security

Signing is the cryptographic foundation of Genome integrity. The private key never leaves the HSM. Signature verification is mandatory before Session instantiation. Key compromise triggers immediate rotation and re-signing. (Physics/008-Security.md)

### Evidence

Every signing and verification operation produces an Event. The complete signing history is recorded for forensic audit. (PHI-008)

### Lifecycle

A Genome transitions from Validated to Signed state after signing. Only Signed Genomes may be used for Session instantiation. (Physics/006-Lifecycles.md)

### Capability Bounds

Signing does not modify capabilities — it cryptographically seals them. An attacker who modifies a capability signature is detected at verification time. (Physics/007-Capabilities.md)

### Communication

HSM communication occurs through a secure, isolated channel. Signature verification requests arrive via ACF. Verification results are returned via ACF. (Law 3 — Communication)

### Design DNA

| Rule | Compliance |
|------|-----------|
| R1 (Modulsingularity) | Signing focused solely on cryptographic integrity |
| R10 (Simpler Over Complex) | Single algorithm (ECDSA P-256), single hash (SHA-256) — no algorithm negotiation |
| R12 (Embrace Errors) | All errors have unique codes (AGS_SIG_001–006) |
| R13 (Design for Failure) | HSM failure blocks signing but not verification (verification can use cached public keys) |
| R14 (Paved Path) | Single paved path: HSM → sign → verify → instantiate |
