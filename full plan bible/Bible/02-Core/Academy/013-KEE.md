## Purpose

The Knowledge Execution Engine (KEE) is the runtime that executes knowledge-driven actions. It consumes Academy knowledge (operational runbooks, playbooks, domain procedures) and turns them into executable plans that the Runtime Engine carries out. KEE bridges knowing and doing: it is how the Academy's knowledge becomes action.

KEE acts within entity capability bounds (R1 modulsingularity, Physics/007-Capabilities.md). It does not grant itself authority — it executes on behalf of authorized entities.

## Execution Model

KEE follows a five-stage execution model:

```
Knowledge → Plan → Execute → Verify → Learn
```

| Stage | Description | Output |
|-------|-------------|--------|
| **Knowledge** | Retrieve relevant knowledge artifact(s) from Academy | Knowledge artifact |
| **Plan** | Transform knowledge into an executable plan | Structured plan |
| **Execute** | Submit plan to Runtime Engine for execution | Execution result |
| **Verify** | Verify that execution outcome matches knowledge intent | Verification result |
| **Learn** | Produce feedback Events for Academy improvement | Feedback Events |

### Stage 1: Knowledge

KEE receives a request to execute a knowledge-driven action.

| Input | Description |
|-------|-------------|
| `entity_id` | Entity requesting execution (must be authorized) |
| `knowledge_id` | Knowledge artifact to execute (or search query) |
| `context` | Execution context (parameters, scope, constraints) |
| `mission_id` | Optional Mission context for the execution |

The KEE retrieves the knowledge artifact from the Academy (via Knowledge API 016).

### Stage 2: Plan

KEE transforms knowledge content into an executable plan:

| Plan Component | Description | Example |
|----------------|-------------|---------|
| Steps | Ordered list of actions | 1. Authenticate 2. Authorize 3. Execute |
| Parameters | Required inputs for each step | username, resource_id, action |
| Conditions | Preconditions and postconditions | Must have `auth.execute` capability |
| Error handlers | Fallback actions on failure | Retry 3 times, then escalate |
| Rollback | Undo actions on critical failure | Revoke token, log failure |

Plan structure:

```
{
    "plan_id": "PLAN-001",
    "knowledge_id": "A-042",
    "entity_id": "AG-042",
    "steps": [
        {
            "step_id": 1,
            "action": "authenticate",
            "parameters": { "method": "jwt", "token": "..." },
            "preconditions": [],
            "error_handler": { "max_retries": 3, "on_failure": "deny" }
        },
        {
            "step_id": 2,
            "action": "authorize",
            "parameters": { "resource": "database", "operation": "query" },
            "preconditions": ["step_1.completed"],
            "error_handler": { "on_failure": "rollback_step_1" }
        }
    ],
    "rollback": [
        { "on": "any_failure", "actions": ["revoke_token", "log_event"] }
    ]
}
```

### Stage 3: Execute

KEE submits the plan to the Runtime Engine for execution:

| Step | Description |
|------|-------------|
| 1 | Plan is validated (structural completeness) |
| 2 | Each step is submitted to Runtime Engine sequentially |
| 3 | Step result is checked before proceeding to next |
| 4 | On failure: error handler is invoked (retry, rollback, or escalate) |
| 5 | On completion: execution result is collected |

KEE does not execute actions itself (R1 modulsingularity). It delegates to the Runtime Engine (Physics/010-Execution.md).

### Stage 4: Verify

KEE verifies that the execution outcome matches the knowledge intent:

| Check | Description |
|-------|-------------|
| Outcome matches expected postconditions | Did the plan achieve what the knowledge said it would? |
| Execution produced expected Events | Are all expected Events in the Event Store? |
| No unintended side effects | Did the execution trigger any unexpected Events? |
| Resource usage within bounds | Did execution stay within entity capability bounds? |

### Stage 5: Learn

KEE produces feedback Events for the Academy:

| Feedback Type | Description | Consumed By |
|---------------|-------------|-------------|
| Execution success | Knowledge was successfully executed | Analytics (012), Confidence scoring |
| Execution failure | Knowledge could not be executed | Validator (005), Knowledge improvement |
| Knowledge gap | Knowledge was incomplete or insufficient | Gap detection (012), Knowledge creation |
| Optimization suggestion | Knowledge could be more efficient | Knowledge improvement pipeline |

## Execution Authorization

KEE verifies authorization at each stage:

| Authorization Check | Description |
|---------------------|-------------|
| Entity identity | Entity is authenticated (ATS) |
| Entity capability | Entity has capability to execute this knowledge type |
| Knowledge access | Entity is authorized to access the knowledge artifact |
| Plan authorization | Each plan step is within entity capability bounds (R1) |
| Resource bounds | Execution stays within entity resource budget (ROS) |

KEE does not bypass the verification pipeline (PHI-005). All execution goes through Identity → Auth → Authorization → Policy → Capability → Risk → Execution Authorization.

## Integration with Runtime Engine

| Concern | KEE | Runtime Engine |
|---------|-----|----------------|
| Role | Transforms knowledge to plans | Executes plan steps |
| Responsibility | Knowledge selection, planning, verification | Action execution, resource management |
| State | Stateless (plans are ephemeral) | Stateful (manages execution state) |
| Output | Execution plan, verification result | Execution result Events |

## KEE Events

| Event Type | Produced When | Fields |
|-----------|--------------|--------|
| `KEE.ExecutionRequested` | Entity requests knowledge execution | request_id, entity_id, knowledge_id, context |
| `KEE.PlanCreated` | Knowledge is transformed to plan | request_id, plan_id, step_count |
| `KEE.PlanValidated` | Plan passes structural validation | request_id, plan_id, valid |
| `KEE.StepExecuted` | A plan step is executed | request_id, step_id, result, duration_ms |
| `KEE.StepFailed` | A plan step fails | request_id, step_id, error_code, retry_count |
| `KEE.ExecutionCompleted` | Plan execution completes | request_id, overall_result, verification_result |
| `KEE.ExecutionFailed` | Plan execution fails irrecoverably | request_id, failed_step, rollback_executed |
| `KEE.LearningFeedback` | Feedback produced for Academy | request_id, feedback_type, details |

## Cross-Cutting Concerns

### Security

KEE does not bypass the verification pipeline. Every execution request goes through the full 7-stage security check (PHI-005). KEE operates with the entity's identity and capabilities — it does not escalate privileges. All execution plans are within entity capability bounds (R1 modulsingularity).

### Evidence

Every KEE operation produces Events. The full execution history (request → plan → execute → verify → learn) is recorded. Execution Events are source Events for Academy learning (feedback loop).

### Lifecycle

KEE follows the LMS lifecycle for execution runs. Each execution request creates a temporary entity with lifecycle: Created → Planned → Running → Completed/Failed → Archived.

### Capability Bounds

| Operation | Required Capability |
|-----------|---------------------|
| Request execution | `knowledge.execute.{type}` |
| Create plan | `knowledge.execute.plan` (implicit) |
| Execute plan | `knowledge.execute` |
| Verify outcome | `knowledge.execute.verify` (implicit) |
| Access knowledge | `knowledge.query` |

All capabilities are scoped by entity, organization, and knowledge type (Physics/007-Capabilities.md).

### Communication

KEE receives requests through ACF. It queries the Academy (016) for knowledge artifacts. It submits plans to the Runtime Engine through ACF. It publishes execution events for audit and Academy feedback.

### Design DNA

| Rule | Application |
|------|-------------|
| R1 | KEE does planning and verification — does not execute actions |
| R4 | Plans are built by KEE, executed by Runtime Engine (builder pattern) |
| R6 | KEE receives dependencies (Academy API, Runtime Engine, ACF) through injection |
| R9 | Plans are deterministic given same knowledge and context |
| R10 | Execution model is linear (5 stages, no branching) |
| R13 | KEE fails closed on execution failure (rollback or deny) |
| R14 | Paved path: Knowledge → Plan → Execute → Verify → Learn |
