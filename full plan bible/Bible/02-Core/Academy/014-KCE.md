## Purpose

The Knowledge Composition Engine (KCE) combines existing knowledge artifacts to produce new insights. While the Knowledge Pipeline creates knowledge from raw Events, KCE creates knowledge from existing knowledge — finding connections, analogies, and patterns across domains. KCE enables cross-domain insight generation, knowledge gap bridging, and hypothesis formation.

KCE operates within constitutional bounds: composed knowledge must pass the same validation pipeline as source-derived knowledge. Composition does not grant authority to create knowledge that would otherwise be rejected.

## Composition Methods

KCE supports five composition methods:

| Method | Description | Example |
|--------|-------------|---------|
| **Union** | Combine knowledge from multiple artifacts into a unified artifact | Merging two operational runbooks |
| **Intersection** | Extract common elements across artifacts | Finding shared steps across authentication procedures |
| **Analogy** | Map knowledge from one domain to another | Applying security patterns to data privacy |
| **Induction** | Generalise patterns from specific knowledge artifacts | Deriving a general authentication rule from specific implementations |
| **Deduction** | Derive specific knowledge from general principles | Applying constitutional principles to a specific scenario |

### 1. Union

Combines two or more knowledge artifacts into a single artifact that contains all their content.

| Input | Output |
|-------|--------|
| Artifact A (authentication methods) + Artifact B (authorization methods) | Artifact C (unified access control knowledge) |

| Rule | Description |
|------|-------------|
| Union is commutative | A ∪ B = B ∪ A |
| Union is associative | (A ∪ B) ∪ C = A ∪ (B ∪ C) |
| Non-contradiction required | Union fails if A and B contradict each other |

### 2. Intersection

Extracts knowledge that is common across multiple artifacts.

| Input | Output |
|-------|--------|
| Artifact A (deployment procedures for AWS) + Artifact B (deployment procedures for Azure) | Artifact C (general deployment principles) |

| Rule | Description |
|------|-------------|
| Minimum artifacts | At least 2 artifacts required |
| Confidence floor | Intersection confidence = min(confidence(A), confidence(B)) |
| Support threshold | Pattern must appear in ≥ 70% of source artifacts |

### 3. Analogy

Maps knowledge from a source domain to a target domain.

| Parameter | Description |
|-----------|-------------|
| `source_artifact_id` | Knowledge in source domain |
| `target_domain` | Target domain for the analogy |
| `mapping_strength` | Confidence that the analogy holds (0.0–1.0) |

Analogy process:

```
1. Extract structural patterns from source artifact
2. Identify equivalent concepts in target domain (via Knowledge Graph)
3. Map patterns to target domain concepts
4. Validate mapped patterns for consistency
5. Produce composed artifact with analogy strength score
```

### 4. Induction

Generalises patterns from multiple specific knowledge artifacts into a general principle.

| Input | Output |
|-------|--------|
| Artifact A (JWT auth) + Artifact B (OAuth flow) + Artifact C (API key validation) | Artifact D (general authentication principles) |

| Prerequisite | Description |
|--------------|-------------|
| Minimum artifacts | At least 3 |
| Pattern consistency | Patterns must be consistent across sources |
| Coverage | Generalisation must cover ≥ 80% of observed cases |
| Confidence penalty | Induction confidence = average(source confidence) × 0.9 |

### 5. Deduction

Derives specific knowledge from general principles or constitutional law.

| Input | Output |
|-------|--------|
| Constitutional principle (PHI-005: Verification-First) | Specific verification procedure for a new entity type |

| Rule | Description |
|------|-------------|
| Constitutional grounding | Deduction must trace to a constitutional source |
| Logical validity | Deduction must follow valid logical reasoning |
| Transparency | Reasoning steps must be recorded in the provenance chain |

## Composition Validation

Composed knowledge must pass the same validation pipeline as source-derived knowledge (Validator 005, Verifier 006, Review 007 if required):

| Validation Stage | Application to Composed Knowledge |
|-----------------|----------------------------------|
| Schema validation | Composed artifact must have valid structure |
| Constitutional consistency | Composition must not violate any Law |
| Evidence provenance | Composed knowledge must trace provenance to source artifacts (and their source Events) |
| Non-contradiction | Composed knowledge must not contradict accepted knowledge |
| Privacy compliance | Composition must not expose private evidence |
| Quality checks | Reproducible, complete, current, non-duplicative (R9) |

### Additional Composition-Specific Validations

| Rule | Description |
|------|-------------|
| Source validity | All source artifacts must be in Accepted or Published state |
| Source freshness | Source artifacts must not be deprecated or archived |
| Composition transparency | Composition method and parameters must be recorded |
| Strength disclosure | Composition confidence must be disclosed in artifact metadata |

## Composition Pipeline

```
Request (method + source artifacts)
    │
    ▼
┌──────────────────────────────────────────────┐
│  KCE Pipeline                                  │
│                                                │
│  1. Load source artifacts from KMS            │
│  2. Validate source artifacts (fresh, valid)  │
│  3. Apply composition method                  │
│  4. Produce composed artifact draft           │
│  5. Validate composed artifact (same pipeline)│
│  6. Register provenance links to sources      │
│  7. Submit for acceptance (if meets threshold)│
└──────────────────────────────────────────────┘
    │
    ▼
Composed Artifact (Proposed state)
```

## Use Cases

| Use Case | Composition Method | Example |
|----------|-------------------|---------|
| Cross-domain insight | Analogy | Apply network security patterns to supply chain security |
| Knowledge gap bridging | Induction | Generalize from 3 domain-specific deployment guides |
| Hypothesis formation | Deduction | Derive expected behavior of new entity type from constitutional principles |
| Knowledge consolidation | Union | Merge 2 operational runbooks from different teams |
| Best practice extraction | Intersection | Find common steps across 5 security audit procedures |
| Constitutional interpretation | Deduction | Apply constitutional amendment to specific operational context |

## KCE Events

| Event Type | Produced When | Fields |
|-----------|--------------|--------|
| `KCE.CompositionRequested` | Composition request submitted | request_id, method, source_artifact_ids |
| `KCE.SourcesLoaded` | Source artifacts are loaded and validated | request_id, source_count, valid_count |
| `KCE.CompositionApplied` | Composition method produces draft | request_id, method, draft_artifact_id |
| `KCE.CompositionValidated` | Composed artifact passes validation | request_id, validation_result, score |
| `KCE.CompositionFailed` | Composition fails (contradiction, invalid sources) | request_id, reason, failed_step |
| `KCE.CompositionSubmitted` | Composed artifact is submitted for acceptance | request_id, artifact_id, target_state |

## Cross-Cutting Concerns

### Security

KCE operates within Security Council oversight. Composed knowledge that touches constitutional interpretation may require Security Council authorization. Composition cannot be used to circumvent validation — composed knowledge must pass the same gates as source-derived knowledge (PHI-001).

### Evidence

Every composition step produces an Event. The provenance chain for composed knowledge includes links to all source artifacts. The full reasoning chain (composition method, parameters, mapping) is recorded. Anyone can verify how composed knowledge was produced.

### Lifecycle

Composed knowledge enters the AKM lifecycle at Generated state (same as source-derived). It must pass validation, verification, and (if required) review before acceptance. Composition does not shortcut the lifecycle.

### Capability Bounds

| Operation | Required Capability |
|-----------|---------------------|
| Request composition | `knowledge.compose` |
| Union composition | `knowledge.compose.union` |
| Intersection composition | `knowledge.compose.intersection` |
| Analogy composition | `knowledge.compose.analogy` |
| Induction composition | `knowledge.compose.induction` |
| Deduction composition | `knowledge.compose.deduction` |

Capabilities are scoped by knowledge type and domain. Sou and Security Council have all composition capabilities.

### Communication

KCE receives composition requests through ACF. It reads source artifacts from KMS (via ACF). It submits composed artifacts to the validation pipeline (Validator 005 via ACF). All composition events are published over ACF.

### Design DNA

| Rule | Application |
|------|-------------|
| R1 | KCE does composition — does not execute or store |
| R3 | Composed knowledge is derived from source artifacts (no duplication) |
| R9 | Composition is deterministic given same sources and method |
| R10 | Five composition methods — no more unless justified |
| R13 | KCE fails closed if source artifacts unavailable or invalid |
| R14 | Paved path: request → load → compose → validate → submit |
| R15 | New composition methods added without modifying existing methods |
