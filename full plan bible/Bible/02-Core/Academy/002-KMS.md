## Purpose

The Knowledge Management System (KMS) is the authoritative storage and retrieval system for knowledge artifacts within the Academy. It manages the full lifecycle of each artifact, maintains version chains, and provides a query interface over the ACF. Every piece of knowledge accepted by the Academy lives in the KMS.

## Knowledge Artifact Structure

Every knowledge artifact is a structured record with the following schema:

| Field | Type | Description | Required |
|-------|------|-------------|----------|
| `id` | UUIDv7 | Unique artifact identifier | Yes |
| `type` | Enum | Knowledge type (operational, domain, constitutional, strategic, experimental) | Yes |
| `content_hash` | SHA-256 | Hash of the artifact content | Yes |
| `source_event_ids` | UUID[] | Event IDs from which this knowledge was derived | Yes (≥ 1) |
| `provenance` | ProvenanceChain | Full provenance record linking to source events | Yes |
| `status` | Enum | Current lifecycle state (AKM states) | Yes |
| `version` | SemVer | Artifact version (Major.Minor.Patch) | Yes |
| `timestamp` | DateTime | When this version was created | Yes |
| `title` | String | Human-readable title | Yes |
| `description` | String | Human-readable description | Yes |
| `content` | JSON | Structured knowledge payload | Yes |
| `metadata` | JSON | Extensible metadata (domain tags, confidence, etc.) | No |
| `signature` | Signature | Cryptographic signature of the artifact | Yes |
| `supersedes` | UUID[] | IDs of artifacts this one supersedes | No |
| `superseded_by` | UUID | ID of artifact that supersedes this one | No |

### Artifact Types

| Type | Content Schema | Typical Size | Retention |
|------|---------------|--------------|-----------|
| Operational | Runbook, playbook, procedure | 1–100 KB | Indefinite (while valid) |
| Domain | Domain model, heuristic, rule | 1–500 KB | Indefinite (while valid) |
| Constitutional | Constitutional interpretation | 1–50 KB | Indefinite (permanent) |
| Strategic | Decision support, planning insight | 1–200 KB | Indefinite (while relevant) |
| Experimental | Tentative finding, hypothesis | 1–50 KB | 90 days unless accepted |

## Knowledge Storage

The KMS uses an append-only event log for the knowledge lifecycle (Event Sourcing pattern, Foundations/005-Architectural-Patterns.md).

```
Knowledge Event Log:

[Created] → [Updated] → [Updated] → [Deprecated] → [Archived]
    │           │            │             │              │
    ▼           ▼            ▼             ▼              ▼
  v1.0.0     v1.1.0      v2.0.0        v2.0.0         v2.0.0
           (minor)     (major)      (deprecated)    (archived)
```

Each event in the log records:

| Event Field | Description |
|-------------|-------------|
| `event_type` | Type of knowledge lifecycle event |
| `artifact_id` | The affected artifact |
| `version` | Version after this event |
| `timestamp` | When the event occurred |
| `authorized_by` | Entity ID that authorized the change |
| `previous_hash` | Hash of previous event in chain |
| `event_hash` | Hash of this event |

### Snapshotting

To prevent unbounded event log growth, the KMS creates snapshots every 100 versions:

- Snapshot contains the full artifact state at that version
- Events before the snapshot may be pruned for storage efficiency
- Snapshot includes the hash chain linking it to the previous snapshot
- Snapshots are stored in the same append-only store

## KMS Operations

### createKnowledge

Creates a new knowledge artifact. The artifact enters the Generated state.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `type` | Yes | Knowledge type |
| `content` | Yes | Structured knowledge payload |
| `source_event_ids` | Yes | Source Events |
| `title` | Yes | Human-readable title |
| `description` | Yes | Human-readable description |
| `metadata` | No | Extensible metadata |

**Response**: `{ artifact_id, version: "1.0.0", status: "Generated" }`

### updateKnowledge

Creates a new version of an existing artifact. Increments version according to change type.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `artifact_id` | Yes | Existing artifact ID |
| `change_type` | Yes | Major, Minor, or Patch |
| `content` | Yes | Updated content |
| `reason` | Yes | Why the update was made |

**Response**: `{ artifact_id, version: "x.y.z", status }`

### deprecateKnowledge

Marks an artifact as deprecated. Deprecated artifacts are not distributed but remain accessible for audit.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `artifact_id` | Yes | Artifact to deprecate |
| `superseded_by` | No | Artifact ID that replaces this one |
| `reason` | Yes | Deprecation reason |

**Response**: `{ artifact_id, status: "Deprecated", superseded_by }`

### archiveKnowledge

Moves a deprecated artifact to the Archived state. Archived artifacts are preserved for audit but excluded from search results.

| Parameter | Required | Description |
|-----------|----------|-------------|
| `artifact_id` | Yes | Artifact to archive |
| `retention_period` | No | Retention in days (default: per-type policy) |

**Response**: `{ artifact_id, status: "Archived", archived_at }`

## Versioning

Each knowledge artifact has a version chain. See 008-Knowledge-Versioning.md for full versioning specification.

| Aspect | Detail |
|--------|--------|
| Scheme | Major.Minor.Patch (SemVer) |
| Initial | 1.0.0 |
| Major increment | Breaking change, supersedes previous |
| Minor increment | Non-breaking addition or refinement |
| Patch increment | Correction, clarification |
| Max versions | No hard limit (practical: snapshot at 100) |
| Branches | Supported for alternative knowledge (resolved on acceptance) |

## Query Interface

The KMS provides a query interface over ACF:

| Query | Description | Filterable By |
|-------|-------------|---------------|
| `getKnowledge(id)` | Retrieve artifact by ID | — |
| `listKnowledge(filters)` | List artifacts matching filters | type, status, version, organization, timerange |
| `queryContent(query)` | Full-text search within artifact content | content fields |
| `getVersionHistory(id)` | Get version chain for artifact | — |
| `listByType(type)` | List all artifacts of a type | type |
| `listByStatus(status)` | List all artifacts in a state | status |
| `listBySourceEvent(event_id)` | Find artifacts derived from an Event | event_id |

## KMS Events

| Event Type | Produced When | Fields |
|-----------|--------------|--------|
| `KMS.ArtifactCreated` | New knowledge artifact created | artifact_id, type, version, source_event_ids |
| `KMS.ArtifactUpdated` | Existing artifact updated | artifact_id, old_version, new_version, change_type |
| `KMS.ArtifactDeprecated` | Artifact deprecated | artifact_id, reason, superseded_by |
| `KMS.ArtifactArchived` | Artifact archived | artifact_id, retention_period |
| `KMS.SnapshotCreated` | Snapshot written for version chain | artifact_id, version, snapshot_hash |
| `KMS.QueryExecuted` | Query was executed against KMS | query_type, filter_count, result_count |

## Cross-Cutting Concerns

### Security

KMS stores signed knowledge artifacts (signature field). Tampering with stored artifacts is detectable via hash chain. Access to KMS operations requires ACF-level authentication and authorization (Physics/008-Security.md).

### Evidence

Every KMS operation produces an Event in the knowledge lifecycle event log. The event log is the authoritative record of all knowledge state changes. No operation modifies state without producing evidence.

### Lifecycle

Knowledge artifacts follow the AKM lifecycle (Governance/006-AKM.md). The KMS enforces lifecycle state transitions: an artifact in Accepted state cannot be directly updated; it must go through a new proposal-validation cycle.

### Capability Bounds

KMS operations are capability-bounded per-entity (Physics/007-Capabilities.md). An entity requires `knowledge.create` capability to create artifacts, `knowledge.update` to update, and `knowledge.query` to query. Capabilities are scoped by knowledge type.

### Communication

All KMS operations are accessed through ACF. Internal consumers (Validator, Registry, Distribution) use ACF topics. External consumers use the Knowledge API (016) which routes through ACF.

### Design DNA

| Rule | Application |
|------|-------------|
| R1 | KMS does storage and query — nothing else |
| R3 | Knowledge is stored once (canonical KMS record) |
| R4 | Artifacts are built by Pipeline, stored by KMS |
| R6 | KMS receives dependencies (Event Store, ACF) through injection |
| R9 | Queries produce deterministic results |
| R13 | KMS fails closed on storage failure |
| R15 | New query types implemented without modifying KMS core |
