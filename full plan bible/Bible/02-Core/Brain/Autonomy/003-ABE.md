## Purpose

The Autonomy Boundary Engine (ABE) handles every situation where an entity encounters the boundary of its assigned autonomy level. ABE owns the escalation chain that routes out-of-bounds actions upward through the authority hierarchy, the Override Authorizer that grants temporary exceptions to autonomy bounds, and the Threshold Manager that controls when escalation triggers automatically. ABE ensures that boundary crossings are never silent â€” they are always escalated, authorized, or denied with full audit trail.

## Core Operations

### Escalation Chain

The Escalation Chain routes out-of-bounds actions upward through the authority hierarchy until an entity with sufficient autonomy approves or denies:

```
Chain Structure (fixed positions):

Position 0: Worker        â† The entity attempting the action
Position 1: Supervisor    â† First-line oversight
Position 2: Manager       â† Department-level authority
Position 3: Director      â† Strategic authority
Position 4: Sou           â† Constitutional intelligence
Position 5: Human         â† Absolute authority (Article I)
```

#### Chain Walking Algorithm

```
Input:  EscalationRequest (entity, action, reason)
Output: EscalationResponse (approved/denied, resolved_by, evidence)

1. RESOLVE CHAIN
   â†’ Query entity hierarchy to populate ChainLink[0..5]
   â†’ For each position, determine if entity exists and is available
   â†’ Handle missing links (see Chain Break Handling)
   â†’ Return resolved_path: string[] of available entity_ids

2. DETERMINE STARTING POSITION
   â†’ Start at entity's current chain_position
   â†’ If emergency mode â†’ skip to position 4 (Sou) or 5 (Human)

3. WALK UP
   â†’ For each position from start â†’ Human:
       Submit request to entity at this position
       Wait for response (with timeout, see thresholds)
       If approved â†’ return EscalationResponse(approved)
       If denied â†’ continue to next position
       If timeout â†’ mark as expired, continue to next
       If entity unavailable â†’ continue (skip)

4. FINAL RESOLUTION
   â†’ If Human responds â†’ final (Human decision is absolute)
   â†’ If all positions exhausted without approval â†’ auto-deny
   â†’ If Sou denies â†’ final (Sou denial is final below Human)

5. RECORD
   â†’ Emit AUT.EscalationResolved
   â†’ Update compliance score
   â†’ Notify all chain participants
   â†’ Store full resolution path in evidence
```

#### Emergency Escalation

Emergency escalation allows an entity to bypass intermediate chain levels when the situation requires immediate higher authority:

```
Emergency Protocol:

Trigger: Action marked with x-aios-emergency: true in ACF header
         AND one of:
         - Imminent constitutional violation
         - Active system threat
         - Human safety concern
         - Critical resource exhaustion

Behavior:
  â†’ Skip positions 0-3 of the chain
  â†’ Submit directly to Sou (position 4)
  â†’ If Sou unavailable within emergency_timeout â†’ route to Human (position 5)
  â†’ Post-event: mandatory audit with evidence_ref

Constraints:
  â†’ False emergency declarations result in -0.50 compliance penalty
  â†’ Emergency without evidence â†’ accepted, flagged for audit (AUT_EMERGENCY_NO_EVIDENCE)
  â†’ Max 1 emergency escalation per entity per 24 hours
```

#### Chain Break Handling

When a position in the chain is unoccupied or unavailable:

```
Chain Break Rules:

1. MISSING ENTITY (position has no assigned entity)
   â†’ Skip the position
   â†’ Log the gap
   â†’ Next available entity receives escalation
   â†’ Chain status set to "broken"

2. ENTITY UNAVAILABLE (offline, timeout, capacity full)
   â†’ Wait for configurable timeout (default: 30s)
   â†’ On timeout, skip to next position
   â†’ Mark chain as "partial"

3. MULTIPLE CONSECUTIVE BREAKS
   â†’ Continue skipping until an available entity is found
   â†’ If no entity found in positions 0-3, route directly to Sou
   â†’ If Sou also unavailable, route to Human

4. CHAIN STATUS
   â†’ "valid" â€” all positions 0-5 are assigned and available
   â†’ "partial" â€” some positions skipped due to unavailability
   â†’ "broken" â€” one or more positions have no assignment
```

### Override Authorizer

| Type | Description | Duration | Required Authority | Evidence Level |
|------|-------------|----------|-------------------|----------------|

### Threshold Manager

The Threshold Manager controls when escalation triggers automatically:

```
Threshold Categories:

ENTITY TYPE THRESHOLDS (per entity_type, applies to all actions):
  - escalation_rate: max escalations per hour before auto-deny
  - risk_tolerance: max DTS risk score before auto-deny vs escalate
  - cost_threshold: max resource cost before escalation required

ACTION TYPE THRESHOLDS (per action_type, overrides entity defaults):
  - scope_size: max scope units before escalation
  - frequency: max occurrences per window before auto-deny
  - risk_score: action-specific risk threshold

CONTEXT THRESHOLDS (dynamic, based on system state):
  - compliance: if entity compliance < threshold, escalate more readily
  - system_load: if system under load, reduce escalation auto-approve
  - override_active: if override active, tighten thresholds
```

Default thresholds:

| Risk tolerance (auto-deny) | 0.7 | Yes |
| Risk tolerance (escalate) | 0.4â€“0.7 | Yes |
| Cost threshold (escalation) | 10,000 tokens | Yes |
| Action frequency limit | 10/hour per action type | Yes |

### Delegation Model

Autonomy includes the right to delegate authority within strict bounds:

```
Delegation Rules:

1. CAPABILITY POSSESSION
   â†’ An entity cannot delegate a capability it does not possess
   â†’ Verified by ACE bounds check before delegation
   
2. BOUNDS PRESERVATION
   â†’ Delegation preserves the delegator's autonomy bounds
   â†’ Recipient operates within min(delegator.level, recipient.max_level)
   
3. SCOPING REQUIREMENT
   â†’ Every delegation must be scoped: resource, action, duration, depth
   â†’ Unscoped delegation is denied
   
4. CHAIN DEPTH LIMIT
   â†’ Maximum delegation chain depth: 3 (A â†’ B â†’ C â†’ D)
   â†’ Depth counted from original delegator
   
5. REVOCATION
   â†’ Delegation can be revoked at any time by the delegator
   â†’ Revocation is immediate and cascading (sub-delegations also revoked)
   â†’ Delegator's supervisor can also revoke

6. AUDIT
   â†’ Every delegation and revocation is recorded as evidence
   â†’ Delegation chain is traceable from origin to leaf
```

Delegation depth limits by level:

| Delegator Level | Max Delegation Depth | Can Delegate To |
|-----------------|---------------------|-----------------|
| L0 | 0 | Nobody |
| L1 | 0 | Nobody |
| L2 | 1 | L0â€“L1 entities only |
| L3 | 2 | L0â€“L2 entities only |
| L4 (Sou) | 3 | Any entity |

## Cross-Cutting Concerns

### Security

Autonomy System operates under Law 8 (Verification-First) and Law 7 (Capability Bounds): every operation is authorized by the Security Kernel before execution, and the component never exceeds its declared capabilities. (Physics/008-Security.md)

### Evidence

Per Law 4 (Evidence), Autonomy System emits an evidence record for each significant state change - what changed, by whom, on what basis, with what outcome - delivered through ACF and persisted by EVS. (Physics/005-Events.md)

### Lifecycle

Per Law 6 (Lifecycle Compliance), Autonomy System instances follow the canonical LMS lifecycle (Draft -> Active -> Suspended -> Archived) and are terminated deterministically; orphan states are prevented. (Physics/006-Lifecycles.md)

### Capability Bounds

Per Law 7 (Capability Bounds), Autonomy System declares its capabilities at creation and operates only within them; capability expansion requires reauthorization through the Security Kernel. (Physics/007-Capabilities.md)
