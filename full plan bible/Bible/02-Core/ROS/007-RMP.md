## Purpose

Resource Management Policies (RMP) define the rules that govern how resources are allocated, preempted, and optimized. Policies are set by the Security Council and may be supplemented by Organization-level sub-policies within constitutional bounds. RMP is evaluated by the Allocator (Stage 4 of the verification pipeline sets policies; ROS applies them).

## Policy Structure

```
Policy {
    policy_id: UUID
    name: String
    description: String
    authority: Enum (security_council, organization)
    entity_type: String (glob pattern, e.g., "org.*", "entity.*")
    resource_type: ResourceType (or "*" for all)
    conditions: List<Condition>
    actions: List<Action>
    priority: Int (higher = evaluated first)
    enabled: Boolean
    valid_from: Timestamp
    valid_until: Timestamp (optional)
}
```

### Conditions

Conditions determine when a policy applies. Multiple conditions are AND-ed.

| Condition Type | Operator | Example |
|----------------|----------|---------|
| `entity_type` | MATCH | `entity_type == "org.research.*"` |
| `resource_type` | EQUALS | `resource_type == "compute"` |
| `usage_threshold` | GT, GTE, LT, LTE | `usage_pct >= 0.8` (80% usage) |
| `time_range` | BETWEEN | `time BETWEEN "22:00" AND "06:00"` (off-peak) |
| `autonomy_level` | GTE | `autonomy_level >= L2` |
| `priority_range` | BETWEEN | `priority BETWEEN 3 AND 5` |

### Actions

Actions define what happens when the policy conditions are met.

| Action Type | Parameters | Description |
|-------------|-----------|-------------|
| `set_allocation_strategy` | strategy: Enum | Override the allocation strategy |
| `set_preemption_priority` | priority: Int | Change preemption priority for matching entities |
| `enable_burst` | multiplier: Float | Allow burst budget for matching entities |
| `set_cost_rate` | rate: Float, resource_type | Override cost rate for matching usage |
| `energy_optimize` | enabled: Boolean | Enable/disable energy-aware allocation |
| `quota_override` | hard_limit: Quantity | Temporarily override quota for matching scope |
| `deny_allocation` | none | Block all allocations for matching scope |

## Policy Types

### Allocation Priority Policy

Controls which entities receive priority during allocation.

```
Example Policy:
  entity_type: "org.research.critical"
  resource_type: "compute"
  conditions: [usage_threshold < 0.9]
  actions: [set_allocation_strategy("priority"), set_preemption_priority(1)]
```

### Preemption Rules

Defines which allocations can be preempted and in what order.

```
Example Policy:
  entity_type: "org.background.*"
  resource_type: "*"
  conditions: [autonomy_level < L2]
  actions: [set_preemption_priority(5)]  // Preempted first
```

### Budget Elasticity Policy

Controls which entities can use elastic budgets.

```
Example Policy:
  entity_type: "org.research.*"
  resource_type: "compute"
  conditions: [entity has elastic budget]
  actions: [enable_burst(multiplier: 2.0)]
```

### Quota Override Policy

Temporarily overrides quota limits for specific scopes during approved events.

```
Example Policy:
  entity_type: "org.*"
  resource_type: "tokens"
  conditions: [time_range("09:00", "17:00")]
  actions: [quota_override(hard_limit: 2000000)]  // Increase token quota during business hours
```

### Energy Optimization Policy

Enables or disables energy-aware allocation for specific scopes.

```
Example Policy:
  entity_type: "org.batch.*"
  resource_type: "compute"
  conditions: [usage_threshold < 0.5]
  actions: [energy_optimize(enabled: true)]
```

## Policy Evaluation

```
function evaluatePolicies(entity, request, context):
    applicable_policies = []
    for policy in all_policies:
        if policy matches entity and resource:
            if all conditions met:
                applicable_policies.append(policy)
    
    // Sort by priority (highest first)
    sort(applicable_policies, by: priority DESC)
    
    result = default_behavior
    for policy in applicable_policies:
        result = apply(policy.actions, result)
        if policy.actions contains deny_allocation:
            return DENY
    
    return result
```

### Policy Precedence

| Precedence | Authority | Scope |
|------------|-----------|-------|
| 1 (highest) | Security Council | System-wide |
| 2 | Security Council | Per-Organization |
| 3 | Organization | Per-Organization (within Council bounds) |
| 4 | Organization | Per-Mission (within Organization bounds) |

Organization sub-policies cannot override Security Council policies. If an Organization policy conflicts with a Security Council policy, the Security Council policy wins.

## Policy Lifecycle

```
Policy Created (draft) → Reviewed → Approved (active) → Applied → Expired / Revoked
```

| State | Description |
|-------|-------------|
| draft | Being authored, not yet active |
| approved | Active and evaluated by Allocator |
| expired | valid_until reached |
| revoked | Manually disabled by authority |
| superseded | Replaced by a newer policy |

## Policy Operations

### setPolicy

Creates a new resource management policy.

| Parameter | Type | Description |
|-----------|------|-------------|
| `policy` | Policy | Policy definition |

**Event produced**: `PolicyCreated { policy_id, name, authority, timestamp }`

### updatePolicy

Updates an existing policy.

| Parameter | Type | Description |
|-----------|------|-------------|
| `policy_id` | UUID | Policy to update |
| `updates` | Partial<Policy> | Fields to update |

**Event produced**: `PolicyUpdated { policy_id, changes, timestamp }`

### deletePolicy

Deletes a policy.

| Parameter | Type | Description |
|-----------|------|-------------|
| `policy_id` | UUID | Policy to delete |

**Event produced**: `PolicyDeleted { policy_id, timestamp }`

### listPolicies

Lists all policies matching criteria.

| Parameter | Type | Description |
|-----------|------|-------------|
| `entity_type` | String (optional) | Filter by entity type |
| `resource_type` | ResourceType (optional) | Filter by resource type |
| `authority` | Enum (optional) | Filter by authority |

**Returns**: List of matching policies with evaluation priorities.

## Cross-Cutting Concerns

### Security

Policy creation requires authorization. Only Security Council may set system-wide policies. Organization policies must not exceed organization authority bounds.

### Evidence

All policy operations produce Events. Policy evaluation history is auditable.

### Lifecycle

Policies have explicit validity periods. Expired policies are automatically disabled. Policy lifecycle is managed by the authority that created them.

### Capability Bounds

Policies cannot override constitutional capability bounds. A policy may reduce but never increase capability resource limits beyond what the entity's template allows.

### Communication

Policies are stored in the Policy Store and distributed to Allocator nodes. Policy changes propagate via ACF to all ROS nodes within 5 seconds.

### Design DNA Compliance

| Rule | Compliance |
|------|------------|
| R1 — Modulsingularity | RMP handles only resource management policies. Separate from allocation logic. |
| R9 — Deterministic | Policy evaluation is deterministic. Same entity + same context always produces same policy result. |
| R10 — Simpler Over Complex | Policy model uses simple condition-action pairs. No complex rule engine. |
| R13 — Design for Failure | If the Policy Store is unavailable, the Allocator defaults to the last known policies. |
| R14 — Paved Path | The paved path is: Security Council Sets Policy → Policy Evaluated → Allocator Applies. |
| R15 — Open/Closed | New action types can be added via the action registry without modifying the evaluator. |
