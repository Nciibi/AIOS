## Purpose

The Resource Exchange Protocol (RXP) enables resource sharing across AIOS instance boundaries. When Instance A has surplus resources and Instance B has deficit, RXP allows Instance B to request and consume resources from Instance A. RXP is built on the inter-instance ACF bridge and requires a federation agreement between instances (see Bible/04-Execution/Security/IDS/004-Federation.md).

## Protocol Overview

```
Instance A (Provider)                  Instance B (Consumer)
       │                                      │
       │     ◄── requestResource ────          │
       │                                      │
       │     ──── offerResource ───►           │
       │                                      │
       │     ◄── acceptOffer ────────          │
       │                                      │
       │     ──── transferResource ──►         │
       │                                      │
       │     ──── completeTransfer ──►         │
       │                                      │
       │     ──── accounting ────────►         │
```

## Protocol Flow

### 1. Request Resource

Instance B discovers that it needs resources beyond its local capacity and sends a resource request to Instance A.

```
Instance B → Instance A:
{
    "message_type": "RXP_REQUEST",
    "request_id": UUID,
    "requester_instance_id": UUID,
    "resource_type": ResourceType,
    "quantity": Quantity,
    "duration": Duration,
    "capability_id": UUID,
    "federation_token": String,          // Signed by Instance B's Security Council
    "priority": Int
}
```

**Authorization**: Instance A validates the federation token against the federation agreement. Only instances with an active federation agreement may make requests.

**Event produced**: `RXPRequestSent { request_id, requester_instance, provider_instance, resource_type, quantity, timestamp }`

### 2. Offer Resource

Instance A evaluates the request against its local capacity, budget, and policies. If resources are available and the federation agreement permits, Instance A sends an offer.

```
Instance A → Instance B:
{
    "message_type": "RXP_OFFER",
    "request_id": UUID,
    "offer_id": UUID,
    "provider_instance_id": UUID,
    "resource_type": ResourceType,
    "quantity_offered": Quantity,
    "cost_rate": Float,                  // Cross-instance cost rate
    "ttl": Duration,                     // Offer valid duration
    "constraints": Map<String, String>   // Provider-specific constraints
}
```

**Authorization**: Instance A's ROS checks local resource availability, quota, and federation policy. The offer is contingent on resources being available at acceptance time.

**Event produced**: `RXPOfferCreated { offer_id, request_id, provider_instance, quantity_offered, cost_rate, ttl, timestamp }`

### 3. Accept Offer

Instance B accepts the offer, committing to consume the offered resources.

```
Instance B → Instance A:
{
    "message_type": "RXP_ACCEPT",
    "offer_id": UUID,
    "request_id": UUID,
    "accepted_quantity": Quantity,
    "execution_token": String
}
```

**Authorization**: Instance B validates that it still needs the resources and that the offer TTL has not expired.

**Event produced**: `RXPOfferAccepted { offer_id, request_id, requester_instance, accepted_quantity, timestamp }`

### 4. Transfer Resource

Instance A allocates the resources and transfers control to Instance B.

```
Instance A → Instance B:
{
    "message_type": "RXP_TRANSFER",
    "offer_id": UUID,
    "transfer_id": UUID,
    "allocation_details": {
        "provider_id": UUID,
        "resource_type": ResourceType,
        "quantity": Quantity,
        "access_credentials": Encrypted<String>,     // Provider-specific access
        "endpoint": String,
        "expires_at": Timestamp
    }
}
```

**Authorization**: Instance A verifies that Instance B has accepted the offer and that resources are still available. Instance B verifies the transfer details.

**Event produced**: `RXPTransferCompleted { transfer_id, offer_id, provider_instance, requester_instance, quantity, timestamp }`

### 5. Complete Transfer

Instance B confirms that the resources have been received and are operational.

```
Instance B → Instance A:
{
    "message_type": "RXP_COMPLETE",
    "transfer_id": UUID,
    "status": Enum (success, partial, failed),
    "received_quantity": Quantity,
    "errors": List<String> (optional)
}
```

**Event produced**: `RXPTransferConfirmed { transfer_id, status, received_quantity, timestamp }`

### 6. Accounting

After resource consumption completes, Instance A sends an accounting record to Instance B.

```
Instance A → Instance B:
{
    "message_type": "RXP_ACCOUNTING",
    "transfer_id": UUID,
    "actual_usage": {
        "quantity_consumed": Quantity,
        "duration": Duration,
        "cost": Float
    },
    "total_charge": Float
}
```

**Event produced**: `RXPAccountingRecorded { transfer_id, provider_instance, requester_instance, cost, timestamp }`

## RXP Operations

### requestResource

Sends a resource request to another AIOS instance.

| Parameter | Type | Description |
|-----------|------|-------------|
| `target_instance_id` | UUID | Instance to request from |
| `resource_type` | ResourceType | Resource type needed |
| `quantity` | Quantity | Amount needed |
| `duration` | Duration | Expected duration |
| `capability_id` | UUID | Capability requiring resources |
| `priority` | Int | Request priority |

**Returns**: `RXPRequest { request_id, status (pending, accepted, rejected) }`

### offerResource

Creates a resource offer in response to a request.

| Parameter | Type | Description |
|-----------|------|-------------|
| `request_id` | UUID | Request being responded to |
| `quantity_offered` | Quantity | Amount offered |
| `cost_rate` | Float | Cross-instance cost rate |
| `ttl` | Duration | Offer validity duration |

### acceptOffer

Accepts a resource offer.

| Parameter | Type | Description |
|-----------|------|-------------|
| `offer_id` | UUID | Offer to accept |
| `accepted_quantity` | Quantity | Amount to accept (may be less than offered) |

### transferResource

Transfers resources from provider instance to consumer instance.

| Parameter | Type | Description |
|-----------|------|-------------|
| `offer_id` | UUID | Accepted offer |
| `transfer_details` | TransferDetails | Provider-specific transfer information |

### completeTransfer

Confirms resource transfer completion.

| Parameter | Type | Description |
|-----------|------|-------------|
| `transfer_id` | UUID | Transfer to confirm |
| `status` | Enum | success, partial, failed |
| `received_quantity` | Quantity | Quantity actually received |

## Federation Requirements

RXP requires an active federation agreement between instances:

| Requirement | Description | Source Document |
|-------------|-------------|----------------|
| Federation agreement | Signed agreement between Security Councils | IDS/004-Federation.md |
| Federation token | Cryptographically signed token included in every request | IDS/004-Federation.md |
| Trust anchor | Both instances trust the same root CA | IDS/003-PKI.md |
| Resource limits | Federation agreement specifies max cross-instance allocation | IDS/004-Federation.md |
| Cost agreement | Agreed cost rates for cross-instance resource usage | IDS/004-Federation.md |

Without a valid federation agreement, RXP requests are denied.

## Cross-Instance Resource Limits

| Limit | Description | Configuration |
|-------|-------------|---------------|
| Max concurrent transfers | Maximum simultaneous cross-instance transfers | Per federation agreement |
| Max transfer quantity | Maximum resources per single transfer | Per federation agreement |
| Max total cross-instance usage | Aggregate resources consumed from remote instances | Per instance (Security Council) |
| Transfer duration limit | Maximum duration for a cross-instance allocation | 24 hours (default) |

## Communication Protocol

RXP uses the inter-instance ACF bridge:

| Layer | Protocol | Details |
|-------|----------|---------|
| Transport | ACF Bridge | Encrypted ACF channel between instances |
| Authentication | Federation token | JWT signed by requesting instance's Security Council |
| Authorization | Federation policy | Evaluated by receiving instance's ROS |
| Serialization | Protocol Buffers | Compact binary serialization |
| Reliability | At-least-once delivery | ACF guarantees message delivery |

## Error Handling

| Error | Code | Recovery |
|-------|------|----------|
| Federation agreement not found | RXP-001 | Verify federation status |
| Offer expired | RXP-002 | Request new offer |
| Insufficient capacity at transfer time | RXP-003 | Request smaller quantity |
| Federation token invalid | RXP-004 | Re-authenticate with Security Council |
| Transfer timeout | RXP-005 | Retry transfer or cancel |
| Accounting mismatch | RXP-006 | Manual reconciliation |

## Cross-Cutting Concerns

### Security

All RXP messages require federation token authentication. Tokens are signed by the requesting instance's Security Council and contain the request scope, quantity, and expiry. TLS encryption is mandatory for inter-instance ACF communication.

### Evidence

Every RXP operation produces an Event on both instances. Cross-instance transfers are fully auditable on both sides. Accounting records are cryptographically signed.

### Lifecycle

Cross-instance resource allocations follow the same lifecycle as local allocations. When the consumer instance recovers its resources, it sends a completion notification to the provider instance.

### Capability Bounds

Cross-instance resource allocation is still bounded by the requesting entity's capability template. RXP does not allow entities to exceed their constitutional capability resource limits.

### Communication

RXP uses the inter-instance ACF bridge with encryption and authentication. Messages use Protocol Buffers for compact serialization. ACF guarantees at-least-once delivery.

### Design DNA Compliance

| Rule | Compliance |
|------|------------|
| R1 — Modulsingularity | RXP handles only cross-instance resource sharing. Separate from local resource management. |
| R5 — Liskov Substitution | Remote providers accessed via RXP implement the same ResourceProvider interface as local providers. |
| R9 — Deterministic | RXP operations are deterministic for same inputs. Transfer outcomes depend on remote instance state, which is outside local control. |
| R10 — Simpler Over Complex | RXP is a simple request-offer-accept-transfer-account protocol. No complex negotiation or auction. |
| R13 — Design for Failure | Offers have TTLs and expire automatically. Transfers have timeouts. If remote instance becomes unreachable, resources are recovered locally. |
| R14 — Paved Path | The paved path is: Request → Offer → Accept → Transfer → Complete → Account. No other path. |
