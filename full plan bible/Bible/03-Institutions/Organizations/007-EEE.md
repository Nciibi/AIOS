## Purpose

The Engine Employment Exchange (EEE) manages how execution Engines are assigned to Organizations. Engines (Sessions instantiated from Worker Genomes) are the executing entities that perform Mission tasks. EEE handles the full employment lifecycle: from posting positions and receiving applications to interviewing, offering, contracting, managing active employment, and handling termination.

## Employment Model

```
Engine â†’ Application â†’ Interview â†’ Offer â†’ Contract â†’ Active â†’ Terminated
```

Employment is a constitutional relationship between an Organization (employer) and an Engine (employee). The Organization provides Mission scope, resources, and governance. The Engine provides execution capability, domain expertise, and operational output.

## Employment States

```
Available â†’ Applied â†’ Interviewing â†’ Offered â†’ Contracted â†’ Active â†’ Terminated â†’ Archived
```

| State | Description | Can Execute? | Capability Access |
|-------|-------------|--------------|-------------------|
| **Available** | Engine is not currently employed. Available for applications. | No | Base capabilities only |
| **Applied** | Engine has applied for a position. Awaiting response. | No | Base capabilities only |
| **Interviewing** | Organization is evaluating the Engine. | No (evaluation tasks only) | Limited (evaluation scope) |
| **Offered** | Organization has extended an offer. Engine reviewing. | No | None |
| **Contracted** | Employment terms accepted. Contract formalized. | No | Pre-activation provisioning |
| **Active** | Engine is employed and executing Mission tasks. | Yes | Full scope per contract |
| **Terminated** | Employment ended. Engine returned to Available pool. | No | Base capabilities only |
| **Archived** | Employment record preserved for audit. | No | None |

## EEE Operations

```
Input:  org_id, position_spec { title, capability_requirements, specialization, culture_fit_criteria, duration }

## Matching Algorithm

EEE matches Engines to positions based on:

| Criterion | Weight (default) | Source | Description |
|-----------|-----------------|--------|-------------|
| Capability Match | 40% | Capability Registry | How well Engine capabilities match position requirements |
| Specialization | 25% | Engine Genome | Engine's domain specialization vs position domain |
| Performance History | 20% | OPE / WHS | Past Mission completion rate, quality scores |
| Culture Fit | 15% | Org Genome | Compatibility with Org culture, working style |

### Matching Flow

```
1. Position posted with requirements
2. EEE queries Available Engine pool
3. Each Engine scored against position criteria
4. Top matches presented to Org Admin/Manager
5. Org selects candidate(s) for interview
6. Interview results feed back into Engine performance history
```

```

## Cross-Cutting Concerns

### Security

Employment operations require role-based authorization. Only Org Admin may extend offers and terminate employment. Engine capabilities are granted on Active state and revoked on termination. All employment Events are recorded for Security Council audit. (Physics/008-Security.md)

### Evidence

Every EEE operation produces an Event. Complete employment history â€” from posting to termination â€” is recorded in the Event Store. Performance evaluation results feed into future matching. (PHI-008)

### Lifecycle

The EEE employment lifecycle (Available â†’ Applied â†’ Interviewing â†’ Offered â†’ Contracted â†’ Active â†’ Terminated â†’ Archived) is an instance of the canonical lifecycle model. Employment is constrained by Organization lifecycle â€” a Suspended Org cannot hire new Engines. (PHI-006)

### Capability Bounds

Engines are employed with specific capability grants defined in the contract. Capabilities are bounded by the Organization's capability scope and the Engine's Genome. Capabilities are activated on Active state and revoked on termination. (Physics/007-Capabilities.md)

### Communication

All EEE operations flow through ACF. Position postings are broadcast to the Available Engine pool. Interview evaluations and offer communications are routed through ACF. (Law 3 â€” Communication)
