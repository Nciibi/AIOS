## Purpose

The Organization Object Model (OOM) defines the structure, schema, roles, and operations of an Organization. OOM is the canonical data model â€” every Organization entity in AIOS is an instance of this model. The model covers organization hierarchy, departmental composition, role-based access, and the constitutional operations for creating, querying, modifying, and dissolving Organizations.

## Organization Schema

```
Organization {
    org_id: UUID                    // Immutable identity assigned by IDS
    name: String                    // Human-readable name
    org_type: OrgType              // root | department | team | committee
    parent_org: UUID | null        // Parent Organization (null for root)
    children: UUID[]               // Direct sub-Organizations
    mission_ids: UUID[]            // Missions owned by this Organization
    engine_ids: UUID[]             // Engines employed via EEE
    resource_budget: BudgetRef     // ROS budget reference
    governance_policy: PolicyRef   // Reference to ORG governance policy
    genome_id: UUID                // AGS Genome that defines this Organization
    lifecycle_state: OrgState      // Current state per OSYS lifecycle
    departments: DeptRecord[]      // Department sub-structure
    roles: RoleAssignment[]        // Role assignments per member
    created_at: Timestamp
    updated_at: Timestamp
}
```

### OrgType Enum

| Type | Description | Parent Required? | Can Own Missions? |
|------|-------------|------------------|-------------------|
| root | Top-level Organization (e.g., AIOS itself) | No | Yes |
| department | Functional or geographic sub-Organization | Yes | Yes (within parent bounds) |
| team | Small operational unit within a department | Yes | Limited |
| committee | Temporary or permanent advisory body | Yes | No |

## Hierarchy Depth

| Level | Type | Description |
|-------|------|-------------|
| 0 (root) | Root | Top of the tree â€” only one per AIOS instance |
| 1 | Executive | Strategic leadership Organizations |
| 2 | Division | Major functional divisions |
| 3 | Department | Specialized operational units |
| 4 | Branch | Geographic or functional sub-divisions |
| 5 | Team | Small coordinated groups |
| 6 | Squad | Minimum operational unit |

**Maximum depth**: 7 levels (root through level 6). Requests to create an Organization deeper than level 6 are rejected with OOM_HIE_001.

## Role Model

| Role | Scope | Capabilities | Can Assign Roles? |
|------|-------|-------------|-------------------|
| **Admin** | Full Organization | All management operations: create/ dissolve departments, modify budget, change governance policy, manage roles | Yes (Admin and below) |
| **Manager** | Assigned departments | Manage department Workers, assign Missions, adjust department budgets, review performance | Yes (Manager and below within department) |
| **Member** | Assigned teams | Execute assigned Missions, access team resources, communicate within scope | No |
| **Observer** | Read-only | View Organization structure, read public Mission status, access non-sensitive evidence | No |

### Role Inheritance

- Admins inherit all Manager capabilities across all departments
- Managers inherit all Member capabilities within their assigned departments
- Members inherit all Observer capabilities within their assigned scope
- Observer capabilities do not extend to sensitive or classified information

### Role Assignment Operations

| Operation | Authorized By | Constraints |
|-----------|--------------|-------------|
| assignAdmin | Security Council | Limited to 1 per Organization (the Director) |
| assignManager | Admin | Must be within Admin's Organization scope |
| assignMember | Manager | Must be within Manager's department scope |
| assignObserver | Admin or Manager | Can be scoped to Organization, department, or Mission |
| revokeRole | Assigning authority or higher | Evidence required for revocation |

## Cross-Cutting Concerns

### Security

Organization creation requires Security Council authorization for root and department-level Organizations. Role assignment is governed by the Role Model â€” only authorized entities may assign roles. Cross-Organization queries are scoped by hierarchy authorization. (Physics/008-Security.md)

### Evidence

Every OOM operation produces an Event. Organization schema changes, role assignments, and structural modifications are all recorded in the Event Store. (PHI-008)

### Lifecycle

OOM operations are lifecycle-state-dependent. Most operations (updateOrg, moveOrg, dissolveOrg) require the Organization to be in Active state. Dissolution requires the Organization to have no active Missions. (PHI-006)

### Capability Bounds

Organization capabilities are defined by its Genome and bounded by its parent's capabilities. An Organization cannot have capabilities beyond its parent's scope. Role capabilities cascade: Admin inherits all, Manager inherits department-scoped, Member inherits team-scoped. (Physics/007-Capabilities.md)

### Communication

All OOM operations communicate through ACF. Creation notifications, updates, and role assignments are broadcast to affected entities via ACF event streams. (Law 3 â€” Communication)
