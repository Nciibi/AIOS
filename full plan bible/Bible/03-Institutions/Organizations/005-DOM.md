## Purpose

The Department Object Model (DOM) defines the sub-organization structure within an Organization. Departments are functional, geographic, or mission-aligned sub-units that inherit the parent Organization's constitutional authority but may have independent sub-policies, specialized capabilities, and dedicated personnel.

## Department Schema

```
Department {
    dept_id: UUID                    // Immutable identity (optional â€” may use parent Org identity)
    org_id: UUID                     // Parent Organization
    name: String                     // Human-readable name
    dept_type: DeptType             // functional | geographic | mission_aligned
    parent_dept: UUID | null         // Parent department (for nesting within Organization)
    children: UUID[]                 // Sub-departments
    capabilities: CapabilityRef[]    // Specialized capabilities (subset of parent Org)
    policies: PolicyRef[]            // Sub-policies (within parent Org bounds)
    members: MemberAssignment[]      // Workers and entities assigned to this department
    resource_budget: BudgetRef       // Sub-budget allocated from parent Org
    lifecycle_state: DeptState       // Created â†’ Active â†’ Dissolved
    created_at: Timestamp
    updated_at: Timestamp
}
```

## Department Types

| Type | Description | Example | Identity Required? |
|------|-------------|---------|-------------------|
| **Functional** | Specialized operational capability | Engineering, Security, Finance, HR | Yes (if >50 members) |
| **Geographic** | Regional or location-based | North America, Europe, Asia-Pacific | Optional |
| **Mission-Aligned** | Temporary or permanent project team | Q3 Optimization Team, Platform Migration | No |

### Functional Departments

Functional departments organize by capability domain. Each functional department owns a specific capability area within the Organization's domain.

| Functional Type | Typical Capabilities | Governance |
|----------------|---------------------|------------|
| Engineering | Code execution, review, deployment | Engineering Manager |
| Security | Audit, compliance, incident response | Security Manager |
| Finance | Budget, cost analysis, resource planning | Finance Manager |
| Operations | Infrastructure, deployment, monitoring | Operations Manager |
| Research | Analysis, experimentation, knowledge | Research Manager |

### Geographic Departments

Geographic departments organize by location to handle regional policies, regulations, and operational requirements.

### Mission-Aligned Departments

Mission-aligned departments are temporary structures created for specific Missions that require cross-functional coordination. They dissolve when the Mission completes.

## Department Creation

```
Input:  org_id, dept_spec { name, dept_type, parent_dept?, capabilities[], policies[] }
Process:
  1. Validate parent Organization exists and is Active
  2. Validate dept_type is recognized
  3. Validate capabilities are subset of parent Organization's capabilities
  4. Validate sub-policies do not exceed parent policy bounds
  5. Check department depth (max nesting: 5 levels within Organization)
  6. If identity required: request IDS identity
  7. Create Department record
  8. Allocate sub-budget from parent Org budget (via ROS)
  9. Register in ODS
Output: Department { dept_id, ... }
Authorization: Admin of parent Organization
Event: DOM.DepartmentCreated
```

## Department Lifecycle

Departments follow a simplified 3-state lifecycle:

```
Created â†’ Active â†’ Dissolved
```

| State | Description | Can Act? | Capability Access |
|-------|-------------|----------|-------------------|
| **Created** | Department record exists. Structure defined but not operational. | No | None |
| **Active** | Department is operational. Members assigned. Capabilities active. | Yes | Full scope per assignment |
| **Dissolved** | Department is permanently dissolved. Members reassigned. Capabilities returned. | No | None |

### Transitions

| Transition | Authorized By | Requires Evidence? | Notes |
|-----------|--------------|-------------------|-------|
| Created â†’ Active | Admin | Yes (readiness confirmation) | Automatic after creation if preconditions met |
| Active â†’ Dissolved | Admin | Yes (dissolution rationale, member reassignment plan) | All members must be reassigned first |
| Active â†’ Dissolved | Parent Org Admin | Yes (restructure rationale) | May be overridden by parent |

### Dissolution Flow

```
1. Dissolution proposed (dept Admin or parent Org Admin)
2. Member reassignment plan created
3. All active members reassigned to other departments or Organization pool
4. Department resources returned to parent Org budget
5. Department capabilities deactivated
6. Department record transitions to Dissolved
7. Event broadcast to affected entities
```

## Department Sub-Policies

Departments may define sub-policies that are more restrictive than the parent Organization's policies. Sub-policies cannot exceed the parent's policy bounds.

| Policy Area | Can Department Override? | Constraint |
|-------------|-------------------------|------------|
| Resource allocation | Yes (more restrictive only) | Must be within Org budget |
| Working hours | Yes (more restrictive only) | Cannot violate constitutional limits |
| Communication rules | Yes (more restrictive only) | Must use ACF |
| Quality standards | Yes (stricter only) | Cannot be laxer than Org standard |
| Security requirements | No | Set by parent Org |
| Constitutional compliance | No | Universal |
| Role definitions | No | Set by OOM |

## Cross-Cutting Concerns

### Security

Department creation and dissolution require Org Admin authorization. Department capabilities are a subset of parent Org capabilities â€” a department cannot exceed its parent's authority. Cross-department operations within the same Org are authorized by Org Admin. (Physics/008-Security.md)

### Evidence

Every DOM operation produces an Event. Department creation, member assignment, policy changes, and dissolution are all recorded in the Event Store. (PHI-008)

### Lifecycle

Departments follow the simplified Created â†’ Active â†’ Dissolved lifecycle. Department lifecycle is constrained by parent Organization lifecycle â€” a Suspended Org means all departments are effectively inactive. (PHI-006)

### Capability Bounds

Department capabilities are always a subset of parent Organization capabilities. Sub-policies are more restrictive or equal â€” never more permissive. Capabilities are assigned to members upon department assignment and revoked upon removal. (Physics/007-Capabilities.md)

### Communication

All DOM operations communicate through ACF. Member assignments, policy changes, and dissolution notifications are broadcast via ACF event streams. (Law 3 â€” Communication)
