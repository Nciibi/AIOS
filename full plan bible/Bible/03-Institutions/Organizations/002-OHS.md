## Purpose

The Organization Health Service (OHS) monitors the operational health of Organizations and performs self-healing actions when health degrades. OHS ensures that Organizations remain productive, compliant, and resilient. It detects issues before they become critical and, where possible, automatically remediates without requiring Security Council intervention.

## Health Metrics

| Metric | Description | Measurement | Target |
|--------|-------------|-------------|--------|
| Mission Completion Rate | Percentage of Missions completed successfully within the Organization | Completed / Total Missions in period | >90% |
| Engine Availability | Percentage of time employed Engines are available and responsive | Uptime / Total time | >99% |
| Resource Utilization | Efficiency of resource budget usage | Resources used / Resources allocated | 70-90% |
| Policy Compliance Score | Percentage of operations passing constitutional compliance checks | Compliant operations / Total operations | >95% |
| Incident Frequency | Number of security incidents or policy violations per period | Count per 30 days | <3/month |
| Worker Satisfaction | Aggregate Worker health and performance metrics | Per-WHS score averaged across Organization | >0.8 (normalized) |

## Health States

```
Healthy → Degraded → Unhealthy → Critical
```

| State | Description | Threshold | Actions |
|-------|-------------|-----------|---------|
| **Healthy** | All metrics within target ranges. Normal operations. | All metrics within bounds | Routine monitoring |
| **Degraded** | One or more metrics below target but above critical threshold. Some operations affected. | Mission completion <85% or Resource utilization >95% or Compliance <90% | Notify Admin, begin diagnostic, initiate self-healing |
| **Unhealthy** | Multiple metrics below critical threshold. Operations significantly affected. | Mission completion <70% or Compliance <80% or Incidents >5/month | Notify Security Council, restrict new Missions, escalate self-healing |
| **Critical** | Organization is at risk of dissolution. Immediate intervention required. | Mission completion <50% or Compliance <60% or Critical incident | Full Security Council review, possible suspension |

### State Transitions

| Transition | Trigger | Authorized By |
|-----------|---------|--------------|
| Healthy → Degraded | Metric threshold breached for >5 minutes | Automatic (OHS) |
| Degraded → Healthy | All metrics return to target for >30 minutes | Automatic (OHS) |
| Degraded → Unhealthy | Metrics worsen beyond critical threshold for >10 minutes | Automatic (OHS) |
| Unhealthy → Degraded | Self-healing actions restore metrics above critical for >1 hour | OHS + Security Council notification |
| Unhealthy → Critical | Mission completion <50% or compliance <60% or Security Council declaration | Automatic (OHS) or Security Council |
| Critical → Unhealthy | Full remediation plan approved and implementation begins | Security Council |

## Self-Healing Actions

| Action | Trigger | Description | Authorization |
|--------|---------|-------------|---------------|
| Reallocate Resources | Resource utilization >95% | Rebalance budget from underutilized Missions to overburdened ones | OHS (automatic) |
| Reassign Engines | Engine availability <95% | Replace failing Engines with alternates from EEE pool | OHS + Admin notification |
| Adjust Budgets | Resource utilization consistently <50% or >95% | Request ROS to adjust budget allocation | OHS (suggest), Admin approval |
| Restructure Departments | Department-level metrics show systematic issues | Recommend department reorganization | OHS (suggest), Admin decision |
| Escalate to Security Council | Unhealthy or Critical state | Submit full health report to Security Council for intervention | OHS (automatic) |
| Suspend Mission Acceptance | Critical state | Pause new Mission assignments until health improves | OHS (automatic), Security Council confirmation |

### Self-Healing Flow

```
1. Metric breach detected (OHS monitor)
2. Health state transitioned (Healthy → Degraded)
3. Diagnostic run: identify root cause
4. Select healing action: highest-impact, lowest-risk
5. Execute healing action (or suggest for manual actions)
6. Monitor for N minutes (configurable per action)
7. If metrics improve → continue monitoring
8. If metrics worsen → escalate to next action or Security Council
9. If metrics return to target → transition to Healthy
10. Log complete healing record as Event
```

## Diagnostic Subsystem

When health degrades, OHS runs diagnostics to identify root causes:

| Diagnostic | Description | Data Sources | Output |
|-----------|-------------|-------------|--------|
| Mission Analysis | Reviews active and recent Mission outcomes | LMS, Missions | Mission health breakdown |
| Resource Audit | Analyzes resource consumption patterns | ROS | Resource efficiency report |
| Compliance Scan | Audits recent operations for policy violations | Security Council | Compliance score breakdown |
| Worker Health Aggregate | Aggregates WHS reports for all Workers in Org | WHS | Worker satisfaction index |
| Dependency Health | Checks health of external dependencies (ROS, ACF) | Infrastructure | Dependency status report |

### Diagnostic Flow

```
1. Degraded state entered → diagnostic triggered automatically
2. Each diagnostic runs in parallel (timeout: 30 seconds each)
3. Results aggregated into DiagnosticSummary
4. Root cause identified (or multiple causes)
5. Healing action selected based on root cause
6. If root cause not identifiable → escalate to Security Council
```

## Health Operations

### getOrgHealth

```
Input:  org_id
Process: aggregate current metrics → compute health state → return
Output: OrgHealthReport { org_id, state, metrics[], last_transition, recommendations[] }
Authorization: Admin or Manager of Organization
Event: OHS.HealthQueried
```

### getHealthHistory

```
Input:  org_id, time_range
Process: query Event Store for health events → aggregate → return
Output: HealthHistory { org_id, events[], metric_timelines{} }
Authorization: Admin of Organization
```

### triggerHealthCheck

```
Input:  org_id
Process: immediate full diagnostic → compare to thresholds → update state → execute healing if needed
Output: HealthCheckResult { org_id, state, diagnostic_summary, actions_taken[] }
Authorization: Admin or Security Council
Event: OHS.HealthCheckTriggered
```

### suppressHealing

```
Input:  org_id, duration, reason
Process: suspend automatic healing actions for specified duration
Output: SuppressionRecord { org_id, suppressed_until, authorized_by }
Authorization: Security Council only
Event: OHS.HealingSuppressed
```

## Health Events

| Event Type | Produced When | Fields |
|-----------|--------------|--------|
| `OHS.HealthChanged` | Health state transitions | org_id, from_state, to_state, metrics_snapshot |
| `OHS.Degraded` | Organization enters Degraded state | org_id, metrics_breached[], threshold_values |
| `OHS.Unhealthy` | Organization enters Unhealthy state | org_id, metrics_breached[], diagnostic_summary |
| `OHS.Critical` | Organization enters Critical state | org_id, diagnostic_summary, recommended_actions |
| `OHS.SelfHealingTriggered` | Self-healing action begins | org_id, action_type, parameters |
| `OHS.SelfHealingComplete` | Self-healing action finishes | org_id, action_type, outcome, metrics_after |
| `OHS.SelfHealingFailed` | Self-healing action fails to improve metrics | org_id, action_type, failure_reason, escalation_path |
| `OHS.MetricBreached` | Individual metric crosses threshold | org_id, metric_name, current_value, threshold |
| `OHS.HealingSuppressed` | Automatic healing is suppressed | org_id, suppressed_by, duration, reason |

## Error Codes (R12)

| Code | Description |
|------|-------------|
| OHS_001 | Organization not found for health query |
| OHS_002 | Health metric data unavailable (insufficient samples) |
| OHS_003 | Healing action not applicable in current health state |
| OHS_004 | Healing action failed — resource unavailable |
| OHS_005 | Cannot suppress healing — authorization insufficient |
| OHS_006 | Health check already in progress for Organization |

## Cross-Cutting Concerns

### Security

Health state transitions that reach Critical trigger automatic Security Council escalation. Self-healing actions that affect resource budgets or Engine assignments require Admin notification. Suppression of healing requires Security Council authorization. (Physics/008-Security.md)

### Evidence

Every health check, state transition, and healing action produces an Event. Complete health history is recorded in the Event Store for audit and trend analysis. Health events are consumed by OPE for performance evaluation. (PHI-008)

### Lifecycle

OHS operates only for Organizations in Active state. Suspended or Dissolved Organizations do not receive health monitoring. Health state affects lifecycle decisions — an Unhealthy Organization may be flagged for suspension review. (PHI-006)

### Capability Bounds

OHS healing actions are bounded by Organization capabilities. OHS cannot reallocate resources beyond the Organization's ROS budget. It cannot reassign Engines without EEE availability. All healing actions operate within constitutional bounds. (Physics/007-Capabilities.md)

### Communication

Health notifications are broadcast via ACF. Degraded and Unhealthy states notify Org Admin. Critical state notifies Security Council. Healing completions and failures are published to the Organization's event stream. (Law 3 — Communication)

### Design DNA

| Rule | Compliance |
|------|-----------|
| R1 (Modulsingularity) | OHS focuses solely on health monitoring and self-healing |
| R9 (Deterministic) | Health state calculation is deterministic — same metrics always produce same state |
| R10 (Simpler Over Complex) | 4 health states with linear transitions — no branching complexity |
| R12 (Embrace Errors) | All errors have unique codes (OHS_001–006) |
| R13 (Design for Failure) | OHS degrades gracefully if metric sources are unavailable — uses last known values |
| R14 (Paved Path) | Paved path: Monitor → Detect → Diagnose → Heal → Verify |
