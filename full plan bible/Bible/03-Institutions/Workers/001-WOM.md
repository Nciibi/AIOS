## Purpose

The Worker Object Model (WOM) defines the structure, capabilities, and lifecycle of a Worker Session. A Worker is a Session instantiated from a Template Genome (AGS) with a unique identity (IDS), executing within a Mission under an Organization's authority. WOM is the canonical data model for all Worker entities in AIOS.

## Worker/Session Schema

```
WorkerSession {
    session_id: UUID                  // Globally unique, immutable identity (IRS)
    template_id: UUID                 // AGS Template Genome this Worker was created from
    identity_id: UUID                 // IDS identity record reference
    mission_id: UUID                  // Mission this Worker executes within
    org_id: UUID                      // Organization that owns this Worker
    runtime_id: UUID                  // Current Runtime executing this Session
    capabilities: CapabilitySet       // Active capabilities (Genome + Org + Mission)
    tokens: TokenSet                  // Authentication and authorization tokens
    lifecycle_state: LifecycleState   // Current state in 10-state model
    context: SessionContext           // Workspace, history, state
    resource_budget: BudgetRef        // ROS resource consumption limits
    health_status: HealthStatus       // Current health (Healthy/Degraded/Unresponsive/Failed)
    created_at: Timestamp
    updated_at: Timestamp
}
```

## Worker Capability Structure

Worker capabilities are determined by three sources applied in order:

```
CapabilityResolutionOrder:
  1. Template Genome (AGS) → Base capability set
  2. Org Policy → Adds restrictions or grants additional capabilities within Org scope
  3. Mission Context → Scopes capabilities to Mission objectives and resource budget
```

### CapabilitySet Schema

```
CapabilitySet {
    base_capabilities: Capability[]       // From Template Genome
    org_grants: CapabilityGrant[]         // Added by Organization policy
    org_restrictions: CapabilityRestriction[]  // Restricted by Organization policy
    mission_scope: ScopeDefinition        // Scoped to Mission objectives
    autonomy_level: AutonomyLevel         // L0-L4 per capability
    resource_bounds: ResourceBounds       // CPU, memory, tokens, storage, network
}
```

### Capability Precedence

| Source | Effect | Overrideable By |
|--------|--------|----------------|
| Template Genome | Defines maximum possible capability scope | Nothing (hard bound) |
| Organization Policy | Narrows or broadens within Genome bounds | Mission context (narrower only) |
| Mission Context | Narrows to Mission-specific scope | Nothing (hard bound for Mission) |

## Worker Lifecycle (10-State)

```
Created → Planned → Assigned → Running → Waiting → Paused → Blocked → Review → Completed → Archived
```

| State | Description | Can Execute? | Capability Access | Modifiable? |
|-------|-------------|-------------|-------------------|-------------|
| **Created** | Session record exists. Identity assigned by IDS. Template loaded. Runtime not yet bound. | No | None | Limited (metadata) |
| **Planned** | Purpose defined. Runtime selected. Resource requirements estimated. | No | Read-only | Yes (plan) |
| **Assigned** | Mission and Organization assigned. Resources allocated via ROS. | No | Read-only, resource discovery | Yes (assignments) |
| **Running** | Actively executing tasks. Evidence being produced. Heartbeat active. | Yes | Full capability scope | Yes (runtime config) |
| **Waiting** | Awaiting external input, dependency resolution, or human approval. | No | Read-only, status queries | No |
| **Paused** | Execution suspended by supervisor. Context preserved. Resumable. | No | Read-only, status queries | No |
| **Blocked** | Cannot proceed due to failure or constraint. Escalation required. | No | Read-only, error reporting | No |
| **Review** | Execution complete. Output being verified against success criteria. | Limited (review actions) | Read-only, review actions | No |
| **Completed** | Purpose fulfilled. Evidence sealed. Tokens revoked. | No | Read-only (evidence access) | No |
| **Archived** | Record preserved for audit. Not operational. | No | None | No |

## WOM Operations

### createWorker

```
Input:  template_id, org_id, mission_id, runtime_preferences?, capability_overrides?
Process:
  1. Validate template_id is an Active AGS Genome
  2. Validate org_id is an Active Organization
  3. Validate mission_id belongs to org_id and is in appropriate state
  4. Request identity from IDS → identity_id assigned
  5. Resolve capabilities: Genome → Org policy → Mission context
  6. Allocate resource budget via ROS
  7. Request LMS to register lifecycle (Created state)
  8. Bind to Runtime
  9. Generate authentication tokens (ATS)
Output: WorkerSession
Authorization: Org Admin or Manager
Event: WOM.WorkerCreated
```

### getWorker

```
Input:  session_id
Process: lookup Session in Registry → verify authorization → return record
Output: WorkerSession
Authorization: Member of owning Organization
```

### updateWorkerCapabilities

```
Input:  session_id, capability_updates { grants[], restrictions[] }
Process: validate updates against Genome bounds → apply → notify Worker
Output: WorkerSession (updated capabilities)
Authorization: Org Admin (for org-level grants), Manager (for mission-level scope)
Event: WOM.WorkerCapabilitiesUpdated
```

### terminateWorker

```
Input:  session_id, reason, evidence_ref
Process: verify authorization → pause execution → save context → revoke tokens → terminate Runtime → archive
Output: TerminationRecord
Authorization: Org Admin, Manager (for department Workers), or Security Council
Event: WOM.WorkerTerminated
```

### listWorkers

```
Input:  org_id, filters { state?, mission_id?, template_id? }
Process: query Worker Registry → filter → return
Output: WorkerSession[]
Authorization: Member of Organization
```

### transitionWorker

```
Input:  session_id, target_state, authorized_by, evidence_ref
Process: validate transition → check authorization → execute → notify → record Event
Output: WorkerSession (new state)
Authorization: Per transition authorization matrix (see Mission Lifecycle pattern)
Event: WOM.WorkerStateChanged
```

## Events

| Event Type | Produced When | Fields |
|-----------|--------------|--------|
| `WOM.WorkerCreated` | New Worker Session is created | session_id, template_id, org_id, mission_id, runtime_id |
| `WOM.WorkerActivated` | Worker enters Running state | session_id, activated_at, runtime_config |
| `WOM.WorkerStateChanged` | Worker lifecycle state transitions | session_id, from_state, to_state, authorized_by |
| `WOM.WorkerCapabilitiesUpdated` | Worker capability set modified | session_id, old_capabilities, new_capabilities, updated_by |
| `WOM.WorkerTerminated` | Worker is terminated | session_id, reason, terminated_by, evidence_ref |
| `WOM.WorkerQuery` | Worker record is queried | session_id, queried_by, query_type |
| `WOM.TransitionDenied` | Lifecycle transition is denied | session_id, from_state, attempted_to, reason, denied_by |

## Error Codes (R12)

| Code | Description |
|------|-------------|
| WOM_001 | Worker creation failed — template not found or not Active |
| WOM_002 | Worker creation failed — Organization not Active |
| WOM_003 | Worker creation failed — Mission not in valid state |
| WOM_004 | Capability update exceeds Genome bounds |
| WOM_005 | Capability update exceeds Organization capability scope |
| WOM_006 | Cannot terminate Worker — active execution in progress (must pause first) |
| WOM_007 | Worker not found |
| WOM_008 | Transition not allowed — invalid from/to state combination |

## Cross-Cutting Concerns

### Security

Worker creation requires Org authorization. Worker termination requires Org Admin or Security Council. Capability updates are bounded by Genome and Org policy. All operation produce Events for Security Council audit. (Physics/008-Security.md)

### Evidence

Every WOM operation produces an Event. The complete lifecycle of every Worker — from creation through termination — is recorded in the Event Store. Evidence chains trace Worker actions to Mission objectives. (PHI-008)

### Lifecycle

Workers follow the canonical 10-state lifecycle (Physics/006-Lifecycles.md, Foundations/008-Object-Lifecycle.md). All lifecycle invariants apply: ordered states, authorized transitions, state-dependent capabilities, hierarchical constraint by Mission lifecycle. (PHI-006)

### Capability Bounds

Worker capabilities are resolved from three sources with strict precedence. Capabilities are state-dependent — a Worker in Running state has full capability scope; a Worker in Paused or Blocked state has none. Capability updates require authorization and are bounded by the Template Genome. (Physics/007-Capabilities.md)

### Communication

Worker creation and state transition notifications are broadcast via ACF. Workers use ACF for all subsequent communication. WCS manages inter-Worker messaging within Missions. (Law 3 — Communication)

### Design DNA

| Rule | Compliance |
|------|-----------|
| R1 (Modulsingularity) | WOM focuses solely on Worker Session structure and lifecycle |
| R4 (Builder) | Worker construction is separate from execution (AGS → IDS → LMS) |
| R5 (Liskov) | All Worker types implement the Session interface |
| R9 (Deterministic) | Capability resolution is deterministic — same inputs always produce same capability set |
| R10 (Simpler Over Complex) | Linear 10-state lifecycle with clear transition rules |
| R12 (Embrace Errors) | All errors have unique codes (WOM_001–008) |
| R14 (Paved Path) | Paved path: Create → Plan → Assign → Run → Complete → Archive |
