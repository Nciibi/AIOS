## Purpose

The Worker Security Service (WSS) enforces session boundaries and isolation for every Worker Session. Each Worker runs in an isolated execution context — no Worker may access another Worker's state, memory, files, or communications. WSS implements the constitutional requirement that Workers are independent, isolated entities that communicate only through authorized ACF channels.

## Security Model

Every Worker/Session runs in an isolated execution context with the following layers:

| Layer | Mechanism | Enforces |
|-------|-----------|----------|
| Process-Level | Operating system process isolation (sandbox/container) | No cross-Worker memory access |
| Filesystem | Chroot/tmpfs per Worker workspace | No cross-Worker file access |
| Network | ACL per Worker — only ACF endpoints accessible | No direct peer-to-peer networking |
| Capability | Limited capability scope per state | No actions outside granted capabilities |
| Memory | Quota enforcement via Runtime | Resource exhaustion prevention |
| Communication | ACF-only messaging with identity verification | No side channels |

## Isolation Mechanisms

### Process-Level Isolation

Each Worker Session runs as an independent operating system process or container. The Runtime ensures:

- Separate process namespace
- No shared memory segments
- No inter-process communication outside ACF
- Process tree isolation (Worker cannot spawn children outside its sandbox)
- Resource quotas (CPU, memory) per process

### Filesystem Isolation

Each Worker gets an isolated filesystem workspace:

- **Root filesystem**: Read-only base image from Template Genome
- **Workspace**: Read-write tmpfs, scoped to this Worker only
- **Scratch space**: Temporary storage, wiped on termination
- **Evidence output**: Write-only evidence channel (Worker writes evidence, cannot read it back)
- **No access**: To other Workers' filesystems, Organization operational intelligence, or system files beyond declared scope

### Network Isolation

Worker network access is strictly controlled:

- **ACF endpoint**: Only ACF message bus is accessible for communication
- **External access**: Configurable per Template — may be blocked entirely or restricted to specified endpoints
- **No listening sockets**: Workers do not expose network services (defense in depth)
- **DNS restricted**: Only configured resolver for ACF service discovery

### Capability Isolation

Worker actions are bounded by capability scope (Physics/007):

- Every action requires a verified capability
- Capabilities are state-dependent (no capabilities in Paused/Blocked states)
- Capability bounds enforce resource limits per action

## Session Boundaries

| Boundary | Policy | Violation Severity |
|----------|--------|-------------------|
| Memory | Worker may not read/write another Worker's memory allocation | S3 — IsolationViolation |
| Filesystem | Worker may not access files outside its workspace | S3 — BoundaryCrossingAttempt |
| Network | Worker may not connect to another Worker directly | S2 — NetworkViolation |
| Process | Worker may not spawn or attach to another Worker's process | S4 — CriticalViolation |
| ACF | Worker may not send messages outside Mission scope without authorization | S2 — CommunicationViolation |
| Capability | Worker may not exceed granted capability bounds | S2 — CapabilityViolation |

## Sandbox Implementation Details

Each Worker sandbox is configured at creation time based on the Template Genome and Org policy:

| Sandbox Dimension | Configuration Source | Default |
|------------------|---------------------|---------|
| Filesystem size | Genome resource_bounds.storage | 1GB tmpfs |
| Memory limit | Genome resource_bounds.memory | 512MB |
| CPU quota | Genome resource_bounds.compute | 1 core |
| Network access | Genome network_policy | ACF-only |
| Process limit | Genome process_policy | 10 child processes |
| Execution timeout | Genome timeout_policy | 1 hour |

### Sandbox Initialization Flow

```
1. Template Genome loaded → sandbox configuration extracted
2. Runtime creates isolated process/container:
   a. Mount read-only root filesystem
   b. Mount writable tmpfs workspace (isolated per Worker)
   c. Apply resource quotas (cgroups or equivalent)
   d. Configure network namespace (ACF endpoint only)
   e. Set process limits
3. WSS validates sandbox integrity:
   a. Verify filesystem isolation (cannot see other Workers)
   b. Verify memory isolation (cannot attach to other processes)
   c. Verify network isolation (cannot reach non-ACF endpoints)
4. If validation passes → Worker is ready for execution
5. If validation fails → Worker creation rejected with WSS_001
```

## Security Monitoring

WSS continuously monitors Worker execution for security violations:

| Monitor | Detection Method | Response |
|---------|-----------------|----------|
| Memory boundary | Runtime memory isolation checks | S3 escalation if cross-Worker access detected |
| Filesystem boundary | Filesystem access logging | S3 escalation if outside-workspace access detected |
| Network boundary | Network connection logging | S2 escalation if non-ACF connection detected |
| Process boundary | Process tree monitoring | S4 escalation if unauthorized child process detected |
| Capability boundary | Capability verification per action | S1-S2 escalation per violation frequency |
| ACF spoofing | Message identity verification per WSS_004 check | S3 escalation if spoofing detected |

## Security Operations

### validateIsolation

```
Input:  session_id
Process: verify process isolation → verify filesystem isolation → verify network isolation → verify capability isolation
Output: IsolationReport { session_id, layers_verified[], status }
Authorization: WSS (internal), Security Council
```

### enforceBoundary

```
Input:  session_id, boundary_type (memory/filesystem/network/process/acf/capability), violation_type
Process: record violation → apply enforcement action (warn/restrict/suspend/terminate) → escalate
Output: EnforcementResult { action_taken, escalation_level }
Authorization: WSS (automatic based on severity)
Event: WSS.BoundaryEnforced
```

### getIsolationStatus

```
Input:  session_id
Process: query isolation layer status → return
Output: IsolationStatus { session_id, layers{}, last_validation }
Authorization: Worker (self), Org Admin, Security Council
```

### configureIsolation

```
Input:  session_id, isolation_config { filesystem_size?, network_rules?, memory_limit?, capability_bound_overrides? }
Process: validate config against Genome bounds → apply → test → confirm
Output: IsolationConfig { applied_rules }
Authorization: Org Admin or Security Council
Event: WSS.IsolationConfigured
```

## ACF Communication Enforcement

Workers communicate only through ACF. WSS enforces:

1. **Source verification**: Every ACF message from a Worker is verified to originate from that Worker's identity
2. **Scope enforcement**: Messages are checked against Mission scope — Worker can only communicate within its assigned Mission
3. **Authorization check**: Every cross-Worker message requires authorization (WCS manages this)
4. **No direct channels**: WSS monitors for direct IPC, shared memory, or filesystem-based communication between Workers

## Security Escalation Levels

| Severity | Code | Violation Examples | Action |
|----------|------|-------------------|--------|
| S0 — Informational | WSS_SEC_001 | Minor resource quota warning | Logged only |
| S1 — Low | WSS_SEC_002 | First capability bound warning | Logged, Worker warned |
| S2 — Medium | WSS_SEC_003 | Repeated capability violation, unauthorized communication attempt | Capability restricted, Org notified |
| S3 — High | WSS_SEC_004 | Isolation violation, memory boundary crossing, filesystem escape attempt | Worker suspended, Security Council notified, investigation initiated |
| S4 — Critical | WSS_SEC_005 | Process injection, cross-Worker memory access, system compromise attempt | Worker force-terminated, full investigation, AI Council notified |

## Cross-Cutting Concerns

### Security

WSS is the enforcement layer for Worker security. It implements Physics/008-Security invariants at the Worker level. Isolation violations trigger S3-S4 escalation. WSS itself is subject to Security Council audit. (Physics/008-Security.md)

### Evidence

Every WSS operation — validation, violation, enforcement, configuration — produces an Event. All isolation violations are recorded as Security Events with full detail for investigation. (PHI-008)

### Lifecycle

Isolation enforcement is lifecycle-state-dependent. A Worker in Running state has full isolation enforcement. A Worker in Paused or Blocked state has reduced enforcement (filesystem and memory preserved but process suspended). A Worker in Archived state has no active isolation — the execution context is destroyed. (PHI-006)

### Capability Bounds

WSS enforces that Worker actions are bounded by capabilities (Physics/007 Invariant 4). Capability violations are detected and escalated. Isolation configuration cannot exceed the Worker's Genome capability bounds. (Physics/007-Capabilities.md)

### Communication

Workers communicate only through ACF. WSS monitors for non-ACF communication (direct IPC, shared memory, filesystem signaling). ACF message identity is verified against the sending Worker's identity. (Law 3 — Communication)

### Design DNA

| Rule | Compliance |
|------|-----------|
| R1 (Modulsingularity) | WSS focuses solely on Worker security isolation |
| R10 (Simpler Over Complex) | Layered isolation with clear boundaries — no complex security models |
| R12 (Embrace Errors) | All security events have unique codes and severity levels (S0-S4) |
| R13 (Design for Failure) | WSS fails closed — if isolation layer cannot be verified, Worker is denied execution |
| R14 (Paved Path) | Paved path: Validate → Monitor → Detect → Escalate → Enforce |
