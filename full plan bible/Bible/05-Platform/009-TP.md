## Purpose

AGS Genome instantiation. TP takes a signed Genome from AGS (Agent Genome System) and produces a runnable Session. TP is the bridge between AGS (template definitions) and LMS (running sessions). It validates the Genome signature, resolves resources, creates identities, registers with LMS, and returns a Session token.

## TP Flow

```
Genome → Validate Signature → Resolve Resources → Create Identity → Register with LMS → Return Session Token
```

### Step-by-Step

1. **Receive Genome**: TP receives signed Genome (genome_id, overrides) via ACF message from AGS
2. **Validate Signature**: Verify genome signature against AGS public key. Reject if invalid, produce security Event
3. **Validate Schema**: Ensure Genome conforms to Genome schema definition. Reject with schema error if invalid
4. **Resolve Resources**: Query ROS to allocate compute resources specified in the Genome (CPU, memory, timeout)
5. **Create Identity**: Request IDS to create a Session identity for the new Session. Receive identity_id
6. **Assign Capabilities**: Request CCA to grant capabilities specified in the Genome
7. **Register with LMS**: Register the Session entity with LMS in Created state
8. **Generate Token**: Create signed Session token encoding session_id, identity_id, and capabilities
9. **Return Response**: Return session_token to requester with full Session metadata

### Flow Diagram

```
AGS ──Genome──► TP ──Validate──► AGS (public key check)
                   │
                   ├──► ROS ──allocate──► Resources
                   │
                   ├──► IDS ──create──► Identity
                   │
                   ├──► CCA ──grant──► Capabilities
                   │
                   ├──► LMS ──register──► Created State
                   │
                   └──► Return Session Token
```

## Validation Steps

TP validates the Genome at every step:

| Step | Validation | Failure Action | Error Code |
|------|-----------|---------------|------------|
| Signature | Verify against AGS public key | Reject, log security Event | TP-001 |
| Schema | Genome conforms to Genome schema | Reject with schema error | TP-002 |
| Resources | Requested resources are available | Reject, provide availability report | TP-003 |
| Capabilities | Requested capabilities exist | Reject with capability error | TP-004 |
| Policy | Genome complies with active policies | Reject with policy violation | TP-005 |
| Duplicate | Genome already instantiated | Reject with duplicate error | TP-006 |

### Validation Result

```
ValidationResult {
  valid: bool,
  genome_id: UUID,
  checks: ValidationCheck[],
  resource_estimate: ResourceEstimate,
  policy_compliance: PolicyResult,
  errors: ValidationError[]
}

ValidationCheck {
  step: string,
  passed: bool,
  details: string,
  error_code?: string
}
```

## Dry Run

`dryRun` performs all validation steps without creating resources, identities, or Sessions.

Report includes: required resources by type, identity type and metadata, LMS lifecycle path (states the Session will traverse), estimated duration, policy compliance status, potential issues, and cost estimate (token budget, resource hours).

## Resource Resolution

TP resolves resources based on the Genome's resource profile:

| Profile | CPU | Memory | Timeout | Storage | Cost Multiplier |
|---------|-----|--------|---------|---------|-----------------|
| standard | 1 core | 512 MB | 30 min | 100 MB | 1.0 |
| high_mem | 2 cores | 4 GB | 60 min | 1 GB | 2.5 |
| compute_heavy | 4 cores | 8 GB | 120 min | 10 GB | 5.0 |
| custom | Per Genome spec | Per Genome spec | Per Genome spec | Per Genome spec | Variable |

## Session Token

The Session token is a signed JWT containing:

```
{
  "session_id": "UUID",
  "identity_id": "UUID",
  "genome_id": "UUID",
  "issued_at": timestamp,
  "expires_at": timestamp,
  "capabilities": ["cap1", "cap2"],
  "resource_budget": {"tokens": 100000, "compute_hours": 2},
  "issuer": "aios:engine:sec:tp:001",
  "signature": "..."
}
```

Token is signed with TP's private key. Verification is performed by ACF Gateway on every message from the Session. Token expires with the Session.

## Cross-Cutting Concerns

### Security

Genome signatures are verified against AGS public key. TP authenticates all requests through ACF. Identity creation requires Security Council authorization. Session tokens are cryptographically signed and verified.

### Evidence

Every step of the instantiation flow produces an Event. Validation Events include the validation result and relevant metadata. Instantiation Events link the Genome to the resulting Session.

### Lifecycle

TP follows Platform service lifecycle. Each instantiation request follows: Received → Validating → Resourcing → Creating → Registering → Completed/Failed. Sessions created by TP are registered with LMS in Created state.

### Capability Bounds

TP only instantiates Genomes. It does not modify Genomes (AGS does), does not manage Sessions (LMS does), does not execute Sessions (Runtime does), and does not store Genome definitions (AGS does). TP is a pipeline.

### Communication

TP communicates through ACF with all dependencies: AGS (receive Genomes), IDS (create identities), ROS (resolve resources), LMS (register Sessions). All responses are delivered via ACF.

### Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | TP does one thing: instantiate Genomes into Sessions |
| R2 — Dependency Order | TP depends on AGS, IDS, ROS, LMS; no upward deps |
| R3 — DRY | Genome validation rules are defined in one place |
| R4 — Builder Pattern | Sessions are built by TP from Genomes |
| R5 — Liskov | All Genome types produce Sessions implementing Session interface |
| R6 — DI over Singletons | TP receives AGS, IDS, ROS, LMS as injected dependencies |
| R7 — Tests Exist | Every validation step and failure mode has tests |
| R8 — Tests Fast | Instantiation pipeline tests complete in <100ms |
| R9 — Deterministic | Same Genome + same overrides always produces same Session |
| R10 — Simpler Over Complex | TP is a linear pipeline; no branching logic |
| R11 — Refactor Over Rewrite | Instantiation flow evolves via RFC |
| R12 — Embrace Errors | Every failure step has a unique error code |
| R13 — Design for Failure | Failed steps produce Events; instantiation can be retried |
| R14 — Paved Path | TP is the only path for Genome instantiation |
| R15 — Open/Closed | New Genome types extend without modifying TP pipeline |

## TP Invariants

1. **TP-INV-001 — Signed Genome Only**: Every Genome must be cryptographically signed by AGS. Unsigned Genomes are rejected at the first validation step.

2. **TP-INV-002 — Complete Validation**: All validation steps complete before any resource allocation begins. Partial validation is not permitted.

3. **TP-INV-003 — Atomic Instantiation**: If any step in the instantiation flow fails, all prior steps are rolled back. Resources are released, identities are not committed.

4. **TP-INV-004 — Identity per Session**: Every Session receives a unique identity from IDS. No two Sessions share an identity.

5. **TP-INV-005 — Token-Bound Capabilities**: Session tokens encode the exact capabilities granted. Capabilities cannot be expanded after token issuance.
