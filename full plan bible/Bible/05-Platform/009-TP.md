## Purpose

AGS Genome instantiation. TP takes a signed Genome from AGS (Agent Genome System) and produces a runnable Session. TP is the bridge between AGS (template definitions) and LMS (running sessions). It validates the Genome signature, resolves resources, creates identities, registers with LMS, and returns a Session token.

## TP Flow

```
Genome → Validate Signature → Resolve Resources → Create Identity → Register with LMS → Return Session Token
```

### Step-by-Step

1. **Receive Genome**: TP receives a signed Genome (genome_id, overrides) via ACF message
2. **Validate Signature**: Verify genome signature against AGS public key — reject if invalid
3. **Resolve Resources**: Query ROS to allocate compute resources specified in the Genome
4. **Create Identity**: Request IDS to create a Session identity for the new Session
5. **Register with LMS**: Register the Session with LMS in Created state
6. **Return Session Token**: Generate and return a Session token to the requester

## TP Operations

```
instantiateSession(genome_id, overrides) → session_token
dryRun(genome_id) → DryRunResult
getTemplate(session_id) → GenomeMetadata
validateGenome(genome_id) → ValidationResult
getSessionStatus(session_token) → SessionStatus
```

### Instantiation Request Schema

```
InstantiationRequest {
  genome_id: UUID,
  overrides: {
    resource_profile?: string,
    runtime_type?: string,
    capabilities?: string[],
    timeout?: duration,
    labels?: map<string,string>
  },
  requested_by: EntityID
}
```

### Instantiation Response

```
InstantiationResponse {
  session_token: string,
  session_id: UUID,
  identity_id: UUID,
  lms_registration_id: UUID,
  resources: ResourceAllocation,
  expires_at: timestamp
}
```

## Validation Steps

TP validates the Genome at every step:

| Step | Validation | Failure Action |
|------|-----------|---------------|
| Signature | Verify against AGS public key | Reject, log security Event |
| Schema | Genome conforms to Genome schema | Reject with schema error |
| Resources | Requested resources are available | Reject, provide availability |
| Capabilities | Requested capabilities are granted | Reject with capability error |
| Policy | Genome complies with active policies | Reject with policy violation |

## Dry Run

dryRun performs all validation steps without actually creating resources, identities, or Sessions. It returns a detailed report of what would happen, including: required resources, identity type, LMS lifecycle path, estimated duration, and policy compliance.

## TP Events

| Event Type | Produced When | Fields |
|-----------|--------------|--------|
| `TP.GenomeReceived` | A Genome is received for processing | genome_id, requested_by, has_overrides |
| `TP.GenomeValidated` | Genome passes all validation | genome_id, validation_result, resource_estimate |
| `TP.SessionInstantiated` | A Session is successfully created | genome_id, session_id, identity_id, resources |
| `TP.InstantiationFailed` | Session creation fails | genome_id, step_failed, reason, error_code |
| `TP.DryRunCompleted` | A dry run is completed | genome_id, validation_passed, resource_report |
| `TP.GenomeRejected` | A Genome is rejected by policy | genome_id, policy_violations, rejected_by |

## Cross-Cutting Concerns

### Security

Genome signatures are verified against AGS public key. TP authenticates all requests through ACF. Identity creation requires Security Council authorization. Session tokens are cryptographically signed.

### Evidence

Every step of the instantiation flow produces an Event. Validation Events include the validation result and relevant metadata. Instantiation Events link the Genome to the resulting Session.

### Lifecycle

TP follows Platform service lifecycle. Each instantiation request follows: Received → Validating → Resourcing → Creating → Registering → Completed/Failed. Sessions created by TP are registered with LMS in Created state.

### Capability Bounds

TP only instantiates Genomes. It does not modify Genomes (AGS does), does not manage Sessions (LMS does), does not execute Sessions (Runtime does), and does not store Genome definitions (AGS does). TP is a pipeline — it transforms Genomes into Sessions through a defined sequence of steps.

### Communication

TP communicates through ACF with all dependencies: AGS (receive Genomes), IDS (create identities), ROS (resolve resources), LMS (register Sessions). All responses are delivered via ACF.

### Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | TP does one thing: instantiate Genomes into Sessions |
| R2 — Dependency Order | TP depends on AGS, IDS, ROS, LMS; no upward deps |
| R3 — DRY | Genome validation rules are defined in one place |
| R4 — Builder Pattern | Sessions are built by TP from Genomes |
| R5 — Liskov | All Genome types produce Sessions implementing Session interface |
| R6 — DI over Singletons | TP receives AGS, IDS, ROS, LMS as injected dependencies |
| R7 — Tests Exist | Every validation step and failure mode has tests |
| R8 — Tests Fast | Instantiation pipeline tests complete in <100ms |
| R9 — Deterministic | Same Genome + same overrides always produces same Session |
| R10 — Simpler Over Complex | TP is a linear pipeline; no branching logic |
| R11 — Refactor Over Rewrite | Instantiation flow evolves via RFC |
| R12 — Embrace Errors | Every failure step has a unique error code |
| R13 — Design for Failure | Failed steps produce Events; instantiation can be retried |
| R14 — Paved Path | TP is the only path for Genome instantiation |
| R15 — Open/Closed | New Genome types extend without modifying TP pipeline |
