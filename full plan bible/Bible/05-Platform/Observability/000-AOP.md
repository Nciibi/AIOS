## Purpose

The AIOS Observability Platform (AOP) is the platform-wide observability subsystem. It provides metrics collection, structured logging, distributed tracing, health checking, dashboards, alerting, anomaly detection, and forensic event capture. Every component in AIOS produces observability data through AOP вҖ” from the Security Council's verification decisions to ROS resource allocations to Sou's reasoning traces.

AOP does not store data itself. It routes observability data to the appropriate storage backends (Event Store for evidence, log store for debugging, metrics store for dashboards) and provides unified query interfaces across all observability domains.

## Core Concepts

### 1. Metrics Pipeline

AOP collects time-series metrics from every component. Metrics follow a consistent structure:

| Component | Metric Type | Examples |
|-----------|-------------|----------|
| Security Council | Counter, Histogram | `verification.requests.total`, `verification.latency.ms`, `denials.by_reason` |
| ROS | Gauge, Counter | `resources.cpu.utilization`, `memory.allocated.bytes`, `budget.remaining` |
| LLMOS | Histogram, Counter | `inference.latency.ms`, `tokens.consumed`, `model.cache.hit_ratio` |
| Sou | Counter, Gauge | `decisions.issued`, `missions.created`, `context.window.size` |
| Conversation OS | Counter, Histogram | `sessions.active`, `turn.latency.ms`, `modality.distribution` |
| ACF | Counter, Gauge | `messages.routed`, `channels.active`, `delivery.latency.ms` |

Metrics collection follows these rules:

| Rule | Behavior |
|------|----------|
| Every metric has a component prefix | e.g., `ros.cpu.utilization`, not `cpu.utilization` |
| Cardinality is bounded | Tag combinations are pre-registered вҖ” unbounded tags are rejected |
| Sampling is configurable | High-volume metrics can be sampled (default 1:10) |
| Aggregation happens at query time | Raw data points are stored; aggregation functions applied at query |
| Retention is tiered | Raw: 7 days, 1m rollup: 30 days, 1h rollup: 365 days |

### 2. Log Pipeline

All components emit structured logs through AOP. Logs follow the canonical Event format:

| Field | Type | Description |
|-------|------|-------------|
| `timestamp` | RFC3339 | Nanosecond precision |
| `level` | enum | `debug`, `info`, `warn`, `error`, `fatal` |
| `component` | string | e.g., `ros.allocator`, `sou.decision` |
| `message` | string | Human-readable description |
| `context` | JSON | Structured key-value pairs |
| `trace_id` | string | Correlation ID for distributed tracing |
| `span_id` | string | Current span within the trace |

Log levels follow a severity escalation policy:

| Level | Storage | Retention | Alert? |
|-------|---------|-----------|--------|
| debug | Log store | 24 hours | No |
| info | Log store | 7 days | No |
| warn | Log store + Event Store | 30 days | Threshold configurable |
| error | Log store + Event Store | 90 days | Yes (immediate) |
| fatal | Event Store (immutable) | Permanent | Yes (page on-call) |

### 3. Distributed Tracing

AOP supports distributed tracing across asynchronous, ACF-mediated workflows. Every message routed through ACF carries trace context.

| Concept | Description |
|---------|-------------|
| Trace | End-to-end workflow (e.g., a full Sou decision cycle) |
| Span | Single operation within a trace (e.g., "resolve context") |
| Span Context | Trace ID + Span ID + parent Span ID propagated via ACF headers |
| Span Kind | `internal`, `client`, `server`, `producer`, `consumer` |

All traces are sampled. Default sampling rate is 1% for production, 100% for development. Traces involving security-critical operations (verification pipeline, capability grants) are always sampled.

### 4. Health Checking

Every ACF-addressable component exposes health endpoints:

| Endpoint | Protocol | Response | Purpose |
|----------|----------|----------|---------|
| `health/liveness` | ACF query | `200 OK` or `503` | Is the process alive? |
| `health/readiness` | ACF query | `200 OK` or `503` | Is the component ready to serve? |
| `health/dependency` | ACF query | Map of dependency status | Are upstream deps healthy? |

Health checks are called on a configurable interval (default 15s). Three consecutive failures mark the component as unhealthy. Unhealthy components trigger alerts and, if configured, automatic restart via the Recovery Engine.

### 5. Dashboards & Alerting

AOP provides a dashboard framework with built-in and custom dashboards:

| Dashboard | Scope | Audience |
|-----------|-------|----------|
| System Health | All components | Security Council, operators |
| Security Posture | Security Council pipelines | Security auditors |
| Resource Utilization | ROS, providers | Organization managers |
| Sou Activity | Sou decisions, missions | Sou supervisor |
| Conversation | Sessions, turns, latency | User experience team |
| Cost & Budget | ROS cost, budget consumption | Finance, org leads |

Alert rules are configured per dashboard:

| Alert Severity | Response Time | Notification Channel |
|----------------|---------------|---------------------|
| critical | < 1 minute | ACF urgent stream + pager |
| high | < 5 minutes | ACF high-priority stream |
| medium | < 30 minutes | ACF message + dashboard banner |
| low | < 24 hours | Dashboard only |

### 6. Anomaly Detection

AOP includes a statistical anomaly detection engine that establishes baselines for all metrics and alerts on deviation:

| Detection Method | Use Case | Example |
|-----------------|----------|---------|
| Static Threshold | Hard limits | CPU > 95% |
| Seasonal Decomposition | Cyclic patterns | Request volume drops 50% below weekly baseline |
| Moving Window | Recent trend | Error rate 3x above rolling 5m average |
| Rate of Change | Spikes | Memory usage jumps 20% in 30 seconds |

Anomaly alerts are always routed through the Alert Evaluator and can trigger automated responses (e.g., scale up resources, throttle requests).

### 7. Black Box Recorder

The Black Box Recorder is an append-only, tamper-evident capture of all system events leading up to a critical failure. It is the forensic record for post-mortem analysis.

| Feature | Behavior |
|---------|----------|
| Capture trigger | Any `fatal` log, unhandled panic, or manual `RECORD` signal |
| Pre-capture buffer | Retains last 60 seconds of all Events and logs in memory |
| Post-capture window | Continues recording for 30 seconds after trigger |
| Storage | Written to Event Store with `blackbox` stream prefix |
| Retention | 90 days; exportable for long-term forensics |

## Data Model

```typescript
interface MetricPoint {
  name: string                    // e.g., "verification.latency.ms"
  value: number
  timestamp: number               // Unix nanos
  tags: Record<string, string>    // e.g., { component: "policy", stage: "capability" }
  type: "counter" | "gauge" | "histogram"
}

interface LogEvent {
  timestamp: number
  level: "debug" | "info" | "warn" | "error" | "fatal"
  component: string
  message: string
  context: Record<string, unknown>
  trace_id?: string
  span_id?: string
}

interface Span {
  trace_id: string
  span_id: string
  parent_span_id?: string
  name: string
  kind: "internal" | "client" | "server" | "producer" | "consumer"
  start_time: number
  end_time?: number
  status: "ok" | "error" | "unset"
  attributes: Record<string, string>
  events: TraceEvent[]
}

interface HealthStatus {
  component: string
  liveness: "healthy" | "unhealthy"
  readiness: "ready" | "not_ready"
  dependencies: Record<string, "healthy" | "degraded" | "down">
  last_checked: number
  consecutive_failures: number
}

interface AlertRule {
  id: string
  name: string
  metric: string
  condition: "above" | "below" | "change" | "anomaly"
  threshold: number
  duration: string         // e.g., "5m" вҖ” must breach for this long
  severity: "critical" | "high" | "medium" | "low"
  channels: string[]       // ACF stream IDs
}
```

## Interfaces

### AOP API (via ACF)

| Method | Auth | Description |
|--------|------|-------------|
| `recordMetric(metric)` | Any component | Submit a metric point |
| `recordLog(event)` | Any component | Submit a structured log entry |
| `startSpan(name, parent?)` | Any component | Begin a distributed trace span |
| `endSpan(span_id, status)` | Any component | End a trace span |
| `queryMetrics(query)` | Sou, Security Council, operators | Query metrics by name, tags, time range |
| `queryLogs(query)` | Sou, Security Council, operators | Search logs by level, component, time range |
| `queryTraces(trace_id)` | Sou, Security Council, operators | Retrieve full trace by ID |
| `getHealth(component)` | Any component | Query component health status |
| `registerDashboard(config)` | Authenticated admin | Register a custom dashboard |
| `createAlertRule(rule)` | Authenticated admin | Add an alert rule |
| `triggerBlackbox()` | Any component | Manually trigger black box recorder |
| `getBlackboxData(session_id)` | Security Council | Retrieve black box recording |

## Invariants

| ID | Invariant | Enforcement |
|----|-----------|-------------|
| AOP-001 | Every component produces at minimum a liveness health check | Architectural вҖ” ACF registration requires health endpoint |
| AOP-002 | All log levels above `info` are also recorded as Events | Architectural вҖ” AOP collector writes dual destinations |
| AOP-003 | Trace context propagates through every ACF message | Architectural вҖ” ACF middleware injects trace headers |
| AOP-004 | Metrics cardinality is bounded by pre-registered tag schemas | Algorithmic вҖ” unknown tag combinations rejected |
| AOP-005 | AOP is non-blocking вҖ” collector failures never affect the producing component | Architectural вҖ” async write with bounded buffer and drop-on-full |
| AOP-006 | Black box recordings are tamper-evident | Cryptographic вҖ” HMAC-chained event sequence |

## Error Cases

| Condition | Error Code | Behavior |
|-----------|------------|----------|
| Metrics store unreachable | `AOP_METRICS_STORE_DOWN` | Buffer in memory; drop oldest points when full |
| Log store unreachable | `AOP_LOG_STORE_DOWN` | Degrade to stdout; retain last 1000 events in memory |
| Unknown alert rule | `AOP_UNKNOWN_ALERT_RULE` | Return error; do not evaluate |
| Cardinality violation | `AOP_CARDINALITY_EXCEEDED` | Reject metric point; log warning |
| Black box buffer overflow | `AOP_BLACKBOX_OVERFLOW` | Drop oldest events in pre-capture buffer |

## Design DNA

| Rule | Assessment |
|------|-----------|
| R1 вҖ” Modulsingularity | AOP does one thing: collect, route, and query observability data |
| R2 вҖ” Dependency Order | Depends on ACF, Event Store; no upward dependencies |
| R3 вҖ” DRY | Metric/log/trace schemas defined once in AOP data model |
| R4 вҖ” Builder Pattern | Dashboards built by composing metric queries with visualization configs |
| R5 вҖ” Liskov Substitution | Any MetricStore/LogStore/TraceStore implements the interface |
| R6 вҖ” DI over Singletons | Storage backends injected via ACF service registration |
| R9 вҖ” Deterministic | Same query always returns same data (data is immutable) |
| R10 вҖ” Simpler Over Complex | Three observability pillars (metrics, logs, traces) cover all use cases |
| R12 вҖ” Embrace Errors | Every ingestion failure has a unique error code |
| R13 вҖ” Design for Failure | AOP never blocks the producing component; drops data before blocking |
| R14 вҖ” Paved Path | All observability flows through AOP collectors |
| R15 вҖ” Open/Closed | New metric types added via MetricStore interface, not by modifying core |
