## Purpose

The Event Store is the immutable append-only log for all AIOS Events. Every Event produced in the system — lifecycle transitions, security actions, mission progress, interaction records — is stored here. The Event Store is the system of record for all state changes. No Event is ever modified or deleted. The Event Store provides the foundation for audit, observability, and state reconstruction.

## Event Schema

Every Event stored in EVS conforms to the following schema:

| Field | Type | Description |
|-------|------|-------------|
| **event_id** | UUIDv7 | Globally unique, time-sortable identifier |
| **event_type** | string | Fully qualified type (e.g., `LMS.StateChanged`) |
| **stream_id** | UUID | The entity this Event belongs to |
| **version** | int64 | Monotonic version number within the stream |
| **data** | JSONB | Event payload |
| **metadata** | JSONB | Timestamp, causation_id, correlation_id |
| **prev_hash** | string | SHA-256 hash of the previous Event in stream |
| **signature** | string | Cryptographic signature of Event content |

## Event Store Operations

```
appendEvent(stream_id, event_type, data, metadata) → event_id
readStream(stream_id, from_version?, to_version?) → Event[]
readEvent(event_id) → Event
subscribeStream(stream_id, subscriber) → subscription_id
queryEvents(query) → Event[]
getEventCount(stream_id) → int
getStreamInfo(stream_id) → StreamInfo
```

### Append Semantics

```
1. Validate event_type is a registered type
2. Assign monotonic version (stream version + 1)
3. Compute prev_hash from last Event in stream
4. Sign Event content
5. Write to append-only log
6. Acknowledge to producer with event_id
7. Publish to subscribers
```

Events are immutable — no delete or update operations. Any correction must be a new Event linked to the original via causation_id.

## Snapshotting

Long event streams are compressed using periodic snapshots. A snapshot captures the projected state at a specific version. New readers load the latest snapshot then apply Events from the snapshot version forward.

| Parameter | Default | Configurable Per Stream |
|-----------|---------|------------------------|
| Snapshot frequency | Every 100 Events | Yes |
| Snapshot retention | Keep last 3 snapshots | Yes |
| Snapshot storage | Separate stream-snapshots bucket | Yes |

## Event Ordering

### Per-Stream Ordering

Within a stream, Events have monotonic version numbers: 1, 2, 3, ... No gaps. No duplicates. EVS enforces that two Events cannot have the same version in the same stream.

### Global Ordering

Global ordering across streams uses Hybrid Logical Clocks (HLC). Each Event's metadata.timestamp is an HLC value that provides causal ordering across the distributed system. HLC ensures that Events from different streams can be ordered relative to each other.

## Clustering

EVS uses Raft consensus for append operations. The Raft leader handles all writes. Follower nodes serve read queries. Read replicas provide additional query capacity.

| Component | Role | Count |
|-----------|------|-------|
| Raft leaders | Handle append operations | 1 (elected) |
| Raft followers | Replicate + serve reads | 2+ |
| Read replicas | Serve read queries only | Configurable |

Raft ensures linearizable writes — once an Event is acknowledged, it is durable across majority of nodes. Network partitions are handled by Raft's leader election: writes are unavailable during election, reads continue on followers.

## EVS Events

| Event Type | Produced When | Fields |
|-----------|--------------|--------|
| `EVS.EventAppended` | An Event is written to the store | event_id, stream_id, version, event_type |
| `EVS.StreamCreated` | A new event stream is initialized | stream_id, entity_type, created_at |
| `EVS.SnapshotCreated` | A snapshot is taken | stream_id, version, event_count |
| `EVS.SubscriptionActivated` | A subscriber subscribes to a stream | subscription_id, stream_id, subscriber_id |
| `EVS.ReadReplicaAdded` | A new read replica comes online | replica_id, endpoint |

## Cross-Cutting Concerns

### Security

Event Store access requires authentication and authorization. Write access is restricted to authorized Platform services. Read access is scoped by entity authorization. EVS validates signatures on all Events.

### Evidence

The Event Store is the foundation of the evidence system. Every Event is immutable, cryptographically chained, and timestamped. EVS supports audit queries across all stored Events.

### Lifecycle

The Event Store follows Platform service lifecycle: Created → Initialized → Active → Recovering → Terminated. Streams have their own lifecycle: Active → Snapshotting → Archived → Purged (after retention period).

### Capability Bounds

EVS only stores and retrieves Events. It does not interpret Event data, does not enforce business rules, does not trigger actions based on Events. Event interpretation is the responsibility of consumers (AUS, EPG, ACF subscriptions).

### Communication

EVS communicates through ACF for all operations. Append requests arrive via ACF. Event subscriptions are delivered through ACF streams. Raft consensus communication uses a dedicated internal channel.

### Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | EVS does one thing: immutable event storage |
| R2 — Dependency Order | EVS depends on ACF; no upward dependencies |
| R3 — DRY | Events are stored once; consumers read from canonical store |
| R4 — Builder Pattern | Events are built by EventBuilders before append |
| R5 — Liskov | All Event storage backends implement the same EventStore interface |
| R6 — DI over Singletons | EVS receives ACF and Raft transport as injected deps |
| R7 — Tests Exist | Every storage backend, Raft state, and query has tests |
| R8 — Tests Fast | Append and read operations test in <10ms |
| R9 — Deterministic | Same Events in same order always produce same stream |
| R10 — Simpler Over Complex | Append-only log; no indexing, no query optimization |
| R11 — Refactor Over Rewrite | Storage schema evolves via versioned migration |
| R12 — Embrace Errors | Every append failure has a unique error code |
| R13 — Design for Failure | Raft consensus; reads continue during leader election |
| R14 — Paved Path | Event Store is the only path for Event persistence |
| R15 — Open/Closed | New storage backends extend without modifying core |
