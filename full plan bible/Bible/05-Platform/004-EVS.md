## Purpose

The Event Store is the immutable append-only log for all AIOS Events. Every Event produced in the system — lifecycle transitions, security actions, mission progress, interaction records — is stored here. The Event Store is the system of record for all state changes. No Event is ever modified or deleted. The Event Store provides the foundation for audit, observability, and state reconstruction.

## Event Schema

Every Event stored in EVS conforms to the following schema:

| Field | Type | Description |
|-------|------|-------------|
| event_id | UUIDv7 | Globally unique, time-sortable identifier |
| event_type | string | Fully qualified type (e.g., `LMS.StateChanged`) |
| stream_id | UUID | The entity this Event belongs to |
| version | int64 | Monotonic version number within the stream |
| data | JSONB | Event payload |
| metadata | JSONB | Timestamp (HLC), causation_id, correlation_id |
| prev_hash | string | SHA-256 hash of the previous Event in stream |
| signature | string | Cryptographic signature of Event content |

### Event Naming Convention

Events follow the naming convention: `{Domain}.{Entity}.{Action}`

Examples: `LMS.StateChanged`, `Security.Authentication.Success`, `ACF.MessageDelivered`.

### Event Type Registration

Event types must be registered before use. Registration includes: event type name, schema (JSON Schema), retention policy, and audit classification.

## Event Store Operations

```
appendEvent(stream_id, event_type, data, metadata) → event_id
readStream(stream_id, from_version?, to_version?) → Event[]
readStreamByTime(stream_id, from_time, to_time) → Event[]
readEvent(event_id) → Event
readEventByVersion(stream_id, version) → Event
subscribeStream(stream_id, subscriber) → subscription_id
queryEvents(query) → Event[]
queryEventsByType(event_type, from_time, to_time) → Event[]
getEventCount(stream_id) → int
getStreamInfo(stream_id) → StreamInfo
listStreams(filter?) → StreamInfo[]
```

### Append Semantics

```
1. Validate event_type is registered
2. Compute version (current stream version + 1)
3. Compute prev_hash from last event in stream
4. Build event record
5. Sign event content with EVS signing key
6. Write to append-only log via Raft consensus
7. Wait for Raft commit (majority of nodes)
8. Assign HLC timestamp
9. Acknowledge to producer with event_id
10. Publish to subscribers asynchronously
```

Events are immutable — no delete or update operations. Any correction must be a new Event linked to the original via causation_id.

### Read Stream

Returns Events in order from from_version to to_version. If from_version is omitted, returns from version 1. If to_version is omitted, returns to the latest version. Maximum returned: 1000 Events per call (paginate for more).

### Query Events

Supports query by:

| Field | Operator | Example |
|-------|----------|---------|
| event_type | eq, in, prefix | `event_type == "LMS.*"` |
| timestamp | gt, gte, lt, lte, range | `timestamp >= "2026-01-01"` |
| metadata.causation_id | eq | `causation_id == "evt-..."` |
| data.* | depends on schema | `data.entity_type == "session"` |

## Snapshotting

Long event streams are compressed using periodic snapshots:

| Parameter | Default | Configurable | Description |
|-----------|---------|--------------|-------------|
| snapshot_interval | 100 Events | Per-stream | Events between snapshots |
| snapshot_retention | 3 | Global | Last N snapshots kept |
| snapshot_compression | zstd | Per-stream | Compression algorithm |
| snapshot_store | separate bucket | Global | Storage location |

### Snapshot Contents

```
Snapshot {
  stream_id: UUID,
  version: int,
  state: JSONB,         // projected entity state
  event_count: int,     // events from version 1 to this snapshot
  prev_snapshot_version: int?,
  checksum: string
}
```

## Event Ordering

### Per-Stream Ordering

Within a stream, Events have monotonic version numbers: 1, 2, 3, ... No gaps. No duplicates. EVS enforces that two Events cannot have the same version in the same stream.

### Global Ordering

Global ordering across streams uses Hybrid Logical Clocks (HLC). Each Event's metadata.timestamp is an HLC value that provides causal ordering across the distributed system. HLC components:

| Component | Description |
|-----------|-------------|
| wall_time | System clock (millisecond precision) |
| logical_counter | Monotonic counter for same-wall-time Events |
| node_id | Node identifier for tiebreaking |

## Clustering

EVS uses Raft consensus for append operations:

| Component | Role | Count |
|-----------|------|-------|
| Raft leader | Handles all append operations | 1 (elected) |
| Raft followers | Replicate log, serve read queries | 2+ (3 or 5 total) |
| Read replicas | Serve read queries only | Configurable |

### Write Path

```
Client → Raft Leader → Log Append → Replicate to Followers → Commit → Acknowledge
```

### Read Path

```
Client → Any Node (Leader, Follower, Replica) → Read from local store → Return
```

### Raft Configuration

| Parameter | Default | Description |
|-----------|---------|-------------|
| election_timeout | 150-300ms | Randomized election timeout |
| heartbeat_interval | 50ms | Leader heartbeat interval |
| snapshot_interval | 10000 | Raft log compaction interval |
| max_log_size | 100MB | Max Raft log before compaction |

## Retention and Archival

| Tier | Duration | Storage | Access |
|------|----------|---------|--------|
| Hot | 90 days | SSDs, replicated | Sub-millisecond |
| Warm | 1 year | Fast HDDs | <10ms |
| Cold | 7 years | Compressed archive | <1 minute |
| Permanent | Indefinite | Cold + backup | Constitutional Events |

| Event Type | Produced When | Fields |
|-----------|--------------|--------|

## Cross-Cutting Concerns

### Security

Event Store access requires authentication and authorization. Write access is restricted to authorized Platform services. Read access is scoped by entity authorization. EVS validates signatures on all Events.

### Evidence

The Event Store is the foundation of the evidence system. Every Event is immutable, cryptographically chained, and timestamped. EVS supports audit queries across all stored Events.

### Lifecycle

The Event Store follows Platform service lifecycle. Streams have their own lifecycle: Active → Snapshotting → Archiving → Archived → Purging (after retention period).

### Capability Bounds

EVS only stores and retrieves Events. It does not interpret Event data, does not enforce business rules, does not trigger actions based on Events. Event interpretation is the responsibility of consumers.

### Communication

EVS communicates through ACF for all operations. Append requests arrive via ACF. Event subscriptions are delivered through ACF streams. Raft consensus communication uses a dedicated internal channel.

### Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | EVS does one thing: immutable event storage |
| R2 — Dependency Order | EVS depends on ACF; no upward dependencies |
| R3 — DRY | Events are stored once; consumers read from canonical store |
| R4 — Builder Pattern | Events are built by EventBuilders before append |
| R5 — Liskov | All Event storage backends implement the same EventStore interface |
| R6 — DI over Singletons | EVS receives ACF and Raft transport as injected deps |
| R7 — Tests Exist | Every storage backend, Raft state, and query has tests |
| R8 — Tests Fast | Append and read operations test in <10ms |
| R9 — Deterministic | Same Events in same order always produce same stream |
| R10 — Simpler Over Complex | Append-only log; no indexing, no query optimization |
| R11 — Refactor Over Rewrite | Storage schema evolves via versioned migration |
| R12 — Embrace Errors | Every append failure has a unique error code |
| R13 — Design for Failure | Raft consensus; reads continue during leader election |
| R14 — Paved Path | Event Store is the only path for Event persistence |
| R15 — Open/Closed | New storage backends extend without modifying core |
