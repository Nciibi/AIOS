## Purpose

Emergency override procedures for exceptional situations. Breaking Glass allows temporary suspension of normal security controls for emergency response. BG is the constitutional safety valve — it provides a controlled, auditable mechanism for bypassing standard procedures when the system is in crisis. Every BG action is time-bounded, fully audited, and subject to mandatory review.

## Trigger Conditions

Breaking Glass may be triggered under the following conditions:

| Trigger | Authorization Required | Max Duration | Auto-Revoke |
|---------|----------------------|--------------|-------------|
| Security Council majority | 2/3 vote of Security Council | 1 hour | Yes |
| Catastrophic system failure | Automatic (any Security Council member) | 30 minutes | Yes |
| Active security breach | Security Council chair | 1 hour | Yes |
| Verified existential threat | Security Council unanimous | 4 hours | Yes |
| Emergency maintenance | Security Council majority | 2 hours | Yes |

### Trigger Evaluation

```
1. Trigger condition detected (automatic monitor or manual report)
2. Evaluate trigger against defined conditions
3. If automatic trigger: notify Security Council within 30s
4. If manual trigger: collect authorization signatures
5. Verify authorization meets threshold
6. Create BG session with time bound
7. Notify all constitutional entities of BG activation
```

## Breaking Glass Actions

| Action | Description | Bypass |
|--------|-------------|--------|
| **Bypass verification** | Skip verification pipeline for specified operation | Stage 3 (Identity) through Stage 6 (Capability) |
| **Override policy** | Temporarily suspend a specific policy | Single policy by ID |
| **Emergency shutdown** | Force-terminate specified entities | LMS force-transition to Terminated |
| **Activate backup** | Fail over to backup systems | PSAP route switch |
| **Resource override** | Allocate resources beyond normal limits | ROS allocation limit override |
| **Emergency access** | Grant temporary access to secured data | ACF authorization bypass |
| **System pause** | Halt non-critical operations | LMS pause all non-critical entities |

### Action Constraints

Every BG action is subject to:

- **Time-bounded**: Auto-revokes after max duration
- **Scope-limited**: Specifies exact entities and operations
- **Audit-logged**: Every action produces detailed audit record
- **Revocable**: Security Council may revoke early
- **Non-recurring**: Each action requires fresh authorization
- **Minimal privilege**: Least privilege necessary for emergency

```

  bg_session_id: UUID,
  trigger: TriggerCondition,
  authorizations: Authorization[],
  duration: duration,
  expires_at: timestamp,
  actions: BGAction[],
  status: active | revoked | expired,
  created_at: timestamp,
  revoked_at?: timestamp,
  review_completed?: boolean
}
```

### Authorization Flow

```
1. Trigger condition detected (automatic or manual)
2. Required authorizing entities identified based on trigger
3. Authorization requests sent to each entity via secure channel
4. Each entity approves/denies with cryptographic signature
5. If threshold met, BG session created with time bound
6. Emergency actions executable within session
7. All actions logged in real-time
8. Session auto-revokes at time bound
9. Post-action review triggered immediately
```

## Post-Action Review

Every BG action requires a post-action review within 24 hours:

| Review Component | Responsible | Deadline | Deliverable |
|-----------------|-------------|----------|-------------|
| Incident report | Security Council secretary | 24 hours | Detailed incident report |
| Constitutional review | CLS | 48 hours | Constitutional compliance assessment |
| Evidence preservation | AUS | Immediate | All BG Events archived at critical tier |
| Lessons learned | Academy | 72 hours | Incident analysis and recommendations |
| Corrective actions | Security Council | 7 days | Action items with owners and deadlines |

### Review Findings

| Finding | Implication |
|---------|-------------|
| Justified — no alternative | No further action, document rationale |
| Justified — process improvement | Update procedures, train operators |
| Partially justified | Revise trigger conditions, improve monitoring |
| Unjustified | Escalate to full Security Council investigation |
| Malicious | Escalate to constitutional violation process |

## BG Invariants

1. **BG-INV-001 — Time-Bound**: Every BG session has a hard expiration. No session may exceed its duration.

2. **BG-INV-002 — Multi-Party Authorization**: No single entity may authorize a BG action alone. Minimum 2/3 of Security Council.

3. **BG-INV-003 — Full Audit**: Every BG action produces at minimum: authorization Event, action Event, revocation/expiry Event, and review Event.

4. **BG-INV-004 — Scoped Impact**: BG actions must specify the minimum scope necessary. Broad scopes require higher authorization.

5. **BG-INV-005 — Mandatory Review**: Every BG activation requires a post-action review. No BG session closes without review.

## Cross-Cutting Concerns

### Security

BG is the highest-security operation in the system. Authorization requires cryptographic signatures from multiple entities. Every action is logged and monitored in real-time. The Security Council can revoke any active BG session instantly.

### Evidence

Every BG action produces multiple Events: authorization, action execution, revocation/expiry, and review. BG Events are classified as critical security Events with 7-year retention. All BG evidence is tamper-proof.

### Lifecycle

BG sessions follow: Requested → Authorized → Active → Revoked/Expired → Reviewed → Closed. BG actions within a session each have their own lifecycle: Queued → Executing → Completed → Audited.

### Capability Bounds

BG is limited to emergency override of security controls. It cannot modify the Constitution, destroy evidence, or permanently alter system configuration without normal governance processes. BG has no authority outside its designated emergency scope.

### Communication

BG communicates through a dedicated secure ACF channel. All BG messages are encrypted end-to-end. Authorization requests and approvals use cryptographic signatures. BG does not communicate with external systems.
