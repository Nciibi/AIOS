## Purpose

Emergency override procedures for exceptional situations. Breaking Glass allows temporary suspension of normal security controls for emergency response. BG is the constitutional safety valve — it provides a controlled, auditable mechanism for bypassing standard procedures when the system is in crisis. Every BG action is time-bounded, fully audited, and subject to mandatory review.

## Trigger Conditions

Breaking Glass may be triggered under the following conditions:

| Trigger | Authorization Required | Max Duration |
|---------|----------------------|--------------|
| **Security Council majority** | 2/3 vote of Security Council | 1 hour |
| **Catastrophic system failure** | Automatic (any Security Council member) | 30 minutes |
| **Active security breach** | Security Council chair | 1 hour |
| **Verified existential threat** | Security Council unanimous | 4 hours |

## Breaking Glass Actions

| Action | Description | Effect |
|--------|-------------|--------|
| **Bypass verification** | Skip verification pipeline for specified operation | Emergency execution without full verification |
| **Override policy** | Temporarily suspend a specific policy | Policy bypassed for specified scope |
| **Emergency shutdown** | Force-terminate specified entities | Immediate termination with evidence preservation |
| **Activate backup** | Fail over to backup systems | System continues on backup infrastructure |
| **Resource override** | Allocate resources beyond normal limits | Emergency resource allocation |

### Action Constraints

Every BG action is subject to:
- **Time-bounded**: Auto-revokes after max duration
- **Scope-limited**: Specifies exact entities and operations affected
- **Audit-logged**: Every action produces a detailed audit record
- **Revocable**: Security Council may revoke early

## BG Operations

```
authorizeBreakGlass(trigger_condition, authorizing_entities) → bg_session_id
executeEmergencyAction(bg_session_id, action, target, params) → result
revokeBreakGlass(bg_session_id) → void
getBreakGlassHistory(filter?) → BGSession[]
getActiveBreakGlassSessions() → BGSession[]
```

### Authorization Flow

```
1. Trigger condition detected (automatic or manual)
2. Required authorizing entities identified
3. Authorization requests sent to each entity
4. Entities approve with digital signature
5. BG session created with time bound
6. Emergency actions executable within session
7. Session auto-revokes at time bound
8. Post-action review triggered
```

## Post-Action Review

Every BG action requires a post-action review within 24 hours:

| Review Component | Responsible | Deadline |
|-----------------|-------------|----------|
| Incident report | Security Council secretary | 24 hours |
| Constitutional review | CLS | 48 hours |
| Evidence preservation | AUS | Immediate |
| Lessons learned | Academy | 72 hours |
| Corrective actions | Security Council | 7 days |

## BG Events

| Event Type | Produced When | Fields |
|-----------|--------------|--------|
| `BG.Authorized` | A BG session is authorized | bg_session_id, trigger, authorizing_entities, duration, expires_at |
| `BG.ActionExecuted` | A BG action is performed | bg_session_id, action, target, params, result |
| `BG.ActionRevoked` | A BG session is revoked early | bg_session_id, revoked_by, reason |

## Cross-Cutting Concerns

### Security

BG is the highest-security operation in the system. Authorization requires cryptographic signatures from multiple entities. Every action is logged and monitored in real-time. The Security Council can revoke any active BG session instantly.

### Evidence

Every BG action produces multiple Events: authorization, action execution, revocation/expiry, and review. BG Events are classified as critical security Events with 7-year retention. All BG evidence is tamper-proof.

### Lifecycle

BG sessions follow: Requested → Authorized → Active → Revoked/Expired → Reviewed → Closed. BG actions within a session each have their own lifecycle: Queued → Executing → Completed → Audited.

### Capability Bounds

BG is limited to emergency override of security controls. It cannot modify the Constitution, destroy evidence, or permanently alter system configuration without normal governance processes. BG has no authority outside its designated emergency scope.

### Communication

BG communicates through a dedicated secure ACF channel. All BG messages are encrypted end-to-end. Authorization requests and approvals use cryptographic signatures. BG does not communicate with external systems.
