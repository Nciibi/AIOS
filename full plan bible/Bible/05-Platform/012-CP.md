## Purpose

Credential issuance and verification for entities that need to authenticate to external systems. CP manages the lifecycle of credentials — API keys, OAuth2 tokens, certificates, and passwords. CP stores credential metadata (not secrets — secrets are stored in SSM). CP ensures that every credential is requested, issued, tracked, rotated, and revoked in a constitutional manner.

## Credential Types

| Type | Description | Use Case |
|------|-------------|----------|
| **API key** | Long-lived key for external service | Service-to-service auth |
| **OAuth2 token** | Short-lived token for SSO | User delegation, OAuth flows |
| **Certificate** | X.509 for mTLS | Mutual TLS authentication |
| **Password** | Legacy credential | Backward compatibility |

## Credential Lifecycle

```
Requested → Issued → Active → Rotated → Revoked → Expired
```

| State | Description |
|-------|-------------|
| **Requested** | Credential requested, pending approval |
| **Issued** | Credential has been generated and delivered |
| **Active** | Credential is valid and can be used |
| **Rotated** | Credential has been replaced with a new one |
| **Revoked** | Credential is explicitly invalidated |
| **Expired** | Credential reached its time-based expiration |

## CP Operations

```
requestCredential(entity_id, type, external_system, config) → request_id
issueCredential(request_id) → credential
rotateCredential(credential_id, reason) → new_credential
revokeCredential(credential_id, reason) → void
listCredentials(entity_id) → CredentialMetadata[]
getCredentialStatus(credential_id) → CredentialStatus
validateCredential(credential_id) → ValidationResult
```

### Security Model

CP stores credential metadata only — actual secrets are stored in SSM (Secret Storage Module). CP holds:

| Stored in CP | Stored in SSM |
|-------------|---------------|
| Credential ID | API key value |
| Entity ID | OAuth2 token secret |
| External system | Certificate private key |
| Issue/expiry dates | Password hash |
| Rotation history | Session keys |
| Status | — |

### Encryption

All credential metadata stored in CP is encrypted at rest using AES-256-GCM. Transmission is over ACF secure channels only. Credentials are never logged, never included in Events, and never exposed in error messages.

## Credential Operations

### Rotation Policy

| Credential Type | Rotation Interval | Grace Period |
|----------------|------------------|--------------|
| API key | 90 days | 7 days overlap |
| OAuth2 token | Per OAuth spec | Token refresh |
| Certificate | 1 year | 30 days overlap |
| Password | 180 days | Immediate |

Rotation is automatic. CP generates new credentials 7 days before expiry for active credentials. Both old and new credentials are valid during the grace period.

### Revocation

Revocation is immediate. Revoked credentials are added to a revocation list. All authentication attempts with revoked credentials are denied. Revocation reasons are logged and auditable.

## Cross-Cutting Concerns

### Security

CP is a security-critical service. Credential metadata is encrypted at rest. Actual secrets are stored in SSM, not CP. All operations require authentication. Unauthorized credential requests are security Events. Credential issuance requires appropriate authorization.

### Evidence

Every credential lifecycle transition produces an Event. Credential metadata is included in Events (not secrets). Rotation and revocation Events are audit Events with 7-year retention.

### Lifecycle

Credentials follow the defined lifecycle. CP follows Platform service lifecycle. Credential type definitions are versioned.

### Capability Bounds

CP only manages credential metadata and lifecycle. It does not store secrets (SSM does), does not authenticate (ATS does), does not issue identities (IDS does). CP capabilities are limited to: request, issue, rotate, revoke, and track credentials.

### Communication

CP communicates through ACF. Credential requests and responses are delivered via ACF. CP communicates with SSM for secret storage through a secure internal channel. All credential transmission is over encrypted channels.

### Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | CP does one thing: credential lifecycle management |
| R2 — Dependency Order | CP depends on ACF and SSM; no upward deps |
| R3 — DRY | Credential metadata is stored once in CP |
| R4 — Builder Pattern | Credentials are built by CredentialBuilders with validation |
| R5 — Liskov | All credential types implement the same Credential interface |
| R6 — DI over Singletons | CP receives ACF and SSM as injected dependencies |
| R7 — Tests Exist | Every credential type and lifecycle transition has tests |
| R8 — Tests Fast | Credential operations complete in <10ms |
| R9 — Deterministic | Same inputs always produce same credential metadata |
| R10 — Simpler Over Complex | Credential lifecycle is linear; no branching state logic |
| R11 — Refactor Over Rewrite | Credential type schemas evolve via versioned updates |
| R12 — Embrace Errors | Every credential failure has a unique error code |
| R13 — Design for Failure | CP fails closed; unavailable = deny |
| R14 — Paved Path | CP is the only path for credential management |
| R15 — Open/Closed | New credential types extend without modifying CP core |
