## Purpose

Credential issuance and verification for entities that need to authenticate to external systems. CP manages the lifecycle of credentials — API keys, OAuth2 tokens, certificates, and passwords. CP stores credential metadata (not secrets — secrets are stored in SSM). CP ensures that every credential is requested, issued, tracked, rotated, and revoked in a constitutional manner.

## Credential Types

| Type | Description | Use Case | Default TTL |
|------|-------------|----------|-------------|
| **API key** | Long-lived key for external service | Service-to-service auth | 90 days |
| **OAuth2 token** | Short-lived token for SSO | User delegation, OAuth flows | Per OAuth spec |
| **Certificate** | X.509 for mTLS | Mutual TLS authentication | 1 year |
| **Password** | Legacy credential | Backward compatibility | 180 days |
| **Session token** | Temporary session credential | Runtime sessions | Session duration |

### Credential Type Schema

```
CredentialType {
  type_name: string,
  metadata_schema: JSON Schema,    // required metadata fields
  rotation_interval: duration,
  grace_period: duration,
  max_active: int,                 // max concurrent active credentials
  requires_approval: bool          // requires Security Council approval
}
```

## Credential Lifecycle

```
Requested → Approved → Issued → Active → Rotating → Rotated → Active
                                              → Revoked → Expired
                                              → Expired
```

| State | Description | Valid for Auth? |
|-------|-------------|-----------------|
| **Requested** | Pending approval | No |
| **Approved** | Approved by authority, not yet issued | No |
| **Issued** | Generated and delivered to requester | Yes (immediately) |
| **Active** | Credential is valid | Yes |
| **Rotating** | Old credential being replaced | Yes (both valid) |
| **Rotated** | Replacement complete, old one deactivated | No |
| **Revoked** | Explicitly invalidated before expiry | No |
| **Expired** | Reached time-based expiry | No |

## Security Model

CP stores credential metadata only — actual secrets are stored in SSM:

### Stored in CP

| Field | Example |
|-------|---------|
| Credential ID | `cred-001-a3f2c9d1` |
| Entity ID | `aios:session:worker:004` |
| External system | `github.com` |
| Issue/expiry dates | `2026-01-01T00:00:00Z` / `2026-04-01T00:00:00Z` |
| Rotation history | `[2026-01-01, 2026-04-01, ...]` |
| Status | `active` |
| Last used | `2026-03-15T12:00:00Z` |

### Stored in SSM (not CP)

| Secret | Description |
|--------|-------------|
| API key value | `sk-abc123...` |
| OAuth2 token secret | `oauth_secret_xyz...` |
| Certificate private key | `-----BEGIN PRIVATE KEY-----...` |
| Password hash | `$2a$12$...` |
| Session key | `session_key_abc...` |

### Encryption

| Data | Encryption | Key |
|------|------------|-----|
| CP metadata at rest | AES-256-GCM | CP master key |
| CP metadata in transit | TLS 1.3 (ACF) | Session key |
| Secrets in SSM | AES-256-GCM | SSM master key |
| Secrets in transit | TLS 1.3 (internal) | mTLS |

## Rotation Policy

| Credential Type | Rotation Interval | Grace Period | Overlap |
|----------------|------------------|--------------|---------|
| API key | 90 days | 7 days | Both valid |
| OAuth2 token | Per OAuth spec | Token refresh | Refresh window |
| Certificate | 1 year | 30 days | Both valid |
| Password | 180 days | Immediate | Immediate switch |

### Automatic Rotation

CP automates rotation:

```
1. Schedule check runs daily
2. Find credentials within 7 days of expiry
3. Generate new credential (new value in SSM, new metadata in CP)
4. Mark old credential as "rotating" (still valid)
5. Deliver new credential to entity
6. Entity switches to new credential
7. Old credential marked "rotated" after grace period
8. SSM secret archived (not deleted)
```

## Revocation

Revocation is immediate. Process:

```
1. Revocation request received with credential_id and reason
2. Verify requesting entity has authority
3. Mark credential as revoked in CP
4. Add credential ID to revocation list (CDN-distributed for performance)
5. Delete secret reference from SSM (optional, configurable)
6. Notify entity of revocation
7. Produce CP.CredentialRevoked Event
```

## Cross-Cutting Concerns

### Security

CP is a security-critical service. Credential metadata is encrypted at rest. Actual secrets are stored in SSM, not CP. All operations require authentication. Unauthorized credential requests are security Events.

### Evidence

Every credential lifecycle transition produces an Event. Credential metadata is included in Events (not secrets). Rotation and revocation Events are audit Events with 7-year retention.

### Lifecycle

Credentials follow the defined lifecycle. CP follows Platform service lifecycle. Credential type definitions are versioned.

### Capability Bounds

CP only manages credential metadata and lifecycle. It does not store secrets (SSM does), does not authenticate (ATS does), does not issue identities (IDS does).

### Communication

CP communicates through ACF. Credential requests and responses are delivered via ACF. CP communicates with SSM for secret storage through a secure internal channel.

### Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | CP does one thing: credential lifecycle management |
| R2 — Dependency Order | CP depends on ACF and SSM; no upward deps |
| R3 — DRY | Credential metadata is stored once in CP |
| R4 — Builder Pattern | Credentials are built by CredentialBuilders with validation |
| R5 — Liskov | All credential types implement the same Credential interface |
| R6 — DI over Singletons | CP receives ACF and SSM as injected dependencies |
| R7 — Tests Exist | Every credential type and lifecycle transition has tests |
| R8 — Tests Fast | Credential operations complete in <10ms |
| R9 — Deterministic | Same inputs always produce same credential metadata |
| R10 — Simpler Over Complex | Credential lifecycle is linear; no branching state logic |
| R11 — Refactor Over Rewrite | Credential type schemas evolve via versioned updates |
| R12 — Embrace Errors | Every credential failure has a unique error code |
| R13 — Design for Failure | CP fails closed; unavailable = deny |
| R14 — Paved Path | CP is the only path for credential management |
| R15 — Open/Closed | New credential types extend without modifying CP core |
