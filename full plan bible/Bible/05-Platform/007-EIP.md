## Purpose

Connectors to external systems for Event import and export. EIP provides a standardized way to send Events to external systems (event sink), receive Events from external systems (event source), or synchronize Events bidirectionally (event bridge). EIP is the gateway between AIOS and the external world — all cross-system Event communication flows through EIP.

## Integration Patterns

| Pattern | Direction | Description |
|---------|-----------|-------------|
| **Event Sink** | AIOS → External | Send Events to external systems |
| **Event Source** | External → AIOS | Import Events from external systems |
| **Event Bridge** | Bidirectional | Synchronize Events both directions |

## Supported Connectors

| Connector | Protocol | Use Case |
|-----------|----------|----------|
| **Webhook** | HTTP/S | Simple push to external endpoints |
| **Kafka** | Kafka protocol | High-throughput event streaming |
| **MQTT** | MQTT 5.0 | IoT and lightweight messaging |
| **AMQP** | AMQP 1.0 | Enterprise message broker integration |
| **REST** | HTTP/S | RESTful API integration |
| **gRPC** | gRPC | High-performance service-to-service |

## Connector Lifecycle

```
Configured → Active → Failed → Disconnected → Reconfigured → Archived
```

| State | Description |
|-------|-------------|
| **Configured** | Connector is defined but not running |
| **Active** | Connector is connected and processing Events |
| **Failed** | Connector encountered an error (auto-retry) |
| **Disconnected** | Connector lost connection (manual reconnect) |
| **Reconfigured** | Connector settings have been updated |
| **Archived** | Connector is decommissioned, config preserved |

## EIP Operations


## Security Requirements

All external connectors require:

1. **Security Council authorization** — explicit approval before activation
2. **mTLS** — mutual TLS for all connections
3. **Rate limiting** — configurable per connector (default 1000 Events/s)
4. **Payload validation** — schema validation for incoming Events
5. **Authentication** — API key or certificate-based auth for outgoing Events

### Security Violations

| Violation | Action |
|-----------|--------|
| Unauthorized activation attempt | Deny, log Security Event, notify Security Council |
| Rate limit exceeded | Throttle, log warning, notify if persistent |
| Invalid payload | Reject, log error, quarantine |
| Certificate expiration | Disconnect, notify, require renewal |
| Suspicious traffic pattern | Disconnect, escalate to Security Council |

## Event Transformation

EIP performs standard transformations at the boundary:

| Transformation | Direction | Description |
|---------------|-----------|-------------|
| **Schema mapping** | Both | Map between internal and external Event schemas |
| **Field renaming** | Both | Map field names to external conventions |
| **Type coercion** | Inbound | Convert external types to internal types |
| **Enrichment** | Outbound | Add metadata (origin, trace_id) before sending |

## Cross-Cutting Concerns

### Security

EIP is the highest-risk Platform service because it bridges internal and external systems. All connectors require Security Council authorization. mTLS is mandatory. Rate limiting prevents abuse. All Event transformations are validated.

### Evidence

Every Event passing through EIP produces a ConnectorEvent — both inbound and outbound. Connector lifecycle Events are recorded. Security violations are immediately escalated.

### Lifecycle

Connectors follow the defined connector lifecycle. Connector definitions are versioned. Active connectors can be updated with zero-downtime reconnection.

### Capability Bounds

EIP only handles Event transport across system boundaries. It does not interpret Event meaning, does not store Events (EVS does), and does not modify Event semantics (only schema mapping). EIP has no authority over internal system state.

### Communication

EIP connects to ACF internally and external systems through configured connector protocols. Internal Event streams are consumed from and published to ACF. External communication uses the configured protocol with mTLS.

### Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | EIP does one thing: external Event integration |
| R2 — Dependency Order | EIP depends on ACF and Security Council; no upward deps |
| R3 — DRY | Connector config is the single source for integration settings |
| R4 — Builder Pattern | Connectors are built by ConnectorBuilders with validation |
| R5 — Liskov | All connectors implement the same Connector interface |
| R6 — DI over Singletons | EIP receives ACF as injected dependency |
| R7 — Tests Exist | Every connector type and transformation has tests |
| R8 — Tests Fast | Connector unit tests complete in <10ms |
| R9 — Deterministic | Same Event + same mapping always produces same output |
| R10 — Simpler Over Complex | Connectors are stateless; no complex state machines |
| R11 — Refactor Over Rewrite | Connector schemas evolve via versioned updates |
| R12 — Embrace Errors | Every connector failure has a unique error code |
| R13 — Design for Failure | Connectors retry with backoff; circuit breaker pattern |
| R14 — Paved Path | EIP is the only path for external Event integration |
| R15 — Open/Closed | New connector types extend without modifying EIP core |
