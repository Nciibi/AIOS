## Purpose

Connectors to external systems for Event import and export. EIP provides a standardized way to send Events to external systems (event sink), receive Events from external systems (event source), or synchronize Events bidirectionally (event bridge). EIP is the gateway between AIOS and the external world — all cross-system Event communication flows through EIP.

## Integration Patterns

| Pattern | Direction | Description | Use Case |
|---------|-----------|-------------|----------|
| **Event Sink** | AIOS → External | Push Events to external system | Webhook notifications, audit export |
| **Event Source** | External → AIOS | Pull Events from external system | Import external events, webhook receiver |
| **Event Bridge** | Bidirectional | Sync Events both directions | Federation with external AIOS instance |

## Supported Connectors

| Connector | Protocol | Authentication | Use Case |
|-----------|----------|---------------|----------|
| **Webhook** | HTTP/S | HMAC signature, API key | Simple push notifications |
| **Kafka** | Kafka protocol | SASL/SSL | High-throughput event streaming |
| **MQTT** | MQTT 5.0 | Certificate | IoT and lightweight messaging |
| **AMQP** | AMQP 1.0 | SASL | Enterprise message broker |
| **REST** | HTTP/S | OAuth2, API key | RESTful API integration |
| **gRPC** | gRPC | mTLS | High-performance service integration |

## Connector Lifecycle

```
Configured → Validated → Connecting → Active → Failed → Disconnected → Reconfigured → Archived
```

| State | Description | Can Communication Flow? |
|-------|-------------|------------------------|
| **Configured** | Connector defined, not running | No |
| **Validated** | Configuration passes validation | No |
| **Connecting** | Connection handshake in progress | No |
| **Active** | Connected and processing Events | Yes |
| **Failed** | Error state, auto-retry in progress | No |
| **Disconnected** | Connection lost, manual intervention needed | No |
| **Reconfigured** | Settings updated, pending reactivation | No |
| **Archived** | Decommissioned, config preserved for audit | No |

## Security Requirements

All external connectors require:

| Requirement | Specification | Enforcement |
|-------------|---------------|-------------|
| Security Council authorization | Explicit approval | Required before activation |
| mTLS | Mutual TLS for all connections | Enforced at transport level |
| Rate limiting | Default 1000 Events/s | Token bucket algorithm |
| Payload validation | JSON Schema validation | Reject invalid payloads |
| Authentication | API key or certificate | Verified on every request |
| Audit logging | All connector Events | Automatic |

### Security Violations

| Violation | Severity | Response |
|-----------|----------|----------|
| Unauthorized activation | Critical | Deny, log, notify Security Council |
| Rate limit exceeded | Warning | Throttle, log, notify if persistent |
| Invalid payload | Warning | Reject, log, quarantine Event |
| Certificate expired | Critical | Disconnect, notify, require renewal |
| Suspicious traffic pattern | Critical | Disconnect, escalate to Security Council |
| Authentication failure | Warning | Log, increment failure counter |
| Data exfiltration attempt | Critical | Block connection, full investigation |

## Event Transformation

EIP performs standard transformations at the boundary:

| Transformation | Direction | Description | Example |
|---------------|-----------|-------------|---------|
| Schema mapping | Both | Map between internal and external schemas | `event_type` → `type` |
| Field renaming | Both | Adapt field names | `entity_id` → `object_id` |
| Type coercion | Inbound | Convert external types to internal | string timestamp → HLC |
| Enrichment | Outbound | Add metadata for external consumption | Add `origin_instance`, `trace_id` |
| Filtering | Both | Include/exclude specific Event types | Only export `audit:true` Events |

## Rate Limiting

| Scope | Default | Enforcement |
|-------|---------|-------------|
| Per connector | 1000 Events/s | Token bucket |
| Per instance | 10000 Events/s | Aggregate across connectors |
| Per external system | 5000 Events/s | Per target endpoint |
| Burst allowance | 2x rate for 10s | Burst bucket |

## Cross-Cutting Concerns

### Security

EIP is the highest-risk Platform service because it bridges internal and external systems. All connectors require Security Council authorization. mTLS is mandatory. Rate limiting prevents abuse. All Event transformations are validated.

### Evidence

Every Event passing through EIP produces a ConnectorEvent — both inbound and outbound. Connector lifecycle Events are recorded. Security violations are immediately escalated.

### Lifecycle

Connectors follow the defined connector lifecycle. Connector definitions are versioned. Active connectors can be updated with zero-downtime reconnection.

### Capability Bounds

EIP only handles Event transport across system boundaries. It does not interpret Event meaning, does not store Events (EVS does), and does not modify Event semantics (only schema mapping). EIP has no authority over internal system state.

### Communication

EIP connects to ACF internally and external systems through configured connector protocols. Internal Event streams are consumed from and published to ACF. External communication uses the configured protocol with mTLS.

### Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | EIP does one thing: external Event integration |
| R2 — Dependency Order | EIP depends on ACF and Security Council; no upward deps |
| R3 — DRY | Connector config is the single source for integration settings |
| R4 — Builder Pattern | Connectors are built by ConnectorBuilders with validation |
| R5 — Liskov | All connectors implement the same Connector interface |
| R6 — DI over Singletons | EIP receives ACF as injected dependency |
| R7 — Tests Exist | Every connector type and transformation has tests |
| R8 — Tests Fast | Connector unit tests complete in <10ms |
| R9 — Deterministic | Same Event + same mapping always produces same output |
| R10 — Simpler Over Complex | Connectors are stateless; no complex state machines |
| R11 — Refactor Over Rewrite | Connector schemas evolve via versioned updates |
| R12 — Embrace Errors | Every connector failure has a unique error code |
| R13 — Design for Failure | Connectors retry with backoff; circuit breaker pattern |
| R14 — Paved Path | EIP is the only path for external Event integration |
| R15 — Open/Closed | New connector types extend without modifying EIP core |
