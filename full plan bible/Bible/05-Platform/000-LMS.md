## Purpose

The Lifecycle Management Service manages the operational lifecycle of every constitutional entity. It is the state machine engine responsible for validating, authorizing, and executing lifecycle transitions, maintaining the authoritative current state for every entity, and producing immutable Events for every transition. LMS is the constitutional authority for lifecycle management — no entity bypasses LMS for state transitions.

## Canonical 10-State Lifecycle

```
Created → Planned → Assigned → Running → Waiting → Paused → Blocked → Review → Completed → Archived
```

| State | Description | Can Act? | Terminal? | Resource Budget |
|-------|-------------|----------|-----------|-----------------|
| **Created** | Entity record exists, identity assigned | No | No | None |
| **Planned** | Purpose, scope, and resources are defined | No | No | None |
| **Assigned** | Resources and capabilities are allocated | No | No | Allocated, not drawable |
| **Running** | Entity is actively performing its function | Yes | No | Active and drawable |
| **Waiting** | Entity waits for external input or dependency | No | No | Frozen |
| **Paused** | Entity execution is suspended, resumable | No | No | Frozen |
| **Blocked** | Entity cannot proceed due to unresolved issue | No | No | Frozen |
| **Review** | Entity requires review before completion | Limited | No | Active (limited) |
| **Completed** | Entity has finished its function successfully | No | No | Returned to pool |
| **Archived** | Entity record preserved for audit, not operational | No | Yes | Evidenced budget |

### State Transition Directionality

Transitions have inherent directionality:

- **Forward**: Move toward natural completion (Created → Running → Completed)
- **Backward**: Move toward earlier states (Running → Paused → Running). Restricted and require higher authorization.
- **Terminal**: Archived is terminal — no transitions from terminal states.

## LMS Operations

### Core Operations

```
createEntity(type, metadata) → entity_id
transitionEntity(entity_id, target_state, authorized_by, rationale) → Event
getState(entity_id) → current_state
getLifecycleHistory(entity_id, from_version?, to_version?) → Event[]
validateTransition(entity_id, target_state) → ValidationResult
getEntitiesByState(state, filter?) → entity_id[]
```

### createEntity

| Parameter | Type | Description |
|-----------|------|-------------|
| type | EntityType | The type of entity (session, mission, organization) |
| metadata | JSON | Initial metadata (name, parent, capabilities) |
| Returns | entity_id | UUIDv7 assigned to the new entity |

Creates a new entity record in the Created state. The entity must have a registered lifecycle definition. Entity creation fails if: the type is not registered, parent entity is in a terminal state, or required metadata is missing.

### transitionEntity

| Parameter | Type | Description |
|-----------|------|-------------|
| entity_id | UUID | The entity to transition |
| target_state | string | The target state |
| authorized_by | EntityID | The entity authorizing the transition |
| rationale | string | Reason for the transition |

Validates the transition against transition rules. If valid and authorized, produces a StateChanged Event, updates the current state, and triggers any side-effect actions (resource freeze/release, cascading transitions).

### getState

Returns the current projection of the entity's event stream. If the entity is not found, returns EntityNotFound error. State is always returned as a single, discrete value.

### getLifecycleHistory

Returns a time-ordered list of lifecycle Events for the entity. Supports pagination via from_version and to_version. Returns an empty list if no Events exist (should not occur for valid entities).

### validateTransition

Checks whether a transition from the entity's current state to the target state would be valid. Returns {valid: bool, reason: string, required_authority: string, preconditions: PreconditionResult[]}.

## Transition Authorization

LMS enforces the transition authorization matrix from Foundations/008-Object-Lifecycle.md:

| Transition | Authorized By | Auth Level | Evidence Required |
|-----------|--------------|------------|-------------------|
| Created → Planned | Creator entity | L2 | No |
| Planned → Assigned | Resource orchestrator | L3 | Yes (resource availability) |
| Assigned → Running | LMS (automatic when ready) | L1 | No |
| Running → Waiting | Entity itself | L2 | Yes (dependency identified) |
| Running → Paused | Supervisor or Security Council | L3 | Yes (pause reason) |
| Running → Blocked | Entity itself | L2 | Yes (error details) |
| Running → Review | Entity or policy trigger | L2 | Yes (check-in reason) |
| Waiting → Running | LMS (automatic) | L1 | Yes (dependency completion) |
| Paused → Running | Supervisor | L3 | No |
| Blocked → Running | Resolver entity/supervisor | L3 | Yes (resolution evidence) |
| Blocked → Review | Entity or supervisor | L2 | Yes (block details) |
| Review → Running | Reviewer | L3 | Yes (review decision) |
| Review → Completed | Reviewer | L3 | Yes (approval) |
| Completed → Archived | LMS (automatic, timer-based) | L1 | Yes (retention policy) |

### Authorization Levels

| Level | Authorizing Entity | Use Case |
|-------|-------------------|----------|
| L1 — Automatic | Condition-based, timer | Idle timeout, resource exhaustion |
| L2 — Entity Self | Session, Mission | Pause, block, waiting |
| L3 — Organization | Parent Organization, Supervisor | Resume, reassign |
| L4 — Security Council | Security Council | Force termination, emergency |
| L5 — Constitutional | Full Council | Dissolution, exemption |

## Event Sourcing Architecture

LMS is fully event-sourced:

```
Entity State = fold(Events [], initial_state, apply_event)
```

Where `apply_event(state, event)` returns the new state after the event is applied. State is never stored directly — it is always a projection of the event log.

### Snapshotting

For long-lived entities, periodic snapshots compress the event stream:

| Parameter | Default | Description |
|-----------|---------|-------------|
| Snapshot interval | 100 Events | New snapshot every N events |
| Snapshot retention | 3 | Keep last N snapshots |
| Snapshot trigger | Configurable | Event count or time-based |

### State Recovery

On LMS startup or failover:

1. Load latest snapshot for each entity
2. Replay Events from snapshot version to latest
3. Verify state consistency (check monotonic version)
4. Resume normal operation

## Hierarchical Lifecycle Constraints

LMS enforces parent-child lifecycle constraints:

| Parent State | Child Constraint |
|-------------|-----------------|
| Organization: Suspended | Children are Paused |
| Organization: Dissolved | Children are terminated |
| Mission: Paused | Sessions are Paused |
| Mission: Completed | Sessions are completed/terminated |

When a parent transitions, LMS cascades the transition to all children. Cascading transitions are processed atomically — all children transition or none.

## Clustering and Fault Tolerance

| Aspect | Configuration |
|--------|--------------|
| Mode | Active-passive with read replicas |
| Writes | Active instance only |
| Reads | Active + read replicas |
| Failover | Automatic, <5 seconds |
| State persistence | Event Store |

### Failover Sequence

1. Active instance loses heartbeat (3 missed)
2. Standby instance detects leader loss
3. Standby loads latest snapshots from Event Store
4. Standby replays Events since snapshot
5. Standby advertises as active via PSAP
6. Read replicas reattach to new active

## LMS Invariants

1. **LMS-INV-001 — State Singularity**: Every entity exists in exactly one lifecycle state at all times.

2. **LMS-INV-002 — Transition Atomicity**: Every transition is atomic — the entity either completes the transition or remains in its original state.

3. **LMS-INV-003 — Event Causality**: Every transition Event references the authorization Event that permitted it.

4. **LMS-INV-004 — Monotonic Progression**: Version numbers within an entity's event stream are strictly increasing with no gaps.

5. **LMS-INV-005 — No Bypass**: No entity may directly modify its lifecycle state. All transitions go through LMS.

6. **LMS-INV-006 — Cascade Integrity**: When cascading transitions occur, all children reach their target states or the parent transition is rolled back.

## Error Codes

| Code | Condition | Description |
|------|-----------|-------------|
| LMS-001 | EntityNotFound | No entity with the given ID exists |
| LMS-002 | InvalidTransition | Transition from current state to target state is not defined |
| LMS-003 | Unauthorized | Requesting entity lacks authority for this transition |
| LMS-004 | PreconditionFailed | A precondition for the transition is not met |
| LMS-005 | EntityInTerminalState | Cannot transition an entity in a terminal state |
| LMS-006 | ParentIncompatible | Parent entity state does not allow this transition |
| LMS-007 | SnapshotNotFound | No snapshot found for entity during recovery |
| LMS-008 | VersionConflict | Event version mismatch (concurrent write detected) |

## LMS Events

| Event Type | Produced When | Fields |
|-----------|--------------|--------|
| `LMS.EntityCreated` | Entity is registered | entity_id, entity_type, created_by, initial_state, metadata, timestamp |
| `LMS.StateChanged` | Entity transitions state | entity_id, from_state, to_state, authorized_by, rationale, transition_version, timestamp |
| `LMS.TransitionDenied` | Transition request rejected | entity_id, from_state, attempted_to, reason, requested_by, rule_evaluated, timestamp |
| `LMS.EntityCompleted` | Entity reaches Completed | entity_id, completed_by, completion_evidence, final_state, timestamp |
| `LMS.EntityArchived` | Entity is archived | entity_id, retention_period, archived_by, archived_at, event_count, timestamp |

| `LMS.SnapshotCreated` | A snapshot is persisted | entity_id, version, event_count, timestamp |

## Cross-Cutting Concerns

### Security

Every transition requires authentication of the requesting entity and authorization by the Security Council. LMS validates the authorization token before executing any transition. Failed authorization attempts produce security Events. LMS itself authenticates to the Event Store for write operations.

### Evidence

Every LMS operation produces at least one Event. Transition Events include from-state, to-state, authorizing entity, rationale, and timestamp. The complete lifecycle history of every entity is queryable from the Event Store. Audit trail is preserved permanently.

### Lifecycle

LMS manages its own lifecycle — it is a constitutional entity with lifecycle states: Created → Active → Suspended → Recovering → Terminated. LMS health is monitored by IRS. LMS failover is managed by the Platform availability system.

### Capability Bounds

LMS can only perform lifecycle operations. It has no authority to modify entity data, allocate resources, or execute domain logic. LMS capabilities are: read state, write transitions, validate transitions, query history. LMS cannot create or destroy entities outside its lifecycle scope.

### Communication

LMS communicates exclusively through ACF. All transition requests arrive via ACF messages. All Events are published to ACF streams. LMS does not expose HTTP or gRPC endpoints — all interaction is through ACF message passing.

### Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | LMS does one thing: lifecycle management |
| R2 — Dependency Order | LMS depends on Event Store and ACF; nothing depends on LMS internals |
| R3 — DRY | Lifecycle definitions are stored once in the Lifecycle Registry |
| R4 — Builder Pattern | Entities are built by their respective factories, not by LMS |
| R5 — Liskov | All entity lifecycles implement the same Lifecycle interface |
| R6 — DI over Singletons | LMS receives ACF and Event Store as injected dependencies |
| R7 — Tests Exist | Every transition path has unit and integration tests |
| R8 — Tests Fast | LMS transition tests complete in <100ms |
| R9 — Deterministic | Same transition inputs always produce same state result |
| R10 — Simpler Over Complex | LMS state machine is a linear DAG, no branching state logic |
| R11 — Refactor Over Rewrite | LMS lifecycle definitions evolve via versioned updates |
| R12 — Embrace Errors | Every transition failure has a unique error code |
| R13 — Design for Failure | LMS active-passive failover; Event replay restores state |
| R14 — Paved Path | LMS is the only path for lifecycle transitions |
| R15 — Open/Closed | New entity lifecycles extend without modifying LMS core |
