## Purpose

Sandboxed plugin execution for third-party code. TPE allows running untrusted code safely within a tightly controlled sandbox. TPE provides process-level isolation, filesystem restrictions, network ACLs, resource limits, and capability restrictions. All TPE jobs require Security Council authorization and are fully audited.

## Isolation Model

| Layer | Technology | Description |
|-------|-----------|-------------|
| **Process** | gVisor, Firecracker microVM | Process-level sandboxing |
| **Filesystem** | Read-only root | Sandboxed filesystem with no access to host |
| **Network** | ACL-based | Restricted network access, allow-list only |
| **Resources** | CPU/memory quota | Hard limits on compute and memory |
| **Capabilities** | Capability-based | Explicit grants for each capability |

## TPE Operations

```
submitJob(code, inputs, resource_limits) → job_id
getJobStatus(job_id) → JobStatus
cancelJob(job_id) → void
getJobResult(job_id) → JobResult
listJobs(filter?) → JobInfo[]
getJobLogs(job_id) → string[]
```

### Job Submission Schema

```
JobSubmission {
  code: Binary,
  runtime: string,       // "python3.11", "node20", "rust", "wasm"
  inputs: map<string, Binary>,
  resource_limits: {
    cpu_cores: float,
    memory_mb: int,
    timeout_seconds: int,
    network_access: string[],
    filesystem_access: string[]
  },
  capabilities: string[],  // granted capabilities
  submitted_by: EntityID
}
```

### Job Lifecycle

```
Submitted → Queued → Validating → Running → Completed → ResultsReady → Retrieved
```

| State | Description |
|-------|-------------|
| **Submitted** | Job received, pending validation |
| **Queued** | Validation passed, waiting for resources |
| **Validating** | Code, inputs, and limits being validated |
| **Running** | Job is executing in sandbox |
| **Completed** | Execution finished successfully |
| **Failed** | Execution encountered an error |
| **Cancelled** | Job was cancelled before completion |
| **ResultsReady** | Output is available for retrieval |
| **Retrieved** | Results have been fetched by submitter |

## Security Requirements

| Requirement | Enforcement |
|-------------|-------------|
| Security Council authorization | Required before any job submission |
| Explicit capability grants | Capabilities must be explicitly listed |
| Time-bounded execution | Max 24 hours per job |
| Resource limits | Hard limits, enforced by kernel |
| Network allow-list | Only specified endpoints accessible |
| Filesystem isolation | Read-only root, temp write space only |
| Audit logging | All job operations produce Events |

### Security Violations

| Violation | Action |
|-----------|--------|
| Resource limit exceeded | Job terminated, escalation to Security Council |
| Network access violation | Connection blocked, security Event logged |
| Filesystem escape attempt | Sandbox destroyed, incident report |
| Capability misuse | Capability revoked, job terminated |
| Timeout exceeded | Job terminated, partial results discarded |

## Cross-Cutting Concerns

### Security

TPE is designed for untrusted code. Every aspect of execution is sandboxed, limited, and audited. Security Council authorization is required. BG (008-BG.md) provides emergency termination capability.

### Evidence

Every job lifecycle transition produces an Event. Security violations are immediately escalated. Job logs are preserved for audit. Sandbox creation and destruction are recorded.

### Lifecycle

Jobs follow the defined job lifecycle. TPE follows Platform service lifecycle. Sandbox types (gVisor, Firecracker) are versioned.

### Capability Bounds

TPE only provides sandboxed execution. It does not interpret code meaning, does not manage resources long-term (ROS does), does not manage job scheduling priorities (Orchestrator does). TPE capabilities are limited to: create sandbox, execute code, enforce limits, return results.

### Communication

TPE communicates through ACF. Job submissions and results are delivered via ACF. Sandbox provisioning is internal to TPE. Security notifications go through ACF.

### Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | TPE does one thing: sandboxed third-party execution |
| R2 — Dependency Order | TPE depends on ACF and Security Council; no upward deps |
| R3 — DRY | Sandbox configurations are defined in one place |
| R4 — Builder Pattern | Sandboxes are built by SandboxBuilders with isolation config |
| R5 — Liskov | All sandbox technologies implement the same Sandbox interface |
| R6 — DI over Singletons | TPE receives ACF as injected dependency |
| R7 — Tests Exist | Every sandbox type and security violation has tests |
| R8 — Tests Fast | Sandbox lifecycle tests complete in <200ms |
| R9 — Deterministic | Same code + same inputs always produces same output |
| R10 — Simpler Over Complex | Sandbox interface is minimal: execute, cancel, get result |
| R11 — Refactor Over Rewrite | Sandbox backends evolve via versioned updates |
| R12 — Embrace Errors | Every execution failure has a unique error code |
| R13 — Design for Failure | Sandbox failures are isolated; host is never affected |
| R14 — Paved Path | TPE is the only path for third-party code execution |
| R15 — Open/Closed | New sandbox types extend without modifying TPE core |
