## Purpose

Sandboxed plugin execution for third-party code. TPE allows running untrusted code safely within a tightly controlled sandbox. TPE provides process-level isolation, filesystem restrictions, network ACLs, resource limits, and capability restrictions. All TPE jobs require Security Council authorization and are fully audited.

## Isolation Model

| Layer | Technology | Description | Bypass Risk |
|-------|-----------|-------------|-------------|
| **Process** | gVisor, Firecracker microVM | Process-level sandboxing with kernel isolation | Low (proven isolation) |
| **Filesystem** | Read-only root with temp write space | Sandboxed filesystem with no access to host | Low (seccomp-bpf enforced) |
| **Network** | ACL-based allow-list | Restricted network access, default deny | Low (veth pair isolation) |
| **CPU** | CFS quota | Hard CPU limits | None (kernel enforced) |
| **Memory** | cgroup limit | Hard memory limits | None (kernel enforced) |
| **Capabilities** | Explicit grant list | Each capability must be explicitly granted | Low (capability check) |

### Sandbox Types

| Sandbox | Isolation Level | Startup Time | Overhead | Use Case |
|---------|----------------|--------------|----------|----------|
| **gVisor** | Application-level kernel | <100ms | ~10% | Multi-tenant, high density |
| **Firecracker** | Full microVM | <200ms | ~3% | Strong isolation, single-tenant |
| **WASM** | Runtime-level sandbox | <10ms | ~1% | Lightweight plugins |

## TPE Operations

```
submitJob(code, inputs, resource_limits, capabilities) → job_id
getJobStatus(job_id) → JobStatus
cancelJob(job_id) → void
getJobResult(job_id) → JobResult
getJobLogs(job_id) → string[]
listJobs(filter?) → JobInfo[]
getJobMetrics(job_id) → JobMetrics
```

### Job Submission

```
JobSubmission {
  code: Binary,                    // compiled or source code
  runtime: string,                 // "python3.11", "node20", "rust", "wasm"
  inputs: map<string, Binary>,     // input files
  resource_limits: {
    cpu_cores: float,              // 0.1 – 8.0
    memory_mb: int,                // 64 – 16384
    timeout_seconds: int,          // 1 – 86400
    network_access: string[],      // allow-list of endpoints
    filesystem_write_mb: int,      // temp space limit
    output_size_mb: int            // output limit
  },
  capabilities: string[],          // explicit capability grants
  submitted_by: EntityID,
  labels?: map<string, string>
}
```

### Job Lifecycle

```
Submitted → Queued → Validating → Scheduled → Running → Completed → Ready
                                                            ↘ Failed
                                     Cancelled (from any state)
```

| State | Description |
|-------|-------------|
| **Submitted** | Job received, pending validation |
| **Queued** | Validation passed, waiting for resources |
| **Validating** | Code, inputs, and limits being validated |
| **Scheduled** | Resources allocated, waiting for execution slot |
| **Running** | Job executing in sandbox |
| **Completed** | Execution finished successfully |
| **Failed** | Execution encountered an error |
| **Cancelled** | Job cancelled by requester or Security Council |
| **Ready** | Output available for retrieval |

## Security Requirements

| Requirement | Enforcement | Circumvention Risk |
|-------------|-------------|-------------------|
| Security Council authorization | Required before first job | Low (cryptographic) |
| Explicit capability grants | Capabilities must be listed | Low (check per op) |
| Time-bounded execution | Max 24 hours per job | None (kernel timer) |
| Resource limits | Hard limits, enforced by kernel | None (cgroup) |
| Network allow-list | Only specified endpoints | Low (veth isolation) |
| Filesystem isolation | Read-only, temp write space | Low (seccomp-bpf) |
| Audit logging | All operations produce Events | Low (immutable) |

### Security Violations

| Violation | Severity | Response | Notification |
|-----------|----------|----------|--------------|
| Resource limit exceeded | Warning | Job terminated | Security Council |
| Network access violation | Critical | Connection blocked, log | Security Council |
| Filesystem escape attempt | Critical | Sandbox destroyed, incident report | Security Council + CLS |
| Capability misuse | High | Capability revoked, job terminated | Security Council |
| Timeout exceeded | Warning | Job terminated | Requester |
| Code injection | Critical | Sandbox destroyed, full investigation | Security Council + CLS |

## Cross-Cutting Concerns

### Security

TPE is designed for untrusted code. Every aspect of execution is sandboxed, limited, and audited. Security Council authorization is required. BG (008-BG.md) provides emergency termination capability.

### Evidence

Every job lifecycle transition produces an Event. Security violations are immediately escalated. Job logs are preserved for audit. Sandbox creation and destruction are recorded.

### Lifecycle

Jobs follow the defined job lifecycle. TPE follows Platform service lifecycle. Sandbox types (gVisor, Firecracker, WASM) are versioned.

### Capability Bounds

TPE only provides sandboxed execution. It does not interpret code meaning, does not manage resources long-term (ROS does), does not manage job scheduling priorities (Orchestrator does).

### Communication

TPE communicates through ACF. Job submissions and results are delivered via ACF. Sandbox provisioning is internal to TPE. Security notifications go through ACF.

### Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | TPE does one thing: sandboxed third-party execution |
| R2 — Dependency Order | TPE depends on ACF and Security Council; no upward deps |
| R3 — DRY | Sandbox configurations are defined in one place |
| R4 — Builder Pattern | Sandboxes are built by SandboxBuilders with isolation config |
| R5 — Liskov | All sandbox technologies implement the same Sandbox interface |
| R6 — DI over Singletons | TPE receives ACF as injected dependency |
| R7 — Tests Exist | Every sandbox type and security violation has tests |
| R8 — Tests Fast | Sandbox lifecycle tests complete in <200ms |
| R9 — Deterministic | Same code + same inputs always produces same output |
| R10 — Simpler Over Complex | Sandbox interface is minimal: execute, cancel, get result |
| R11 — Refactor Over Rewrite | Sandbox backends evolve via versioned updates |
| R12 — Embrace Errors | Every execution failure has a unique error code |
| R13 — Design for Failure | Sandbox failures are isolated; host is never affected |
| R14 — Paved Path | TPE is the only path for third-party code execution |
| R15 — Open/Closed | New sandbox types extend without modifying TPE core |

## TPE Invariants

1. **TPE-INV-001 — Full Isolation**: No job may access the host system, other jobs' data, or resources outside its allocated limits. Isolation is enforced at the kernel level.

2. **TPE-INV-002 — Capability Gating**: Every operation requires an explicitly granted capability. No implicit capabilities are available.

3. **TPE-INV-003 — Time-Bound Execution**: Every job has a hard timeout. Jobs exceeding the timeout are terminated. No runaway jobs are permitted.

4. **TPE-INV-004 — Audit Trail**: Every job lifecycle transition produces an Event. Security violations produce critical Events with full context.

5. **TPE-INV-005 — Resource Enforcement**: CPU and memory limits are enforced by the kernel (cgroups). Jobs cannot exceed their allocated limits by any means.

## Sandbox Cleanup

When a job completes or is terminated:

```
1. Kill all processes in the sandbox
2. Close all network connections
3. Unmount filesystem
4. Wipe temporary storage
5. Destroy sandbox (gVisor/VM/WASM)
6. Release CPU/memory quotas
7. Verify cleanup (check for remaining processes)
8. Produce JobCompleted/JobFailed Event
```
