## Purpose

Secure compute environment for sensitive operations. TEE provides hardware-backed isolation for high-security workloads. TEE ensures that code and data are protected from the host operating system, hypervisor, and other tenants. TEE is used for cryptographic key operations, high-security execution, and sensitive data processing.

## TEE Types

| Type | Technology | Isolation Model | Memory Encryption | Use Case |
|------|-----------|-----------------|-------------------|----------|
| **Intel SGX** | Enclave | CPU-level memory encryption | Yes (MEE) | In-process secure computation |
| **AMD SEV** | Encrypted VM | Full VM memory encryption | Yes (AES-256) | Secure VM workloads |
| **AWS Nitro** | Isolated instance | Hardware virtualized | Yes (hardware) | Cloud-native secure execution |

### TEE Comparison

| Aspect | SGX | SEV | Nitro |
|--------|-----|-----|-------|
| Memory limit | 512 MB (EPC) | Full VM memory | Full instance memory |
| Attestation | Intel Attestation Service | AMD SEV-SNP | AWS KMS attestation |
| TCB size | Small (CPU only) | Large (full hypervisor) | Medium (Nitro hypervisor) |
| Performance overhead | ~5% | ~2% | <1% |
| Code changes required | Significant | Minimal | None |

## Enclave Lifecycle

```
Created → Loading → Measuring → Initialized → Ready → Executing → Completed → Destroyed
```

| State | Description | Operations Allowed |
|-------|-------------|-------------------|
| **Created** | Enclave metadata recorded, no code loaded | Destroy |
| **Loading** | Code and data being loaded into protected memory | Continue loading, Destroy |
| **Measuring** | Enclave contents being hashed for attestation | Finalize load, Destroy |
| **Initialized** | Enclave is initialized with measurement value | Get measurement, Destroy |
| **Ready** | Enclave ready to accept execution requests | Execute, Attest, Destroy |
| **Executing** | Enclave actively processing a request | Continue, Cancel |
| **Completed** | Enclave execution finished, results extracted | Get result, Destroy |
| **Destroyed** | Memory and state are securely wiped | None |

## Attestation

TEE produces a signed attestation report proving:

1. **Identity**: The enclave is running the expected code (measured hash)
2. **Integrity**: The enclave has not been tampered with
3. **Isolation**: The enclave is running in genuine TEE hardware
4. **Freshness**: The attestation is current (includes nonce)

### Attestation Report Schema

```
AttestationReport {
  enclave_id: UUID,
  measurement: string,         // SHA-256 hash of enclave contents
  signer: string,             // code signing key fingerprint
  tee_type: string,           // sgx, sev, nitro
  platform_claims: {
    tee_version: string,
    platform_id: string,
    debug_mode: bool
  },
  nonce: string,              // freshness proof
  timestamp: HLC,
  signature: string,          // TEE hardware signature
  cert_chain: string[]        // TEE vendor certificate chain
}
```

### Attestation Verification

```
1. Request attestation from enclave
2. Verify signature against TEE hardware vendor's root certificate
3. Verify certificate chain is valid and not revoked
4. Compare measurement hash against expected code hash
5. Verify nonce matches request
6. Check timestamp is within acceptable skew (30s)
7. Verify debug_mode is false (production enclaves only)
8. If all pass, enclave is trustworthy
```

## Security Considerations

| Concern | Mitigation | Implementation |
|---------|-----------|----------------|
| Side-channel attacks | Constant-time operations | All crypto uses constant-time implementations |
| Rollback attacks | Monotonic counters | Hardware monotonic counters for state |
| IO channels | Encrypted channels | All TEE IO uses TLS 1.3 |
| Attestation reuse | Nonces, timestamps | Unique nonce per attestation request |
| Memory exhaustion | Resource limits | Per-enclave memory limit |
| Init poisoning | Measurement verification | Verify measurement before trusting |

## TEE Invariants

1. **TEE-INV-001 — Hardware Isolation**: Enclave memory is inaccessible to host OS, hypervisor, and other tenants.

2. **TEE-INV-002 — Measured Boot**: Every enclave's code and initial data are measured (hashed) before execution. The measurement is included in the attestation.

3. **TEE-INV-003 — Attested Identity**: Every enclave can prove its identity through hardware attestation. The attestation is verifiable by third parties.

4. **TEE-INV-004 — Secure Destruction**: Enclave destruction securely wipes all memory and state. No residual data remains.

5. **TEE-INV-005 — No Persistence**: Enclaves are ephemeral. State is not preserved across enclave destruction. Persistent state must be explicitly handled.

## Cross-Cutting Concerns

### Security

TEE is the highest-security execution environment. All enclave operations require Security Council authorization. Attestation verification is mandatory before trusting enclave output. Enclave destruction must securely wipe all memory and state.

### Evidence

Every enclave operation produces an Event. Attestation reports are stored as evidence. Security violations are immediately escalated.

### Lifecycle

Enclaves follow their defined lifecycle. TEE itself follows Platform service lifecycle. Enclave types (SGX, SEV, Nitro) are versioned and may be deprecated.

### Capability Bounds

TEE only provides isolated execution. It does not manage keys (HSM/KMS does), does not store data long-term, and does not interpret the code it executes.

### Communication

TEE communicates through ACF. Enclave creation and execution requests arrive via ACF. Attestation reports are delivered through ACF. TEE does not expose external endpoints.

### Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | TEE does one thing: hardware-backed secure execution |
| R2 — Dependency Order | TEE depends on ACF and Security Council; no upward deps |
| R3 — DRY | Enclave measurement verification is defined in one place |
| R4 — Builder Pattern | Enclaves are built by EnclaveBuilders with hardware validation |
| R5 — Liskov | All TEE types implement the same Enclave interface |
| R6 — DI over Singletons | TEE receives ACF as injected dependency |
| R7 — Tests Exist | Every TEE type and attestation flow has tests |
| R8 — Tests Fast | Enclave lifecycle tests complete in <500ms |
| R9 — Deterministic | Same code + same data always produces same measurement |
| R10 — Simpler Over Complex | Enclave interface is minimal: create, execute, attest, destroy |
| R11 — Refactor Over Rewrite | TEE backends evolve via versioned updates |
| R12 — Embrace Errors | Every enclave failure has a unique error code |
| R13 — Design for Failure | Enclave failures are isolated; attestation verification fails safe |
| R14 — Paved Path | TEE is the only path for hardware-backed secure execution |
| R15 — Open/Closed | New TEE types extend without modifying TEE core |
