## Purpose

EPG processes Event streams through a directed acyclic graph (DAG) of processors. Each node in the graph performs a specific transformation — filter, transform, enrich, aggregate, or route. EPG is used by Academy (knowledge derivation), AUS (audit filtering), Risk (risk event processing), and Analytics (metric derivation). EPG provides a declarative way to define Event processing pipelines.

## EPG DAG

```
                    ┌────────┐
                    │ Events │
                    └───┬────┘
                        │
                   ┌────▼────┐
                   │  Filter │
                   └────┬────┘
                        │
                   ┌────▼──────┐
                   │ Transform │
                   └────┬──────┘
                        │
                   ┌────▼─────┐
                   │  Enrich  │
                   └────┬─────┘
                        │
                   ┌────▼────────┐
                   │  Aggregate  │
                   └────┬────────┘
                        │
                   ┌────▼─────┐
                   │  Route   │
                   └────┬─────┘
                        │
              ┌─────────┴──────────┐
              ▼                    ▼
      ┌──────────────┐    ┌──────────────┐
      │ Output Stream│    │ Output Stream│
      │      A       │    │      B       │
      └──────────────┘    └──────────────┘
```

### Node Types

| Node Type | Description | Input → Output |
|-----------|-------------|----------------|
| **filter** | Include/exclude Events based on predicate | Event[] → Event[] (subset) |
| **transform** | Modify Event structure | Event → Event (modified) |
| **enrich** | Add data from external sources | Event → Event (enriched) |
| **aggregate** | Combine multiple Events into one | Event[] → Event (summary) |
| **route** | Send Events to different output streams | Event → Event (routed) |

### Node Configuration

```
ProcessorNode {
  id: string,
  type: NodeType,
  config: map<string, any>,
  retry: RetryConfig,
  metrics: MetricsConfig
}
```

### Edge Configuration

```
Edge {
  id: string,

  type: EdgeType,   // passthrough, conditional, split, merge

}
```

### Graph Definition

```
EPGGraph {
  id: string,
  name: string,
  version: int,
  source_streams: string[],       // ACF subscription topics
  nodes: ProcessorNode[],
  edges: Edge[],
  output_streams: string[],       // ACF publish topics
  error_stream: string,            // failed events topic
  max_retries: int,               // default 3
  backoff_ms: int,                // default 100
  enabled: bool
}
```

## Fault Tolerance

EPG is fault-tolerant at the node level:

| Fault | Handling |
|-------|----------|
| Node processing failure | Retry up to 3 attempts with exponential backoff (100ms, 500ms, 2s) |
| Enrichment source unavailable | Return stale cached data if available; else retry |
| Downstream stream unavailable | Buffer up to 10000 Events; apply backpressure |
| Graph crash | State persisted every 1000 Events; recovery replays from checkpoint |
| Persistent node failure | Event sent to dead letter stream after max retries |

### Dead Letter Event Schema

```
DeadLetterEvent {
  original_event: Event,
  failed_node: string,
  retry_count: int,
  errors: ProcessingError[],
  first_failed_at: timestamp,
  last_failed_at: timestamp,
  graph_id: string
}
```

## Node Type Details

### Filter Nodes

Filter predicate supports:

| Operator | Description |
|----------|-------------|
| eq, neq | Equality comparison |
| gt, gte, lt, lte | Numeric comparison |
| in, not_in | Set membership |
| exists, not_exists | Field presence |
| matches | Regex match (string fields) |
| contains | Array containment |
| and, or, not | Logical composition |

### Transform Nodes

| Operation | Description | Example |
|-----------|-------------|---------|
| add_field | Add new field | `add_field("processed_at", now())` |
| remove_field | Remove field | `remove_field("internal_data")` |
| rename_field | Rename field | `rename_field("user_id", "actor_id")` |
| cast_field | Change field type | `cast_field("count", "int")` |
| compute | Compute derived value | `compute("duration_ns", "end_time - start_time")` |

### Enrich Nodes

Enrichment sources:

| Source | Protocol | Cache TTL |
|--------|----------|-----------|
| IDS (entity metadata) | ACF query | 5 minutes |
| PSAP (service info) | ACF query | 1 minute |
| EVS (event history) | ACF query | 10 minutes |
| EIP (external data) | ACF via EIP | Configurable |

### Aggregate Nodes

| Window | Description | Trigger |
|--------|-------------|---------|
| tumbling | Fixed windows, non-overlapping (e.g., 5min) | Window end |
| sliding | Fixed windows, overlapping (e.g., 5min, slide 1min) | Every slide |
| session | Activity-based windows (e.g., 30min inactivity) | Inactivity timeout |
| count | Fixed count of Events | Every N Events |

### Route Nodes

Route rules:

```
RouteRule {
  predicate: string,    // Event field condition
  target_stream: string // Output stream name
  priority: int         // Rule evaluation order
}
```

## Cross-Cutting Concerns

### Security

Graph definitions are protected. Only authorized entities (Academy, AUS, Risk, Analytics) may define graphs. Graph activation requires Security Council authorization. Processing nodes cannot access unauthorized data.

### Evidence

Every Event processed through EPG produces a NodeProcessed Event. Failures produce NodeFailed Events. Dead letter Events are preserved for audit. Graph lifecycle Events are recorded.

### Lifecycle

EPG graphs follow: Draft → Active → Paused → Updating → Deactivated → Archived. Graph definitions are versioned. Active graphs can be updated without stopping processing (hot reload).

### Capability Bounds

EPG only processes Events. It does not store Events long-term (EVS does). It does not interpret Event meaning (consumers do). It has no authority over source systems.

### Communication

EPG consumes Event streams from ACF subscriptions. Processed Events are published to ACF output streams. EPG communicates with enrichment sources through ACF queries.

### Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | EPG does one thing: DAG-based Event processing |
| R2 — Dependency Order | EPG depends on ACF and EVS; no upward deps |
| R3 — DRY | Graph definitions are the single source for processing logic |
| R4 — Builder Pattern | Graphs are built by GraphBuilders with validation |
| R5 — Liskov | All processor nodes implement the same Processor interface |
| R6 — DI over Singletons | EPG receives ACF as injected dependency |
| R7 — Tests Exist | Every node type and graph configuration has tests |
| R8 — Tests Fast | Node processing tests complete in <5ms |
| R9 — Deterministic | Same Event + same node always produces same output |
| R10 — Simpler Over Complex | DAG is explicit; no hidden control flow |
| R11 — Refactor Over Rewrite | Graph definitions evolve via versioned updates |
| R12 — Embrace Errors | Every node failure has a unique error code |
| R13 — Design for Failure | Retry logic, dead letter queue, checkpoint recovery |
| R14 — Paved Path | EPG is the only path for Event stream processing |
| R15 — Open/Closed | New node types extend without modifying EPG core |
