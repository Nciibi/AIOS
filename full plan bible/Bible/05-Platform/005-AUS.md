## Purpose

Queryable audit trail over the Event Store. AUS provides filtered, searchable access to audit Events — all Events tagged with `audit: true`. Audit Events include security, identity, lifecycle, governance, and policy Events. AUS enables the Security Council, Organization managers, and entity supervisors to investigate system activity.

## Audit Events

Audit Events are all Events with the `audit: true` tag. The following Event categories are always audit Events:

| Category | Examples |
|----------|---------|
| **Security** | Authentication attempts, authorization decisions, verification failures |
| **Identity** | Identity creation, suspension, retirement, credential changes |
| **Lifecycle** | State transitions, entity creation, entity completion |
| **Governance** | RFC submissions, constitutional reviews, policy changes |
| **Policy** | Policy evaluations, capability grants, resource allocations |

## AUS Operations

```
queryAuditLog(filter) → AuditRecord[]
getAuditTrail(entity_id) → AuditRecord[]
getSecurityEvents(time_range) → AuditRecord[]
exportAuditLog(format, filter) → export_handle
getAuditStats(time_range) → AuditStats
searchAuditLog(query) → AuditRecord[]
```

### Filters

| Filter | Type | Description |
|--------|------|-------------|
| **time_range** | [timestamp, timestamp] | Filter by Event time |
| **event_types** | string[] | Include only specified types |
| **entity_id** | UUID | Filter by entity |
| **actor_id** | UUID | Filter by acting entity |
| **severity** | enum | info, warning, critical |
| **tags** | string[] | Filter by metadata tags |

### Audit Trail

The audit trail for an entity shows every Event involving that entity in chronological order. The trail includes Events where the entity was the target, the actor, or referenced in the payload.

## Audit Retention

| Retention Tier | Duration | Storage |
|---------------|----------|---------|
| **Hot** | 90 days | Active Event Store |
| **Warm** | 1 year | Fast-access archive |
| **Cold** | 7 years | Compressed archival storage |
| **Permanent** | Indefinite | Constitutional Events only |

Retention is configurable per Organization. Minimum retention is 7 years for security Events. After retention period expires, Events are moved to cold storage. Events are never deleted — only moved to progressively colder storage tiers.

## Authorization

Audit access is tiered by role:

| Role | Scope | Access Level |
|------|-------|-------------|
| Security Council | Full system | Read all, export all |
| Organization manager | Org-scoped | Read Org Events, export Org Events |
| Entity supervisor | Entity-scoped | Read entity Events |
| Entity self | Self only | Read own Events |
| External auditor | Granted scope | Read/export granted scope only |

## AUS Events

| Event Type | Produced When | Fields |
|-----------|--------------|--------|
| `AUS.AuditQueryExecuted` | An audit query is run | query_id, filter, requesting_entity, result_count |
| `AUS.AuditExportInitiated` | An audit export is started | export_id, format, filter, requested_by |
| `AUS.AuditExportCompleted` | An audit export is ready | export_id, record_count, size_bytes |
| `AUS.AuditRetentionTriggered` | Events are moved to cold storage | tier_from, tier_to, event_count, cutoff_timestamp |
| `AUS.SecurityEventFlagged` | A critical security Event is flagged | event_id, severity, reason, notified_entities |

## Cross-Cutting Concerns

### Security

AUS access is strictly authorized. Unauthorized audit queries are denied and logged as security Events. Export controls prevent bulk data exfiltration. AUS authenticates to EVS for read operations.

### Evidence

Every audit query produces an AuditQueryExecuted Event. Audit exports are tracked with export Events. Retention actions are recorded. AUS itself is auditable — all AUS operations produce Events.

### Lifecycle

AUS follows Platform service lifecycle. Audit retention policies follow their own lifecycle: Defined → Active → Reviewing → Updated. Retention tier transitions are lifecycle events for the stored data.

### Capability Bounds

AUS is read-only over the Event Store. It cannot modify or delete Events. It cannot bypass EVS to access storage directly. It has no authority over entity state or system configuration.

### Communication

AUS communicates through ACF. Audit queries arrive via ACF messages. Query results are returned through ACF response streams. Export files are delivered through ACF streaming.

### Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | AUS does one thing: queryable audit trail |
| R2 — Dependency Order | AUS depends on EVS and ACF; no upward deps |
| R3 — DRY | Audit events are stored once in EVS; AUS indexes |
| R4 — Builder Pattern | Audit queries are built by QueryBuilders |
| R5 — Liskov | All audit storage tiers implement same AuditStore interface |
| R6 — DI over Singletons | AUS receives EVS and ACF as injected dependencies |
| R7 — Tests Exist | Every query filter, export format, and auth scope has tests |
| R8 — Tests Fast | Audit queries complete in <100ms for hot data |
| R9 — Deterministic | Same query + same data always produces same results |
| R10 — Simpler Over Complex | Audit queries are filtered scans; no complex joins |
| R11 — Refactor Over Rewrite | Query schemas evolve via versioned updates |
| R12 — Embrace Errors | Every query failure has a unique error code |
| R13 — Design for Failure | AUS fails closed; unavailable = deny query |
| R14 — Paved Path | AUS is the only path for audit queries |
| R15 — Open/Closed | New audit query types extend without modifying core |
