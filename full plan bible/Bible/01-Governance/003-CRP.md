## Purpose

The Change Request Pipeline governs the RFC lifecycle — how changes to AIOS are proposed, reviewed, approved, implemented, and verified. Every change to the Bible, to the Constitution, or to the system architecture must go through the CRP. The CRP is the operational heart of AIOS governance.

## RFC Lifecycle

```
Draft → Submitted → Review → Approved → Implemented → Verified → Active
                             → Rejected
                             → Withdrawn
```

| State | Description | Owner |
|-------|-------------|-------|
| **Draft** | RFC is being written. Not yet formally submitted. | Author |
| **Submitted** | RFC is submitted for review. Immutable from this point. | Author |
| **Review** | RFC is being evaluated by the Security Council. | Security Council |
| **Approved** | RFC is approved for implementation. | Security Council |
| **Rejected** | RFC is rejected. Will not be implemented. | Security Council |
| **Withdrawn** | Author withdraws the RFC before completion. | Author |
| **Implemented** | RFC has been implemented in code/specification. | Implementation team |
| **Verified** | Implementation has been verified against the RFC. | Security Council |
| **Active** | RFC's changes are live in the system. | Security Council |

## RFC Types

| Type | Description | Requires | Approval Level |
|------|-------------|----------|----------------|
| **Constitutional** | Amends a Law | Sou + Security Council | Highest |
| **Bible** | Amends a Bible specification | Security Council | High |
| **Architecture** | Changes system architecture | Security Council + ADG | High |
| **Feature** | Adds new functionality within existing spec | Security Council | Medium |
| **Bugfix** | Fixes a deviation from specification | Security Council | Medium |
| **Clerical** | Typo, formatting, clarification | Security Council (simple majority) | Low |

## RFC Structure

Every RFC must contain:

```
RFC-NNNN: Title
Type: [Constitutional | Bible | Architecture | Feature | Bugfix | Clerical]
Status: [Draft | Submitted | Review | Approved | ...]
Author: [identity_id]
Source Law: [Law(s) this RFC derives from]
Source Physics: [Physics document(s) affected]
Supersedes: [RFC-NNNN if applicable]
Superseded By: [RFC-NNNN if applicable]

## Problem Statement
## Proposed Solution
## Impact Analysis
  - Affected Documents
  - Affected Services
  - Breaking Changes (Yes/No)
## Evidence
  - Evidence supporting the change
  - Evidence of need (Academy analysis, incident reports, etc.)
## Constitutional Review
  - Does this RFC violate any Law?
  - Does this RFC require a Law amendment?
## Design DNA Compliance
  - R1 through R15: how each rule is satisfied or why it does not apply
## Migration Plan
  - How existing state is migrated
  - Backward compatibility period
## Review Notes
  - Security Council review outcome
  - ADG review outcome (if architecture type)
  - Conditions or amendments
```

## RFC Submission Requirements

### Mandatory
- Valid author identity (must be a constitutional entity with identity)
- Problem statement with supporting evidence
- Proposed solution with design alternatives considered (R10)
- Source Law citation (every change traces to a Law)
- Design DNA compliance checklist (R1–R15)
- Impact analysis
- Migration plan (if breaking change)

### Evidence Requirements by Type

| RFC Type | Required Evidence |
|----------|------------------|
| Constitutional | Academy analysis (minimum 3 independent analyses), real-world incident report or gap analysis |
| Bible | Reference implementation, test results, impact analysis |
| Architecture | ADG template, dependency graph, performance impact analysis |
| Feature | Use case evidence, user/stakeholder request, mock or prototype |
| Bugfix | Bug report, reproduction steps, fix evidence |
| Clerical | Diff of the change, justification |

## Review Process

### Standard Review (Bible, Feature, Bugfix)

1. RFC is submitted (status: Submitted)
2. CRP assigns to Security Council review queue
3. Security Council reviews within 7 calendar days (SLA)
4. Council may request amendments (RFC returns to Draft)
5. Council approves or rejects
6. If approved: status → Approved, implementation may begin

### Constitutional Review

1. RFC is submitted (status: Submitted)
2. CRP routes to CLS for constitutional classification
3. Evidence package assembled
4. Sou reviews and votes (majority)
5. Security Council reviews and votes (unanimous)
6. If both approve: status → Approved

### Expedited Review (Bugfix, Emergency)

1. RFC is submitted with `priority: critical` flag
2. Security Council reviews within 24 hours
3. Same approval process as standard

## Implementation Verification

After an RFC is Approved:

1. Implementation team implements the change
2. Implementation is verified against the RFC specification
3. Verification includes: unit tests, integration tests, contract tests (R7)
4. If verification passes: status → Verified
5. Security Council activates the change: status → Active
6. If verification fails: RFC returns to Draft with failure evidence

| `CRP.RFCImplemented` | Implementation completes | rfc_id, implementation_hash, affected_docs |
