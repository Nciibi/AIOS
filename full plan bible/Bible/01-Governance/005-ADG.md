## Purpose

The Architectural Decision Gateway governs decisions that affect AIOS's system architecture — module boundaries, interface contracts, service dependencies, and structural patterns. ADG ensures that every architectural decision is evidenced, reviewed against Design DNA (R1–R15), and consistent with the existing architecture.

## When ADG Review Is Required

An architectural review is required when a change affects:

| Concern | Examples |
|---------|----------|
| Module boundaries | Splitting a service, merging two services, extracting a new service |
| Interface contracts | Adding/modifying/removing a public API, changing ACF message schema |
| Dependencies | Adding a new dependency, changing dependency direction, introducing a cycle |
| Data model | Changing the identity schema, event schema, or stored data format |
| Security model | Changing authentication, authorization, or verification flow |
| Deployment topology | Changing cluster setup, replication model, or failover strategy |
| Protocol | Introducing a new communication protocol, changing existing protocol |

## ADG Decision Lifecycle

```
Identified → Assessed → Documented → Reviewed → Approved → Recorded → Deprecated → Superseded
```

| State | Description | Owner |
|-------|-------------|-------|
| **Identified** | An architectural decision is needed | Proposer |
| **Assessed** | ADG classifies the decision's impact level | ADG Engine |
| **Documented** | Decision is written in the ADG template | Proposer |
| **Reviewed** | ADG reviews the decision against Design DNA | ADG Review Board |
| **Approved** | Decision is approved; may proceed to RFC | ADG Review Board |
| **Recorded** | Decision is stored in the ADG index | CKR |
| **Deprecated** | Decision is no longer recommended for new work | ADG Review Board |
| **Superseded** | Decision has been replaced by a newer ADG | ADG Review Board |

## ADG Template

Every architectural decision must be documented using this template:

```
ADG-NNNN: Title
Date: YYYY-MM-DD
Status: [Proposed | Approved | Recorded | Deprecated | Superseded]
Author: [identity_id]
Source Law: [Law(s)]
Source Physics: [Physics/011-Design-DNA.md — specific R#]

## Context
  - What is the architectural problem?
  - What constraints apply?
  - What is the current architecture?

## Decision
  - What is the chosen approach?
  - Why is this approach correct?

## Alternatives Considered
  - Alternative 1 (with rationale for rejection)
  - Alternative 2 (with rationale for rejection)

## Design DNA Compliance
  - R1 (Modulsingularity): How does this decision respect single responsibility?
  - R2 (Dependency Order): Is the dependency graph still acyclic?
  - R10 (Simpler Over Complex): Why is this the simplest valid solution?
  - R15 (Open/Closed): Does this decision allow future extension without modification?

## Consequences
  - Positive consequences
  - Negative consequences
  - Mitigations for negative consequences

## References
  - Related ADGs
  - Related RFCs
  - Related Bible documents
```

## ADG Approval Process

### Tier 1 — Low Impact

Change is within existing module boundaries, no API change, no data model change.

**Process**: ADG documentation only. No formal review. Recorded in ADG index.

### Tier 2 — Medium Impact

Change affects a single module's public interface or internal structure.

**Process**:
1. ADG documented using template
2. Reviewed by ADG Review Board (2 architecture reviewers)
3. Approved if consistent with Design DNA and existing architecture
4. Recorded in ADG index

### Tier 3 — High Impact

Change affects multiple modules, changes security model, or changes deployment topology.

**Process**:
1. ADG documented using template
2. Reviewed by ADG Review Board (3 architecture reviewers)
3. Reviewed by Security Council for security implications
4. Approved if:
   - Consistent with Design DNA (R1–R15)
   - No circular dependencies introduced (R2)
   - Security model is maintained or improved
   - Migration path exists
5. ADG becomes prerequisite for RFC submission

## ADG Index

All ADGs are indexed in the CKR (Constitutional Knowledge Repository). The index is queryable:

| Query | Description |
|-------|-------------|
| `getADG(adg_id)` | Retrieve a specific ADG |
| `listADGs(status?)` | List all ADGs, optionally filtered by status |
| `findADGs(module)` | Find all ADGs affecting a specific module |
| `getADGImpact(adg_id)` | Get the impact analysis of a specific ADG |

## ADG Events

| Event Type | Produced When | Fields |
|-----------|--------------|--------|
| `ADG.DecisionProposed` | An architectural decision is identified | adg_id, proposer_id, module, impact_level |
| `ADG.DecisionApproved` | ADG review passes | adg_id, reviewer_ids, conditions |
| `ADG.DecisionSuperseded` | A newer ADG replaces this one | old_adg_id, new_adg_id, reason |
| `ADG.DecisionDeprecated` | ADG is no longer applicable | adg_id, reason, recommended_replacement |
