## Purpose

The Verification Pipeline is the constitutional gate through which every action in AIOS must pass before execution. The pipeline is the operational heart of the Security Council. It enforces Law 8 (Verification-First): no action executes without verification. Every action passes through 7 sequential stages. A denial at any stage prevents execution entirely.

## Architecture

### Pipeline Overview

```
Action Request
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│                   Verification Pipeline                       │
│  ┌──────────┐  ┌──────────────┐  ┌──────────┐  ┌──────────┐ │
│  │ Stage 1  │  │  Stage 2      │  │ Stage 3  │  │ Stage 4  │ │
│  │ Identity │─►│ Authentication│─►│Authorizatn│─►│  Policy  │ │
│  └──────────┘  └──────────────┘  └──────────┘  └────┬─────┘ │
│                                                       │       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐           │       │
│  │ Stage 7  │  │ Stage 6  │  │ Stage 5  │           │       │
│  │Execution │◄─│  Risk    │◄─│Capability│◄──────────┘       │
│  │Authoriztn│  │Assessmnt │  │  Verify  │                   │
│  └──────────┘  └──────────┘  └──────────┘                   │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
Authorization Token (allow) or Denial (deny + reason + escalation)
```

### Pipeline Invariants

1. **Total Order**: Stages execute in order 1→7. No stage executes before the previous stage completes.
2. **Fail-Fast**: Any stage denial immediately terminates the pipeline. No subsequent stages are evaluated.
3. **Mandatory**: All 7 stages execute for every action. No action type is exempt from any stage.
4. **Stateless**: Each stage is independent (except for context passed from previous stages).
5. **Evidence-Producing**: Every stage produces an Event. The cumulative result is recorded as an Execution Verification Event.

## Stage 1 — Identity Verification

**Purpose**: Verify the requesting entity has a valid, active identity registered with IDS.

**Process**:
1. Extract sender identity_id from ACF envelope
2. Call IDS.resolveIdentity(identity_id)
3. Verify: identity exists, status is Active, entity_id matches, signature valid
4. On success: populate context.entity
5. On failure: produce Identity.Denied Event

| Result | Action |
|--------|--------|
| Identity exists and Active | Pass to Stage 2 |
| Identity does not exist | Deny. `VERIFY_ID_001` |
| Identity not Active | Deny. `VERIFY_ID_002` |
| Signature invalid | Deny. `VERIFY_ID_003`. Escalate S3 |
| Identity mismatch (spoofing) | Deny. `VERIFY_ID_004`. Escalate S4 |

**Caching**: Identity records cached for 60s. Denials for non-existent identities cached for 300s. Invalidated on identity status change.

**Event**: `Verification.IdentityCheck — { identity_id, result, reason? }`

## Stage 2 — Authentication Verification

**Purpose**: Verify the entity can prove it is who it claims to be.

**Process**:
1. Extract auth_token and auth_type from ACF envelope
2. Call Authentication.validateToken(auth_token)
3. Validate: signature valid, not expired, not revoked, sub matches entity, auth_type matches expected
4. On success: populate context.authentication
5. On failure: produce Authentication.Denied Event

| Result | Action |
|--------|--------|
| Token valid and matching | Pass to Stage 3 |
| Invalid signature | Deny. `VERIFY_AUTH_001`. Escalate S3 |
| Token expired | Deny. `VERIFY_AUTH_002` |
| Token revoked | Deny. `VERIFY_AUTH_003` |
| Subject mismatch | Deny. `VERIFY_AUTH_004`. Escalate S3 |
| Auth type mismatch | Deny. `VERIFY_AUTH_005` |

**Caching**: Token verification cached for remaining TTL (max 5 min). Invalidated on token revocation.

**Event**: `Verification.AuthenticationCheck — { identity, token_type, result, reason? }`

## Stage 3 — Authorization Verification

**Purpose**: Verify the entity is authorized to perform the requested action.

**Process**:
1. Determine required authorization (action type, target, scope, context)
2. Resolve entity's authorization scope (direct capabilities, delegated capabilities, contextual authority)
3. Verify action is within authorization scope
4. On success: populate authorization
5. On failure: produce Authorization.Denied Event

**Authorization Sources** (checked in order): Template capability, Organization policy, Mission scope, CCA grant, Delegated token.

| Result | Action |
|--------|--------|
| Action within scope | Pass to Stage 4 |
| No capability for this type | Deny. `VERIFY_AUTHZ_001` |
| No capability for target | Deny. `VERIFY_AUTHZ_002` |
| Exceeds delegation | Deny. `VERIFY_AUTHZ_003` |
| Capability suspended/revoked | Deny. `VERIFY_AUTHZ_004` |

**Event**: `Verification.AuthorizationCheck — { identity, action, capability_id?, result }`

## Stage 4 — Policy Verification

**Purpose**: Verify action complies with all applicable policies.

**Process**:
1. Load applicable policies (constitutional, organizational, Mission, safety, compliance)
2. Evaluate action against each policy
3. On all pass: populate policy result
4. On any fail: produce Policy.Denied Event

**Conflict Resolution**: More restrictive policy wins. Safety > Organization > Mission. Constitutional always takes precedence.

| Result | Action |
|--------|--------|
| All policies pass | Pass to Stage 5 |
| Constitutional violation | Deny. `VERIFY_POL_001`. Escalate S4 |
| Safety violation | Deny. `VERIFY_POL_002`. Escalate S3 |
| Organization violation | Deny. `VERIFY_POL_003` |
| Compliance violation | Deny. `VERIFY_POL_004`. Escalate S3 |

**Event**: `Verification.PolicyCheck — { identity, action, policies_applied, result, violated_policy? }`

## Stage 5 — Capability Verification

**Purpose**: Verify entity has a valid, non-expired, bounds-adequate capability for this action.

**Process**:
1. Resolve capability from CCA — verify it is Active
2. Verify capability bounds (resource, duration, scope, frequency, autonomy)
3. Verify capability authorization chain
4. On all pass: populate capability result
5. On failure: produce Capability.Denied Event

**Resource Bounds Check** (calls ROS):

| Resource | Check | Denial Code |
|---------|-------|-------------|
| Token budget | `ROS.budgetRemaining > action.estimated_tokens` | `VERIFY_CAP_001` |
| Memory | `ROS.budgetRemaining > action.estimated_memory` | `VERIFY_CAP_002` |
| Compute | `ROS.budgetRemaining > action.estimated_compute` | `VERIFY_CAP_003` |
| Storage | `ROS.budgetRemaining > action.estimated_storage` | `VERIFY_CAP_004` |
| Call frequency | `callCount < bound.max_calls` | `VERIFY_CAP_005` |

| Result | Action |
|--------|--------|
| Capability valid, bounds sufficient | Pass to Stage 6 |
| Capability not found | Deny. `VERIFY_CAP_001` |
| Capability expired | Deny. `VERIFY_CAP_002` |
| Capability revoked | Deny. `VERIFY_CAP_003` |
| Resource bounds exceeded | Deny. `VERIFY_CAP_004` |
| Autonomy insufficient | Deny. `VERIFY_CAP_005` |
| Grant chain invalid | Deny. `VERIFY_CAP_006`. Escalate S2 |

**Event**: `Verification.CapabilityCheck — { identity, action, capability_id, resource_usage_projections, result }`

## Stage 6 — Risk Assessment

**Purpose**: Assess risk of executing this action. Deny if risk exceeds threshold.

**Process**:
1. Calculate risk score: action type risk + entity risk factor + target risk + scope risk + historical risk
2. Compare against entity's max risk level, action's max risk level, system-wide cap
3. On within threshold: populate risk result
4. On exceedance: deny or require additional authorization based on severity

**Risk Tiers**:

| Tier | Score | Actions | Required Action |
|------|-------|---------|----------------|
| L0 — Negligible | 0.0–0.1 | None | Execute without additional checks |
| L1 — Low | 0.1–0.3 | Read non-sensitive | Execute, log |
| L2 — Medium | 0.3–0.5 | Write non-sensitive, common tools | Require User confirmation |
| L3 — High | 0.5–0.7 | Write sensitive, system commands, modify Missions | Require MFA + explicit approval |
| L4 — Critical | 0.7–1.0 | Delete data, terminate entities, modify policies | Require Security Council approval |

| Result | Action |
|--------|--------|
| Risk ≤ threshold | Pass to Stage 7 |
| Risk exceeds (low) | Require additional authorization |
| Risk exceeds (medium) | Deny. `VERIFY_RISK_001`. Escalate to parent |
| Risk exceeds (high) | Deny. `VERIFY_RISK_002`. Escalate to Security Council |

**Event**: `Verification.RiskAssessment — { identity, action, risk_score, risk_tier, result, required_approval? }`

## Stage 7 — Execution Authorization

**Purpose**: Issue the final execution authorization token, or deny.

**Process**:
1. Verify all prior stages passed
2. Calculate execution token scope (action, entity, capability, time window, resource budget, risk tier)
3. Reserve resources in ROS
4. Generate execution authorization token (signed by Security Council)
5. Produce Verification.Complete Event
6. Return token

**Execution Authorization Token**:
```json
{
  "token_id": "UUIDv7",
  "sub": "identity_id",
  "action_id": "action.id",
  "capability_id": "capability.id",
  "iat": "current_timestamp",
  "exp": "current_timestamp + token_ttl",
  "resource_budget": { "tokens": 0, "memory": 0, "compute": 0, "storage": 0 },
  "risk_tier": "L1",
  "stages": [ "all stage results" ],
  "scope_hash": "SHA256(action_serialized)"
}
```

**Token TTL**:

| Action Type | Default TTL | Max TTL |
|------------|-------------|---------|
| Tool (read) | 30s | 60s |
| Tool (write) | 10s | 30s |
| Tool (execute) | 5s | 15s |
| Model call | 60s | 300s |
| Resource allocation | 10s | 30s |
| Lifecycle transition | 5s | 15s |
| Batch operation | Scoped to batch | Max 5 min |

| Result | Action |
|--------|--------|
| All stages pass | Return ExecutionToken |
| Any stage denied | Return Denial with error code, reason, escalation level |

## Token Validation (Runtime Side)

The Runtime Engine validates before executing:
1. Call SecurityCouncil.verifyToken(execution_token) — validates signature + not revoked
2. Validate scope: action matches token's action_id, resource consumption within budget, current time within window
3. Execute action
4. On completion: return unused resources to ROS, produce execution Event
5. On failure: return unused resources, produce failure Event, mark token consumed

## Pipeline Performance

### Latency Targets

| Stage | Max Latency | P99 Target |
|-------|-------------|------------|
| Stage 1 — Identity | 5ms | 15ms |
| Stage 2 — Authentication | 2ms | 5ms |
| Stage 3 — Authorization | 10ms | 30ms |
| Stage 4 — Policy | 20ms | 60ms |
| Stage 5 — Capability | 15ms | 40ms |
| Stage 6 — Risk | 10ms | 25ms |
| Stage 7 — Execution Auth | 5ms | 10ms |
| **Total Pipeline** | **67ms** | **185ms** |

### Caching

| Cache | Store | TTL | Invalidated By |
|-------|-------|-----|----------------|
| Identity records | In-memory | 60s | IDS Event stream |
| Token verification | In-memory | Token TTL (max 5 min) | TRL Event stream |
| Authorization decisions | In-memory | 30s | CCA Event stream |
| Policy results | In-memory | 60s | Policy change Event |
| Risk assessments | In-memory | 15s | Violation Event |
| Capability records | In-memory | 30s | CCA Event stream |

### Parallelism

Strictly sequential. No stage executes in parallel. Rationale: dependency ordering, fail-fast optimization, simplified audit. Pipeline latency (67ms p50, 185ms p99) is acceptable relative to action execution time (1s–30s).

## Cross-Cutting Concerns

### Security
- Pipeline itself is secured. Pipeline configuration changes require MFA.
- Verification tokens signed with Security Council's private key.
- Pipeline subject to same invariants it enforces. Council does not skip stages for itself.

### Evidence
- Every stage produces an Event. Complete pipeline result recorded as Verification.Complete or Verification.Denied.
- Pipeline Events are immutable, stored in Event Store.

### Lifecycle
- Pipeline checks entity lifecycle state (Stage 1 — identity status, Stage 5 — state-dependent capabilities).
- State transitions are verified through the pipeline if initiated by an action.

### Capability Bounds
- Stage 5 (Capability Verification) is the primary bounds enforcement point.
- Resource pre-reservation (Stage 7) prevents resource exhaustion.

### Communication
- Pipeline called by ACF before routing any message. Failed messages not delivered.
- Results communicated back through ACF.

### Design DNA Compliance
- **R1**: Pipeline does one thing: verify actions before execution.
- **R2**: Stages depend on simpler primitives. Pipeline depends on IDS, Auth, CCA, ROS, LMS.
- **R10**: 7 stages, linear pipeline. No branching, no parallelism, no optimization tricks.
- **R13**: Fail closed — if any dependency unreachable, deny rather than allow.
- **R14**: Paved path is 7-stage pipeline. No shortcuts.
- **R15**: Stages are closed for modification. New verification via new stages (extending), not modifying existing ones.
