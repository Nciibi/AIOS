## Purpose

Certificate Authority Management (CAM) provides PKI (Public Key Infrastructure) services for AIOS. CAM issues, manages, and revokes X.509 certificates used by constitutional entities for identity assertion, TLS/mTLS communication, code signing, and document signing. CAM operates as a private Certificate Authority rooted in the AIOS platform Root CA, with intermediate CAs for organizational and functional domains.

CAM is the bridge between AIOS identity (Law 5) and cryptographic verification (Law 8). While IDS manages identity as a constitutional concept (who is this entity?), CAM manages the cryptographic credentials that prove identity (how does this entity prove it?). Every certificate issued by CAM is linked to an IDS identity record. Certificate revocation triggers identity status changes in IDS, and vice versa.

CAM does not replace the CSP (000-CSP.md). CSP provides the raw cryptographic operations; CAM builds certificate management on top of CSP's signing and key generation capabilities. Separation of concerns ensures that cryptographic operations are centrally audited while certificate lifecycle management follows PKI standards.

## Certificate Hierarchy

### CA Structure

```
AIOS Root CA (self-signed, offline)
  └── AIOS Platform CA (online, cross-signed)
       ├── Organizational CAs (one per Organization)
       │    ├── Identity Certificates (Workers, Sessions)
       │    ├── Code Signing Certificates (Templates, Runtimes)
       │    └── Document Signing Certificates (Evidence, Policies)
       ├── Security Council CA
       │    └── Inter-institutional Certificates
       └── Federation CA
            └── Cross-instance Trust Certificates
```

### Root CA Protection

- The Root CA private key is generated in an air-gapped HSM
- Root CA key activation requires 3-of-5 multi-party authorization from Security Council members
- Root CA key is stored offline, physically secured, and never connected to any network
- Root CA certificate has a 25-year validity with 5-year re-signing ceremonies
- Root CA usage is restricted to signing Platform CA and Security Council CA certificates

### Certificate Profiles

| Profile | Purpose | Validity | Key Usage | Extensions |
|---------|---------|----------|-----------|------------|
| Identity | Entity identity authentication | 1 year | digitalSignature, keyEncipherment | SubjectAltName: entity ID, entity type |
| TLS Server | ACF endpoint TLS | 2 years | serverAuth, digitalSignature | SAN: DNS names, IP addresses |
| TLS Client | mTLS client authentication | 1 year | clientAuth, digitalSignature | SAN: entity ID |
| Code Signing | Template and Runtime signing | 3 years | codeSigning | OID for template hash |
| Document Signing | Evidence and policy signing | 6 months | nonRepudiation | OID for document type |
| Federation | Cross-instance trust | 2 years | digitalSignature, keyCertSign | Path length: 1 |

## Certificate Lifecycle

### Issuance Flow

```
Subject → CAM Gateway → Validation → Approval → CSP Signing → Certificate → Distribution
```

1. **Subject**: Entity requests certificate via ACF with identity proof (IDS-verified)
2. **CAM Gateway**: Validates requestor has `cert.request` capability
3. **Validation**: CAM verifies identity, certificate profile eligibility, and CSR (Certificate Signing Request) integrity
4. **Approval**: Certificate issuance is authorized by the issuing CA's policy (automated for standard profiles, manual for elevated)
5. **CSP Signing**: CSP generates the certificate using the issuing CA's private key
6. **Distribution**: CAM delivers the signed certificate and CA chain to the requestor via encrypted ACF message
7. **Registration**: Certificate fingerprint and serial number registered with IDS and the Certificate Transparency log

### Renewal

- Certificates may be renewed up to 30 days before expiry
- Renewal reuses the same subject public key (key unchanged) or requests a new key pair
- Renewed certificates have a new serial number; the old certificate remains in the CT log
- Renewal re-validates identity — if the entity's status has changed, renewal is denied

### Revocation

| Revocation Reason | Trigger | CRL Distribution |
|------------------|---------|------------------|
| Key Compromise | CSP reports key compromised | Immediate, delta CRL within 5 minutes |
| Entity Termination | OSYS terminates entity | Immediate, full CRL within 15 minutes |
| Certificate Misuse | Security Council investigation | Immediate, full CRL within 5 minutes |
| Affiliation Change | Entity changes Organization | Standard CRL schedule |
| Supersession | New certificate issued | Standard CRL schedule |

### Certificate Transparency

All issued certificates are logged in an append-only Certificate Transparency (CT) log. The CT log is a public, verifiable record of all AIOS certificates. Any entity may audit the CT log to detect mis-issuance. CAM monitors the CT log for unauthorized certificates. The CT log is backed by the EAS evidence chain for integrity.

### OCSP Responder

CAM operates an Online Certificate Status Protocol (OCSP) responder for real-time certificate status checks:
- OCSP responses are pre-signed with 4-hour validity for performance
- Revocation updates are reflected in OCSP within 5 minutes
- OCSP responses are cached by ACF for the response validity period
- OCSP stapling is required for all TLS connections

## Certificate Validation

### Validation Process

When an entity presents a certificate for authentication:

1. **Chain Building**: Build certificate path from leaf to trusted Root CA
2. **Signature Verification**: Verify each certificate's signature using the issuer's public key (via CSP)
3. **Validity Period**: Verify current time is within each certificate's validity window
4. **Revocation Status**: Check CRL or OCSP for each certificate in the chain
5. **Key Usage**: Verify the certificate's key usage permits the requested operation
6. **Name Constraints**: Verify the entity ID in the certificate matches the presented identity
7. **Path Length**: Verify the chain length does not exceed the path length constraint

### Validation Caching

| Cache | TTL | Invalidated By |
|-------|-----|----------------|
| Chain building | 1 hour | New CA certificate issuance |
| Revocation status (CRL) | CRL nextUpdate | New CRL publication |
| Revocation status (OCSP) | OCSP response validity | Revocation event |
| Path validation | 1 hour | CA hierarchy change |

## Cross-Cutting Concerns

### Security

- Root CA private key is air-gapped. No network access. Physical access requires 3-of-5 multi-party authorization.
- All CA signing keys are HSM-protected. Private keys never exist outside HSM memory.
- Certificate issuance requires identity verification through IDS. No anonymous certificates.
- CAM enforces algorithm policy via CSP. Weak algorithms are rejected at the profile level.

### Evidence

- Every certificate issuance, renewal, and revocation is recorded as evidence in EAS.
- CRLs are signed by the issuing CA and logged in EAS for auditability.
- The Certificate Transparency log is backed by the EAS evidence Merkle-DAG.

### Lifecycle

- Certificates have strict validity periods. Expired certificates are rejected by all validators.
- Entity lifecycle transitions trigger certificate revocation. A terminated entity's certificates are revoked.
- CA certificates have shorter validity than their subjects to encourage rotation.

### Capability Bounds

- Certificate issuance requires the `cert.request` capability. This capability is scoped to the certificate profile.
- An entity may only request certificates for its own identity. Proxy requests require explicit delegation.
- Rate limiting: max 100 certificate requests per entity per hour.

### Communication

- Certificate requests and deliveries flow through ACF encrypted channels.
- CRL distribution uses ACF publish-subscribe for real-time updates.
- OCSP responses are delivered through ACF with low-latency routing.

### Design DNA

| Rule | Assessment | Rationale |
|------|-----------|-----------|
| R1 — Modulsingularity | Compliant | CAM does one thing: manage X.509 certificate lifecycle |
| R2 — Dependency Order | Compliant | CAM depends on CSP and IDS; no circular dependencies |
| R3 — DRY | Compliant | Certificate profiles defined once per profile type |
| R4 — Builder Pattern | Compliant | Certificates constructed by CertificateBuilder with chain validation |
| R5 — Liskov Substitution | Compliant | Any CA implementation conforms to the CertificateAuthority interface |
| R6 — DI over Singletons | Compliant | CAM service injected into ACF TLS, IDS, and verification pipeline |
| R7 — Tests Exist | Compliant | Unit tests for chain building, integration tests for CSP integration |
| R8 — Tests Fast | Compliant | Issuance test suite <60s; CRL generation <30s |
| R9 — Deterministic Tests | Compliant | Same CSR + same CA key always produces deterministic certificate |
| R10 — Prefer Simpler | Compliant | Standard X.509 PKI — no custom certificate formats or extensions |
| R11 — Refactor over Rewrite | Compliant | PKI evolution happens through profile updates, not CAM rewrites |
| R12 — Embrace Errors | Compliant | Each issuance failure has a validation step code and CA-issued error message |
| R13 — Design for Failure | Compliant | Root CA is offline; Platform CA is clustered for HA; OCSP responders are load-balanced |
| R14 — Paved Path | Compliant | All certificate operations go through the single CAM→CSP→HSM path |
| R15 — Open/Closed | Compliant | New profiles are added via profile definitions, not CAM core modifications |
