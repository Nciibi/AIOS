## Purpose

The Cryptographic Service Provider (CSP) is the central cryptographic operations service for AIOS. It provides standardized, auditable, and verifiable cryptographic primitives to all constitutional entities. CSP does not implement cryptographic algorithms from scratch — it wraps and manages FIPS 140-3 validated cryptographic libraries, enforcing consistent algorithm selection, key management, and operational security for all system components.

CSP serves three constitutional roles. First, it provides identity cryptography: generating key pairs, signing identity assertions, and verifying signatures for Law 5 (Identity). Second, it provides evidence cryptography: computing hashes for Merkle-DAG evidence chains, sealing records, and verifying chain integrity for Law 4 (Evidence). Third, it provides channel cryptography: establishing encrypted communication sessions between entities for secure information exchange.

CSP is not a general-purpose encryption library for arbitrary use. Every cryptographic operation is authenticated, logged, and bounded. CSP enforces algorithm policy: which algorithms are permitted, their minimum key strengths, and their approved use cases. No entity may request a cryptographic operation that violates CSP policy. This ensures that the entire AIOS platform operates within a consistent cryptographic posture.

## Cryptographic Operations

### Supported Operations

| Operation | Algorithm(s) | Min Key/Output Strength | Use Case |
|-----------|-------------|------------------------|----------|
| Key Generation (Asymmetric) | ECDSA P-384, Ed25519 | P-384 | Identity keys, signing keys |
| Key Generation (Symmetric) | AES-256-GCM | 256-bit | Encryption keys, session keys |
| Key Generation (KEM) | ML-KEM (CRYSTALS-Kyber) | Level 3 | Post-quantum key encapsulation |
| Signing | ECDSA P-384, Ed25519 | P-384 | Identity assertions, capability signing, evidence sealing |
| Signature Verification | ECDSA P-384, Ed25519 | P-384 | Identity verification, token verification |
| Hash | SHA-384, SHA3-384 | 384-bit | Evidence chain hashing, content integrity |
| Symmetric Encryption | AES-256-GCM | 256-bit, 96-bit nonce | Session encryption, payload encryption |
| Asymmetric Encryption | RSA-OAEP (4096-bit), X25519 | 4096-bit / 256-bit | Key encapsulation for secure channels |
| Key Derivation | HKDF-SHA384 | 384-bit | Key derivation from shared secrets |
| Random Number | DRBG (CTR_DRBG with AES-256) | 256-bit entropy | Nonces, salts, capability IDs |

### Prohibited Operations

The following operations are denied by CSP policy:

- MD4, MD5 (any purpose)
- SHA-1 (except for legacy verification, logged)
- RC4, DES, 3DES
- RSA with key size < 2048-bit
- DSA with key size < 2048-bit
- ECDSA with curve size < P-256
- Any algorithm not on the approved list
- Any operation with a key that has been compromised or revoked

## Key Lifecycle

### Key Hierarchy

```
Master Key (HSM-protected, air-gapped)
  ├── Signing Key (Identity, evidence, capabilities)
  ├── Encryption Key (Session encryption, channel encryption)
  ├── KEM Key (Post-quantum encapsulation)
  ├── Evidence Sealing Key (Merkle-DAG chain sealing)
  └── Recovery Key (Emergency decryption, multi-party controlled)
```

Each key type follows its own lifecycle:

| Stage | Description | CSP Action |
|-------|-------------|------------|
| Pre-Activation | Key generated but not yet active | Stored in encrypted form, not usable |
| Active | Key is operational | Allowed for specified operations |
| Suspended | Key temporarily disabled | Operations return error, key retained |
| Compromised | Key is suspected compromised | Immediate revocation, alert, audit |
| Destroyed | Key is cryptographically erased | All material zeroized, certificate revoked |

### Key Rotation

| Key Type | Rotation Period | Trigger Events |
|----------|----------------|----------------|
| Identity Signing Key | 1 year | Entity creation, entity role change |
| Evidence Sealing Key | 6 months | Chain milestone, security audit |
| Session Encryption Key | Per-session | Session creation |
| KEM Key | 3 months | Post-quantum security review |
| Master Key | 5 years | Security Council directive, annual audit |

### Key Storage

- Private keys are stored in an HSM (Hardware Security Module) or secure enclave at rest.
- Keys in use are held in HSM memory only — never in system RAM outside the HSM boundary.
- Key backups are encrypted under the Master Key and split using Shamir's Secret Sharing (5-of-9 threshold).
- Key recovery requires multi-party authorization from the Security Council.

## CSP Operations

### Signing Flow

```typescript
interface SigningRequest {
  keyId: string;
  payload: Buffer;            // Content to sign
  purpose: "identity" | "evidence" | "capability" | "token";
  context?: string;           // Optional domain separation string
}

interface SigningResult {
  signature: Buffer;
  algorithm: string;
  keyId: string;
  signingTimestamp: Timestamp;
  signatureId: string;         // Unique signature identifier for audit
}
```

### Verification Flow

```typescript
interface VerificationRequest {
  signature: Buffer;           // Signature to verify
  payload: Buffer;             // Original content
  algorithm: string;           // Algorithm used
  publicKey: Buffer;           // Signer's public key
}

interface VerificationResult {
  isValid: boolean;
  algorithm: string;
  verifiedAt: Timestamp;
  details?: string;            // If invalid, reason for failure
}
```

## Cross-Cutting Concerns

### Security

- CSP never exposes private key material to callers. All operations are performed inside the HSM boundary.
- Cryptographic policy is enforced by CSP, not by callers. A caller cannot override algorithm selection.
- Rate limiting is applied per-caller: max 1000 signing operations per second per entity.
- Side-channel resistance is required for all implementations. Timing attacks must be mitigated.

### Evidence

- Every cryptographic operation produces an audit event logged by EAS.
- Key lifecycle events are independently verifiable through the evidence chain.
- The CSP policy itself is evidence — any change to approved algorithms is recorded and auditable.

### Lifecycle

- Keys follow the lifecycle stages defined above. A key in any stage other than Active returns an error for operations.
- When an entity is terminated, its keys are suspended and then destroyed according to the key rotation schedule.
- Session keys are destroyed when the session ends.

### Capability Bounds

- Entities must have the `crypto.sign` or `crypto.verify` capability to invoke CSP operations.
- Key generation is restricted to entities with `crypto.key_generate` capability (typically IDS, CCA, EAS).
- Resource budgets apply: each signature consumes from the entity's token and compute budget.

### Communication

- All CSP requests and responses flow through ACF. No direct HSM access is permitted.
- CSP responses include a signature over the result for non-repudiation of CSP's own operations.
- CSP health is published through ACF for monitoring.

### Design DNA

| Rule | Assessment | Rationale |
|------|-----------|-----------|
| R1 — Modulsingularity | Compliant | CSP does one thing: provide audited cryptographic operations |
| R2 — Dependency Order | Compliant | CSP depends on HSM firmware; nothing depends on CSP for its own cryptographic operations |
| R3 — DRY | Compliant | Algorithm policy defined in one place, enforced by CSP for all callers |
| R4 — Builder Pattern | Compliant | Key objects constructed by KeyBuilder within HSM boundary |
| R5 — Liskov Substitution | Compliant | Any HSM implementation conforms to CSP interface |
| R6 — DI over Singletons | Compliant | CSP service injected into IDS, CCA, EAS, and other consumers |
| R7 — Tests Exist | Compliant | Unit tests for each operation, integration tests for HSM interaction |
| R8 — Tests Fast | Compliant | Deterministic tests use software keystore; full HSM tests run nightly |
| R9 — Deterministic Tests | Compliant | Seeded DRBG ensures deterministic test outputs |
| R10 — Prefer Simpler | Compliant | Operations are thin wrappers over HSM functions — no complex orchestration |
| R11 — Refactor over Rewrite | Compliant | Algorithm additions are configuration changes, not CSP rewrites |
| R12 — Embrace Errors | Compliant | Every cryptographic failure has a unique error code and audit trail |
| R13 — Design for Failure | Compliant | HSM cluster provides redundancy; if primary HSM fails, failover is automatic |
| R14 — Paved Path | Compliant | All cryptographic operations go through CSP — no entity calls crypto libraries directly |
| R15 — Open/Closed | Compliant | New algorithms are added via HSM firmware updates and CSP policy changes |
