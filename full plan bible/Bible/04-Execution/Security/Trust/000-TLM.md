## Purpose

The Trust Lifecycle Manager (TLM) is the constitutional authority for establishing, maintaining, evaluating, and revoking trust relationships within AIOS. Trust in AIOS is not a binary state — it is a continuous, quantitative score that reflects the system's confidence in an entity's identity, behavior, and compliance with constitutional principles. Under Law 8 (Verification-First), trust scores are consumed by the Risk Engine (000-RE.md) as a primary input to entity risk assessment.

Trust in AIOS has three dimensions. First, identity trust: confidence that the entity is who it claims to be, established through identity verification (Law 5) and cryptographic attestation (Law 8). Second, behavioral trust: confidence based on the entity's historical actions — compliance with policies, absence of violations, consistency with expected behavior patterns. Third, contextual trust: confidence modulated by the operational context — threat level, regulatory environment, and peer consensus.

TLM does not grant or deny authorization. It produces trust scores that other services consume. The Risk Engine uses trust as an input to risk scoring. The verification pipeline uses trust to inform authentication requirements (low trust → step-up authentication). Organizations use trust to make delegation decisions (high trust → broader delegation allowed). TLM's role is to maintain an accurate, evidence-based assessment of how much each entity should be trusted at any given time.

## Trust Model

### Trust Score

```typescript
interface TrustScore {
  entityId: string;
  overall: number;                          // 0.0 (no trust) to 1.0 (absolute trust)
  components: {
    identityTrust: number;                  // Identity verification strength
    behavioralTrust: number;                // Historical behavior assessment
    contextualTrust: number;                // Current context modulation
  };
  confidence: number;                       // Confidence in the score (0.0 to 1.0)
  lastUpdated: Timestamp;
  expiryTime: Timestamp;                    // Score expires and must be refreshed
  scoreHistory: TrustScoreSnapshot[];       // Last 10 score snapshots
}
```

### Component Weights

| Component | Default Weight | Range | Description |
|-----------|---------------|-------|-------------|
| Identity Trust | 0.35 | 0.0–1.0 | Based on authentication strength, identity age, key attestation |
| Behavioral Trust | 0.50 | 0.0–1.0 | Based on policy compliance, violation history, action consistency |
| Contextual Trust | 0.15 | −0.3–+0.3 | Modulator based on threat level, regulatory zone, peer signals |

### Score Calculation

```
overall = clamp(
    w_identity * identityTrust +
    w_behavioral * behavioralTrust +
    w_contextual * contextualTrust,
    0.0, 1.0
)
```

## Trust Components

### Identity Trust

Identity trust measures confidence in the entity's identity:

| Factor | Weight | Description | Decay |
|--------|--------|-------------|-------|
| Authentication strength | 0.30 | MFA > cryptographic > password > API key | None (per session) |
| Identity age | 0.25 | Older identities are more trusted | Increases with age, cap at 30 days |
| Key attestation | 0.20 | HSM-backed keys > software keys | Per attestation cycle |
| Identity chain | 0.15 | Direct issuance > delegated > self-issued | None |
| Revocation history | 0.10 | Number of identity revocations | Negative factor, decays over 90 days |

### Behavioral Trust

Behavioral trust measures confidence based on the entity's actions:

| Factor | Weight | Description | Decay |
|--------|--------|-------------|-------|
| Policy compliance rate | 0.30 | Ratio of compliant actions to total actions | 7-day sliding window |
| Violation severity | 0.25 | Weighted sum of violation severities | Negative factor, decays over 30 days |
| Action consistency | 0.20 | Variance from entity's normal action pattern | 14-day sliding window |
| Escalation history | 0.15 | Number of actions requiring escalation | Negative factor |
| Peer comparison | 0.10 | Trust relative to peer group (same type, org) | 7-day recalculation |

### Contextual Trust

Contextual trust modulates the score based on current environment:

| Factor | Modulation | Trigger |
|--------|-----------|---------|
| System threat level | −0.10 per level above normal | Threat level = elevated, high, severe |
| Regulatory zone | −0.05 per restricted zone | GDPR, HIPAA, PCI actions |
| New location | −0.10 | Entity is acting from previously unseen location |
| New device | −0.05 | Entity is using previously unseen client |
| Peer trust consensus | ±0.05 | Average trust of peer entities |

## Trust Lifecycle

### Score Refresh

| Entity Type | Refresh Interval | Trigger Events |
|-------------|-----------------|----------------|
| Worker | Per-action (real-time) | Every action through verification pipeline |
| Session | Every 5 minutes | Activity detected, token refresh |
| Engine | Every 15 minutes | Periodic health check |
| Organization | Every 1 hour | Periodic, or on member violation event |
| User | Every 24 hours | Daily recalculation |

### Score Decay

Trust scores decay over time without positive signals:

| Component | Decay Rate | Floor |
|-----------|------------|-------|
| Identity trust | 0.01 per day without authentication | 0.3 |
| Behavioral trust | 0.02 per day without action | 0.2 |
| Contextual trust | 0.05 per hour (returns to 0.0) | 0.0 |
| Overall | Fastest decay among components | 0.1 |

### Trust Events That Change Scores

| Event | Impact | Duration |
|-------|--------|----------|
| Successful authentication | Identity trust +0.05 | Permanent (slow decay) |
| MFA verification | Identity trust +0.10 | Permanent |
| Policy violation (minor) | Behavioral trust −0.10 | 7-day recovery |
| Policy violation (major) | Behavioral trust −0.30 | 30-day recovery |
| Constitutional violation | Behavioral trust −0.50 | 90-day recovery |
| Consistent compliance (7 days) | Behavioral trust +0.02/day | Cumulative |
| Successful evidence verification | Behavioral trust +0.05 | Permanent |
| Key rotation completed | Identity trust +0.03 | Permanent |

## Trust Evaluation

### Evaluation Triggers

Trust scores are evaluated and updated under these conditions:

1. **Pre-action (Stage 6)**: The Risk Engine requests the entity's current trust score as an input to risk assessment
2. **Post-action**: After an action completes, the outcome updates the entity's behavioral trust
3. **Periodic refresh**: Scores are refreshed according to the refresh intervals above
4. **Event-driven**: Certain events (violation, authentication, escalation) trigger immediate trust recalculation

### Trust Thresholds

| Trust Range | Label | Implication |
|-------------|-------|-------------|
| 0.90–1.00 | Fully Trusted | Full delegation, minimal additional controls |
| 0.70–0.89 | Trusted | Standard operations, normal controls |
| 0.50–0.69 | Marginally Trusted | Enhanced monitoring, step-up auth required |
| 0.30–0.49 | Low Trust | Restricted delegation, frequent re-verification |
| 0.00–0.29 | Untrusted | Maximum controls, Security Council notification |

### Trust Recovery

Entities with low trust scores may recover through consistent positive behavior:

```
Recovery Time = (base_damage / daily_positive_capacity) * recovery_factor
```

- Minor violation recovery: ~3-7 days of consistent compliance
- Major violation recovery: ~14-30 days
- Constitutional violation recovery: ~60-90 days
- Recovery is automatic — no manual trust restoration

## Cross-Cutting Concerns

### Security

- Trust scores cannot be directly modified by any entity, including the entity itself. Trust is earned through behavior, not assigned.
- Trust score computation is deterministic and auditable. Any score change can be traced to specific events and evidence.
- A compromised entity's trust score drops rapidly as violation events are processed, limiting the blast radius.

### Evidence

- Every trust score calculation is recorded as evidence with all contributing factors.
- Trust-affecting events are traceable to the original evidence (violation event, authentication event, etc.).
- Trust score history is retained for 90 days for trend analysis and forensic investigation.

### Lifecycle

- Trust scores are reset when an entity is re-created (new identity, new lifecycle). The new entity starts at the default trust score for its type (0.50 for Workers, 0.70 for Organizations).
- Entity suspension freezes the trust score. Entity termination archives the trust history.
- Trust scores above 0.95 require sustained positive behavior over a minimum 30-day period.

### Capability Bounds

- Trust scores are consumed by the Risk Engine but are not capability bounds. An entity with low trust may still execute actions, but with enhanced controls.
- Entities with trust scores below 0.30 may have their delegation capabilities restricted.
- Trust recovery is bounded — an entity cannot recover trust faster than the recovery formulas allow.

### Communication

- Trust scores are distributed through ACF to consumers (Risk Engine, verification pipeline, organizations).
- Trust-affecting events are consumed from the ACF event stream (EAS evidence events, violation events).
- Trust threshold crossing events are published to ACF for alerting and monitoring.

### Design DNA

| Rule | Assessment | Rationale |
|------|-----------|-----------|
| R1 — Modulsingularity | Compliant | TLM does one thing: maintain quantitative entity trust scores |
| R2 — Dependency Order | Compliant | TLM depends on EAS and IDS; Risk Engine depends on TLM |
| R3 — DRY | Compliant | Trust scoring formulas defined once per entity type |
| R4 — Builder Pattern | Compliant | TrustScore constructed by TrustCalculator from component factors |
| R5 — Liskov Substitution | Compliant | Any trust model implementation conforms to TrustModel interface |
| R6 — DI over Singletons | Compliant | TLM injected into Risk Engine and verification pipeline |
| R7 — Tests Exist | Compliant | Unit tests for each trust component, integration tests for score combination |
| R8 — Tests Fast | Compliant | Trust calculation target <5ms; test suite <30s |
| R9 — Deterministic Tests | Compliant | Same event history always produces same trust score |
| R10 — Prefer Simpler | Compliant | Weighted linear combination with decay — no ML, no probabilistic trust |
| R11 — Refactor over Rewrite | Compliant | Trust model evolves through weight configuration, not engine rewrites |
| R12 — Embrace Errors | Compliant | Each trust calculation records the contributing events and factor weights |
| R13 — Design for Failure | Compliant | If TLM is unreachable, Risk Engine uses last cached trust score (degraded) |
| R14 — Paved Path | Compliant | All trust evaluations go through the single TLM scoring path |
| R15 — Open/Closed | Compliant | New trust factors added through factor registration, not engine changes |
