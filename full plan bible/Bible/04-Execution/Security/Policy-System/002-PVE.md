## Purpose

The Policy Validation Engine (PVE) is the gatekeeper for all policy definitions in the Policy System. Before any policy becomes active, PVE validates it across four dimensions: syntactic correctness (is it valid PDG?), semantic consistency (does it make sense?), constitutional compliance (does it respect higher-order policies?), and safety (is it bounded and safe to evaluate?). A policy that fails any dimension is rejected and cannot progress past the Review stage.

PVE serves two operational modes. In **pre-authoring mode**, PVE provides real-time feedback to policy authors as they write policies, flagging errors, conflicts, and compliance issues through the authoring interface. In **pre-activation mode**, PVE performs a comprehensive validation pass before a policy transitions from Review to Active. Both modes use the same validation pipeline to ensure consistency.

PVE is not optional. Every policy, regardless of domain (constitutional, organizational, mission, safety, regulatory), must pass PVE validation before activation. There is no bypass mechanism. The PVE itself is secured — its validation rules are defined at the constitutional level and cannot be modified by policy authors.

## Validation Pipeline

### Pipeline Overview

```
PDG Input → Syntax → Semantic → Compliance → Safety → Report
```

The pipeline is sequential and fail-fast. If any stage produces errors, subsequent stages are not executed.

### Stage 1: Syntax Validation

**Purpose**: Verify the policy is syntactically valid PDG (001-PDG.md).

**Checks performed**:
- Tokenization succeeds (no unrecognized characters or malformed literals)
- Grammar production rules are followed (no structural errors)
- All parentheses, brackets, and braces are balanced
- All strings are properly terminated
- Identifiers conform to naming rules
- Reserved keywords are not used as identifiers

**Output**: Validated AST or list of syntax errors with line, column, and expected token information.

### Stage 2: Semantic Validation

**Purpose**: Verify the policy is semantically consistent and well-typed.

**Checks performed**:
- Type checking: all operators receive compatible operand types
- Attribute resolution: all referenced attributes exist in the attribute registry
- Function resolution: all called functions exist with matching signatures
- Scope validation: attribute references are within the policy domain's allowed scope
- Uniqueness: no duplicate rule effects for the same condition
- Reference integrity: all set literals contain elements of consistent type

**Semantic Rules**:

| Rule | Description | Violation |
|------|-------------|-----------|
| Type Safety | No implicit type coercion | `"admin" == 1` is an error |
| Attribute Exists | Referenced attribute must be registered | `subject.nonexistent` is an error |
| Attribute Scoping | Policy domain may restrict accessible attributes | Organizational policy cannot reference `env.threat_level` |
| Effect Completeness | Policy must have at least one rule | Empty policy is an error |
| No Self-Reference | Policy cannot reference its own ID | Circular reference detection |
| Budget Compliance | Estimated cost must not exceed budget | Cost exceeds 10000 units is a warning (pre-authoring) |

### Stage 3: Constitutional Compliance

**Purpose**: Verify the policy does not violate constitutional policies or Physics invariants.

**Checks performed**:
- Policy priority is within the allowed range for its domain
- Policy does not attempt to override a higher-priority domain
- Policy does not violate constitutional invariant rules (Physics/008-Security.md)
- Policy effect is compatible with constitutional rights
- Policy obligations do not require constitutionally prohibited actions

**Constitutional Rules**:

| Rule | Description | Enforcement |
|------|-------------|-------------|
| Priority Protection | Organizational policies cannot set priority < 300 | Priority clamped at domain minimum |
| Law 4 Compliance | Policy cannot authorize invisible decisions | Evidence recording obligation is required |
| Law 5 Compliance | Policy cannot bypass identity verification | Subject identity must be verified |
| Law 7 Compliance | Policy cannot expand capability bounds | Policy can restrict but never expand |
| Law 8 Compliance | Policy cannot bypass verification pipeline | Verification cannot be waived |
| Rights Preservation | Policy cannot violate entity constitutional rights | Reviewed against Rights Registry |

### Stage 4: Safety Validation

**Purpose**: Verify the policy is safe to evaluate — bounded, non-recursive, and non-exploitable.

**Checks performed**:
- **Boundedness**: All loops, recursions, and iterations are statically bounded
- **Non-Recursion**: No direct or indirect rule recursion
- **Determinism**: Policy evaluation is deterministic for the same input
- **Non-Interference**: Policy evaluation has no side effects
- **DoS Resistance**: Evaluation cost is within budget and attribute resolution is bounded
- **ReDoS Resistance**: All regex patterns are checked for exponential backtracking

**Safety Guarantees**:

| Property | Guarantee | Mechanism |
|----------|-----------|-----------|
| Termination | Every policy evaluation terminates | No loops, no recursion, bounded cost |
| Determinism | Same input → same result | No external state, no randomness |
| No Side Effects | Evaluation does not modify system state | Grammar forbids mutation |
| Bounded Memory | Evaluation allocates bounded memory | Static analysis of allocation paths |
| Bounded Time | Evaluation completes within budget | Cost budget enforced at runtime |

## Conflict Detection

### Intra-Policy Conflicts

Detected within a single policy definition:

| Conflict | Detection | Resolution |
|----------|-----------|------------|
| Contradictory rules | Same condition with allow and deny | Error: contradictory rules |
| Redundant rules | Condition subsumed by another rule | Warning: redundant rule |
| Unreachable rules | Condition is always false | Warning: unreachable rule |
| Tautological rules | Condition is always true | Warning: overly broad rule |

### Inter-Policy Conflicts

Detected across policies in the same domain:

| Conflict | Detection | Resolution |
|----------|-----------|------------|
| Direct conflict | Same action/resource with different effects | Error: conflict with Policy X |
| Partial overlap | Overlapping condition scopes with different effects | Warning: potential conflict with Policy X |
| Subsumption | One policy's conditions fully contain another's | Warning: Policy X is subsumed by Policy Y |
| Priority inversion | Lower-priority policy contradicts higher-priority | Error: Policy X violates higher-priority Policy Y |

### Conflict Resolution Requirements

For a policy to pass validation, all intra-policy conflicts must be resolved (errors). Inter-policy conflicts are reported as warnings but do not block activation unless they involve constitutional or safety policies. The policy author must acknowledge all warnings before activation.

## Validation Output

### Validation Report Structure

```typescript
interface ValidationReport {
  policyId: string;
  version: string;
  result: "pass" | "fail" | "pass_with_warnings";
  stages: {
    syntax: StageResult;
    semantics: StageResult;
    compliance: StageResult;
    safety: StageResult;
  };
  conflicts: ConflictReport[];
  warnings: ValidationWarning[];
  errors: ValidationError[];
  metrics: {
    astNodes: number;
    estimatedCost: number;
    attributeCount: number;
    ruleCount: number;
    validationTimeMs: number;
  };
}
```

### Severity Levels

| Level | Description | Action Required |
|-------|-------------|----------------|
| Error | Validation fails. Policy cannot proceed. | Must be fixed |
| Warning | Validation passes but issue is noted. | Must be acknowledged |
| Info | Informational observation. | Optional review |

## Events

| Event Type | Produced When | Fields |
|-----------|---------------|--------|
| `PVE.ValidationStarted` | Validation of a policy begins | policy_id, version, validation_mode |
| `PVE.ValidationCompleted` | Validation completes with result | policy_id, version, result, error_count, warning_count |
| `PVE.SyntaxError` | A syntax error is detected | policy_id, location, expected_tokens, actual_token |
| `PVE.SemanticError` | A semantic error is detected | policy_id, error_type, context |
| `PVE.ComplianceViolation` | A constitutional compliance violation | policy_id, violated_rule, severity |
| `PVE.SafetyViolation` | A safety violation is detected | policy_id, violation_type, details |
| `PVE.ConflictDetected` | A conflict between policies is found | policy_id_a, policy_id_b, conflict_type, scope |
| `PVE.PolicyRejected` | A policy is rejected by PVE | policy_id, version, rejection_reason, validation_report_id |

## Cross-Cutting Concerns

### Security

- PVE is tamper-proof — its validation rules are defined at the constitutional level and cannot be modified through policy authoring interfaces.
- The PVE itself is validated by a separate instance before deployment (self-validation).
- Validation rules are versioned and immutable after deployment.

### Evidence

- Every validation run produces a comprehensive report stored in EAS.
- Historical validation reports are retained for audit and regression detection.
- Policy rejections include full validation context for post-mortem analysis.

### Lifecycle

- PVE validation is a mandatory gateway in the policy lifecycle (Draft → Review → **PVE** → Active).
- A policy that fails validation returns to Draft status with the validation report attached.
- Re-validation is required when a policy is modified after initial validation.

### Capability Bounds

- PVE enforces that policies cannot exceed their domain's authority bounds.
- Policy evaluation cost budgets are validated to prevent resource exhaustion.
- Attribute access scoping is enforced at the domain level.

### Communication

- PVE communicates validation results through ACF to the policy author and Policy System.
- Real-time feedback in pre-authoring mode uses ACF streaming channels.
- Validation reports are published as evidence through EAS.

### Design DNA

| Rule | Assessment | Rationale |
|------|-----------|-----------|
| R1 — Modulsingularity | Compliant | PVE does one thing: validate policies before activation |
| R2 — Dependency Order | Compliant | PVE depends on PDG parser; nothing depends on PVE |
| R3 — DRY | Compliant | Validation rules defined in one registry, applied uniformly |
| R4 — Builder Pattern | Compliant | ValidationReport constructed by ReportBuilder from stage results |
| R5 — Liskov Substitution | Compliant | Any validator implementation conforms to the Validator interface |
| R6 — DI over Singletons | Compliant | PVE injected into Policy Builder for pre-activation validation |
| R7 — Tests Exist | Compliant | Unit tests for each validation rule, integration tests for pipeline stages |
| R8 — Tests Fast | Compliant | Validation target <50ms for average policy; test suite <60s |
| R9 — Deterministic Tests | Compliant | Same policy always produces same validation result |
| R10 — Prefer Simpler | Compliant | 4-stage linear pipeline — no branching, no iterative refinement |
| R11 — Refactor over Rewrite | Compliant | Validation rules are added through registry updates, not engine rewrites |
| R12 — Embrace Errors | Compliant | Each error identifies the stage, rule, and exact location in the policy |
| R13 — Design for Failure | Compliant | If PVE crashes during validation, the policy remains in Draft status |
| R14 — Paved Path | Compliant | PVE is the only path from Draft to Active — no policy bypasses validation |
| R15 — Open/Closed | Compliant | New validation rules added through rule registration, not PVE engine changes |
