## Purpose

The Policy System (PS) is the central authority for policy definition, storage, evaluation, and lifecycle management within AIOS. Under Law 8 (Verification-First), Stage 4 of the verification pipeline evaluates every action against applicable policies. PS provides the policy storage, retrieval, and evaluation engine that powers Stage 4. It is the institutional memory of what is permitted, prohibited, and required across the platform.

PS is not a single monolithic policy engine. It is a policy framework that supports multiple policy domains (constitutional, organizational, mission, safety, regulatory) expressed through multiple policy languages (Policy Definition Grammar for custom policies, attribute predicates for ABAC, and capability constraints for delegation). Each policy domain has its own authoring authority, review process, and lifecycle, but all policies are stored in the same policy repository and evaluated by the same evaluation engine.

Policies in PS are first-class constitutional entities. They have identities (issued by IDS), lifecycles (managed by LMS), capabilities (assigned by CCA), and produce evidence (recorded by EAS). A policy is not a configuration file — it is a constitutional directive with full lifecycle management, audit trail, and enforcement guarantees.

## Policy Model

### Policy Structure

```typescript
interface Policy {
  id: string;                          // UUIDv7
  name: string;                        // Human-readable name
  domain: PolicyDomain;                // Constitutional, Organizational, Mission, Safety, Regulatory
  version: string;                     // Semantic version
  priority: number;                    // Lower number = higher priority (0 = constitutional)
  effect: "allow" | "deny" | "obligation";
  rules: RuleExpression[];             // Conditions expressed in PDG
  obligations?: Obligation[];          // Actions that must be performed
  validFrom: Timestamp;
  validTo?: Timestamp;                 // Optional expiration
  authority: string;                   // Entity ID of the authorizing body
  signature: string;                   // Authority's cryptographic signature
}
```

### Policy Domains

| Domain | Priority Range | Authoring Authority | Scope | Enforcement |
|--------|---------------|---------------------|-------|-------------|
| Constitutional | 0–99 | Security Council | System-wide | Always enforced, immutable at runtime |
| Safety | 100–199 | Safety Officer | System-wide | Always enforced, emergency suspendable |
| Regulatory | 200–299 | Compliance Officer | Per-jurisdiction | Enforced within jurisdiction scope |
| Organizational | 300–399 | Organization Admin | Per-organization | Enforced for org entities and resources |
| Mission | 400–499 | Mission Owner | Per-mission | Enforced for mission-scoped actions |

### Policy Resolution

At evaluation time, PS resolves the applicable policy set for the action:

1. Collect all policies matching the action's domain scope
2. Filter by time: `validFrom <= now < validTo` (or no expiry)
3. Sort by priority (ascending)
4. Sort by version within priority (newer version wins if priorities equal)
5. Evaluate in order: `deny` policies first (fail-fast), then `obligation` policies, then `allow` policies
6. Return the aggregate result: any `deny` match denies; an `allow` match may allow if no `deny` matched

### Conflict Resolution

| Conflict | Resolution |
|----------|-----------|
| Allow vs Deny (same scope) | Deny wins |
| Higher priority vs Lower priority | Higher priority wins (constitutional overrides organizational) |
| More specific vs Less specific | More specific scoping wins |
| Same priority, same scope | Deny wins (fail-safe) |

## Policy Storage

### Repository Structure

Policies are stored in a versioned, immutable repository backed by the EAS evidence chain:

- Each policy version is an immutable document in the repository
- Policy updates create new versions; old versions are retained for audit
- The repository supports query by domain, authority, scope, and effective time
- Policy snapshots are materialized for the evaluation engine at configurable intervals

### Replication

The policy repository is replicated across all Security Council nodes:
- Read replicas for policy evaluation (low latency)
- Write master for policy authoring and updates
- Replication lag target: <1 second for policy activation
- Policy changes are broadcast via ACF event streams

## Policy Evaluation Engine

### Evaluation Pipeline

```
Action → Load Applicable Policies → Sort → Evaluate Deny → Evaluate Obligation → Evaluate Allow → Result
```

The evaluation engine is invoked at Stage 4 of the verification pipeline:

1. **Load**: Retrieve all applicable policies from the local policy cache or repository
2. **Sort**: Order by priority, then version (newest first within same priority)
3. **Deny Pass**: Evaluate deny policies. Any match → immediate denial with policy reference
4. **Obligation Pass**: Evaluate obligation policies. Collect required actions.
5. **Allow Pass**: Evaluate allow policies. If no allow policy matches → fall back to RBAC/ABAC default
6. **Return**: Aggregate result with matched policy IDs, matched rule expressions, and obligations

### Caching

| Cache | TTL | Strategy |
|-------|-----|----------|
| Applicable policy set | 30s | LRU, invalidated on policy change |
| Policy evaluation result | 15s | LRU, keyed by (entity_id, action_hash) |
| Policy document (raw) | 60s | LRU, invalidated on version change |
| Policy domain index | 60s | Periodic refresh |

### Performance Targets

| Metric | Target |
|--------|--------|
| Policy load (cold) | <20ms |
| Policy evaluation (cached) | <5ms |
| Policy evaluation (uncached) | <20ms |
| Policy activation (change to evaluate) | <5s |
| P99 pipeline latency (Stage 4) | <60ms |

## Policy Enforcement

### Enforcement Points

Policies are enforced at multiple points in the system:

1. **Verification Pipeline (Stage 4)**: Pre-execution policy evaluation for all actions
2. **ACF Gateway**: Policies restricting communication (data sovereignty, routing restrictions)
3. **Resource Allocation (ROS)**: Policies governing resource consumption limits
4. **Entity Creation (OSYS/LMS)**: Policies governing entity lifecycle (what types may be created, with what capabilities)
5. **Evidence Access (EAS)**: Policies governing evidence retention and query permissions

### Violation Handling

When a policy violation is detected:
1. Action is denied (pre-execution) or flagged (post-execution)
2. Violation event is produced with policy ID, rule, and context
3. For constitutional and safety violations: immediate escalation to Security Council
4. For organizational and mission violations: notification to domain authority
5. Repeated violations by the same entity trigger capability review and possible suspension

## Cross-Cutting Concerns

### Security

- Policy definitions are themselves evaluated against constitutional policy before activation
- Policy evaluation engine is sandboxed — a malformed policy cannot crash the engine
- Policy authoring requires authentication and authorization; only designated authorities may create policies
- Policy priority is immutable after creation; a constitutional policy cannot be downgraded to organizational

### Evidence

- Every policy evaluation is recorded as evidence with the matched rules and final decision
- Policy change history is maintained as immutable evidence for audit
- Policy violation events include the full evaluation context for forensic investigation

### Lifecycle

- Policies follow a defined lifecycle: Draft → Review → Active → Superseded → Archived
- Active policies may be emergency-suspended by the Security Council without full lifecycle
- Expired policies (validTo reached) are automatically moved to Archived

### Capability Bounds

- Policy evaluation occurs after RBAC/ABAC authorization (Stage 3) but before capability verification (Stage 5)
- A policy may deny an action that RBAC allows, but cannot allow an action that RBAC denies
- Policy obligations may create resource consumption that is accounted in Stage 5 capability verification

### Communication

- Policy loading requires ACF communication with the policy repository
- Policy changes are broadcast through ACF event streams for cache invalidation
- Evaluation results are communicated through the pipeline context to Stage 5

### Design DNA

| Rule | Assessment | Rationale |
|------|-----------|-----------|
| R1 — Modulsingularity | Compliant | Policy System does one thing: manage and evaluate policies |
| R2 — Dependency Order | Compliant | PS depends on IDS and EAS; pipeline depends on PS |
| R3 — DRY | Compliant | Policy definitions are stored once in the repository, referenced by ID |
| R4 — Builder Pattern | Compliant | Policies are constructed by PolicyBuilder with validation and signing |
| R5 — Liskov Substitution | Compliant | Any policy evaluation engine implements the PolicyEngine interface |
| R6 — DI over Singletons | Compliant | PS engine injected into verification pipeline Stage 4 |
| R7 — Tests Exist | Compliant | Unit tests for evaluation logic, integration tests for pipeline Stage 4 |
| R8 — Tests Fast | Compliant | Evaluation target <20ms; test suite <60s |
| R9 — Deterministic Tests | Compliant | Same policy set + same action always produces same result |
| R10 — Prefer Simpler | Compliant | Linear priority-sorted evaluation — no inference chains or recursion |
| R11 — Refactor over Rewrite | Compliant | Policy evolution through versioned updates, not engine rewrites |
| R12 — Embrace Errors | Compliant | Each violation has policy ID, rule reference, and evaluation context |
| R13 — Design for Failure | Compliant | If repository unreachable, use cached policy snapshot (fail closed if cache empty) |
| R14 — Paved Path | Compliant | Stage 4 evaluation is the single paved path for policy enforcement |
| R15 — Open/Closed | Compliant | New policy domains added via domain registration, not engine modification |
