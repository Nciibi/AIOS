## Purpose

The Policy Definition Grammar (PDG) defines the formal language used to express policy rules within the AIOS Policy System (000-PS.md). PDG is a declarative, domain-specific language designed for expressing authorization conditions, obligations, and constraints. It is not a general-purpose programming language — it is a limited, verifiable grammar for policy expression that balances expressiveness with analyzability.

PDG serves three constituencies. Policy authors (Security Council, Organization Admins, Mission Owners) use PDG to write policies in a syntax that maps directly to the authorization model. The Policy Validation Engine (002-PVE.md) uses PDG to statically analyze policies for conflicts, errors, and constitutional compliance. The Policy Evaluation Engine (000-PS.md) compiles PDG expressions into evaluation plans for fast runtime execution.

PDG is designed for formal verification. Every PDG expression has a well-defined semantics, bounded evaluation cost, and no side effects. A PDG policy cannot trigger external actions, cannot loop infinitely, and cannot access runtime state beyond the declared attribute sources. This makes PDG policies safe to evaluate in the hot path of the verification pipeline.

## Grammar Specification

### Lexical Structure

```
Comment    ::= "#" [^\n]* "\n"
Identifier ::= [a-zA-Z_][a-zA-Z0-9_]*
String     ::= "\"" ([^"\\] | "\\" .)* "\""
Number     ::= "-"? [0-9]+ ("." [0-9]+)?
Boolean    ::= "true" | "false"
Duration   ::= Number ("ms" | "s" | "m" | "h" | "d")
Timestamp  ::= "@" Number
```

### Types

```
Type ::= "bool" | "int" | "float" | "string" | "duration" | "timestamp"
        | "set<" Type ">" | "enum<" Identifier { "," Identifier } ">"
```

### Expressions

```
Expr ::= Literal
       | Identifier                                                    # Attribute reference
       | Identifier "." Identifier                                     # Scoped attribute (e.g., subject.clearance)
       | "(" Expr ")"
       | Expr BinOp Expr                                               # Binary operators
       | UnOp Expr                                                     # Unary operators
       | Expr "?" Expr ":" Expr                                        # Ternary conditional
       | "[" Expr { "," Expr } "]"                                     # Set literal
       | FuncName "(" [Expr { "," Expr }] ")"                          # Built-in function call

BinOp ::= "==" | "!=" | "<" | "<=" | ">" | ">="
        | "&&" | "||"
        | "in" | "not_in" | "contains" | "matches"

UnOp ::= "!" | "-"
```

### Rules

```
Policy ::= "policy" Identifier "{" Rule+ "}"
Rule   ::= Effect "when" Expr [ "unless" Expr ]
Effect ::= "allow" | "deny" | "require" | "forbid"
```

### Obligations

```
Policy       ::= "policy" Identifier "{" TopLevel+ "}"
TopLevel     ::= Rule | Obligation
Obligation   ::= "obligation" Identifier "when" Expr "do" Action
Action       ::= "log" "(" Expr ")"
               | "notify" "(" Expr "," Expr ")"
               | "escalate" "(" Expr ")"
               | "require_mfa" "(" ")"

Attribute    ::= Identifier [ "." Identifier ]

FuncName     ::= "len" | "count" | "contains_any" | "contains_all"
               | "time_between" | "day_of_week" | "in_range"
               | "matches_regex" | "hash_eq"
```

## Semantics

### Evaluation Model

PDG expressions are evaluated in a context that provides:
- `subject`: attributes of the requesting entity (from IDS, CCA, Trust)
- `resource`: attributes of the target resource (from IDS, LMS)
- `action`: attributes of the requested action (from Pipeline)
- `env`: environment attributes (time, system load, threat level)

Attribute resolution is lazy — an attribute is only fetched from its source if referenced. Undefined attributes evaluate to `null`. Comparison with `null` returns `false` for all operators except `==` and `!=`.

### Operator Semantics

| Operator | Types | Semantics |
|----------|-------|-----------|
| `==` | Any | Equality (deep for sets) |
| `!=` | Any | Inequality |
| `<`, `<=`, `>`, `>=` | int, float, duration, timestamp | Ordering |
| `&&` | bool, bool | Logical AND (short-circuit) |
| `||` | bool, bool | Logical OR (short-circuit) |
| `!` | bool | Logical NOT |
| `in` | T, set\<T\> | Set membership |
| `not_in` | T, set\<T\> | Set exclusion |
| `contains` | set\<T\>, T | Set contains element |
| `matches` | string, string | Regex match |

### Rule Evaluation

```
when  Expr is true  + unless  Expr is false  → Effect applies
when  Expr is true  + unless  Expr is true   → Effect does not apply
when  Expr is false                           → Effect does not apply
```

## Built-in Functions

| Function | Signature | Description | Example |
|----------|-----------|-------------|---------|
| `len` | `len(set\<T\>) → int` | Cardinality of a set | `len(subject.roles) > 0` |
| `contains_any` | `contains_any(set\<T\>, set\<T\>) → bool` | Sets intersect | `contains_any(subject.roles, ["admin", "auditor"])` |
| `contains_all` | `contains_all(set\<T\>, set\<T\>) → bool` | First set superset of second | `contains_all(resource.tags, ["encrypted", "audited"])` |
| `time_between` | `time_between(timestamp, timestamp) → bool` | Current time between two timestamps | `time_between(@0900, @1700)` |
| `day_of_week` | `day_of_week() → int` | Current day of week (0=Sun) | `day_of_week() >= 1 && day_of_week() <= 5` |
| `matches_regex` | `matches_regex(string, string) → bool` | Regex match | `matches_regex(subject.id, "^aios:worker:")` |
| `hash_eq` | `hash_eq(string, string) → bool` | Constant-time string comparison | `hash_eq(action.type, "delete")` |

## Example Policies

```policies
# Constitutional: No evidence deletion
policy evidence_integrity {
    deny when action.type == "delete" && resource.type == "evidence"
}

# Organizational: Business hours only
policy business_hours_org_resources {
    deny when resource.classification == "internal"
          && !time_between(@0900, @1700)
          && day_of_week() >= 1
          && day_of_week() <= 5
}

# Safety: High-risk actions require MFA
policy high_risk_mfa {
    require when action.risk_score > 0.7
    obligation notify_security when action.risk_score > 0.7
        do notify("security_council", "High risk action: " + action.type)
}
```

## Grammar Constraints

### Safety Constraints

The following constraints are enforced by the PDG parser, validator, and compiler:

| Constraint | Enforcement | Rationale |
|-----------|-------------|-----------|
| No recursion | Static analysis | Prevent infinite evaluation |
| No loops | Grammar forbids iteration | PDG is declarative, not imperative |
| No mutation | Grammar forbids assignment | Policies are pure predicates |
| No I/O | Grammar forbids external calls | Side-effect-free evaluation |
| Bounded evaluation | `len()` only on declared sets | Prevent denial-of-service through set explosion |
| No type coercion | Strict type checking | Prevent subtle evaluation errors |
| No dynamic dispatch | All functions resolved statically | Deterministic evaluation |

### Evaluation Cost Budget

Each PDG expression has a static cost estimate computed at compile time:
- Attribute reference: 1 unit
- Function call: 5 + parameter cost units
- Binary operator: 2 units
- Set membership: 10 + set size cost units
- Maximum per-rule budget: 1000 units
- Maximum per-policy budget: 10000 units

If evaluation would exceed the budget, the engine returns an error rather than proceeding.

| Event Type | Produced When | Fields |

| `PDG.TypeChecked` | A PDG expression passes type checking | policy_id, expression_hash, attribute_references |
| `PDG.Compiled` | A PDG expression is compiled to evaluation plan | policy_id, instruction_count, estimated_cost |

## Cross-Cutting Concerns

### Security

- PDG expressions are validated before activation. A syntactically valid but semantically dangerous expression is caught by the validator (002-PVE.md).
- Attribute references are scoped — a policy may only reference attributes from declared sources.
- The `matches_regex` function uses a ReDoS-safe regex engine with complexity limits.

### Evidence

- PDG compilation produces structured representations that are stored as evidence.
- Policy evaluation traces include the specific PDG expression branches that were taken.
- Parse errors and validation results are recorded for audit.

### Lifecycle

- PDG versioning is tied to policy versioning. A new version of a policy may use new PDG features.
- The PDG specification itself is versioned (this document is v1.0). Future versions may add features but must maintain backward compatibility.

### Capability Bounds

- PDG expressions are bounded by the cost budget. A policy cannot consume excessive evaluation resources.
- Attribute resolution is bounded — a policy cannot cause unbounded attribute chain resolution.

### Communication

- PDG compilation is local to the Policy System node. No ACF communication is needed at evaluation time.
- Compiled policies are distributed through ACF replication.

### Design DNA

| Rule | Assessment | Rationale |
|------|-----------|-----------|
| R1 — Modulsingularity | Compliant | PDG does one thing: define policy expressions |
| R2 — Dependency Order | Compliant | PDG depends on nothing; the evaluation engine depends on PDG |
| R3 — DRY | Compliant | Grammar defined once in this specification, implemented by parser |
| R4 — Builder Pattern | Compliant | PDG AST is constructed by Parser, not directly instantiated |
| R5 — Liskov Substitution | Compliant | Any PDG implementation produces equivalent ASTs for identical input |
| R6 — DI over Singletons | Compliant | PDG parser injected into Policy Builder |
| R7 — Tests Exist | Compliant | Parser tests for every grammar construct, edge case inputs |
| R8 — Tests Fast | Compliant | All grammar constructs parse in <1ms |
| R9 — Deterministic Tests | Compliant | Same input always produces same AST |
| R10 — Prefer Simpler | Compliant | 4 operators, 8 built-in functions — minimal surface area |
| R11 — Refactor over Rewrite | Compliant | Grammar extensions are backward-compatible additions |
| R12 — Embrace Errors | Compliant | Every parse error has line, column, token, and expected alternatives |
| R13 — Design for Failure | Compliant | Parse failure returns structured error, never crashes |
| R14 — Paved Path | Compliant | PDG is the only way to write policies — no alternative DSL |
| R15 — Open/Closed | Compliant | New functions added by registration, not parser modification |
