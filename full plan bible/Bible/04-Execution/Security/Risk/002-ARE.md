## Purpose

The Automated Risk Evaluator (ARE) enhances the Risk Engine (000-RE.md) with machine-learning-based risk scoring. While the Risk Engine provides deterministic, rule-based scoring using weighted component factors, ARE applies trained models to detect complex risk patterns, emerging threats, and subtle anomalies that rule-based systems cannot capture. ARE does not replace the deterministic Risk Engine — it provides a supplementary risk signal that is combined with the deterministic score.

ARE operates under a strict constitutional constraint: it may augment risk scoring but never override it. The deterministic Risk Engine establishes the baseline risk score. ARE may increase the score (if the ML model detects additional risk) but may not decrease it below the deterministic baseline. This ensures that ML failures (false negatives, concept drift, adversarial inputs) cannot result in under-scored risk.

ARE models are trained on historical risk data from the Risk Knowledge Graph (001-RKG.md) and are validated against constitutional risk invariants before deployment. Every model version is tested for fairness, calibration, and robustness. Models that exhibit bias against entity types, organizations, or other protected attributes are rejected. Model predictions are explainable — every score adjustment by ARE includes a feature attribution vector showing which input features drove the adjustment.

## Model Architecture

### Model Stack

```
Risk Features (from RKG)
    │
    ▼
┌─────────────────────────────┐

│  (normalization, encoding,   │

└────────────┬────────────────┘
             │

│  Ensemble Model             │
│  ├─ Gradient Boosted Trees  │

             │

│  Score Calibration          │

             │

└─────────────────────────────┘
```

### Model Components

| Component | Algorithm | Purpose | Training Frequency |
|-----------|-----------|---------|-------------------|
| Gradient Boosted Trees | XGBoost/LightGBM | Feature-based risk classification | Weekly |
| Isolation Forest | Isolation Forest | Anomaly detection on action features | Daily |
| Temporal CNN | 1D CNN over timestamped sequences | Pattern-of-life anomaly detection | Weekly |

### Feature Set

| Feature Category | Count | Examples |
|-----------------|-------|----------|
| Entity features | 25 | trust_score, violation_count_7d, action_velocity, autonomy_level_encoded |
| Action features | 15 | action_type_encoded, base_risk, scope_depth, delegation_level |
| Resource features | 10 | classification_encoded, owner_type, lifecycle_state_encoded |
| Temporal features | 20 | hour_sin, hour_cos, day_of_week, days_since_last_action, actions_this_hour |
| Historical features | 30 | similar_action_violation_rate, entity_risk_trend, peer_comparison |

Total: 115 features per evaluation.

### Score Combination

```
final_risk = max(deterministic_risk, are_adjusted_risk)
```

Where:
- `deterministic_risk` = output of the rule-based Risk Engine (000-RE.md)
- `are_adjusted_risk` = `clamp(base_model_score + anomaly_boost, deterministic_risk, 1.0)`
- `anomaly_boost` = anomaly detection score (Isolation Forest + Temporal CNN), applied only if > 0

## Model Lifecycle

### Development

```
Data Extraction (RKG) → Feature Engineering → Training → Validation → Deployment
```

- Training data: 90 days of historical risk events from RKG
- Validation split: 80/20 temporal (train on days 1-72, validate on days 73-90)
- Evaluation metrics: AUC-ROC > 0.85, Precision at P95 > 0.90, Calibration error < 0.05
- Fairness audit: Demographic parity across entity types, organizations, and autonomy levels

### Deployment

| Stage | Duration | Validation | Traffic |
|-------|----------|------------|---------|
| Shadow | 24 hours | Compare against production model | 100% shadow, 0% serving |
| Canary | 48 hours | AUC, calibration, latency | 5% serving |
| Gradual rollout | 72 hours | Monitor incident rate | 25% → 50% → 100% |
| Full deployment | Until next model | Standard monitoring | 100% |

### Monitoring

| Metric | Alert Threshold | Action |
|--------|----------------|--------|
| Prediction latency > 20ms | P99 | Fall back to deterministic score, investigate |
| Calibration error > 0.10 | Mean | Trigger model retraining |
| AUC < 0.80 | Sliding 7-day | Roll back to previous model version |
| Fairness violation | Any group | Pause model, investigate bias |
| Feature drift > 3 sigma | Per feature | Retrain or retune feature pipeline |

### Rollback

If monitoring triggers a critical alert:
1. ARE immediately returns 0.0 adjustment (pure deterministic scoring)
2. Model is marked as failed in the model registry
3. Incident investigation determines root cause
4. Previous known-good model is re-deployed or model is retrained

## Explainability

### Feature Attribution

Every ARE score includes a SHAP-based feature attribution vector:

```typescript
interface AREAttribution {
  score: number;
  baseline: number;             // Expected score without entity-specific features
  contributions: {
    feature: string;
    contribution: number;       // Positive = increased risk, negative = decreased
    featureValue: string;       // Actual feature value for this prediction
  }[];
  topContributors: string[];    // Top 5 features driving the score
  anomalySignals?: {
    signalType: string;
    strength: number;
    description: string;
  }[];
}
```

### Explanation Storage

Every ARE prediction and its attribution vector are stored as evidence in EAS for auditability. The explanation must be available for at least 90 days. If a model version is retired, its explanations remain accessible.

## Model Governance

### Approval Gate

No model version may serve predictions without passing the Model Governance gate:

1. **Data scientist**: Trains model, documents performance
2. **Validator**: Independent validation of metrics and fairness
3. **Constitutional reviewer**: Verifies compliance with Physics and Constitution
4. **Security Council**: Approves model for deployment

## Cross-Cutting Concerns

### Security

- ARE models are signed by the Security Council before deployment. Unsigned models are rejected.
- Model weights are stored encrypted. Access requires `ml.model_load` capability.
- Adversarial robustness testing is required for all model versions. Models must maintain AUC > 0.80 under perturbation of up to 10% of features.

### Evidence

- Every ARE prediction is recorded as evidence with full attribution vector.
- Model performance metrics are recorded daily as evidence.
- Fairness audits are stored as immutable evidence for external review.

### Lifecycle

- Models have defined lifetimes (max 90 days without retraining). Expired models are auto-rolled-back.
- Model training consumes compute resources that are budgeted through ROS.
- Model version upgrades follow the staged deployment lifecycle.

### Capability Bounds

- ARE model inference consumes compute budget. Each evaluation is budgeted at 10ms of compute time.
- Models are sandboxed from the production Risk Engine. A model crash does not affect deterministic scoring.
- Feature extraction queries to RKG are subject to RKG query rate limits.

### Communication

- ARE communicates through ACF. Feature extraction queries RKG, predictions returned to Stage 6.
- Model deployment commands flow through ACF with multi-party authorization.
- Model monitoring metrics are published via ACF event streams.

### Design DNA

| Rule | Assessment | Rationale |
|------|-----------|-----------|
| R1 — Modulsingularity | Compliant | ARE does one thing: provide ML-enhanced risk adjustment |
| R2 — Dependency Order | Compliant | ARE depends on RKG and RE; nothing depends on ARE (fallback is pure RE) |
| R3 — DRY | Compliant | Feature definitions in one registry, model config in model registry |
| R4 — Builder Pattern | Compliant | PredictionResult constructed by PredictionBuilder with attribution |
| R5 — Liskov Substitution | Compliant | Any ML model implements the RiskModel interface for scoring |
| R6 — DI over Singletons | Compliant | ARE injected into Risk Engine as an optional scorer |
| R7 — Tests Exist | Compliant | Unit tests for each model component, integration tests for score combination |
| R8 — Tests Fast | Compliant | Inference target <10ms; training runs as nightly batch |
| R9 — Deterministic Tests | Compliant | Seeded models produce deterministic predictions for same input |
| R10 — Prefer Simpler | Compliant | Ensemble of 3 proven algorithms — no deep learning, no reinforcement learning |
| R11 — Refactor over Rewrite | Compliant | Model evolution through retraining, not architecture rewrites |
| R12 — Embrace Errors | Compliant | Every prediction includes confidence score; low-confidence predictions flagged |
| R13 — Design for Failure | Compliant | ARE failure → fallback to deterministic scoring (degraded but functional) |
| R14 — Paved Path | Compliant | All risk assessments use combined deterministic + ARE score — single path |
| R15 — Open/Closed | Compliant | New models added through model registry, not ARE core changes |
