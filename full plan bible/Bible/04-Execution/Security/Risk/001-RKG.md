## Purpose

The Risk Knowledge Graph (RKG) is the structured, queryable representation of all risk-related information within AIOS. It models entities, actions, resources, policies, and their risk relationships as a property graph. The RKG is not a risk scoring engine — it is the data substrate that the Risk Engine (000-RE.md) and Automated Risk Evaluator (002-ARE.md) query to compute risk scores, detect anomalies, and identify risk patterns.

The RKG answers questions like: what actions has this entity performed that are similar to the requested action? What resources have been the target of recent violations? What is the historical risk trajectory of this entity type? Which policies are most frequently violated? These queries would be expensive or impossible with flat event logs — the graph structure enables efficient traversal of risk relationships.

The RKG is built from evidence streams. Every event recorded by EAS is analyzed and materialized as nodes and edges in the knowledge graph. Nodes represent entities, actions, resources, policies, and sessions. Edges represent relationships: performed, targeted, governed_by, violated, escalated, delegated. Each edge carries metadata (timestamp, risk score, outcome) that enables temporal analysis and risk path tracing.

## Graph Model

### Node Types

| Node Type | Label | Properties | Source |
|-----------|-------|------------|--------|
| Entity | `:Entity` | id, type, trust_score, autonomy_level, org, dept, created_at | IDS, Trust |
| Action | `:Action` | id, action_type, base_risk, resource_type, scope | Pipeline |
| Resource | `:Resource` | id, resource_type, classification, owner, lifecycle_state | IDS, LMS |
| Policy | `:Policy` | id, domain, effect, priority, version, status | Policy System |
| Session | `:Session` | id, entity_id, started_at, ended_at, risk_tier | SSM |
| Violation | `:Violation` | id, policy_id, severity, action_id, timestamp, outcome | EAS |
| RiskScore | `:RiskScore` | id, score, confidence, tier, computed_at | Risk Engine |

### Edge Types

| Edge Type | Source → Target | Properties | Description |
|-----------|----------------|------------|-------------|
| `:PERFORMED` | Entity → Action | timestamp, risk_score, outcome | Entity performed an action |
| `:TARGETED` | Action → Resource | access_type, duration | Action accessed a resource |
| `:GOVERNED_BY` | Entity → Policy | effective_at, expires_at | Entity is governed by a policy |
| `:VIOLATED` | Action → Policy | violation_type, severity | Action violated a policy |
| `:ESCALATED_TO` | Action → Entity | escalation_reason, resolution | Action was escalated to an entity |
| `:DELEGATED_FROM` | Entity → Entity | capability_id, depth | Capability delegation relationship |
| `:MEMBER_OF` | Entity → Entity | relationship_type | Organizational membership |
| `:SIMILAR_TO` | Action → Action | similarity_score, basis | Actions deemed similar by feature analysis |

```cypher
(:Entity {

```

## Graph Operations

### Risk Path Queries

The RKG supports path-based risk queries that traverse the graph to identify risk propagation:

```
Entity → [:PERFORMED] → Action → [:TARGETED] → Resource
                                                 ↑
                                            [:TARGETED]
                                                 |
    Violation ← [:VIOLATED] ← Action2 ← [:PERFORMED] ← Entity2
```

Example query: find all resources accessed by entities that have violated policies in the last hour.

```cypher
MATCH (e:Entity)-[:PERFORMED]->(a:Action)-[:TARGETED]->(r:Resource)
WHERE EXISTS {
  MATCH (e)-[:PERFORMED]->()-[:VIOLATED]->(:Policy)
  WHERE violation.timestamp > now() - duration('1h')
}
RETURN r.id, r.classification, count(a) as accessCount
```

### Similarity Queries

The RKG maintains similarity edges between actions based on feature vectors:

| Feature | Description | Weight |
|---------|-------------|--------|
| action_type | One-hot encoded action type | 0.35 |
| resource_classification | Sensitivity level | 0.25 |
| entity_type | Type of requesting entity | 0.15 |
| scope_depth | Delegation depth | 0.10 |
| time_of_day | Hour of execution (0-23) | 0.10 |
| previous_outcome | Success/failure of similar actions | 0.05 |

Similarity queries enable the Risk Engine to find historical precedents: "has an action similar to this one resulted in a violation before?"

### Anomaly Detection Queries

The RKG supports statistical anomaly detection through graph pattern analysis:

- **Degree anomaly**: Entity performs actions at a rate >3 standard deviations from its baseline
- **Path anomaly**: Entity accesses a resource that is disconnected from its normal access graph
- **Timing anomaly**: Entity performs actions at hours it has never acted before
- **Chain anomaly**: Delegation chain length exceeds historical maximum for this entity type

## Data Ingestion

### Ingestion Pipeline

```
EAS Event Stream → RKG Gateway → Event Processor → Graph Update → Index
```

1. **EAS Event Stream**: RKG subscribes to all risk-relevant event types from EAS
2. **RKG Gateway**: Validates event schema and filters non-risk events
3. **Event Processor**: Transforms flat events into graph operations (create/update nodes and edges)
4. **Graph Update**: Applies changes to the property graph store
5. **Index**: Updates graph indexes for query performance

### Event to Graph Mapping

| EAS Event Type | Graph Operation |
|----------------|-----------------|
| `Verification.RiskAssessment` | Create RiskScore node. Connect Entity→Action edge. |
| `Verification.Complete` | Update Action node with outcome. |
| `Verification.Denied` | Create Violation node. Connect Action→Policy edge. |
| `POL.PolicyViolation` | Create Violation node. Update entity risk profile. |
| `RSK.AnomalyDetected` | Create Anomaly node. Connect to affected entity. |
| `EAS.EvidenceSealed` | If action evidence, update Action node. |

### Performance Targets

| Operation | P50 | P99 |
|-----------|-----|-----|
| Event ingestion latency | 50ms | 200ms |
| Simple node lookup (by ID) | 5ms | 20ms |
| Risk path query (depth 3) | 50ms | 150ms |
| Similarity query (k=10) | 100ms | 300ms |
| Anomaly detection scan | 500ms | 2s |

## Data Retention & Pruning

| Data | Retention | Pruning Strategy |
|------|-----------|-----------------|
| Entity nodes | Entity lifetime + 30 days | Removed when entity is archived |
| Action nodes | 90 days | Aggregated into statistical profiles after 90d |
| RiskScore nodes | 30 days | Aggregated into entity risk profile |
| Violation nodes | 1 year | Archived to cold storage |
| Similarity edges | 7 days | Recalculated daily |
| Anomaly nodes | 30 days | Summarized into entity risk profile |

## Cross-Cutting Concerns

### Security

- RKG data is classified based on its source. Evidence-derived data inherits the evidence classification.
- Graph queries are subject to the same authorization pipeline as any data access.
- The RKG does not store private keys, secrets, or session content — only metadata and risk attributes.

### Evidence

- The RKG is derived from evidence but is not itself evidence. The authoritative record is always EAS.
- Graph pruning does not delete evidence — it only removes derived representations.
- Query results include traceability back to the source evidence records.

### Lifecycle

- Entity lifecycle transitions trigger graph updates: suspension marks entity nodes as inactive; termination triggers node archiving.
- Policy lifecycle changes are reflected in the graph: deactivated policies are marked inactive.
- Session end triggers cleanup of session edges and risk score nodes.

### Capability Bounds

- RKG is a read-heavy data store. Write operations are limited to ingestion (max 1000 writes/second).
- Query resource consumption is bounded. Complex traversals are limited to 5 hops by default.
- Each entity has a maximum RKG query rate of 10 queries/second.

### Communication

- RKG ingestion consumes the EAS event stream through ACF.
- Query results are returned through ACF with configurable response size limits.
- Graph updates are propagated to replica nodes through ACF.

### Design DNA

| Rule | Assessment | Rationale |
|------|-----------|-----------|
| R1 — Modulsingularity | Compliant | RKG does one thing: maintain a risk-focused property graph |
| R2 — Dependency Order | Compliant | RKG depends on EAS event stream; Risk Engine depends on RKG |
| R3 — DRY | Compliant | Graph model defined once; entities/actions/resources are canonical nodes |
| R4 — Builder Pattern | Compliant | Graph constructed by GraphBuilder from event streams |
| R5 — Liskov Substitution | Compliant | Any graph database implementation conforms to RKG query interface |
| R6 — DI over Singletons | Compliant | RKG injected into Risk Engine and ARE for query access |
| R7 — Tests Exist | Compliant | Unit tests for graph operations, integration tests for event ingestion |
| R8 — Tests Fast | Compliant | Graph queries target <50ms; test suite <60s |
| R9 — Deterministic Tests | Compliant | Same event stream always produces same graph state |
| R10 — Prefer Simpler | Compliant | Property graph with 7 node types and 10 edge types — minimal schema |
| R11 — Refactor over Rewrite | Compliant | Graph schema evolves through property additions, not rewrites |
| R12 — Embrace Errors | Compliant | Every ingestion error traces to the source event and processing step |
| R13 — Design for Failure | Compliant | RKG is replicated; if write replica fails, read replicas serve queries |
| R14 — Paved Path | Compliant | All risk queries go through RKG — no direct EAS queries for risk purposes |
| R15 — Open/Closed | Compliant | New node and edge types added through schema registration, not graph engine changes |
