## Purpose

The Risk Engine (RE) is the system responsible for quantifying, evaluating, and responding to risk associated with every action in AIOS. Under Law 8 (Verification-First), Stage 6 of the verification pipeline performs risk assessment for every action before execution authorization. RE provides the risk scoring, tier classification, and mitigation recommendations that power Stage 6.

Risk in AIOS is not subjective. Every action has a quantifiable risk score computed from multiple factors: the action type's inherent risk, the requesting entity's trust and autonomy profile, the target resource's sensitivity classification, the operational context (system load, threat level, regulatory zone), and historical patterns (entity behavior, anomaly signals, past violations). The risk score is a deterministic float in the range [0.0, 1.0] mapped to a risk tier (L0–L4).

Risk assessment is not punitive — it is protective. A high-risk action is not necessarily denied; it may proceed with additional controls (confirmation, MFA, escalation). The Risk Engine recommends the appropriate control level based on the risk score and configurable thresholds. The verification pipeline makes the final authorization decision using the risk assessment as one input among the seven stages.

## Risk Model

### Risk Score Components

```typescript
interface RiskScore {
  overall: number;                         // 0.0 to 1.0
  components: {
    actionRisk: number;                    // Inherent risk of the action type
    entityRisk: number;                    // Entity trust and history factor
    resourceRisk: number;                  // Target resource sensitivity
    contextRisk: number;                   // Environmental risk factors
    historicalRisk: number;                // Pattern-based risk
    anomalyFactor: number;                 // Deviation from baseline
  };
  confidence: number;                      // Confidence in the score (0.0 to 1.0)
}
```

### Risk Score Calculation

```
overall = clamp(
    w_action * actionRisk +
    w_entity * entityRisk +
    w_resource * resourceRisk +
    w_context * contextRisk +
    w_history * historicalRisk +
    w_anomaly * anomalyFactor,
    0.0, 1.0
)
```

Weights are configurable per entity type and are stored in the Risk Engine configuration:

| Entity Type | w_action | w_entity | w_resource | w_context | w_history | w_anomaly |
|------------|----------|----------|------------|-----------|-----------|-----------|
| Worker | 0.30 | 0.25 | 0.20 | 0.10 | 0.10 | 0.05 |
| Session | 0.25 | 0.30 | 0.20 | 0.10 | 0.10 | 0.05 |
| Engine | 0.20 | 0.35 | 0.20 | 0.10 | 0.10 | 0.05 |
| Organization | 0.15 | 0.30 | 0.25 | 0.15 | 0.10 | 0.05 |
| Security Council | 0.10 | 0.40 | 0.20 | 0.15 | 0.10 | 0.05 |

### Risk Tiers

| Tier | Score Range | Label | Default Control |
|------|-------------|-------|-----------------|
| L0 | 0.000–0.100 | Negligible | No additional controls |
| L1 | 0.100–0.300 | Low | Log and monitor |
| L2 | 0.300–0.500 | Medium | Confirm with entity |
| L3 | 0.500–0.700 | High | Require MFA + explicit approval |
| L4 | 0.700–1.000 | Critical | Escalate to Security Council |

Thresholds are configurable per entity type. A Worker with autonomy level L4 may have a higher L3/L4 threshold than an L1 Worker.

## Risk Scoring Engines

### Action Risk Engine

Calculates inherent risk of the action type:

| Action Type | Base Risk | Risk Factors |
|-------------|-----------|--------------|
| `read` | 0.05 | resource sensitivity modifier: +0.1 per classification level above `public` |
| `write` | 0.20 | write scope modifier: +0.1 for cross-org writes, +0.2 for system writes |
| `execute` | 0.35 | execute privilege modifier: +0.15 for system commands, +0.25 for lifecycle |
| `delete` | 0.60 | delete scope modifier: +0.2 for entity deletion, +0.3 for evidence deletion |
| `delegate` | 0.40 | delegation depth modifier: +0.1 per delegation level beyond 1 |
| `create` | 0.25 | creation scope modifier: +0.1 for entity creation, +0.2 for policy creation |

### Entity Risk Engine

Calculates risk based on the requesting entity's profile:

| Factor | Weight | Source | Description |
|--------|--------|--------|-------------|
| Trust score | 0.40 | Trust/000-TLM.md | 1.0 = fully trusted → risk multiplier (1 - trust) |
| Autonomy level | 0.25 | CCA | L4 = 0.1 multiplier, L0 = 1.0 multiplier |
| Violation history | 0.20 | EAS | Recent violations increase risk |
| Entity age | 0.10 | IDS | New entities (<1 hour) have elevated risk |
| Entity type | 0.05 | IDS | Workers > Sessions > Engines > Organizations |

### Resource Risk Engine

Calculates risk based on the target resource:

| Classification | Base Risk Modifier | Examples |
|---------------|-------------------|----------|
| `public` | 0.0 | Public documentation, open data |
| `internal` | 0.1 | Internal documentation, metrics |
| `confidential` | 0.3 | Session data, mission plans |
| `restricted` | 0.5 | Policies, evidence records, identity data |
| `critical` | 0.7 | Security Council keys, constitution, root CA |

### Context Risk Engine

Calculates environmental risk factors:

| Factor | Modifier | Source |
|--------|----------|--------|
| System load > 0.8 | +0.05 | ROS |
| Threat level = elevated | +0.10 | Environment |
| Threat level = high | +0.20 | Environment |
| Threat level = severe | +0.35 | Environment |
| Outside business hours | +0.05 | Environment |
| Regulatory zone = restricted | +0.10 | Policy |
| Federated request | +0.15 | Cross-instance |

### Historical Risk Engine

Analyzes historical patterns for the entity:

| Pattern | Risk Delta | Detection Window |
|---------|-----------|-----------------|
| Recent denial | +0.05 | Last 10 minutes |
| Repeated similar violations | +0.10 per violation | Last hour |
| Escalating risk trend | +0.15 | Last 24 hours |
| Anomalous geographic access | +0.20 | Real-time |
| New device or location | +0.10 | First occurrence |

## Risk Response

### Control Actions

When Stage 6 identifies a risk score that triggers a control requirement, the Risk Engine recommends one of the following:

| Control | Description | Typical Tier |
|---------|-------------|--------------|
| `none` | No additional controls | L0, L1 |
| `log` | Ensure action and context are logged | L0, L1 |
| `confirm` | Require entity to confirm the action | L2 |
| `mfa` | Require multi-factor authentication | L3 |
| `approval` | Require explicit approval from authority | L3 |
| `escalate` | Escalate to Security Council for decision | L4 |

### Dynamic Threshold Adjustment

Risk thresholds may be dynamically adjusted based on:

- **System-wide threat level**: During elevated threat, all thresholds shift down by one tier
- **Entity-specific trend**: Entities with improving risk scores may have thresholds relaxed
- **Mission criticality**: Actions for high-priority missions may have relaxed thresholds
- **Regulatory requirements**: Certain jurisdictions require lower thresholds for specific actions

## Cross-Cutting Concerns

### Security

- Risk scores are computed deterministically. The same inputs always produce the same score.
- Risk Engine configuration is immutable at runtime. Changes require policy-level authorization.
- Risk scoring cannot be bypassed — Stage 6 is mandatory in the verification pipeline.

### Evidence

- Every risk assessment produces an `RSK.RiskScoreComputed` event recorded in EAS.
- Risk score components are logged for auditability and explainability.
- Threshold violations are recorded as high-priority evidence.

### Lifecycle

- Risk scores are computed per-action and are not cached beyond the action's TTL.
- Entity risk trends are updated after each action and are retained for 30 days.
- When an entity's lifecycle status changes (suspend, terminate), its risk profile is recalculated.

### Capability Bounds

- Risk controls may require additional resource consumption (e.g., MFA consumes tokens, evidence recording consumes storage). These are accounted in Stage 5.
- Entities with autonomy level L0 are not subject to risk-based controls — their actions are always verified with maximum scrutiny.

### Communication

- Risk assessment requests flow through ACF as part of the pipeline context.
- Risk Engine publishes anomaly events through ACF for downstream consumers (Security Council, Trust).
- Threshold adjustment events are broadcast to all pipeline instances.

### Design DNA

| Rule | Assessment | Rationale |
|------|-----------|-----------|
| R1 — Modulsingularity | Compliant | Risk Engine does one thing: compute and respond to action risk |
| R2 — Dependency Order | Compliant | RE depends on Trust, EAS, and Environment; pipeline depends on RE |
| R3 — DRY | Compliant | Risk scoring weights defined once per entity type in configuration |
| R4 — Builder Pattern | Compliant | RiskScore constructed by RiskCalculator from component scores |
| R5 — Liskov Substitution | Compliant | Any risk scoring model implements the RiskModel interface |
| R6 — DI over Singletons | Compliant | RE injected into verification pipeline Stage 6 |
| R7 — Tests Exist | Compliant | Unit tests for each scoring engine, integration tests for combined scoring |
| R8 — Tests Fast | Compliant | Risk scoring target <10ms; test suite <30s |
| R9 — Deterministic Tests | Compliant | Same inputs always produce same risk score |
| R10 — Prefer Simpler | Compliant | Linear weighted scoring model — no ML, no probabilistic inference |
| R11 — Refactor over Rewrite | Compliant | Scoring models evolve through weight configuration, not engine rewrites |
| R12 — Embrace Errors | Compliant | Each scoring failure identifies the factor and source |
| R13 — Design for Failure | Compliant | If a risk component source is unreachable, use default weight (fail conservative) |
| R14 — Paved Path | Compliant | All actions go through Stage 6 risk assessment — no bypass path |
| R15 — Open/Closed | Compliant | New risk factors added through factor registration, not engine changes |
