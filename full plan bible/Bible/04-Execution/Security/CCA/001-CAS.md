## Purpose

The Capability Assignment System (CAS) assigns capabilities to entities based on their autonomy level, role, declared bounds, and template definition. CAS is the authoritative decision point for capability entitlements — it determines which capabilities an entity may hold, at what constraint level, and for how long.

CAS works in concert with CCA: CAS determines the capability assignment, and CCA issues the capability certificate. This separation ensures that assignment decisions are auditable independently from certificate issuance.

## Assignment Model

### Assignment Sources

A capability assignment originates from one of four sources:

| Source | Authority | Scope | Example |
|--------|-----------|-------|---------|
| Template | AGS Genome definition | All entities of a type | Every Coding Worker gets `fs:read` for its workspace |
| Role | Organization role definition | Entities in a role | All Backend developers get `net:connect` to staging |
| Delegation | Entity-to-entity grant | Specific delegate | Mission Owner delegates `fs:write` to a specific Worker |
| Override | Sou or Security Council directive | Named entity override | Sou grants temporary `fs:read /etc` for diagnostics |

### Assignment Resolution

When CAS receives an assignment request, it resolves the capability through this priority-ordered lookup:

```
1. Override assignments (highest priority — explicit overrides)
2. Delegation assignments (active delegation grants)
3. Role-based assignments (entity's current role bindings)
4. Template-based assignments (entity type defaults)
```

Multiple sources may contribute to the same capability. The most specific source wins. If a capability is present in multiple sources at the same specificity level, the strictest constraint set applies.

### Assignment Constraints

Every CAS assignment includes constraint parameters:

| Parameter | Source | Default |
|-----------|--------|---------|
| Resource scope | Template + Role | Template scope |
| Duration | Template + Delegation | Template default (typically 24h for Workers) |
| Rate limit | Role | Template default |
| Delegation depth | Role | 0 (cannot delegate) |
| Autonomy floor | Autonomy System | L1 |
| Cost budget | Role + Resource budget | Organization budget allocation |

### Assignment Lifecycle

```
Template definition → Role binding → CAS evaluation → CCA issuance → Active → Renewal → Expiry/Revocation
```

| State | Description | Next |
|-------|-------------|------|
| Pending | Capability declared but not yet assigned | CAS evaluates → Assigned or Rejected |
| Assigned | CAS decision recorded, pending certificate issuance | CCA issues → Active |
| Active | Capability certificate issued | Time passes → Expired; or revoked |
| Expired | Certificate time limit reached | Re-evaluate → new assignment or drop |
| Revoked | CAS assignment invalidated | Cannot be reassigned without new evaluation |

## Assignment Evaluation

CAS evaluates each assignment request against five gates:

| Gate | Check | Failure Behavior |
|------|-------|-----------------|
| 1 — Template validity | Is the requested capability in the entity's Genome template? | Deny; entity type does not include this capability |
| 2 — Autonomy compatibility | Is the entity's autonomy level sufficient? | Deny; entity must progress autonomy level |
| 3 — Resource availability | Does the entity's resource budget allow this capability? | Deny; insufficient budget |
| 4 — Dependency satisfaction | Are all CDG dependencies satisfied? | Deny; dependency chain broken |
| 5 — Conflict detection | Does this capability conflict with existing assignments? | Warn; override required to proceed |

### Autonomy-Capability Matrix

| Capability Category | L0 | L1 | L2 | L3 | L4 |
|--------------------|----|----|----|----|----|
| `fs:read` | Template-scoped | Template + role | Delegatable | Full namespace | Full |
| `fs:write` | None | Template-scoped | Template + role | Delegatable | Full |
| `net:connect` | None | Whitelist only | Whitelist + role | Namespace | Full |
| `sys:capability` | None | None | None | CCA delegate | Full |
| `mission:create` | None | None | None | Sou delegate | Full |
| `org:modify` | None | None | None | None | Sou only |

## Interfaces

### CAS API (via ACF)

| Method | Auth | Description |
|--------|------|-------------|
| `evaluateAssignment(entity_id, capability, context)` | CCA | Evaluate whether entity should receive a capability |
| `assignFromTemplate(entity_id)` | Sou, OSYS | Assign all template capabilities to newly created entity |
| `assignFromRole(entity_id, role_id)` | Organization director | Assign capabilities associated with a role |
| `revokeAssignment(capability_id, reason)` | Security Council, Sou | Invalidate a capability assignment |
| `getAssignments(entity_id)` | entity, Sou, Security Council | List all capability assignments for entity |
| `resolveConflicts(entity_id, capability_a, capability_b)` | Sou | Resolve conflicting capability assignments |
| `getAssignmentSource(capability_id)` | Security Council | Trace which source produced this assignment |

```typescript
interface AssignmentEvaluator {

```

## Invariants

| ID | Invariant | Enforcement |
|----|-----------|-------------|
| CAS-001 | Every assignment passes all five evaluation gates | Algorithmic — gate sequence cannot be skipped |
| CAS-002 | An entity cannot hold a capability not declared in its template | Architectural — template validity gating |
| CAS-003 | Capability breadth is bounded by autonomy level | Algorithmic — autonomy-compatibility check |
| CAS-004 | Override assignments are always logged with evidence reference | Architectural — every override produces an Event |
| CAS-005 | Assignment conflicts are always reported, never silently resolved | Algorithmic — conflict detection gate blocks auto-resolution |

## Error Cases

| Condition | Error Code | Behavior |
|-----------|------------|----------|
| Entity not registered with IDS | `CAS_UNKNOWN_ENTITY` | Deny assignment; entity must be registered first |
| Capability not in template | `CAS_NOT_IN_TEMPLATE` | Deny; capability not available for this entity type |
| Insufficient autonomy level | `CAS_AUTONOMY_TOO_LOW` | Deny; entity must progress to required level |
| Insufficient resource budget | `CAS_BUDGET_INSUFFICIENT` | Deny; entity or organization budget exhausted |
| Dependency chain broken | `CAS_DEPENDENCY_MISSING` | Deny; prerequisite capability not held |
| Assignment conflict detected | `CAS_CONFLICT_EXISTS` | Warn; requires Sou directive to override |

## Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | CAS does one thing: determine which capabilities an entity should have |
| R2 — Dependency Order | Depends on AGS, Autonomy, ROS, CDG; consumed by CCA |
| R3 — DRY | Template capabilities defined in AGS, referenced by ID from CAS |
| R9 — Deterministic | Same entity + same request → same assignment result |
| R10 — Simpler Over Complex | Four priority sources (Override > Delegation > Role > Template) cover all cases |
| R13 — Design for Failure | If CDG unavailable, dependency gate degrades to safe-pass (allow, log warning) |
| R14 — Paved Path | Template-based assignment at entity creation is the standard path |
