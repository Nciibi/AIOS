## Purpose

The Capability Dependency Graph (CDG) is a directed acyclic graph (DAG) that maps which capabilities depend on which. It is used by CAS (during assignment evaluation) and CCA (during verification) to ensure that capability grants do not create circular dependencies, violate bound hierarchies, or leave dangling dependency chains.

CDG answers three questions:
1. **Pre-requisite**: What capabilities must an entity already hold before receiving this one?
2. **Transitive closure**: What capabilities will be affected if this one is revoked?
3. **Circularity**: Would granting this capability create a dependency cycle?

## Graph Model

### Edge Semantics

```
Capability A ──requires──→ Capability B
  A cannot be exercised unless B is held

Capability A ──conflicts──→ Capability B  
  A and B cannot be held simultaneously by the same entity
```

| Edge Type | Symbol | Meaning | Example |
|-----------|--------|---------|---------|
| Requires | `→` | Target must be active for source to be usable | `fs:write /etc` requires `fs:read /etc` |
| Extends | `⇢` | Target is a superset of source | `net:*` extends `net:connect` |
| Conflicts | `—` | Source and target cannot coexist | `fs:read` conflicts with `fs:deny` |
| Delegates | `↷` | Source can delegate to create target | `fs:*` delegates `fs:read` |

### Graph Structure

```
Network namespace
  │
  ├── requires ──→ net:connect ──requires──→ identity:resolved
  │
  └── net:listen ──requires──→ net:connect
                               │
                               └── conflicts ──→ net:disable

Filesystem namespace
  │
  ├── fs:write ──extends──→ fs:read
  │               │
  │               requires
  │               │
  │               ▼
  │          resource:resolved ──requires──→ identity:resolved
  │
  └── fs:delete ──requires──→ fs:write
                    │
                    └── requires ──→ resource:ownership

System namespace
  │
  ├── sys:capability ──requires──→ identity:verified
  │                     │
  │                     └── requires ──→ audit:evidenced
  │
  └── mission:create ──requires───→ autonomy:L2
                      │
                      └── requires ──→ resource:budget_allocated
```

### Node Properties

Every capability in the CDG has the following properties:

| Property | Description |
|----------|-------------|
| `capability_id` | Unique identifier matching the capability name |

## Operations

### 1. Pre-requisite Resolution

When CAS evaluates an assignment, it queries CDG for the transitive closure of required capabilities:

```
resolvePrerequisites(capability_id):
  visited = set()
  result = []
  queue = [capability_id]
  
  while queue is not empty:
    current = queue.pop()
    if current in visited:
      continue
    visited.add(current)
    for dep in graph[current].requires:
      result.add(dep)
      queue.add(dep)
  
  return result
```

CDG guarantees that prerequisite resolution completes in O(n) where n is the number of edges traversed.

### 2. Circularity Detection

Before any new capability is added or assignment granted, CDG verifies the graph remains acyclic:

```
detectCycle(capability_id, new_dependency):
  visited = set()
  stack = [(new_dependency, [])]
  
  while stack is not empty:
    current, path = stack.pop()
    if current == capability_id:
      return (true, path + [current])    // cycle detected
    if current in visited:
      continue
    visited.add(current)
    for next_dep in graph[current].requires:
      stack.append((next_dep, path + [current]))
  
  return (false, [])                      // no cycle
```

If a cycle is detected, the assignment is denied. Cycle resolution requires a Sou directive or capability definition refinement.

### 3. Impact Analysis

When a capability is revoked, CDG computes the full set of affected capabilities:

```
impactAnalysis(capability_id):
  affected = set()
  visited = set()
  
  // Outbound revocations: capabilities that depend on this one
  for edge in graph.incoming_edges(capability_id):
    if edge.type == "requires":
      affected.add(edge.source)
      // Check if alternative dependency path exists
      if not has_alternative_path(edge.source, capability_id):
        affected.union(impactAnalysis(edge.source))
  
  // Delegation chain invalidation
  for edge in graph.outgoing_edges(capability_id):
    if edge.type == "delegates":
      for descendant in delegation_tree(edge.target):
        affected.add(descendant)
  
  return affected
```

## Graph Management

### Registration

New capabilities register their dependencies when defined:

| Action | Trigger | Behavior |
|--------|---------|----------|
| Add node | New capability type registered | Node created with requires, extends, conflicts edges |
| Remove node | Capability type deprecated | All edges removed; existing grants grandfathered |
| Add edge | Capability dependency declared | Circularity check runs before commit |
| Remove edge | Capability definition updated | Impact analysis runs before commit; affected grants may need re-verification |

### Versioning

CDG is versioned. Each change produces:

| Artifact | Description | Retention |
|----------|-------------|-----------|
| Graph snapshot | Complete CDG state at version N | Permanent in ADG |
| Diff log | Edges added/removed between versions | 90 days |
| Impact report | Capabilities affected by each change | 90 days |

### Caching

CDG maintains an in-memory cache for low-latency verification:

| Cache | Contents | TTL | Invalidation |
|-------|----------|-----|-------------|
| Prerequisite cache | `{capability → prerequisite_list}` | 60 seconds | On graph change |
| Circularity cache | `{capability → no_cycle_confirmed}` | 60 seconds | On graph change |
| Impact cache | `{capability → impact_set}` | 60 seconds | On graph change |

CCA verification reads from the cache first, then falls back to live graph traversal. Cache hit target: >99%.

## Invariants

| ID | Invariant | Enforcement |
|----|-----------|-------------|
| CDG-001 | The capability graph is always a DAG — no circular dependencies | Algorithmic — circularity check on every edge addition |
| CDG-002 | Every capability's prerequisites are a subset of its delegator's capabilities | Algorithmic — prerequisite closure check on grant |
| CDG-003 | CDG is the single source of truth for dependency relationships | Architectural — no other component stores capability dependency data |
| CDG-004 | Impact analysis runs before every revocation | Algorithmic — enforced in CCA `revokeCapability` |
| CDG-005 | Cache invalidation is immediate on graph change | Architectural — write-through cache pattern |
| CDG-006 | CDG operations complete in <5ms for cache hits, <50ms for graph traversal | Architectural — indexed graph representation |

## Error Cases

| Condition | Error Code | Behavior |
|-----------|------------|----------|
| Circular dependency detected | `CDG_CYCLE_DETECTED` | Deny edge addition; return cycle path |
| Unknown capability node | `CDG_UNKNOWN_NODE` | Return error; capability must be registered first |
| Edge already exists | `CDG_EDGE_EXISTS` | Return existing edge; no duplicate created |
| Impact analysis timeout | `CDG_IMPACT_TIMEOUT` | Deny operation; large graphs must be versioned |
| Cache stale | `CDG_CACHE_STALE` | Fall back to graph traversal; async cache refresh |

## Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | CDG does one thing: model capability dependency relationships |
| R2 — Dependency Order | Depends on capability definitions; consumed by CAS and CCA |
| R3 — DRY | Dependency relationships stored once in CDG; resolved by ID everywhere |
| R4 — Builder Pattern | Dependencies discovered by traverse → accumulate → verify |
| R9 — Deterministic | Same graph state always produces same traversal results |
| R10 — Simpler Over Complex | Four edge types (requires, extends, conflicts, delegates) cover all relationships |
| R13 — Design for Failure | If CDG is unavailable, CAS falls back to safe-pass (allow with warning) |
| R14 — Paved Path | Prerequisite resolution via CDG is the standard dependency verification path |
| R15 — Open/Closed | New edge types added via GraphEdge interface, not by modifying traversal |
