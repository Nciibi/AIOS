## Purpose

The Capability Certification Authority (CCA) is the constitutional authority for all capability operations in AIOS. It owns the capability lifecycle: definition, registration, grant, verification, renewal, revocation, and certification. CCA is the stage 5 verifier in the Security Council's 7-stage pipeline — it verifies that the requesting entity has the required capabilities before execution authorization is issued.

Every entity in AIOS operates under Law 7 (Capability Bounds). CCA ensures those bounds are declared, verified, and enforced. No entity can perform an action without an active, verified capability grant from CCA.

Three subsystems comprise the CCA family:

| Subsystem | Document | Responsibility |
|-----------|----------|---------------|
| **CAS** — Capability Assignment System | `001-CAS.md` | Assigns capabilities to entities based on autonomy level, role, and declared bounds |
| **CDG** — Capability Dependency Graph | `002-CDG.md` | Maintains the directed graph of capability dependencies for circularity detection |
| **CCA** — Capability Certification | `000-CCA.md` | Issues, verifies, renews, and revokes capability certificates |

## Core Concepts

### 1. Capability Model

A capability is a first-class constitutional entity:

```typescript
interface Capability {
  id: string                          // UUIDv7
  name: string                        // e.g., "fs:read", "net:connect"
  type: CapabilityType
  entity_id: string                   // The entity that holds this capability
  grantor_id: string                  // The entity that delegated this capability
  source: CapabilitySource
  constraints: CapabilityConstraint[]
  status: "pending" | "active" | "suspended" | "revoked" | "expired"
  issued_at: timestamp
  expires_at?: timestamp
  parent_cap_id?: string              // For delegation chains
  evidence_ref: string                // EVS event ID for the grant decision
}
```

### 2. Three-Stage Certification

Capability certification follows a three-stage process defined in ADG-011:

| Stage | Step | Responsible | Description |
|-------|------|-------------|-------------|
| 1 — Application | Declare | Entity or Sou | Entity declares desired capabilities within template bounds |
| 2 — Verification | Validate | Security Council | Security Council validates against entity template, autonomy level, resource budget |
| 3 — Granting | Issue | CCA | CCA issues time-limited capability certificate; registers in Capability Registry |

### 3. Capability Hierarchy

Capabilities form a three-level hierarchy that governs assignment and delegation:

```
Level 1: ALL capabilities
  │
  ▼
Level 2: NAMESPACE capabilities (e.g., all fs:* operations)
  │
  ▼
Level 3: ACTION capabilities (e.g., fs:read, fs:write)
```

| Level | Scope | Assigned By | Example |
|-------|-------|-------------|---------|
| ALL | All actions across all namespaces | CCA (Sou only) | Sou has ALL |
| NAMESPACE | All actions within a namespace | CAS | `fs:*` — all filesystem operations |
| ACTION | Single named action | Delegation | `fs:read /etc/config` — read a specific file |

### 4. Capability Constraints

Every capability carries constraints that bound its usage:

| Constraint | Type | Example |
|------------|------|---------|
| Resource scope | Path, resource ID | `fs:read /var/log/*` |
| Duration | Time window | Expires in 24 hours |
| Rate limit | Operations per second | Max 10 requests/second |
| Depth | Delegation depth | Max delegation depth: 2 |
| Cost budget | Token/resource budget | Max 5000 tokens per invocation |
| Autonomy floor | Minimum autonomy level | Requires L2+ to exercise |

Constraints are AND-ed — all must be satisfied for the capability to authorize an action.

### 5. Capability Verification (Pipeline Stage 5)

CCA operates as Stage 5 in the Security Council verification pipeline. When a component calls CCA for verification:

```
1. Resolve capability from Registry by (entity_id, action)
2. Verify capability status is "active"
3. Verify all constraints are satisfied (scope, duration, rate, budget, autonomy)
4. Verify capability chain (delegation ancestry) is intact and unbroken
5. Verify parent capabilities (if delegated) are still active
6. Return CapabilityGrant or InsufficientCapability error
```

Verification is a synchronous ACF call with a <10ms latency target. If CCA is unreachable, the pipeline fails closed — the action is denied.

### 6. Capability Lifecycle

```
Declaration → Verification → Grant → Active → (Renewal cycle) → Revoked or Expired
```

| State | Description | Enter | Exit |
|-------|-------------|-------|------|
| pending | Declared, awaiting verification | Entity declares | Verification completes |
| active | Verified, ready for use | CCA issues grant | Expiry or revocation |
| suspended | Temporarily disabled | Security Council directive | Security Council reinstates |
| revoked | Permanently invalidated | Violation, regression, or Sou directive | Cannot be reinstated |
| expired | Time limit reached | `expires_at` reached | Must go through new certification |

### 7. Capability Revocation

Revocation is the most security-critical operation. Rules:

| Rule | Behavior |
|------|----------|
| Revocation is immediate | Active capabilities are invalidated synchronously |
| Revocation cascades | Revoking a parent capability revokes all descendant grants |
| Revocation produces evidence | `CapabilityRevoked` event written to EVS |
| Revocation is irreversible | A revoked capability ID cannot be reused |
| Revocation notification | Entity notified via ACF; in-flight actions denied at next verification |

### 8. Capability Delegation

CCA supports capability delegation with the following constraints:

| Rule | Description |
|------|-------------|
| An entity can only delegate capabilities it actively holds | Cannot delegate a capability it doesn't possess |
| Delegation cannot exceed the grantor's own constraints | The delegate's capability constraints are a subset of the grantor's |
| Delegation chain depth is bounded | Default max depth: 3 (configurable per entity type) |
| Delegation is always scoped | Scope must be equal or narrower than the source capability |

## Data Model

```typescript
interface CapabilityGrant {
  capability_id: string
  entity_id: string
  action: string                    // e.g., "fs:read"
  resource_pattern?: string         // e.g., "/var/log/**"
  constraints: {
    max_rate?: number               // operations/second
    max_depth?: number              // delegation depth
    max_cost?: number               // token/resource budget
    expires_at?: timestamp
    min_autonomy_level?: number
  }
  grantor_id: string
  parent_grant_id?: string
  issued_at: timestamp
  evidence_ref: string
  signature: string                 // CSP-signed
}

interface CapabilityVerificationRequest {
  entity_id: string
  action: string
  resource?: string
  context: {
    autonomy_level: number
    mission_id?: string
    timestamp: number
  }
}

interface CapabilityVerificationResponse {
  verified: boolean
  grant?: CapabilityGrant
  rejection_reason?: string
  verified_at: timestamp
  verifier_id: string
}
```

## Interfaces

### CCA API (via ACF)

| Method | Auth | Description |
|--------|------|-------------|
| `grantCapability(entity_id, action, constraints)` | Sou, Security Council | Issue a new capability grant |
| `verifyCapability(request)` | Security Council (Stage 5) | Verify entity has capability for action |
| `revokeCapability(capability_id, reason)` | Security Council, Sou | Revoke an active capability |
| `suspendCapability(capability_id, reason)` | Security Council | Temporarily disable a capability |
| `reinstateCapability(capability_id)` | Security Council | Reactivate a suspended capability |
| `delegateCapability(capability_id, target_entity, constraints)` | Entity holding the cap | Delegate a bounded capability |
| `getEntityCapabilities(entity_id)` | Sou, Security Council, entity itself | List all capabilities for an entity |
| `getCapabilityChain(capability_id)` | Security Council | Resolve full delegation ancestry |
| `registerCapabilityType(name, schema)` | Sou | Register a new capability type in the Registry |

### Internal Interfaces

```typescript
interface CapabilityRegistry {
  store(grant: CapabilityGrant): Promise<void>
  get(capability_id: string): Promise<CapabilityGrant>
  findByEntity(entity_id: string): Promise<CapabilityGrant[]>
  findByAction(action: string, resource?: string): Promise<CapabilityGrant[]>
  revoke(capability_id: string): Promise<void>
}

interface CapabilityVerifier {
  verify(request: CapabilityVerificationRequest): Promise<CapabilityVerificationResponse>
  checkConstraints(grant: CapabilityGrant, context: VerificationContext): ConstraintResult
  checkChain(grant: CapabilityGrant): Promise<ChainVerification>
}
```

## Invariants

| ID | Invariant | Enforcement |
|----|-----------|-------------|
| CCA-001 | Every action executed in AIOS maps to at least one active capability | Constitutional — Law 7 (Capability Bounds) |
| CCA-002 | Capability verification is synchronous and runs in <10ms | Architectural — in-memory Registry + CDG cache |
| CCA-003 | Capability grants are time-limited; permanent grants require constitutional amendment | Algorithmic — `expires_at` is mandatory |
| CCA-004 | Revocation cascades through the full delegation chain | Algorithmic — CDG traversal on revoke |
| CCA-005 | A revoked capability ID cannot be reused | Architectural — ID space is append-only |
| CCA-006 | No entity can verify its own capabilities | Architectural — CCA is independent of all other Security Council services |
| CCA-007 | Capability grants require evidence (EVS event reference) | Architectural — `evidence_ref` is mandatory |
| CCA-008 | Delegation chain depth is bounded (default: 3) | Algorithmic — enforced on `delegateCapability` |

## Error Cases

| Condition | Error Code | Behavior |
|-----------|------------|----------|
| Entity has no active capability for action | `CCA_INSUFFICIENT_CAPABILITY` | Deny; return required capability in error details |
| Capability expired | `CCA_CAPABILITY_EXPIRED` | Deny; entity must request renewal |
| Capability revoked | `CCA_CAPABILITY_REVOKED` | Deny; include revocation reason |
| Delegation chain broken | `CCA_CHAIN_BROKEN` | Deny; require re-delegation from grantor |
| Capability exceeds entity autonomy level | `CCA_LEVEL_TOO_LOW` | Deny; entity must progress autonomy level first |
| CCA unavailable | `CCA_SERVICE_UNAVAILABLE` | Pipeline Stage 5 fails closed (deny all) |
| Constraint violation | `CCA_CONSTRAINT_VIOLATED` | Deny; include violated constraint details |
| Registry full (cardinality limit) | `CCA_REGISTRY_FULL` | Deny; oldest inactive capabilities must be pruned |

## Design DNA

| Rule | Assessment |
|------|-----------|
| R1 — Modulsingularity | CCA does one thing: manage capability lifecycle and verification |
| R2 — Dependency Order | Depends on IDS, EVS; consumed by Execution-Auth; no circular deps |
| R3 — DRY | Capability definitions stored in Registry; referenced by ID everywhere |
| R4 — Builder Pattern | Capability grants built by Application → Verification → Grant pipeline |
| R5 — Liskov Substitution | Any CapabilityVerifier implements the verification interface |
| R6 — DI over Singletons | Registry backend and verifier injected; CCA core is dependency-free |
| R9 — Deterministic | Same entity + same action + same context → same verification result |
| R10 — Simpler Over Complex | Three-level hierarchy (ALL → NAMESPACE → ACTION) covers all use cases |
| R13 — Design for Failure | Unreachable CCA causes denial, not bypass (fail closed) |
| R14 — Paved Path | Three-stage certification (declare → verify → grant) is the only path |
| R15 — Open/Closed | New capability types registered via `registerCapabilityType`, not by modifying core |
