## Purpose

The Evidence Audit Service (EAS) is the constitutional mechanism for recording, preserving, and querying evidence of every decision and action within AIOS. Under Law 4 (Law of Evidence), every decision must be recorded with evidence — no invisible decisions. EAS is the institutional implementation of this law: it provides an immutable, tamper-evident store for all system events and exposes query capabilities for audit, investigation, compliance, and learning.

EAS is not a general-purpose log. It is an evidence store. Every event recorded by EAS is a constitutional evidence record: it carries identity attribution (who), timestamp (when), action and context (what), authorization chain (why), and outcome (result). Evidence records are immutable after sealing — they cannot be modified, deleted, or reordered. The integrity of the evidence chain is cryptographically verifiable.

EAS serves multiple consumers: the Security Council reads evidence for post-execution audit and forensic investigation; Sou reads evidence for system learning and strategic analysis; external auditors read evidence for compliance verification; and the system itself reads evidence for automated anomaly detection and risk assessment. Every consumer reads the same immutable evidence — there is no separate audit trail for different audiences.

## Evidence Model

### Evidence Record Structure

```typescript
interface EvidenceRecord {
  id: string;                           // UUIDv7, globally unique
  eventType: string;                    // Fully qualified event type
  producer: string;                     // Entity ID of the producing component
  subject: string;                      // Entity ID that the event is about
  timestamp: Timestamp;                 // When the event occurred (nanosecond precision)
  sequence: number;                     // Monotonic sequence per producer
  data: Record<string, unknown>;        // Event payload
  parentId?: string;                    // Parent evidence record (for causal chains)
  signature: string;                    // Producer's signature over (id + timestamp + sequence + data hash)
  sealHash: string;                     // Merkle-DAG seal linking this record to the chain
}
```

### Evidence Chain

Evidence records are organized as a cryptographic Merkle-DAG:

- Each record's `sealHash` is computed over `(parent.sealHash || record.signedContent)`
- The chain is append-only. No record can be inserted, deleted, or reordered without breaking the hash chain.
- The current chain tip is published to the Evidence Registry for distributed verification.
- Chain integrity is verified by recomputing hashes from genesis to tip.

## Evidence Ingestion

### Ingestion Pipeline

```
Producer → ACF → EAS Gateway → Validation → Sealing → Storage → Registry
```

1. **Producer**: Any constitutional entity emits an event via ACF
2. **ACF**: Routes event to EAS Gateway with verified identity and authentication
3. **EAS Gateway**: Validates event schema, producer authorization, and required fields
4. **Validation**: EAS validates that the event conforms to its declared schema, all required fields are present, and the producer is authorized to emit this event type
5. **Sealing**: EAS computes the record hash, links it to the Merkle-DAG chain tip, and produces the sealed evidence record
6. **Storage**: Record is written to the immutable evidence store (distributed append-only log)
7. **Registry**: The chain tip hash is published to the Evidence Registry for external verification

### Ingestion Guarantees

- **Durability**: Once EAS acknowledges ingestion, the record is durably stored. Acknowledgment is not sent until the record is in at least 3 storage nodes.
- **Ordering**: Records from a single producer are ordered by sequence number. Cross-producer ordering is best-effort based on timestamp.
- **Exactly-Once**: EAS deduplicates records by (producer, sequence) pair. Duplicate submissions return the existing record ID.
- **Latency**: P99 ingestion latency <50ms from producer emission to registry publication.

## Evidence Query & Retrieval

### Query API

```typescript
interface EvidenceQuery {
  eventTypes?: string[];           // Filter by event type
  subjects?: string[];             // Filter by subject entity ID
  producers?: string[];            // Filter by producer entity ID
  timeRange?: { from: Timestamp; to: Timestamp };
  sequenceRange?: { producer: string; from: number; to: number };
  parentId?: string;               // Find causal children
  sealHash?: string;               // Find by chain link
  limit?: number;                  // Max results (default 100, max 10000)
  offset?: number;                 // Pagination offset
}
```

### Chain Verification

```typescript
interface ChainVerificationResult {
  isValid: boolean;
  chainTip: string;
  verifiedRange: { from: Timestamp; to: Timestamp };
  brokenLinks: { index: number; expected: string; actual: string }[];
  verifiedAt: Timestamp;
}
```

Any consumer can request chain verification, which recomputes the Merkle-DAG hashes from a specified start to the current tip. A broken link indicates tampering and triggers an immediate Security Council alert.

### Access Control

Evidence queries are subject to the same verification pipeline as any other action:
- Stage 1-2: Requestor identity and authentication verified
- Stage 3: Authorization checked — `auditor` and `admin` roles may query all evidence; other roles may query only evidence where they are the subject or producer
- Stage 4: Policy evaluated — evidence query policies restrict bulk export and cross-category joins
- Stages 5-7: Capability, risk, and execution authorization completed

## Cross-Cutting Concerns

### Security

- Evidence records are immutable. Even the Security Council cannot modify sealed records.
- Evidence access is authorized through the standard verification pipeline. No backdoor access exists.
- The cryptographic keys used for chain sealing are held by the Evidence Registry, not by EAS itself — separation of powers.
- Bulk export is rate-limited to prevent exfiltration. Maximum 10,000 records per export, maximum 1 export per 5 minutes.

### Evidence

- Self-referential: every EAS action (ingest, query, verify, archive) itself produces evidence records.
- The evidence chain is verifiable by any third party. AIOS publishes the chain tip hash for external verification.
- Retention policies are themselves evidence records, subject to the same immutability guarantees.

### Lifecycle

- Evidence retention is governed by the category-specific retention periods listed above.
- Archived evidence is moved to cold storage but remains queryable (with higher latency).
- When an entity is terminated, its evidence records are not deleted — they are retained under the producer's identity.
- At the end of retention period, evidence is cryptographically erased (seal hashes preserved, content destroyed).

### Capability Bounds

- Evidence producers must have the `emit_evidence` capability. This capability is assigned at entity creation time.
- Evidence query operations are bounded by rate limits and result size limits to prevent resource exhaustion.
- Bulk export consumes a significant resource budget that must be pre-approved.

### Communication

- All evidence flows through ACF. EAS does not accept direct ingestion from any entity.
- Evidence query results are returned through ACF message envelopes with integrity verification.
- Evidence Registry publication uses ACF fan-out to all registered verifiers.

### Design DNA

| Rule | Assessment | Rationale |
|------|-----------|-----------|
| R1 — Modulsingularity | Compliant | EAS does one thing: seal and serve immutable evidence |
| R2 — Dependency Order | Compliant | EAS depends on ACF and IDS; no evidence consumer depends on EAS internals |
| R3 — DRY | Compliant | Evidence schema defined once in Event Registry, referenced by event type |
| R4 — Builder Pattern | Compliant | EvidenceRecord constructed by EvidenceBuilder, validated before sealing |
| R5 — Liskov Substitution | Compliant | Any storage backend implements the EvidenceStore interface |
| R6 — DI over Singletons | Compliant | EAS is injected as a service into pipeline, audit console, and reporting |
| R7 — Tests Exist | Compliant | Unit tests for sealing, integration tests for chain verification, chaos tests for integrity |
| R8 — Tests Fast | Compliant | Ingestion test suite <30s; chain verification benchmarks <60s |
| R9 — Deterministic Tests | Compliant | Same input always produces same seal hash; chain verification is deterministic |
| R10 — Prefer Simpler | Compliant | Append-only log with Merkle-DAG — no indexing, no query optimization beyond filters |
| R11 — Refactor over Rewrite | Compliant | Storage backends can be swapped via EvidenceStore interface without schema changes |
| R12 — Embrace Errors | Compliant | Every ingestion failure has a unique error code identifying the exact validation step |
| R13 — Design for Failure | Compliant | Evidence is replicated across nodes; if primary storage fails, replica serves reads |
| R14 — Paved Path | Compliant | All evidence ingestion follows the single paved ACF→Gateway→Seal→Store path |
| R15 — Open/Closed | Compliant | New event types are added via schema registration, not by modifying EAS core |
