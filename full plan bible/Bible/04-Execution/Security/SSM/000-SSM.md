## Purpose

Session & Secret Management (SSM) is the constitutional authority for session lifecycle management and secrets governance within AIOS. SSM handles two distinct but related responsibilities. First, it manages session creation, maintenance, and termination for all interactive and programmatic sessions. Second, it manages secrets — API keys, tokens, passwords, cryptographic material — across their entire lifecycle from generation to destruction.

Sessions in AIOS are constitutional entities. A session represents a period of interaction between an entity and the system, carrying identity (Law 5), producing evidence (Law 4), and bounded by lifecycle (Law 6). SSM ensures that every session is properly created with authenticated identity, maintained with periodic re-verification, and terminated with clean teardown. Sessions that exceed their maximum idle time or lifetime are force-terminated.

Secrets are the sensitive credentials that entities use to prove identity and authenticate actions. SSM manages secrets with the highest security posture: secrets are encrypted at rest (via CSP, 000-CSP.md), never logged, never transmitted in plaintext, and automatically rotated according to policy. SSM does not store secrets in plaintext — it stores only encrypted blobs and cryptographic hashes. The plaintext secret is shown to the entity once at creation and is never retrievable afterward.

## Session Management

### Session Model

```typescript
interface Session {
  id: string;                    // UUIDv7
  entityId: string;              // Identity of the session owner
  sessionType: "interactive" | "programmatic" | "system";
  state: SessionState;
  authentication: {
    method: string;              // E.g., "password", "api_key", "mfa"
    authenticatedAt: Timestamp;
    expiresAt: Timestamp;
    mfaVerified: boolean;
  };
  metadata: {
    clientInfo: string;          // User agent, client version
    sourceIp: string;
    startTime: Timestamp;
    lastActivity: Timestamp;
  };
  capabilities: string[];        // Active capabilities for this session
  policies: string[];            // Policies governing this session
}
```

### Session Types

| Type | Description | Max Lifetime | Max Idle | Refreshable |
|------|-------------|--------------|----------|-------------|
| Interactive | User-facing session (CLI, GUI, Voice) | 24 hours | 30 minutes | Yes (within lifetime) |
| Programmatic | API session (service-to-service) | 7 days | None | Yes (token rotation) |
| System | System-internal session (Engine/Worker) | Worker lifetime | None | No (scoped to Worker) |

### Session Lifecycle

```
Create → Authenticate → Active → [Refresh] → Terminate
                                    ↓
                              Suspended → Restore
```

| State | Description | Transitions To |
|-------|-------------|----------------|
| Created | Session record created, not yet authenticated | Authenticated |
| Authenticated | Identity verified, session token issued | Active |
| Active | Session is operational, actions may be performed | Refreshing, Suspended, Terminated |
| Refreshing | Token is being refreshed | Active, Terminated |
| Suspended | Session temporarily suspended (security event) | Active (restored), Terminated |
| Terminated | Session ended, all resources released | None (terminal) |

### Session Token

```typescript
interface SessionToken {
  tokenId: string;
  sessionId: string;
  entityId: string;
  issuedAt: Timestamp;
  expiresAt: Timestamp;
  lastRefresh: Timestamp;
  signature: string;             // Signed by Security Council via CSP
}
```

The session token is presented by the entity for every action within the session. It is verified at Stage 2 of the verification pipeline. Token expiry triggers re-authentication.

## Secret Management

### Secret Model

```typescript
interface Secret {
  id: string;                    // UUIDv7
  name: string;                  // Human-readable name
  type: "api_key" | "password" | "token" | "certificate" | "encryption_key";
  entityId: string;              // Owning entity
  status: SecretStatus;
  algorithm?: string;            // For cryptographic secrets
  createdAt: Timestamp;
  expiresAt?: Timestamp;
  lastUsedAt?: Timestamp;
  rotationPolicy: RotationPolicy;
  accessPolicy: AccessPolicy;
}
```

### Secret Lifecycle

```
Generate → Store → Active → [Rotate] → Revoke → Destroy
                                  ↓
                            Compromised → Emergency Revoke
```

| State | Description | Transitions To |
|-------|-------------|----------------|
| Generating | Secret is being created in HSM | Stored |
| Stored | Secret is encrypted and stored | Active |
| Active | Secret is available for use | Rotating, Revoked, Compromised |
| Rotating | Old secret is being replaced | Active (new secret) |
| Revoked | Secret is revoked, no longer valid | Destroyed |
| Compromised | Secret is suspected compromised | Revoked (emergency) |
| Destroyed | Secret material is cryptographically erased | None (terminal) |

### Secret Storage

Secrets are stored using a layered encryption model:

1. **Secret material** is encrypted with a data encryption key (DEK) using AES-256-GCM
2. The **DEK** is encrypted with a key encryption key (KEK) stored in the HSM
3. The **encrypted blob** is stored in the SSM secret store (backed by a secure database)
4. **Access control** is enforced at the SSM service layer — a caller must present a valid authorization token and have the `secret.read` capability

### Auto-Rotation

| Secret Type | Rotation Period | Grace Period | Notification |
|-------------|----------------|--------------|--------------|
| API Key | 90 days | 7 days (both keys valid) | 14 days before expiry |
| Password | 90 days | None (old password invalidated) | 14 days before expiry |
| Token (session) | Per-session | None | Automatic |
| Token (service) | 7 days | 1 hour | 24 hours before expiry |
| Certificate | Per certificate validity | 30 days | 60 days before expiry |

### Emergency Revocation

When a secret is suspected compromised:

1. Secret status is immediately set to Compromised
2. Secret is revoked — all authentications using this secret fail
3. Security Council is notified with full context
4. New secret is generated if authorized
5. Post-mortem investigation records the compromise as evidence

## Access Control

### Secret Access Levels

| Level | Description | Examples |
|-------|-------------|----------|
| Owner | Full access to the secret | Create, read, rotate, revoke, destroy |
| Manager | Administrative access | Rotate, revoke, view metadata (not secret material) |
| User | Operational access | Use secret for authentication |
| Auditor | Read-only access to metadata | View creation time, rotation history, access logs |

### Audit Trail

Every secret access is logged with:

| Field | Description |
|-------|-------------|
| secret_id | Target secret |
| entity_id | Requesting entity |
| operation | Read, rotate, revoke, destroy |
| timestamp | When the operation occurred |
| result | Success or failure |
| ip_address | Source IP of the request |
| authorization | Authorization evidence for the operation |

## Cross-Cutting Concerns

### Security

- Secret material is never logged, never included in error messages, and never transmitted in plaintext.
- Secrets are zeroized in memory after use. The SSM client library guarantees `mlock`/`memset` patterns.
- Session token signatures are verified at every use through the verification pipeline.
- SSM enforces rate limiting: max 10 authentication attempts per session per second; max 5 secret read attempts per entity per second.

### Evidence

- Every session lifecycle transition is recorded as evidence in EAS.
- Every secret access (read, rotate, revoke, destroy) is recorded with full audit context.
- Secret access denials are recorded as security events for investigation.

### Lifecycle

- Session lifetime is bounded by the maximum lifetime for the session type. Force-termination occurs at expiry.
- Secret rotation is automatic. Secrets that exceed their rotation period without rotation are force-rotated.
- Entity termination triggers cascade: all active sessions for the entity are force-terminated; all secrets owned by the entity are revoked.

### Capability Bounds

- Session creation requires the `session.create` capability. The session's capabilities are derived from the creating entity's capabilities.
- Secret access is scoped by the entity's capabilities. An entity may only access secrets it owns or that have been explicitly delegated.
- Session token issuance consumes from the entity's token budget.

### Communication

- Session tokens are transmitted in the ACF envelope, never in URLs or headers outside ACF.
- Secret material is delivered through encrypted ACF channels only.
- Secret access events are published through ACF event streams for monitoring.

### Design DNA

| Rule | Assessment | Rationale |
|------|-----------|-----------|
| R1 — Modulsingularity | Compliant | SSM does two closely related things: sessions and secrets — unified by identity lifecycle |
| R2 — Dependency Order | Compliant | SSM depends on CSP and IDS; sessions depend on SSM |
| R3 — DRY | Compliant | Session and secret schemas defined once; entity type configs are canonical |
| R4 — Builder Pattern | Compliant | Session and Secret constructed by their respective Builders with validation |
| R5 — Liskov Substitution | Compliant | Any secret store implementation conforms to SecretStore interface |
| R6 — DI over Singletons | Compliant | SSM injected into ACF, Runtime, and verification pipeline |
| R7 — Tests Exist | Compliant | Unit tests for session lifecycle, integration tests for secret rotation |
| R8 — Tests Fast | Compliant | Session operations target <5ms; secret operations target <10ms; test suite <60s |
| R9 — Deterministic Tests | Compliant | Same session inputs always produce same lifecycle transitions |
| R10 — Prefer Simpler | Compliant | Sessions are linear state machines; secrets are encrypted blobs with access control |
| R11 — Refactor over Rewrite | Compliant | Secret store backends are swappable via SecretStore interface |
| R12 — Embrace Errors | Compliant | Every access denial identifies the secret, operation, and authorization failure |
| R13 — Design for Failure | Compliant | Secret store is replicated; if primary fails, read replicas serve metadata |
| R14 — Paved Path | Compliant | All session and secret operations follow the single SSM path |
| R15 — Open/Closed | Compliant | New secret types added through type registration, not SSM core changes |
