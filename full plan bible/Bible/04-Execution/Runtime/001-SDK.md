## Purpose

The Runtime SDK is the framework for building Execution Providers — the domain-specific plugins that carry out verified actions within the AIOS Runtime Engine. Every provider integrates with the Runtime Manager through the `ExecutionProvider` interface defined in this SDK. The SDK enforces constitutional compliance (token validation, bounds enforcement, Event production, lifecycle management) so that provider authors focus on execution logic.

The SDK guarantees that every provider adheres to Law 8 (Verification-First) — no provider executes without a validated token. Law 7 (Capability Bounds) is enforced through the SDK's resource monitoring and budget checking middleware.

## Provider Interface

All Execution Providers implement the following interface:

```
interface ExecutionProvider {
    // Identity
    providerId(): string
    providerVersion(): string

    // Capability declaration
    supportedActionTypes(): ActionType[]
    capabilityDeclaration(): CapabilityDeclaration

    // Lifecycle
    initialize(config: ProviderConfig): Result<void>
    health(): ProviderHealth
    shutdown(): Result<void>

    // Execution (the core contract)
    execute(context: ExecutionContext): ExecutionResult

    // Streaming (where supported)
    executeStream(context: ExecutionContext): Stream<ExecutionChunk>
}
```

### Provider Identity

Each provider has a unique `providerId` (reverse domain notation, e.g., `aios.provider.claude`) and a semantic version. Identity is registered with the Runtime Manager at startup and is included in all execution Events.

### Capability Declaration

Every provider declares what it can execute and under what resource constraints:

| Field | Type | Description |
|-------|------|-------------|
| action_types | ActionType[] | The action types this provider handles |
| max_parallelism | u32 | Maximum concurrent executions |
| default_timeout_ms | u64 | Default timeout for executions |
| resource_requirements | ResourceProfile | CPU, memory, network, storage baselines |
| supported_autonomy_levels | AutonomyLevel[] | L0–L4 levels this provider supports |

The Runtime Manager uses this declaration to route execution requests. A provider that declares `action_types: ["model.inference"]` will only receive model inference execution requests.

## Execution Context

When the Runtime Manager dispatches an execution to a provider, it passes an `ExecutionContext`:

| Field | Source | Description |
|-------|--------|-------------|
| execution_id | Runtime Manager | Unique execution identifier |
| token | VerificationToken | Validated execution token from Security Kernel |
| action | Action | The action to execute (parameters, inputs) |
| capability_bounds | CapabilityBounds | Resource limits from the verified capability |
| autonomy_level | AutonomyLevel | L0–L4 level for this execution |
| parent_entity_id | EntityID | The entity that requested execution |
| deadline | Timestamp | Hard deadline — execution must complete by this time |

The provider uses the context to configure its execution environment. It must not exceed `capability_bounds` and must respect the `deadline`. If the deadline is reached, the provider must terminate execution and return a timeout error.

## Execution Result

Every execution produces an `ExecutionResult`:

| Field | Type | Description |
|-------|------|-------------|
| execution_id | string | Matching the execution context |
| status | Status | Completed, Failed, Cancelled, TimedOut |
| output | Output | Structured output data (type-specific) |
| metrics | ExecutionMetrics | Tokens consumed, duration, resource usage |
| error | ErrorInfo | Present only if status is Failed or TimedOut |
| events | Event[] | Provider-produced diagnostic Events |

The Runtime Manager wraps the result in a standard Runtime Event before returning it to the requesting entity via ACF.

## Streaming Execution

Providers that support streaming (model inference, long-running tool chains) implement `executeStream`. The stream produces `ExecutionChunk` objects:

| Field | Type | Description |
|-------|------|-------------|
| sequence | u64 | Monotonic chunk sequence number |
| data | bytes | Streaming payload (token, frame, batch) |
| progress | f64 | 0.0–1.0 estimated progress |
| is_final | bool | True if this is the last chunk |
| metrics | ExecutionMetrics | Cumulative metrics at this point |

The Runtime Manager relays streaming chunks to the requesting entity through ACF streaming channels. Chunks are produced as Events for observability.

## Resource Monitoring Middleware

The SDK provides a built-in middleware layer that intercepts every `execute` call to enforce resource bounds:

1. Pre-execution: Verify that the execution token's scope matches the provider's declared action types
2. During execution: Sample resource consumption (token count, memory, duration) at configurable intervals
3. At 90% of any bound: Produce a warning Event
4. At 100% of any bound: Terminate execution and return a `BoundsExceeded` error
5. Post-execution: Record final metrics and release provider-level resources

This middleware is automatically applied by the SDK. Provider authors cannot disable it.

## Provider Lifecycle

| State | Callback | Description |
|-------|----------|-------------|
| Registered | `constructor` | Provider instantiated by Runtime Manager |
| Initialized | `initialize()` | Provider sets up connections, authenticates, allocates resources |
| Healthy | `health()` returns ok | Provider is accepting executions |
| Degraded | `health()` returns degraded | Provider is operating with reduced capacity |
| Unhealthy | `health()` returns error | Provider is not accepting executions |
| Shutdown | `shutdown()` | Provider releases resources, closes connections |

The Runtime Manager polls `health()` at a configurable interval. A provider in Degraded or Unhealthy state is removed from the routing table until health is restored.

## Error Handling

All provider errors use the SDK's error framework:

| Error Code Range | Category | Example |
|------------------|----------|---------|
| PRV-0000–0999 | Validation | Invalid parameters, unsupported action type |
| PRV-1000–1999 | Authentication | Provider API key expired, unauthorized |
| PRV-2000–2999 | Resource | Rate limited, quota exceeded, timeout |
| PRV-3000–3999 | Execution | Model error, tool failure, internal error |
| PRV-4000–4999 | Boundary | Bounds exceeded, scope violation, quarantine |

Every error must include: error code, human-readable message, execution_id, provider_id, timestamp, and suggested escalation path. Errors are automatically wrapped in Failed execution Events by the SDK.

## Testing Framework

The SDK includes a test harness that provides:

- Mock Runtime Manager for unit testing provider logic in isolation
- Fake verification tokens for testing token validation paths
- Resource bound simulation for testing bounds enforcement
- Deterministic clock for testing timeout and deadline behavior
- Event capture for asserting correct Event production

Provider authors must use this harness for all tests. The test harness enforces R7 (Tests Exist) and R8 (Fast Tests).

## Cross-Cutting Concerns

### Security

The SDK never exposes raw API keys or secrets to provider implementation code. Secrets are injected as references resolved by the Runtime Manager's secret store. The SDK validates every execution token before invoking the provider's `execute` method. Providers cannot bypass token validation.

### Evidence

The SDK automatically produces Events for every execution lifecycle transition. Provider authors do not need to manually produce Events — the middleware layer handles Event production. Provider-authored Events (diagnostic chunks, progress updates) are forwarded through the same Event pipeline.

### Lifecycle

The SDK manages provider lifecycle through the `initialize/health/shutdown` contract. Providers are expected to be stateless across executions — all execution state is contained in the `ExecutionContext`. The SDK tears down per-execution resources automatically after `execute` returns.

### Capability Bounds

Bounds enforcement is built into the SDK middleware. Provider authors declare their resource requirements in `capabilityDeclaration()`. The middleware enforces token budgets, duration limits, and memory caps without provider involvement. A provider cannot accidentally exceed its declared bounds.

### Communication

Providers communicate with external services (APIs, model endpoints, hardware interfaces) through provider-defined channels. All provider-to-Runtime Manager communication uses the SDK's internal interface — not ACF. ACF is reserved for entity-to-Runtime communication. Providers must not listen on ACF channels directly.

### Design DNA

| Rule | Assessment |
|------|------------|
| R1 — Modulsingularity | The SDK does one thing: standardize Execution Provider construction. |
| R2 — Dependency Order | SDK depends on Runtime Manager interfaces. Providers depend on SDK. No cycles. |
| R3 — DRY | Token validation, bounds enforcement, Event production are in the SDK — not duplicated across providers. |
| R4 — Builder Pattern | ExecutionContext is built by the Runtime Manager. Providers receive pre-built contexts. |
| R5 — Liskov Substitution | Every provider implements the same `ExecutionProvider` interface. Providers are drop-in replaceable. |
| R6 — DI over Singletons | Provider dependencies (config, secret references, HTTP clients) are injected through `initialize()`. |
| R7 — Tests Exist | SDK includes a test harness. Every provider must have tests using the harness. |
| R8 — Fast Tests | Harness tests complete in < 1s. Mock backends eliminate network latency. |
| R9 — Deterministic | The SDK's execution path is deterministic for a given context and action. Provider nondeterminism is contained. |
| R10 — Simpler Over Complex | The execute/executeStream dual-path is the simplest interface that covers all execution types. |
| R11 — Refactor over Rewrite | SDK interface evolves through additive versioning. Existing providers are not broken by new SDK features. |
| R12 — Embrace Errors | Error codes are structured, unique, and categorized. The SDK enforces error code usage in all providers. |
| R13 — Design for Failure | Middleware enforces deadlines and bounds. Health checks detect provider failure. Circuit breakers prevent cascading failure. |
| R14 — Paved Path | The paved path is: register → initialize → execute → produce Event → return. No alternative execution flow. |
| R15 — Open/Closed | New action types and providers implement existing interfaces without modifying the SDK or Runtime Manager. |

## Provider Configuration Contract

Providers receive a `ProviderConfig` object on `initialize()`:

| Field | Type | Source |
|-------|------|--------|
| provider_id | string | Assigned by Runtime Manager |
| secret_refs | Map<string, SecretReference> | References to secrets in the Runtime Manager's secret store |
| endpoint_map | Map<string, string> | Service endpoints (API URLs, broker addresses, robot controllers) |
| capabilities | CapabilityDeclaration | Provider's own declared capabilities |
| log_level | LogLevel | Provider-specific log verbosity |
| tags | Map<string, string> | Metadata tags for observability and routing |

Providers resolve secrets at invocation time by calling `resolveSecret(ref)` on the Runtime Manager interface. Secrets are never stored in provider memory between executions.

## Provider Registry Interaction

When a provider registers with the Runtime Manager, the following sequence occurs:

1. Provider calls `RuntimeManager.registerProvider(providerId, capabilities)`
2. Runtime Manager validates capability declaration against the provider's declared action types
3. Runtime Manager assigns a routing key based on action_types
4. Runtime Manager calls `provider.initialize(config)` with resolved configuration
5. Provider returns success or failure with error details
6. If success, Runtime Manager adds provider to the routing table
7. If failure, Runtime Manager logs the error and retries registration on a configurable interval

## Migration and Versioning

The SDK follows semantic versioning. Providers declare which SDK version they target in their capability declaration. The Runtime Manager supports multiple SDK versions concurrently through an adapter layer:

| SDK Version | Status | Provider Compatibility |
|-------------|--------|----------------------|
| 1.x | Current | All new providers must target 1.x |
| 2.x (planned) | RFC stage | Adds batch execution, predictive resource allocation |

Providers pinned to an older SDK version continue to function until the SDK version's end-of-life date. The Runtime Manager logs deprecation warnings for providers using end-of-life SDK versions.
