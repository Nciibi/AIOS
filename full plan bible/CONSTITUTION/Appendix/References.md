# Purpose

This appendix serves as the master reference for the AIOS documentation ecosystem.

Its purpose is to provide a centralized index of constitutional documents, architectural specifications, implementation guides and supporting materials while preserving clear relationships between them.

This appendix is informational.

It does not establish constitutional authority.

---

# Documentation Hierarchy

The AIOS documentation is organized into multiple constitutional layers.

```

Vision

↓

DNA

↓

Constitution

↓

Architecture

↓
```

Each layer refines the one above it.

No lower layer may contradict a higher constitutional layer.

---

# Documentation Layers

## Vision

Defines the long-term purpose and direction of AIOS.

The Vision explains why AIOS exists.

---

## DNA

Defines the fundamental architectural philosophy.

DNA establishes immutable design principles from which the Constitution is derived.

---

## Constitution

Defines the constitutional governance of AIOS.

The Constitution establishes

- Authority
- Responsibilities
- Institutions
- Society
- Rights
- Obligations
- Constitutional Principles

The Constitution is the highest normative document after the DNA.

---

## Architecture

Architecture documents describe how constitutional principles are realized through system design.

Architecture documents SHALL remain consistent with the Constitution.

---

## Bibles

Bible documentation provides complete implementation specifications for major AIOS subsystems.

Examples include

- Sou Bible
- Academy Bible
- OSYS Bible
- ROS Bible
- DTS Bible
- Organizations Bible
- Security Bible
- Infrastructure Bible

Bibles define implementation architecture.

They do not redefine constitutional responsibilities.

---

## RFCs

Request for Comments (RFCs) describe proposed architectural changes before constitutional adoption.

RFCs encourage discussion and review.

Accepted RFCs may become constitutional amendments or implementation specifications.

---

## ADRs

Architecture Decision Records (ADRs) preserve significant engineering decisions.

Each ADR records

- Context
- Problem
- Decision
- Alternatives
- Consequences

ADRs preserve implementation history.

They do not modify the Constitution.

---

## SDK Documentation

SDK documentation defines developer-facing APIs, libraries and integration guidance.

SDKs implement constitutional capabilities without altering constitutional meaning.

---

## Implementation

Implementation consists of executable software, configuration and runtime components.

Implementation SHALL conform to all higher documentation layers.

# Constitutional Dependency Rules

Every document SHALL depend only upon documents above its own layer.

Examples

An implementation MAY reference an SDK.

An SDK MAY reference a Bible.

A Bible MAY reference the Constitution.

The Constitution MAY reference the DNA.

The reverse SHALL NOT occur.

---

# Amendment References

Constitutional amendments SHALL

- Preserve document history.
- Maintain version continuity.
- Record rationale.
- Preserve compatibility where practical.

Historical Notes SHALL record significant constitutional evolution.
