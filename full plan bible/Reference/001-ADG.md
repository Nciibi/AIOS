## Purpose

The Architectural Decision Gateway (ADG) is the governance mechanism for all architectural decisions in AIOS. Every decision that affects module boundaries, interface contracts, service dependencies, structural patterns, or cross-cutting concerns must pass through the ADG before implementation. This document is the quick-reference guide to the ADG process, entry types, and lifecycle.

## When an ADG Is Required

An ADG entry is required when a decision affects:

- **Module boundaries**: Splitting, merging, or introducing new modules
- **Interface contracts**: Public APIs, ACF topic schemas, SDK interfaces
- **Service dependencies**: Adding, removing, or changing service relationships
- **Structural patterns**: Design patterns, layering, communication topologies
- **Cross-cutting concerns**: Security, evidence, lifecycle, capability bounds
- **Standards**: Naming conventions, document formats, coding standards

## When an ADG Is NOT Required

- Routine implementation decisions within an existing module
- Bug fixes that do not change interfaces or behavior
- Performance optimizations that preserve semantics
- Documentation updates that do not change specifications

## ADG Entry Format

```
ADG-NNNN: Title
Date: YYYY-MM-DD
Status: [Proposed | Approved | Recorded | Deprecated | Superseded | RFC Pending]
Author: [identity_id]
Source Laws: [Law(s)]
Source Physics: [Physics document(s)]

## Context
  - What is the architectural problem?
  - What constraints apply?
  - What is the current architecture?
  - What prompted this decision?

## Decision
  - What is the chosen approach?
  - Why is this approach correct?
  - What alternatives were considered and rejected?

## Impact
  - What changes are required?
  - What modules are affected?
  - What is the migration path?

## Design DNA Compliance
  - R1: Single responsibility
  - R2: Acyclic dependencies
  - R10: Simplicity
  - R14: Paved path
  - R15: Extensibility
```

## ADG Lifecycle

```
Proposed → [Review] → Approved → [Implementation] → Recorded
                                              ↓
                                         Deprecated
                                              ↓
                                         Superseded
```

| Stage | Description |
|-------|-------------|
| Proposed | Entry drafted, submitted for review |
| Approved | Reviewed and accepted by Architecture Review Board |
| Recorded | Implemented and stored in CKR (Constitutional Knowledge Repository) |
| Deprecated | No longer recommended for new work; existing implementations maintained |
| Superseded | Replaced by a newer ADG entry |
| RFC Pending | Requires an RFC for implementation planning |

## ADG to RFC Handoff

When an ADG entry reaches "Approved" status and requires implementation planning, an RFC is initiated:

1. **ADG Approved** → Architecture Review Board confirms the decision
2. **RFC Drafted** → Implementation plan, timeline, affected modules, migration steps
3. **RFC Reviewed** → Through the CRP process (`Bible/01-Governance/003-CRP.md`)
4. **RFC Approved** → Implementation begins
5. **ADG Status → Recorded** → Implementation complete, decision stored in CKR

ADGs with "RFC Pending" status (e.g., ADG-013) have been approved architecturally but require an RFC for implementation scoping before work can begin.

## ADG Review Criteria

The Architecture Review Board evaluates each ADG entry against:

1. **Constitutional compliance**: Does this violate any Law or Physics invariant?
2. **Architectural coherence**: Does this fit the existing architecture or justify the change?
3. **Evidence**: Is the decision supported by evidence (practical experience, analysis, or simulation)?
4. **Minimality**: Is this the simplest solution that satisfies requirements?
5. **Extensibility**: Does this decision enable or constrain future changes?

## Cross-Cutting Concerns

### Security
ADGs affecting the Security Kernel, identity, or capability boundaries require Security Council review. Security-critical ADGs are flagged in the index.

### Evidence
All ADG decisions are recorded in CKR with full context. Status changes are evidenced. Superseded ADGs remain indexed with a reference to the superseding entry.

### Lifecycle
ADGs follow the governance lifecycle defined in `Bible/01-Governance/005-ADG.md`. Each ADG progresses through review, approval, and implementation stages.

### Capability Bounds
ADGs may define capability boundaries. Capability-bound ADGs must declare scope, limits, and verification mechanisms.
