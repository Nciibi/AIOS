## Purpose

The Document Quality Checklist (DQC) defines quality criteria for all AIOS documentation. Every document must pass this checklist before it is considered complete. The checklist is enforced during document review and can be automated where possible.

## Mandatory Checklist

### Structure
- [ ] Follows the BAS template (metadata table, Purpose, required sections)
- [ ] H1 has correct format: `# AIOS <Section> — <Title>`
- [ ] H2 has correct format: `## <Sequence> — <Subtitle>`
- [ ] All sections are present (Purpose, body, Cross-Cutting Concerns, Related Documents)
- [ ] Document ID follows naming convention (`AIOS-BBL-{NNN}-{ABBR}-{NNN}`)
- [ ] Length is within guidelines (see BAS)

### Content
- [ ] Purpose section clearly defines scope in 1–3 paragraphs
- [ ] Every concept is clearly explained, not assumed
- [ ] No contradictory statements within the document
- [ ] No contradictory statements with sibling documents in the same directory
- [ ] Implementation-relevant details include concrete examples
- [ ] Edge cases are documented (failure modes, error states, boundary conditions)
- [ ] Performance characteristics are specified (latency SLOs, throughput targets)

### Cross-References
- [ ] Every external reference resolves to an existing document
- [ ] All `.md` extensions are included in cross-references
- [ ] No directory-only references (e.g., `Security/IDS/` without a filename)
- [ ] Related Documents table includes all substantive dependencies
- [ ] Cross-references to sibling documents use relative paths
- [ ] Cross-references to Physics documents exist where laws are cited
- [ ] Source Laws in metadata match the laws referenced in the body

### Events (where applicable)
- [ ] Every state transition produces at least one Event
- [ ] Event table has columns: Event Type, Description, Payload fields
- [ ] Event types use PascalCase with dot prefix (e.g., `Session.Created`)

### Error Codes (where applicable)
- [ ] Every failure mode has a unique error code
- [ ] Error codes follow format: `{SOURCE}_{NNN}`
- [ ] Error table has columns: Code, Description
- [ ] All error codes appear in at least one test

### Design DNA
- [ ] Design DNA Compliance table is present
- [ ] At minimum R1, R10, R12, R14, R15 are assessed
- [ ] Additional rules (R2, R3, R4, R6, R13) are included where relevant
- [ ] Each rule assessment includes a meaningful justification

### Cross-Cutting Concerns
- [ ] Security section documents security implications
- [ ] Evidence section documents Event production
- [ ] Lifecycle section documents state machine and transitions
- [ ] Capability Bounds section documents boundary implications
- [ ] Interoperability section documents integration points

### Naming and Terminology
- [ ] All acronyms are defined on first use
- [ ] Acronyms match the canonical definitions in Bible/09-Reference/001-Glossary.md
- [ ] Law references use format `Law N — Name` (not `Law/N`)
- [ ] Document ID category matches the directory location

## Quality Levels

| Level | Criteria | Use Case |
|-------|----------|----------|
| L0 — Draft | Structure and purpose defined, body incomplete | Work in progress |
| L1 — Complete | All sections present, all items checked | Ready for review |
| L2 — Reviewed | Peer review completed, issues resolved | Ready for merge |
| L3 — Audited | Cross-document audit passed, no broken links | Published |

## Automation

The following DQC items can be automated:
- Cross-reference validity (check all paths resolve)
- Metadata table format (regex check)
- Document ID format (regex check)
- Event table structure (check columns exist)
- Error code format (regex check)
- Law reference format (regex check)
- H1/H2 format (regex check)
- Acronym definition on first use (check acronym + definition pair)
