## Purpose

Bible Authoring Standards (BAS) define the format, structure, and conventions for writing Bible documents. Every Bible document must follow these standards to ensure consistency, discoverability, and tooling compatibility across the entire documentation set.

## Required Sections

Every substantive document (not overviews) must include:

- **Purpose**: What this document defines
- **Core specification**: The main content (interface, algorithm, lifecycle, etc.)
- **Events**: Table of Events this component produces (Event Type, Description, Payload)
- **Error Codes**: Table of error codes (Code, Description)
- **Cross-Cutting Concerns**: Security, Evidence, Lifecycle, Capability Bounds, Design DNA Compliance, Interoperability
- **Related Documents**: Cross-references to other Bible docs, Physics, Standards

Overview documents (000-Overview.md) must include:
- **Purpose**: Scope of the directory
- **Component map**: List of documents and their relationships
- **Related Documents**: Cross-references

## Document Status Lifecycle

Documents follow this status lifecycle:

```
Draft → [Review] → Active → [Amendment] → Active (new version)
                                    → Deprecated → Superseded
```

| Status | Meaning | Rules |
|--------|---------|-------|
| Draft | Work in progress, not yet reviewed | May contain TBD/TODO placeholders |
| Active | Current and authoritative | All placeholders resolved, passed DQC |
| Deprecated | Still valid but not recommended for new work | Existing implementations maintained |
| Superseded | Replaced by a newer document | Reference to superseding doc required in metadata |

**TBD/TODO Handling**: Draft documents may use TBD/TODO markers. Before promoting to Active, all TBD/TODO markers must be resolved or tracked as GitHub issues with links in the document.

## Cross-Reference Format

Cross-references use paths relative to the project root (`full plan bible/`):

```
Bible/<Section>/<Subdirectory>/<File>.md
Physics/<File>.md
```

Examples:
- `Bible/04-Execution/Security/IDS/000-Overview.md`
- `Bible/05-Platform/004-EVS.md`
- `Physics/005-Events.md`

**Rules**:
- Always include the `.md` extension
- Use full path from project root with `Bible/` or `Physics/` prefix
- Never use directory-only references (e.g., `Security/IDS/` is wrong)
- Never use paths relative to the current file's location — always from project root
- Link to the specific `.md` file, not a directory

## Design DNA Compliance

Every document must include a Design DNA Compliance table listing at least R1, R10, R12, R14, and R15. Include additional rules where relevant.

| Rule | Compliance |
|------|------------|
| R1 — Modulsingularity | Document covers exactly one concern |
| R10 — Minimalism | Document describes the simplest valid solution |
| R12 — Embrace Errors | Error codes are defined for every failure mode |
| R14 — Paved Path | Document defines the single paved path |
| R15 — Open/Closed | Specification allows extension without modification |
